# MongoDB 事务与一致性

## 单文档原子性

MongoDB 对单个文档内的写入天然原子。合理使用嵌入式建模，可以把需要强一致的字段放在同一文档内，减少跨文档事务需求。

例如订单状态和订单明细若在同一订单文档中，更新订单状态、支付信息和局部明细可作为单文档原子操作完成。

## 多文档事务

MongoDB 支持复制集和分片集群上的多文档 ACID 事务。事务适合确实需要跨多个文档、多个集合保持强一致的场景，但不应作为默认建模手段。

事务成本：

- 持有更多锁和快照资源。
- 增加 oplog、WiredTiger cache 和网络往返成本。
- 长事务会影响复制、缓存回收和冲突概率。
- 分片事务涉及多参与者协调，延迟和失败处理更复杂。

## WriteConcern

WriteConcern 控制写入被确认的持久化和复制级别。

- `w: 1`：主节点写入成功即确认，延迟低，但主节点故障时可能回滚。
- `w: "majority"`：写入复制到多数节点后确认，可靠性更高。
- `j: true`：要求写入 journal 后确认，增强单节点崩溃恢复能力。
- `wtimeout`：等待复制确认的超时时间，超时不一定代表写入失败，需要结合结果处理。

生产关键写入通常使用 `majority`，低价值日志可使用较低级别换吞吐。

## ReadConcern

ReadConcern 控制读取可见性。

- `local`：读取节点本地已写入数据，不保证多数提交。
- `majority`：读取多数节点确认的数据，降低读到将来被回滚数据的风险。
- `snapshot`：事务中提供快照读。
- `linearizable`：读取前确认主节点仍是合法主节点，语义最强但代价高。

一致性要求高的读写链路通常组合 `writeConcern: majority` 和 `readConcern: majority`。

## ReadPreference

ReadPreference 决定读请求发往哪个节点。

- `primary`：默认，从主节点读，读到最新数据概率最高。
- `primaryPreferred`：优先主节点，主节点不可用时读从节点。
- `secondary`：只读从节点，适合可接受延迟的数据。
- `secondaryPreferred`：优先从节点。
- `nearest`：按网络延迟选择节点。

从节点读可能受到复制延迟影响。涉及刚写后读、权限、余额、订单状态等强一致链路，应避免随意使用 secondary read。

## 因果一致性

驱动 session 可提供因果一致性，使同一会话后续读能看到之前已确认写入。它适合用户请求链路内的 read-your-writes 语义，但仍需正确配置 ReadConcern 和 WriteConcern。

## 乐观并发控制

可通过版本号或状态条件避免并发覆盖：

```javascript
db.orders.updateOne(
  { _id: orderId, status: "CREATED", version: 3 },
  { $set: { status: "PAID" }, $inc: { version: 1 } }
)
```

如果 `matchedCount` 为 0，表示状态已变化或版本冲突，应由业务决定重试、忽略或报错。

## Change Streams

Change Streams 基于 oplog 提供变更订阅能力，适合缓存失效、搜索索引同步、异步投影、审计流水等场景。

实践注意：

- 需要保存 resume token，支持消费者重启后继续消费。
- 消费延迟不能超过 oplog 保留窗口，否则可能无法恢复。
- 下游处理应幂等，因为网络重试或恢复可能导致重复处理。

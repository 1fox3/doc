# MongoDB 存储与运维

## 存储引擎

MongoDB 默认使用 WiredTiger 存储引擎。WiredTiger 提供文档级并发控制、压缩、缓存管理、journal 和 checkpoint。旧版本中的 MMAPv1 已不再是现代部署重点。

MongoDB Enterprise 还提供 In-Memory 存储引擎，适合极低延迟且数据可由外部系统恢复的场景，但需要严格评估成本和可用性。

## WiredTiger

WiredTiger 的核心关注点：

- Cache：保存热数据、索引页和脏页，缓存压力直接影响读写延迟。
- Checkpoint：周期性把内存中的脏数据刷到数据文件，用于缩短恢复时间。
- Journal：记录崩溃恢复所需日志，提高异常宕机后的数据安全性。
- 压缩：集合和索引可使用压缩算法降低磁盘占用。

WiredTiger 内部使用 B-Tree 结构组织数据和索引。实践中更重要的是理解索引选择、缓存命中、磁盘 I/O 和写放大，而不是只记结构名称。

## 压缩

常见压缩算法：

- Snappy：默认常用选择，压缩和解压速度快，压缩率适中。
- zlib：压缩率较高，但 CPU 开销更大。
- Zstandard：在压缩率和速度之间更均衡，适合较新版本评估。

选择压缩算法时要看瓶颈：如果磁盘空间和 I/O 是瓶颈，可考虑更高压缩率；如果 CPU 已经紧张，应优先使用低开销方案。

## 容量规划

容量规划不只看原始数据大小，还要考虑：

- 索引大小。
- 副本数量。
- oplog 保留窗口。
- journal 和 checkpoint 峰值空间。
- 分片迁移和备份过程中的临时空间。
- 数据增长率和冷热数据比例。

索引和热数据工作集应尽量能被内存覆盖，否则读延迟会明显受磁盘性能影响。

## 慢查询分析

排查慢查询常用手段：

- 查看慢查询日志。
- 使用 profiler 捕获集合级或数据库级慢操作。
- 对目标查询执行 `explain("executionStats")`。
- 关注 `keysExamined`、`docsExamined`、`nReturned`、排序、锁等待和返回文档大小。
- 检查索引是否支持过滤、排序和投影。

常见优化方向：

- 调整复合索引字段顺序。
- 使用覆盖查询减少回表。
- 避免大 offset 分页。
- 限制返回字段和文档大小。
- 把复杂聚合拆成异步预计算或物化视图。

## 备份恢复

备份策略应同时覆盖全量备份、增量或 oplog 备份、恢复演练和权限隔离。

常见方式：

- 文件系统快照：速度快，但需要保证一致性。
- `mongodump`/`mongorestore`：逻辑备份，适合中小规模或迁移，超大集群成本较高。
- 云厂商或 Ops Manager/Cloud Manager 备份：适合托管和自动化场景。

备份是否可用只能通过恢复验证确认。生产应定期演练恢复到新环境，并记录 RPO、RTO。

## 安全配置

基础安全项：

- 开启认证，不允许匿名访问。
- 使用最小权限角色，不用业务账号执行管理命令。
- 限制监听地址和网络访问来源。
- 开启 TLS 保护传输链路。
- 对备份文件和快照做访问控制。
- 定期轮换密码和审计高危操作。

## 监控指标

关键监控项：

- 连接数、请求速率、读写延迟。
- opcounters、锁等待、队列长度。
- WiredTiger cache 使用率、dirty bytes、eviction。
- 磁盘 IOPS、吞吐、延迟和空间剩余。
- 复制延迟、oplog window、选举次数。
- 分片集群 Chunk 分布、Balancer 状态、迁移耗时。

## 故障排查思路

- 先判断影响范围：单集合、单 Shard、单节点还是全集群。
- 再判断瓶颈类型：CPU、内存、磁盘、网络、锁、复制或路由。
- 对慢操作定位查询形态、索引命中、扫描量和返回量。
- 对写入抖动检查 journal、checkpoint、复制确认和磁盘延迟。
- 对读旧数据检查 ReadPreference、复制延迟和 ReadConcern。

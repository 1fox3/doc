# 连接的原理

## 核心模型

- MySQL 执行连接时会选择驱动表和被驱动表。
- 驱动表输出的每一批或每一行会用于匹配被驱动表。
- 被驱动表连接列是否有索引，通常决定连接查询能否稳定扩展。

## 连接算法

- Nested Loop Join：最基础的嵌套循环连接。
- Index Nested Loop Join：被驱动表通过索引查找，常见且高效。
- Block Nested Loop Join：使用 join buffer 缓冲驱动表记录，减少被驱动表扫描次数。
- Hash Join：MySQL 8.0 支持等值连接的哈希执行方式。

## SQL 设计

- 连接条件必须写完整，避免笛卡尔积。
- 被驱动表的 `ON` 条件列应有合适索引。
- 外连接中右表过滤条件放在 `ON` 或 `WHERE` 会影响结果语义。

## 排查重点

- `EXPLAIN` 中表访问顺序、`type`、`key`、`rows`、`Extra`。
- `Using join buffer` 通常表示被驱动表缺少高效索引或优化器认为缓冲更划算。
- 大表连接慢时先确认驱动表过滤后行数和被驱动表索引命中情况。

## 半连接优化

- `EXISTS` 和 `IN` 不一定逐行执行，优化器可能改写为半连接。
- 半连接只关心是否存在匹配记录，不需要返回右表全部匹配行。
- 常见策略包括 FirstMatch、LooseScan、Materialization、Duplicate Weedout。
- 如果子查询包含复杂聚合、LIMIT 或非确定条件，优化空间可能受限。

## 连接性能案例

```sql
-- payments.order_id 如果没有索引，会对每个订单扫描 payments。
SELECT o.id
FROM orders o
WHERE EXISTS (
  SELECT 1 FROM payments p
  WHERE p.order_id = o.id AND p.status = 'SUCCESS'
);

CREATE INDEX idx_payments_order_status ON payments(order_id, status);
```

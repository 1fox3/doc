# 事务

## 实现基础

- undo log 保证回滚能力和 MVCC 历史版本。
- redo log 保证提交后的崩溃恢复。
- 锁保证当前读和写写冲突下的数据一致性。
- ReadView 决定快照读可见性。

## 隔离级别

- RC 每条语句生成 ReadView，适合减少锁冲突但允许不可重复读。
- RR 事务内复用 ReadView，是 InnoDB 默认隔离级别。
- 当前读在 RR 下结合 next-key lock 防止范围幻读。

## 提交一致性

- 未开启 binlog 时，InnoDB 主要依赖 redo 完成崩溃恢复。
- 开启 binlog 时，redo 和 binlog 通过两阶段提交保持一致。
- `innodb_flush_log_at_trx_commit` 和 `sync_binlog` 决定崩溃时的数据丢失窗口。

## 实践要点

- 避免长事务、大事务和事务内外部调用。
- 批量变更要分批提交，控制 undo、redo、锁和复制延迟。
- 排查事务问题优先看 `information_schema.innodb_trx` 和 `SHOW ENGINE INNODB STATUS`。

## 事务模板

```sql
START TRANSACTION;

SELECT stock FROM inventory WHERE sku_id = 1001 FOR UPDATE;
UPDATE inventory SET stock = stock - 1 WHERE sku_id = 1001 AND stock > 0;
INSERT INTO order_item(order_id, sku_id, quantity) VALUES (1, 1001, 1);

COMMIT;
```

实际业务中应尽量用单条条件更新替代“先查再改”，减少锁持有时间和并发窗口。

## 事务与消息一致性

- 数据库事务提交成功但消息发送失败，会导致下游不知道变更。
- 消息发送成功但数据库事务回滚，会导致下游消费不存在的变更。
- 常见方案是 outbox 表：在同一数据库事务中写业务数据和消息表，再由异步任务投递消息。

## 长事务来源

- 应用连接池未关闭事务。
- 后台任务一次处理过多数据。
- 人工会话 `BEGIN` 后忘记提交。
- 备份或导出工具开启一致性快照后运行时间过长。

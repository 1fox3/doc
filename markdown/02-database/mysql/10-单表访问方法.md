# 单表访问方法

## 访问方法

- `const`：通过主键或唯一索引等值定位一条记录，优化器可把表当作常量处理。
- `ref`：普通二级索引等值匹配，可能返回多条记录。
- `ref_or_null`：在 `ref` 基础上额外扫描 `NULL` 值记录。
- `range`：使用索引扫描一个或多个范围区间，例如 `BETWEEN`、`IN`、`>`。
- `index`：扫描整个索引树，通常比全表扫描少读列，但仍可能扫描大量记录。
- `ALL`：全表扫描，数据量大时风险最高，但小表或低选择性条件下可能是合理选择。

## 索引合并

- `Intersection` 合并多个索引的交集，常见于多个 `AND` 条件分别命中不同索引。
- `Union` 合并多个索引的并集，常见于 `OR` 条件。
- `Sort-Union` 先对 Row ID 排序再合并，额外排序代价更高。
- 索引合并不是联合索引的替代品，高频查询通常应设计更贴合条件顺序的复合索引。

## 判断方法

```sql
EXPLAIN SELECT * FROM t WHERE id = 1;
EXPLAIN SELECT * FROM t WHERE status = 1 AND created_at >= '2026-01-01';
```

## 实践要点

- `type` 从优到劣大致为 `system`、`const`、`eq_ref`、`ref`、`range`、`index`、`ALL`，但不能只按 type 判断好坏。
- `rows` 和 `filtered` 反映优化器估算值，统计信息不准会导致访问方法选择错误。
- 当 `ALL` 出现在大表热点查询中，应优先检查谓词是否可索引、是否存在隐式转换和是否需要复合索引。

## 访问方法示例

```sql
-- const：主键等值。
EXPLAIN SELECT * FROM user WHERE id = 100;

-- ref：普通索引等值。
EXPLAIN SELECT * FROM user WHERE status = 1;

-- range：范围扫描。
EXPLAIN SELECT * FROM user WHERE created_at >= '2026-01-01';

-- index：可能扫描整个二级索引。
EXPLAIN SELECT name FROM user ORDER BY name;
```

## 选择访问方法的关键

- 条件能否转换为索引扫描区间。
- 扫描区间预计返回多少行。
- 返回行是否需要回表。
- 是否需要额外排序、分组或临时表。
- 表和索引统计信息是否准确。

## 优化动作

- 如果 `key` 为空，检查是否缺索引、条件是否写法不支持索引、字段类型是否一致。
- 如果 `rows` 很大，检查联合索引列顺序是否符合过滤条件。
- 如果 `filtered` 很低，说明扫描后丢弃大量记录，可考虑把过滤列加入索引。

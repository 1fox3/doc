# 基于规则的优化

## 常见改写

- 常量传递：`a = b AND b = 1` 可推导出 `a = 1`。
- 条件化简：移除恒真条件，提前识别恒假条件。
- 外连接消除：当 `WHERE` 条件要求右表列非空时，左连接可能被改写为内连接。
- 子查询优化：`IN`、`EXISTS` 可能被改写为半连接或物化表。
- 谓词下推：将过滤条件尽量下推到更早阶段，减少中间结果。

## 索引相关规则

- 使用最左前缀构造扫描区间。
- 将多个等值条件、范围条件转换为索引区间。
- 对覆盖索引查询避免回表。
- 对可下推条件启用 Index Condition Pushdown，减少回表。

## 实践要点

- SQL 写法影响优化器能否识别规则，例如函数包裹列、隐式转换和复杂表达式会降低可优化性。
- 外连接条件应区分写在 `ON` 还是 `WHERE`，两者语义和优化空间不同。
- 对复杂 SQL 使用 `EXPLAIN FORMAT=JSON` 或 `optimizer_trace` 验证具体改写结果。

## 可优化写法示例

```sql
-- 不利于索引：函数作用在列上。
SELECT * FROM orders WHERE DATE(created_at) = '2026-01-01';

-- 更利于索引：转换成范围条件。
SELECT * FROM orders
WHERE created_at >= '2026-01-01'
  AND created_at < '2026-01-02';
```

## 子查询处理

- 非相关子查询可能先执行并物化结果，再被外层查询使用。
- 相关子查询可能被改写为半连接，也可能按外层行反复执行。
- `EXISTS` 适合表达存在性判断，查询列通常不重要。
- `IN` 遇到 NULL 语义时更复杂，改写前必须确认结果等价。

## 谓词下推边界

- 派生表、视图和聚合查询不一定都能完全下推条件。
- 外连接中被保留表和非保留表的条件下推要保持 NULL 补齐语义。
- 条件包含非确定函数时，优化器不能随意改写执行顺序。

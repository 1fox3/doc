# InnoDB数据统计

## 统计信息用途

- 优化器依赖统计信息估算表大小、索引基数、过滤率和访问成本。
- 统计信息会影响索引选择、连接顺序、是否全表扫描和是否使用索引合并。

## 统计信息类型

- 持久化统计信息由 `innodb_stats_persistent` 控制，默认会保存到系统表中，重启后仍可用。
- 非持久化统计信息会随实例运行动态变化，计划稳定性较差。
- MySQL 8.0 直方图可描述非索引列或低均匀度列的数据分布，改善估算。

## 常用命令

```sql
SHOW VARIABLES LIKE 'innodb_stats_persistent';
ANALYZE TABLE t_user;
SHOW INDEX FROM t_user;
ANALYZE TABLE t_user UPDATE HISTOGRAM ON status WITH 16 BUCKETS;
ANALYZE TABLE t_user DROP HISTOGRAM ON status;
```

## 实践要点

- 数据倾斜严重时，单纯索引基数可能不足以支撑准确估算，可考虑直方图。
- 大批量导入或删除后，统计信息可能滞后，执行计划可能突然变化。
- 线上更新统计信息要评估元数据锁、I/O 和计划变化风险。

## 统计信息偏差表现

- 优化器选择低选择性索引，导致大量回表。
- 多表连接选择了错误驱动表，造成中间结果膨胀。
- 明明有合适索引却选择全表扫描，或相反选择了成本很高的索引扫描。
- `EXPLAIN` 估算 `rows` 和 `EXPLAIN ANALYZE` 实际行数差距很大。

## 直方图适用场景

- 列没有索引，但经常作为过滤条件。
- 列数据分布严重倾斜，例如状态、地区、租户 ID。
- 优化器长期误判选择性，且不适合通过新增索引解决。

## 注意事项

- 直方图会影响执行计划，添加或删除前应在测试环境验证核心 SQL。
- 直方图不是实时更新的，数据分布变化后需要重新分析。
- 高并发线上执行 `ANALYZE TABLE` 需避开高峰，并观察执行计划变化。

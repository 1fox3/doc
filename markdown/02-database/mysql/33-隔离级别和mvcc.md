# 隔离级别和MVCC

## 隔离级别对比

- RU 允许脏读，不适合常规业务。
- RC 每条一致性读语句创建 ReadView，能看到其他事务已提交的新版本。
- RR 事务内第一次一致性读创建 ReadView，后续复用，InnoDB 默认使用该级别。
- Serializable 通过更强约束减少并发异常，但吞吐量最低。

## 快照读和当前读

- 快照读：普通 `SELECT`，基于 MVCC 读取可见历史版本。
- 当前读：`SELECT ... FOR UPDATE`、`SELECT ... LOCK IN SHARE MODE`、`UPDATE`、`DELETE`，读取最新版本并加锁。
- 当前读会参与锁冲突，快照读通常不阻塞写入。

## 关键结论

- MVCC 主要解决读写并发，不替代所有锁。
- RR 下快照读避免幻读现象，当前读依赖 next-key lock 防止范围内插入。
- RC 锁范围通常更小，适合高并发写入，但业务需接受不可重复读。

## 排查关注

- 事务隔离级别是否被连接池或框架修改。
- 是否存在长事务导致 undo 无法清理。
- 当前读范围条件是否命中索引，避免锁范围扩大。

## 异常读现象

- 脏读：读到其他事务未提交数据，InnoDB 常规隔离级别下应避免使用 RU。
- 不可重复读：同一事务内两次读同一行结果不同，RC 下可能出现。
- 幻读：同一事务内两次范围查询出现新增或消失的记录，当前读需要依赖锁约束。

## 业务选择建议

- 账户余额、库存扣减、唯一资源分配等写冲突敏感场景，应明确使用当前读或原子更新。
- 报表查询和普通列表查询优先快照读，避免加锁影响写入。
- 如果使用 RC，需要确认代码没有依赖“事务内多次普通 SELECT 结果稳定”的语义。

## 原子更新示例

```sql
UPDATE inventory
SET stock = stock - 1
WHERE sku_id = 1001 AND stock > 0;
```

该写法通过单条当前读更新完成条件判断和扣减，通常比先查再更新更安全。

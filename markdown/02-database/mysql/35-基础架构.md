# 基础架构

## Server 层

- 连接器负责连接建立、认证、权限和会话状态。
- 分析器负责词法、语法解析和语义检查。
- 优化器负责选择索引、访问路径、连接顺序和执行算法。
- 执行器按计划调用存储引擎接口。
- binlog 位于 Server 层，用于复制和时间点恢复。

## 存储引擎层

- InnoDB 负责 B+Tree 索引、Buffer Pool、事务、行锁、MVCC、redo log 和 undo log。
- MySQL 通过存储引擎接口屏蔽不同引擎实现差异。
- 同一 SQL 在不同引擎上的锁、事务和崩溃恢复语义可能不同。

## 一条更新 SQL 的链路

- 连接器校验账号权限。
- 分析器和优化器生成执行计划。
- 执行器调用 InnoDB 定位并修改记录。
- InnoDB 写 undo、修改 Buffer Pool 页、写 redo。
- Server 层写 binlog，提交阶段协调 redo 和 binlog。

## 实践要点

- 分析问题时先判断发生在 Server 层还是 InnoDB 层。
- 慢 SQL 多看优化器和执行器，锁等待多看 InnoDB 和 MDL，复制问题多看 binlog 和事务提交。

## 一条查询 SQL 的链路

- 连接器复用或建立连接，检查账号权限。
- 分析器解析 SQL 并检查表、列和函数是否存在。
- 优化器选择访问方法、索引、连接顺序和排序策略。
- 执行器调用 InnoDB 读取索引页和数据页。
- InnoDB 从 Buffer Pool 命中或从磁盘读取页，返回记录给 Server 层。
- Server 层做剩余过滤、排序、聚合或返回结果集。

## 一条更新 SQL 的关键日志

- undo log 先记录修改前版本，用于回滚和 MVCC。
- Buffer Pool 中的数据页被修改为脏页。
- redo log 记录页修改，保证崩溃恢复。
- binlog 记录逻辑变更，用于复制和恢复。
- 提交时通过 redo/binlog 两阶段提交协调一致性。

## 定位分层

- 连接数打满多在连接层或应用连接池。
- 语法错误和权限错误多在 Server 层。
- 执行计划差多在优化器和统计信息。
- 磁盘 I/O、锁、事务、崩溃恢复多在 InnoDB 层。

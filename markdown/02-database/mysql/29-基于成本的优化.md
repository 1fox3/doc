# 基于成本的优化

## 优化器决策

- 选择访问方法：全表扫描、索引扫描、范围扫描、索引合并。
- 选择索引：比较扫描行数、回表成本、排序成本和过滤率。
- 选择连接顺序：估算不同表顺序的中间结果规模。
- 选择是否物化子查询、是否使用临时表或额外排序。

## 成本偏差来源

- 统计信息过期。
- 数据分布倾斜。
- 条件之间存在相关性，但优化器按独立性估算。
- 缓存命中、磁盘延迟和并发锁等待不是静态成本模型能完全反映的因素。

## 处理方式

- `ANALYZE TABLE` 刷新统计信息。
- 为倾斜列建立直方图。
- 改写 SQL，让条件更可索引。
- 增加覆盖索引或更符合查询模式的联合索引。
- 谨慎使用 hint 或 `FORCE INDEX` 作为最后手段。

## Hint 使用边界

- Hint 可用于临时绕过优化器误判，但应记录原因和退出条件。
- 数据分布变化后，原先强制的索引可能变差。
- MySQL 支持 optimizer hint，例如 `JOIN_ORDER`、`INDEX`、`NO_INDEX` 等，版本差异需要确认。

## 成本与真实耗时差异

- 成本模型不能完整反映锁等待。
- 成本模型不能准确反映磁盘瞬时抖动和 Buffer Pool 命中状态。
- 成本模型对列相关性和复杂表达式估算有限。
- 因此慢 SQL 分析需要结合执行计划、实际耗时、扫描行数、等待事件和业务并发。

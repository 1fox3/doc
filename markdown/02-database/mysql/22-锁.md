# 锁

## 锁类型

- 表锁：锁定整张表，粒度大，并发低，常见于 DDL、元数据锁和部分存储引擎。
- 行锁：InnoDB 基于索引记录加锁，粒度小，并发高。
- 共享锁 `S`：允许其他事务读，不允许写。
- 排他锁 `X`：阻塞其他事务读写同一加锁资源。
- 意向锁：表级锁，表示事务将对表中记录加 S 或 X 行锁，用于表锁和行锁兼容性判断。

## InnoDB 行锁

- Record Lock 锁定已有索引记录。
- Gap Lock 锁定索引记录之间的间隙，阻止插入。
- Next-Key Lock 是 Record Lock 与 Gap Lock 的组合，常用于 RR 下范围当前读防幻读。
- Insert Intention Lock 是插入意向锁，多个事务插入不同位置通常可并发。

## 死锁与等待

- 死锁是多个事务循环等待对方持有的锁，InnoDB 检测到后会回滚其中一个事务。
- 锁等待不一定是死锁，可能只是等待另一个长事务提交。
- 没有命中索引的更新或范围条件过宽，会导致锁范围扩大。

## 排查命令

```sql
SHOW ENGINE INNODB STATUS\G
SELECT * FROM performance_schema.data_locks\G
SELECT * FROM performance_schema.data_lock_waits\G
SELECT * FROM information_schema.innodb_trx\G
```

## 实践要点

- 更新和删除必须尽量走明确索引，避免扫描过程中锁住大量记录。
- 多表更新保持固定访问顺序，减少死锁概率。
- 事务内操作尽量短，避免把锁持有时间扩展到业务逻辑或外部调用。

## 加锁示例

```sql
-- 主键等值当前读，通常锁定单条记录。
SELECT * FROM orders WHERE id = 100 FOR UPDATE;

-- 范围当前读，在 RR 下可能加 next-key lock，阻止范围内插入。
SELECT * FROM orders
WHERE user_id = 10 AND created_at >= '2026-01-01'
FOR UPDATE;

-- 无索引条件更新，可能扫描大量记录并扩大锁范围。
UPDATE orders SET status = 2 WHERE remark = 'timeout';
```

## 死锁处理原则

- 死锁不是数据库 bug，而是并发访问顺序冲突的结果。
- 应用层必须对死锁错误做有限次数重试，重试前回滚当前事务。
- 从 `SHOW ENGINE INNODB STATUS` 的 latest deadlock 中找出两个事务的 SQL、索引和等待锁。
- 通过固定访问顺序、缩小事务范围、补充索引和减少批量锁定记录降低死锁概率。

## MDL 风险

- 长事务即使只执行过普通 SELECT，也可能持有表的 MDL 读锁直到事务结束。
- DDL 等待 MDL 写锁时，后续 DML 可能排队在 DDL 后面，引发业务雪崩。
- 线上 DDL 前应检查长事务，并设置合理锁等待超时或使用在线变更工具。

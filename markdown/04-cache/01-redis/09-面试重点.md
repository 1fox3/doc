# Redis 面试重点

## Redis 为什么快

- 纯内存访问为主。
- 单线程执行命令避免大部分锁竞争和上下文切换。
- IO 多路复用高效管理连接。
- 数据结构紧凑，针对不同规模使用不同编码。
- Redis 6 之后支持多线程 IO，但命令执行主路径仍强调串行语义。

## 数据结构

- String 可用于缓存、计数、分布式锁值，但大 value 会阻塞网络和内存。
- Hash 适合对象字段，但大 hash 要警惕 big key。
- List 适合简单队列，但可靠消费建议看 Stream 或 MQ。
- Set/ZSet 适合去重、交集、排行榜，ZSet 底层常见跳表和哈希表。
- Stream 支持消费者组、ack、pending list，更适合轻量消息流。

## 持久化

- RDB 是快照，恢复快但可能丢最近数据。
- AOF 记录写命令，安全性更好但文件可能更大，需要 rewrite。
- 混合持久化结合 RDB 恢复速度和 AOF 增量安全性。

## 缓存问题

- 穿透：查询不存在数据，使用布隆过滤器、空值缓存、参数校验。
- 击穿：热点 key 过期，大量请求打 DB，使用互斥重建、逻辑过期、热点不过期。
- 雪崩：大量 key 同时过期或 Redis 故障，使用随机 TTL、多级缓存、限流降级。
- 一致性：常见策略是先更新 DB 再删缓存，配合重试、延迟双删或订阅 binlog 补偿。

## 分布式锁

- `SET key value NX PX` 是基础，value 必须唯一，释放锁用 Lua 校验 value 后删除。
- 锁过期时间要覆盖业务执行时间，必要时续期。
- Redis 锁适合一般互斥，不适合强一致金融级互斥，RedLock 也存在争议。

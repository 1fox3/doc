# 过期淘汰与内存管理

## 过期时间

Redis 通过 `EXPIRE`、`PEXPIRE`、`EXPIREAT`、`PEXPIREAT` 设置 key 的过期时间，通过 `TTL`、`PTTL` 查看剩余时间，通过 `PERSIST` 移除过期时间。

过期时间保存在 `redisDb.expires` 字典中，key 指向过期时间戳。过期不代表立即删除，删除时机由策略决定。

## 过期删除策略

- 定时删除：到期立即删除，内存释放及时，但需要大量定时器，CPU 成本高。
- 惰性删除：访问 key 时检查是否过期，过期则删除，CPU 友好但可能滞留内存。
- 定期删除：后台周期性抽样检查过期 key，在 CPU 和内存之间折中。

Redis 主要采用惰性删除加定期删除。线上看到已过期 key 未立即消失是正常现象。

## 内存淘汰策略

当 `used_memory` 超过 `maxmemory` 时，Redis 按 `maxmemory-policy` 选择淘汰行为。

- `noeviction`：不淘汰，写命令报错，适合不能丢数据的场景。
- `volatile-ttl`：只从设置过期时间的 key 中淘汰即将过期的 key。
- `volatile-random`：只从设置过期时间的 key 中随机淘汰。
- `volatile-lru`：只从设置过期时间的 key 中近似淘汰最近最少使用。
- `volatile-lfu`：只从设置过期时间的 key 中近似淘汰最低频访问。
- `allkeys-random`：从全部 key 中随机淘汰。
- `allkeys-lru`：从全部 key 中近似淘汰最近最少使用。
- `allkeys-lfu`：从全部 key 中近似淘汰最低频访问。

缓存场景常用 `allkeys-lru` 或 `allkeys-lfu`。如果只有缓存 key 设置了 TTL，而元数据 key 不应被淘汰，可考虑 `volatile-*`，但要确保所有可淘汰 key 都设置了过期时间。

## LRU 与 LFU

Redis 的 LRU 和 LFU 都是近似算法，不维护全量精确链表。这样能降低内存和 CPU 成本，但淘汰结果不一定完全符合理论顺序。

- LRU 基于最近访问时间，适合访问热点随时间变化较快的场景。
- LFU 基于访问频率衰减，适合长期热点稳定的场景。
- `maxmemory-samples` 影响采样精度，越大越接近真实淘汰，但 CPU 成本越高。

## 内存碎片

内存碎片通常来自频繁分配释放、对象大小变化、jemalloc 分配粒度和 copy-on-write。碎片率可通过 `INFO memory` 查看。

关键指标：

- `used_memory`：Redis 分配器视角已使用内存。
- `used_memory_rss`：操作系统视角 Redis 占用物理内存。
- `mem_fragmentation_ratio`：约等于 `used_memory_rss / used_memory`。
- `allocator_frag_ratio`、`allocator_frag_bytes`：分配器碎片指标。

碎片率略高于 1 通常正常。明显偏高且 RSS 压力大时，可以考虑主动碎片整理。

## 主动碎片整理

相关配置包括：

- `activedefrag yes`：开启主动碎片整理。
- `active-defrag-ignore-bytes`：碎片字节数超过阈值才整理。
- `active-defrag-threshold-lower`：低阈值，达到后开始整理。
- `active-defrag-threshold-upper`：高阈值，超过后更积极整理。
- `active-defrag-cycle-min`、`active-defrag-cycle-max`：控制 CPU 占用范围。

开启整理会消耗 CPU，应结合延迟和碎片收益评估。

## 大 key 治理

大 key 治理优先级高于单纯调参数。

- String 过大：拆分字段、压缩、外部对象存储。
- Hash 字段过多：按业务维度分片，例如 `user:attrs:{uid}:base`、`user:attrs:{uid}:stats`。
- List/Set/ZSet 元素过多：按时间、范围或 hash tag 分片。
- 删除大 key：优先使用 `UNLINK`，避免 `DEL` 同步释放阻塞。

排查命令：`redis-cli --bigkeys`、`MEMORY USAGE key`、`SCAN` 配合类型和长度命令。

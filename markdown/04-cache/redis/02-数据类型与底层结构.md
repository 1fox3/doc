# 数据类型与底层结构

## 对象模型

Redis 的值对象由 `redisObject` 描述，核心字段包括 `type`、`encoding`、`ptr`、`refcount` 和 `lru/lfu` 信息。`type` 决定对外数据类型，`encoding` 决定内部实现方式。

同一种数据类型会根据元素数量、元素大小和版本差异选择不同编码。Redis 7 中大量紧凑结构使用 `listpack`，旧资料中的 `ziplist` 需要按版本区分理解。

## String

- 适合缓存对象序列化结果、计数器、分布式锁 token、简单状态位。
- 底层可能使用 `int`、`embstr`、`raw` 编码。
- `embstr` 将对象头和 SDS 连续分配，适合较短字符串。
- `raw` 适合较长字符串，扩容和修改更灵活。
- 常用命令有 `GET`、`SET`、`MGET`、`INCR`、`DECR`、`SETEX`、`SET NX PX`。

## List

- 适合简单队列、最近列表和固定窗口数据。
- 现代 Redis 主要使用 `quicklist`，它由多个紧凑节点组成，兼顾链表插入删除和压缩存储。
- 头尾操作通常是 O(1)，按索引访问或范围过大时可能退化。
- 常用命令有 `LPUSH`、`RPUSH`、`LPOP`、`RPOP`、`BLPOP`、`BRPOP`、`LTRIM`。

## Hash

- 适合存储对象的多个字段，例如用户资料、配置项、商品属性。
- 小对象通常使用 `listpack`，字段或值过大、元素过多时转为哈希表。
- 相比把整个对象序列化为 String，Hash 可以局部更新字段，但字段过多也会形成大 key。
- 常用命令有 `HGET`、`HSET`、`HMGET`、`HINCRBY`、`HSCAN`。

## Set

- 适合去重、标签集合、共同好友、黑白名单。
- 全整数且数量较小时可使用 `intset`，否则使用哈希表。
- 集合运算 `SINTER`、`SUNION`、`SDIFF` 可能消耗大量 CPU，生产中要控制集合大小。

## Sorted Set

- 适合排行榜、延迟队列、按分值排序的时间线。
- 小规模可使用 `listpack`，大规模使用 `skiplist` 加 `dict`。
- `dict` 支持按 member O(1) 查 score，`skiplist` 支持按 score 范围查询和排名。
- 常用命令有 `ZADD`、`ZRANGE`、`ZREVRANGE`、`ZRANGEBYSCORE`、`ZREM`、`ZCARD`。

## Bitmap

- 本质是 String 的位操作，适合签到、在线状态、布尔标记和简单布隆过滤器实现。
- `SETBIT`、`GETBIT` 单点操作成本低，`BITCOUNT`、`BITOP` 会扫描字符串范围，超大 Bitmap 要分片。

## HyperLogLog

- 用于基数统计，例如 UV 估算。
- 优点是内存固定且很小，缺点是有标准误差，不支持精确去重列表。
- 常用命令有 `PFADD`、`PFCOUNT`、`PFMERGE`。

## Geo

- 基于 Sorted Set 存储地理位置，经纬度会编码为 GeoHash 分值。
- 适合附近的人、门店检索等粗粒度空间查询。
- 常用命令有 `GEOADD`、`GEODIST`、`GEOSEARCH`。

## Stream

- Redis 5 引入，适合轻量消息流、事件日志和消费者组模型。
- 消息 ID 由毫秒时间戳和序列号组成。
- 消费者组通过 PEL 记录已投递未确认消息，支持 `XACK`、`XPENDING`、`XCLAIM` 或 `XAUTOCLAIM` 处理异常消费者。
- Stream 不是 Kafka 替代品，消息堆积、分区扩展、跨机房复制和长期存储能力都更弱。

## 底层结构重点

- SDS：记录长度，二进制安全，避免 C 字符串 O(N) 求长度和缓冲区溢出。
- dict：Redis 数据库 keyspace、Hash、大 Set 等核心结构，采用渐进式 rehash 降低一次性迁移阻塞。
- intset：整数集合，按整数宽度升级，不降级。
- listpack：紧凑连续内存结构，替代 ziplist，降低连锁更新风险。
- quicklist：List 的主要实现，由多个 listpack 节点组成。
- skiplist：有序集合的范围查询和排名核心结构。

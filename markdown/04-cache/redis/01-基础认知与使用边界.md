# 基础认知与使用边界

## Redis 是什么

Redis 是基于内存的键值数据库，也可以理解为数据结构服务器。它以单线程命令执行为核心，结合 IO 多路复用、高效对象编码、持久化和复制能力，提供低延迟的数据访问。

常见用途包括缓存、计数器、排行榜、分布式锁、会话存储、限流、去重、延迟队列和轻量消息流。

## 为什么快

- 数据主要在内存中读写，避免了常规磁盘随机 IO。
- 命令执行线程避免了大量锁竞争和上下文切换。
- 基于 `epoll`、`kqueue` 等 IO 多路复用处理大量连接。
- 数据结构针对常见场景做了紧凑编码，例如 `listpack`、`intset`、`quicklist`。
- 协议 RESP 简单，解析和序列化成本低。

## 基础限制

- String 值最大为 512 MB，但生产环境不应接近该上限。
- 单个 key 过大容易造成命令阻塞、复制延迟、AOF 重写膨胀和 Cluster 迁移抖动。
- 单实例默认有 16 个逻辑库，Cluster 模式只使用 0 号库。
- Redis 不是强一致数据库，复制、故障转移和缓存场景都可能出现短暂不一致。
- Redis 不适合作为无限容量存储，必须设置容量规划、过期策略和淘汰策略。

## 常见配置

- `maxmemory`：限制实例最大可用内存，缓存场景建议显式设置。
- `maxmemory-policy`：内存达到上限后的淘汰策略。
- `databases`：逻辑数据库数量，生产中更推荐通过不同实例隔离业务。
- `timeout`：客户端空闲连接超时。
- `tcp-keepalive`：检测异常 TCP 连接。
- `protected-mode`、`requirepass`、`aclfile`：访问控制相关配置。

## 选择 Redis 前要确认的问题

- 数据是否允许丢失：如果不能丢失，只用 Redis 做唯一存储风险较高。
- 是否需要跨 key 强事务：Redis 的事务不支持回滚，复杂一致性应放在业务或数据库层。
- 容量是否可控：缓存必须有过期时间、淘汰策略和降级方案。
- 访问是否集中：热 key 和大 key 会把单线程模型的优势变成瓶颈。
- 是否需要水平扩展：数据量或 QPS 超过单实例能力时需要 Cluster、代理或多级缓存。

## 生产使用原则

- key 命名要包含业务域、对象类型和主键，例如 `user:profile:10001`。
- 避免无过期时间的缓存 key，除非它属于明确的状态或索引数据。
- 避免在主链路使用 `KEYS`、大范围 `SMEMBERS`、大 `LRANGE` 等 O(N) 命令。
- 批量操作优先使用 pipeline、Lua 或批量命令，但要控制单次 payload 大小。
- 所有缓存方案都要设计降级路径，不能让 Redis 故障直接拖垮数据库。

# GC 日志与调优

GC 调优的目标是让应用在可接受的停顿、吞吐、内存成本和稳定性之间达到平衡。调优必须先看日志和指标，再决定是否修改参数。

## 先看什么

- GC 类型：Young GC、Mixed GC、Full GC、Concurrent Cycle、Allocation Failure、Metadata GC Threshold。
- 停顿时间：单次暂停、P95/P99 暂停、最大暂停，以及暂停发生的业务时间点。
- 回收效果：GC 前后 Eden、Survivor、Old、Humongous、Metaspace 的变化。
- 分配速率：Eden 填满速度、TLAB refill、对象分配热点。
- 晋升情况：对象年龄分布、Survivor 是否放得下、老年代增长速度。
- 并发阶段：并发标记是否及时完成，是否发生 to-space exhausted、concurrent mode failure 或 evacuation failure。

## JDK 9+ 统一日志

推荐开启 GC 与安全点日志：

```bash
-Xlog:gc*,safepoint:file=/path/to/gc-%t.log:time,uptime,level,tags:filecount=10,filesize=100m
```

需要观察对象年龄或堆更详细信息时可临时增加：

```bash
-Xlog:gc+age=trace,gc+heap=debug,gc+ergo=trace:file=/path/to/gc-detail-%t.log:time,uptime,level,tags:filecount=5,filesize=100m
```

## JDK 8 常见日志

## 常见日志信号

| 信号 | 常见原因 | 处理方向 |
| --- | --- | --- |
| Young GC 很频繁但 Old 不涨 | 分配速率高、短命对象多、Eden 偏小 | 查分配热点，评估对象创建、批量大小、Eden 或堆大小 |
| Young GC 后 Old 快速上涨 | 存活对象多、Survivor 不足、晋升过早 | 看对象年龄分布和引用链，检查缓存、队列、批处理 |
| Full GC 后 Old 降不下来 | 老年代真实存活对象多、泄漏 | 导出 hprof，分析 Retained Size 和 GC Roots |
| Metadata GC Threshold | 元空间接近阈值 | 查类数量、ClassLoader、动态代理、脚本或热部署 |
| Humongous 对象多 | 大数组、大字符串、大 byte buffer | 降低单对象大小，分块处理，调整 G1 Region 需谨慎 |
| evacuation failure | G1 复制存活对象时空间不足 | 增大堆、提前并发标记、降低晋升压力或检查泄漏 |
| to-space exhausted | G1 evacuation 目标空间不足 | 增大堆，降低分配峰值，检查 Humongous 和老年代占用 |
| concurrent mode failure | CMS 并发回收赶不上分配 | 提前启动 CMS、降低老年代增长或迁移到 G1/ZGC |
| Allocation Stall | ZGC/Shenandoah 等并发回收跟不上分配 | 增大堆、降低分配速率、增加 CPU 或减少 throttling |
| System.gc() | 代码或框架显式触发 GC | 查调用来源，评估 `-XX:+DisableExplicitGC` 或改为并发显式 GC 策略 |

## 日志字段理解

- `Pause Young` 通常表示年轻代停顿，关注停顿耗时、Eden 回收量、Survivor 和 Old 变化。
- `Pause Full` 表示 Full GC，通常要重点追查触发原因和回收效果。
- `Concurrent Mark` 表示并发标记阶段，不完全 STW，但会消耗 CPU。
- `User` 是所有 GC 线程消耗的 CPU 时间总和，`Sys` 是内核态时间，`Real` 是墙钟时间。
- `User + Sys` 明显大于 `Real` 通常说明 GC 多线程并行；`Real` 异常大可能和 CPU 抢占、IO、虚拟化或容器 throttling 有关。

## G1 调优思路

- 不优先固定新生代大小，固定 `-Xmn` 会削弱 G1 按停顿目标自适应调整的能力。
- `-XX:MaxGCPauseMillis` 是目标不是承诺，设置过低会导致年轻代过小、GC 过频繁。
- 老年代上涨快时先看 Mixed GC 是否发生、回收收益是否低、并发标记是否太晚。
- Humongous 对象超过 Region 一半大小会占用连续 Region，容易造成空间碎片和提前触发并发周期。
- `-XX:InitiatingHeapOccupancyPercent` 可让并发标记更早启动，但不能解决真实泄漏。

### G1 常见调整

- 停顿超目标且 Young GC 频繁：检查停顿目标是否过低、分配速率是否异常、对象是否过早晋升。
- Mixed GC 回收少：检查老年代 Region 垃圾比例、`G1HeapWastePercent`、Mixed GC 次数和是否存在大对象。
- 并发标记启动太晚：可降低 `InitiatingHeapOccupancyPercent`，但要确认不是泄漏。
- to-space 不足：可增大堆、提高 `G1ReservePercent`、降低晋升压力或减少 Humongous 分配。

## ZGC 调优思路

- ZGC 以低停顿为目标，主要成本从 STW 转移到并发阶段和读屏障。
- 堆需要留出足够余量让并发回收追上分配速度，低延迟场景不要把堆压得太满。
- 重点观察 Allocation Stall、Concurrent GC 周期耗时、Live Set、分配速率和 CPU 余量。
- JDK 17 以后 ZGC 可用于大堆低延迟服务，JDK 21 的分代 ZGC 能更好处理大量短命对象。

## 调优流程

1. 明确目标：吞吐优先、延迟优先、成本优先或稳定性优先。
2. 收集基线：GC 日志、JFR、系统 CPU、RSS、容器 limit、请求延迟和错误率。
3. 判断瓶颈：分配过快、存活对象过多、老年代不足、元空间膨胀、直接内存或线程过多。
4. 每次只改少量参数，并保留修改前后的同口径指标。
5. 压测和生产回放都要覆盖峰值流量、批任务、缓存预热和发布重启场景。

## 面试回答模板

1. 先确认目标：降低延迟、提高吞吐、减少 OOM，还是降低机器成本。
2. 说明证据：GC 日志、JFR、jstat、heap dump、NMT、容器指标。
3. 判断原因：分配速率、存活对象、晋升、元空间、直接内存、CPU 不足。
4. 给出方案：代码层减少对象或泄漏，参数层调整堆、GC、并发线程和日志。
5. 验证结果：对比 P99、Full GC、分配速率、RSS、CPU、错误率，保留回滚方案。

## 常见误区

- 只看平均 GC 时间，忽略最大停顿和业务尖刺。
- 只增大堆，不分析对象为什么存活。
- 只设置 `MaxGCPauseMillis`，不看分配速率和 CPU 余量。
- 把 Full GC 次数作为唯一指标，忽略吞吐下降、Allocation Stall 和容器 OOMKill。
- 在容器内只配置 `-Xmx`，忽略元空间、直接内存、线程栈、代码缓存和 native 内存。

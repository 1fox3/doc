# 面试重点

这份文档用于面试前串联 JVM 高频问题。回答时不要只背结论，要能解释机制、边界条件和排查方法。

## 对象从 new 到回收

1. 编译期 `new` 语句生成 `new`、`dup`、`invokespecial <init>` 等字节码。
2. 执行 `new` 时先检查常量池中的类符号引用是否已加载、解析、初始化。
3. JVM 在堆上分配内存，常见快路径是 Eden 中 TLAB 指针碰撞。
4. 内存清零，设置对象头，包括 Mark Word、类型指针、数组长度等。
5. 执行构造方法 `<init>`，完成实例字段赋值和构造逻辑。
6. 对象被引用时成为可达对象；没有从 GC Roots 出发的引用链时成为不可达候选。
7. GC 根据可达性分析、引用类型、finalize/Cleaner、分代或 Region 策略决定回收。

## JVM 内存区域高频题

| 问题 | 回答要点 |
| --- | --- |
| 堆和栈区别 | 堆线程共享、存对象、GC 管理；栈线程私有、存栈帧、方法调用出入栈 |
| 方法区和元空间 | 方法区是规范逻辑概念；HotSpot JDK 8+ 用本地内存元空间实现类元数据 |
| 运行时常量池和字符串常量池 | 运行时常量池属于方法区逻辑内容；字符串常量池 JDK 7+ 主要在堆中 |
| 直接内存 | 不属于 Java 堆，但属于进程内存，受容器 limit 和 `MaxDirectMemorySize` 影响 |
| 程序计数器为什么线程私有 | 线程切换后需要恢复各自字节码执行位置 |

## 类加载高频题

- 类加载过程：加载、验证、准备、解析、初始化、使用、卸载；验证、准备、解析合称连接。
- 准备阶段给类变量赋零值，初始化阶段执行 `<clinit>()`，编译期常量可能通过 `ConstantValue` 在准备阶段赋值。
- 双亲委派避免核心类被篡改，保证同名核心类只有一份类型定义。
- 判断两个类是否相同，要同时看类全限定名和定义类加载器。
- 破坏双亲委派的场景包括 SPI、Tomcat WebApp 隔离、OSGi、插件系统。
- 类卸载要求实例不可达、`Class` 对象不可达、ClassLoader 不可达。

## GC 高频题

| 问题 | 回答要点 |
| --- | --- |
| 为什么不用引用计数 | 无法解决循环引用，维护计数也有开销 |
| GC Roots 有哪些 | 栈引用、JNI、静态字段、运行时常量池、锁对象、JVM 内部引用 |
| 三种算法区别 | 标记清除有碎片，复制需要额外空间，整理移动对象成本高但无碎片 |
| 分代假说 | 大部分对象朝生夕死，少量对象长期存活，所以新生代和老年代用不同策略 |
| Minor/Major/Full GC | Minor 常指新生代；Major 常被不同资料混用；Full 通常涉及整个堆和方法区相关回收 |
| CMS 问题 | 浮动垃圾、碎片、并发失败、CPU 敏感 |
| G1 特点 | Region、RSet、CSet、Young/Mixed GC、可预测停顿目标、复制整理 |
| ZGC 特点 | 低停顿、并发标记和转移、读屏障、着色指针、需要 CPU 和堆余量 |

## JMM 高频题

- JMM 解决原子性、可见性、有序性问题。
- happens-before 是可见性和有序性的规则，不等于真实时间先后。
- volatile 保证可见性和有序性，不保证复合操作原子性。
- synchronized 同时提供互斥、可见性和有序性。
- final 字段有初始化安全保证，但构造期间 this 逸出会破坏保证。
- CAS 依赖比较并交换，问题包括 ABA、自旋成本和单变量限制。

## synchronized 和锁优化

- synchronized 锁普通对象、`this` 或 `Class` 对象，支持可重入。
- 同步代码块使用 `monitorenter`/`monitorexit`，同步方法使用 `ACC_SYNCHRONIZED`。
- 轻量级锁适合低竞争，通过 CAS 操作对象头和锁记录。
- 重量级锁依赖 Monitor 和操作系统阻塞/唤醒，竞争高时避免无意义自旋。
- JIT 可通过逃逸分析做锁消除和锁粗化。
- JDK 15 起偏向锁默认禁用，面试回答要说明版本差异。

## JIT 高频题

- 解释器启动快，JIT 峰值性能好；分层编译平衡启动速度和长期性能。
- 热点探测基于方法调用计数器和回边计数器。
- 常见优化包括内联、逃逸分析、标量替换、锁消除、公共子表达式消除、边界检查消除。
- JIT 优化依赖运行时 Profile，假设失效会去优化。
- 微基准必须使用 JMH，避免预热不足、死代码消除、常量折叠和逃逸分析误导。

## 调优题回答框架

1. 先问目标：降低 P99、减少 Full GC、提升吞吐、减少 OOM，还是降低成本。
2. 先采集证据：GC 日志、JFR、jstat、jcmd、heap dump、NMT、容器指标。
3. 判断瓶颈：分配速率、存活对象、晋升失败、元空间、直接内存、线程、CPU throttling。
4. 优先改代码和容量模型，再谨慎改 JVM 参数。
5. 修改后用同口径指标验证，并准备回滚。

## 高频追问

### 为什么 `i++` 不是原子操作

`i++` 包含读取、加一、写回多个步骤。volatile 只能保证单次读写可见，不能把复合操作变成原子操作。需要用锁、AtomicInteger 或 LongAdder。

### 为什么双重检查锁要加 volatile

对象创建可拆成分配内存、初始化对象、引用赋值。没有 volatile 时，引用赋值可能被其他线程先观察到，导致读到未完全初始化对象。

### 为什么 G1 不建议固定 Xmn

G1 会根据停顿目标动态调整年轻代 Region 数量。固定新生代会削弱自适应能力，可能导致停顿目标和回收效率失衡。

### 为什么堆不大也会 OOMKill

容器限制的是进程总内存。RSS 包括堆、元空间、直接内存、线程栈、代码缓存、GC native 内存、JNI 库和 mmap，不只看 `-Xmx`。

### Full GC 后内存降不下来怎么办

先判断是正常高存活还是泄漏。导出 heap dump，用 MAT 看 Dominator Tree、Retained Heap 和 Path To GC Roots，再结合业务缓存、队列、ThreadLocal 和静态集合定位。

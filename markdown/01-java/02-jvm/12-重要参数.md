# 重要参数

JVM 参数调优应先明确目标，再基于 GC 日志、JFR、线程栈、堆转储和系统指标验证。不要只根据经验值修改参数。

## 堆内存

| 参数 | 说明 |
| --- | --- |
| `-Xms<size>` | 初始堆大小 |
| `-Xmx<size>` | 最大堆大小 |
| `-XX:InitialRAMPercentage` | 按容器或机器内存百分比设置初始堆 |
| `-XX:MaxRAMPercentage` | 按容器或机器内存百分比设置最大堆 |

生产环境常把 `-Xms` 和 `-Xmx` 设置为相同值，减少运行期堆扩缩容带来的抖动。容器环境需要确认 JVM 版本对 cgroup 的支持。

## 新生代与晋升

| 参数 | 说明 |
| --- | --- |
| `-Xmn<size>` | 新生代大小，G1 下通常不建议固定 |
| `-XX:NewSize` | 新生代初始大小 |
| `-XX:MaxNewSize` | 新生代最大大小 |
| `-XX:NewRatio` | 老年代与新生代比例 |
| `-XX:SurvivorRatio` | Eden 与单个 Survivor 的比例 |
| `-XX:MaxTenuringThreshold` | 对象晋升年龄阈值 |
| `-XX:PretenureSizeThreshold` | 大对象直接进入老年代阈值，主要对 Serial/ParNew 有效 |

## 栈

| 参数 | 说明 |
| --- | --- |
| `-Xss<size>` | 单线程 Java 栈大小 |

`-Xss` 越大，递归深度越高，但同等进程内存下可创建线程数越少。

## 元空间与直接内存

| 参数 | 说明 |
| --- | --- |
| `-XX:MetaspaceSize` | 触发元空间 GC 的初始阈值 |
| `-XX:MaxMetaspaceSize` | 元空间最大值 |
| `-XX:MaxDirectMemorySize` | 最大直接内存 |
| `-XX:ReservedCodeCacheSize` | 代码缓存大小 |

元空间持续上涨通常要排查动态类生成和 ClassLoader 泄漏，不应只盲目增大上限。

## 容器参数

| 参数 | 说明 |
| --- | --- |
| `-XX:+UseContainerSupport` | 启用容器资源感知，新版本通常默认开启 |
| `-XX:InitialRAMPercentage` | 按可用内存百分比设置初始堆 |
| `-XX:MaxRAMPercentage` | 按可用内存百分比设置最大堆 |
| `-XX:MinRAMPercentage` | 小内存场景下的最小堆百分比策略 |
| `-XX:ActiveProcessorCount=<n>` | 指定 JVM 看到的处理器数量 |

容器内 `-Xmx` 不能贴近 memory limit，需要给元空间、直接内存、线程栈、代码缓存、GC native 内存和系统开销留余量。

## 收集器选择

| 参数 | 说明 |
| --- | --- |
| `-XX:+UseSerialGC` | Serial + Serial Old |
| `-XX:+UseParallelGC` | Parallel Scavenge + Parallel Old |
| `-XX:+UseG1GC` | G1 收集器，JDK 9+ 默认 |
| `-XX:+UseZGC` | ZGC，低延迟场景 |
| `-XX:+UseShenandoahGC` | Shenandoah，低延迟场景，取决于发行版支持 |

CMS 相关参数仅适用于旧 JDK，JDK 14 已移除 CMS。

## 版本默认值

| JDK 版本 | 默认收集器趋势 |
| --- | --- |
| JDK 7/8 Server 常见 | Parallel GC，具体取决于机器和启动模式 |
| JDK 9+ | G1 通常为默认收集器 |
| JDK 17/21 | G1 默认，ZGC/Shenandoah 可按场景启用 |

实际默认值必须以 `java -XX:+PrintCommandLineFlags -version` 或 `jcmd <pid> VM.flags` 为准，不同发行版可能有差异。

## G1 常用参数

| 参数 | 说明 |
| --- | --- |
| `-XX:MaxGCPauseMillis=<n>` | 目标最大停顿时间，默认通常为 200ms |
| `-XX:G1HeapRegionSize=<size>` | Region 大小，通常由 JVM 自动选择 |
| `-XX:InitiatingHeapOccupancyPercent=<n>` | 触发并发标记的堆占用阈值，默认常见为 45 |
| `-XX:ConcGCThreads=<n>` | 并发 GC 线程数 |
| `-XX:ParallelGCThreads=<n>` | STW 阶段并行 GC 线程数 |
| `-XX:G1ReservePercent=<n>` | G1 预留空间比例，降低 to-space 不足风险 |
| `-XX:G1HeapWastePercent=<n>` | 控制 Mixed GC 可接受的堆浪费比例 |
| `-XX:G1MixedGCCountTarget=<n>` | 一轮标记后目标 Mixed GC 次数 |

G1 调优优先观察 Mixed GC、Humongous 对象、晋升失败和并发标记是否及时。

## ZGC 常用参数

| 参数 | 说明 |
| --- | --- |
| `-XX:+UseZGC` | 启用 ZGC |
| `-XX:+ZGenerational` | 启用分代 ZGC，JDK 21+ 可用 |
| `-XX:SoftMaxHeapSize=<size>` | ZGC 软堆上限，允许必要时超过但倾向控制在该值内 |
| `-XX:ConcGCThreads=<n>` | 并发 GC 线程数 |

ZGC 调优重点不是压低单次停顿，而是保证并发回收有足够 CPU 和堆余量，避免 Allocation Stall。

## GC 日志

JDK 9+ 推荐统一日志：

```bash
-Xlog:gc*,safepoint:file=/path/to/gc-%t.log:time,uptime,level,tags:filecount=10,filesize=100m
```

JDK 8 常见参数：

```bash
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-XX:+PrintTenuringDistribution
-XX:+PrintHeapAtGC
-XX:+PrintReferenceGC
-XX:+PrintGCApplicationStoppedTime
-Xloggc:/path/to/gc-%t.log
-XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFiles=10
-XX:GCLogFileSize=100M
```

## OOM 处理

| 参数 | 说明 |
| --- | --- |
| `-XX:+HeapDumpOnOutOfMemoryError` | OOM 时生成堆转储 |
| `-XX:HeapDumpPath=<path>` | 堆转储路径 |
| `-XX:OnOutOfMemoryError="<cmd>"` | OOM 时执行命令 |
| `-XX:+ExitOnOutOfMemoryError` | OOM 后退出进程 |
| `-XX:+CrashOnOutOfMemoryError` | OOM 后崩溃并生成错误日志 |
| `-XX:+UseGCOverheadLimit` | 启用 GC 开销限制 |

## 诊断参数

| 参数 | 说明 |
| --- | --- |
| `-XX:+PrintCommandLineFlags` | 打印最终 JVM 参数 |
| `-XX:+UnlockDiagnosticVMOptions` | 解锁诊断参数 |
| `-XX:NativeMemoryTracking=summary` | 开启 NMT 概览，需配合 `jcmd VM.native_memory` |
| `-XX:StartFlightRecording=...` | 启动时开启 JFR |
| `-XX:+FlightRecorder` | JDK 8 中启用 JFR，具体可用性取决于发行版和版本 |
| `-XX:+UnlockCommercialFeatures` | 旧 Oracle JDK 8 启用商业特性，现代 JDK 不再需要 |

## 参数使用原则

- 先确认瓶颈是 CPU、堆、元空间、直接内存、锁、线程还是 IO。
- 每次只修改少量参数，并保留修改前后的日志和指标。
- 容器部署要同时考虑 JVM 堆、元空间、直接内存、线程栈、代码缓存和 native 内存。
- 停顿目标、吞吐目标和内存成本互相制约，不能只优化单一指标。
- 面试回答调优题时，先说观测和证据，再说参数；直接报一组固定参数通常是不合格答案。

# 虚拟机字节码执行引擎

字节码执行引擎负责执行 Class 文件中的指令。HotSpot 同时具备解释执行和即时编译执行能力，解释器用于快速启动，JIT 用于优化热点代码。

## 栈帧结构

每次方法调用都会创建栈帧，栈帧随方法调用入栈，随返回或异常退出出栈。

### 局部变量表

- 存放方法参数和局部变量。
- 以 Slot 为单位，`long` 和 `double` 占两个 Slot，其余类型占一个 Slot。
- 实例方法的第 0 个 Slot 是 `this`。

### 操作数栈

- 字节码指令计算的工作区。
- 大多数指令从操作数栈弹出操作数，再把结果压回栈。
- 最大深度由编译器写入 `Code` 属性中的 `max_stack`。

### 动态连接

- 每个栈帧持有指向运行时常量池中当前方法引用的连接。
- 方法调用、字段访问等指令通过动态连接支持符号引用解析和运行期绑定。

### 方法返回地址

- 正常返回时记录调用者程序计数器位置。
- 异常退出时通过异常表和调用链传播，不一定使用正常返回地址。

## 方法调用指令

- `invokestatic`：调用静态方法。
- `invokespecial`：调用构造方法、私有方法和父类方法。
- `invokevirtual`：调用普通虚方法，运行期根据接收者实际类型分派。
- `invokeinterface`：调用接口方法，运行期搜索实现类。
- `invokedynamic`：动态调用点，由引导方法在运行期决定目标方法。

### 调用指令边界

- `invokestatic` 和 `invokespecial` 目标通常可在解析阶段静态确定。
- `invokevirtual` 需要根据接收者实际类型做动态分派，但 `final`、`private`、构造方法和静态方法不存在普通虚分派。
- `invokeinterface` 需要按接口方法解析并在运行期定位实现类方法，早期成本更高，现代 JVM 会通过内联缓存和 JIT 优化降低开销。
- `invokedynamic` 的调用目标不是由 JVM 固定规则决定，而是由 Bootstrap Method 链接出 `CallSite`。

## 解析与分派

- 解析调用在类加载解析阶段或首次调用时确定目标，例如静态方法、私有方法、构造方法、父类方法。
- 静态分派发生在编译期，典型场景是方法重载，依据静态类型选择方法版本。
- 动态分派发生在运行期，典型场景是方法重写，依据对象实际类型选择方法版本。
- Java 是静态多分派、动态单分派语言：重载选择考虑多个参数静态类型，重写选择只根据接收者实际类型。

### 重载与重写

- 重载是编译期行为，选择依据是变量静态类型、参数数量、装箱拆箱、可变参数和最具体类型规则。
- 重写是运行期行为，选择依据是接收者实际类型。
- 字段不参与多态，字段访问在编译期确定；静态方法也不参与运行期重写，只是隐藏。

## 虚方法表

- JVM 可为类维护虚方法表 vtable，为接口维护接口方法表 itable 或类似结构，加速动态分派。
- 子类重写父类方法时，虚方法表中对应槽位指向子类实现。
- JIT 可基于类型 Profile 将虚调用去虚化，甚至内联成直接调用。

## 动态语言支持

- `java.lang.invoke.MethodHandle` 是可直接执行的、类型化的方法句柄，权限和调用链路与反射不同。
- `invokedynamic` 把动态调用点链接逻辑交给引导方法，lambda、字符串拼接和动态语言运行时都可使用。
- 引导方法返回 `CallSite`，其中包含实际调用目标。

### Lambda 与 invokedynamic

- Java 8 lambda 通常不会直接编译成匿名内部类，而是生成 `invokedynamic` 调用点。
- `LambdaMetafactory` 作为引导方法创建函数式接口实例。
- 具体实现可能使用隐藏类、动态生成类或缓存策略，取决于 JDK 版本和 lambda 形态。

## 异常处理

- 字节码中的异常表记录 try-catch 覆盖范围、处理器入口和捕获类型。
- 抛出异常后 JVM 在当前方法异常表中查找匹配处理器，找不到则弹出栈帧向调用者传播。
- `finally` 通常通过编译器复制代码或异常路径插入实现，不是单独的 JVM 指令。

## 同步指令

- 同步代码块编译为 `monitorenter` 和 `monitorexit`，异常路径也必须保证退出监视器。
- 同步方法通过方法访问标志 `ACC_SYNCHRONIZED` 表示，进入方法时隐式获取监视器。
- 实例同步方法锁的是 `this`，静态同步方法锁的是对应 `Class` 对象。

## 解释执行与编译执行

- 解释执行逐条解释字节码，启动快、无需编译成本。
- JIT 将热点方法或热点循环编译成本地机器码，执行快但需要编译成本。
- 分层编译结合 C1 和 C2，先快速生成较低优化代码，再对热点代码做更激进优化。

## 排查用途

- 方法调用行为异常时，用 `javap -c` 查看具体调用指令。
- lambda、反射、MethodHandle、动态代理问题可结合 `invokedynamic` 和类加载信息分析。
- 性能问题需要结合解释执行、JIT 编译、去优化和内联情况判断。

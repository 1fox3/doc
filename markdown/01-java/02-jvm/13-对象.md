# 对象

对象相关机制包括创建流程、内存分配、对象头、实例数据、对齐填充和引用访问方式。它连接类加载、堆内存、GC、锁优化和 JIT 逃逸分析。

## 创建流程

1. 类加载检查：执行 `new` 指令时，JVM 先在常量池中定位类的符号引用，并检查类是否已加载、解析和初始化。
2. 分配内存：根据堆是否规整选择指针碰撞或空闲列表。
3. 初始化零值：把对象实例字段初始化为数据类型默认值，不包含对象头。
4. 设置对象头：写入 Mark Word、类型指针、数组长度等元数据。
5. 执行 `<init>`：执行构造方法和实例初始化代码。

### new 指令之后

- `new` 指令只负责分配对象并完成零值初始化和对象头设置。
- 构造方法调用由随后的 `dup`、`invokespecial <init>` 等字节码完成。
- 如果构造方法中 `this` 逸出，其他线程可能观察到尚未完全构造完成的对象。

## 内存分配方式

- 指针碰撞：堆内存规整时，只需移动分配指针，速度快，常见于带压缩整理的收集器。
- 空闲列表：堆存在碎片时，JVM 维护可用内存块列表，分配时查找合适空间。
- TLAB：每个线程在 Eden 中预分配一小块私有分配缓冲区，减少分配时的同步竞争。
- CAS + 失败重试：TLAB 不够或慢路径分配时，通过 CAS 保证并发分配正确性。

## 对象内存布局

### 对象头

- Mark Word：保存哈希码、GC 年龄、锁标志、偏向锁信息或锁记录指针等。
- 类型指针：指向类元数据，用于确定对象类型。
- 数组长度：只有数组对象包含。

### Mark Word 内容

- 无锁状态可存储 identity hash code、GC 分代年龄和锁标志。
- 轻量级锁状态可存储指向栈中锁记录的指针。
- 重量级锁状态可存储指向 ObjectMonitor 的指针。
- 偏向锁相关位在旧版本 HotSpot 中使用，现代 JDK 中默认禁用或逐步废弃。

### 实例数据

- 保存对象字段内容。
- 字段布局会受字段类型、继承关系、压缩指针、对象对齐和 JVM 布局策略影响。
- 父类字段通常排在子类字段之前。

### 对齐填充

- HotSpot 对象大小通常按 8 字节对齐。
- 如果对象头和实例数据不足对齐边界，会插入填充字节。

## 压缩指针

- 压缩普通对象指针由 `-XX:+UseCompressedOops` 控制，压缩类指针由 `-XX:+UseCompressedClassPointers` 控制。
- 在合适堆大小下，压缩指针可把 64 位引用压缩为 32 位偏移，降低对象引用字段和对象头成本。
- 堆过大可能无法使用压缩指针，引用变宽会增加内存占用并影响缓存命中。
- 实际状态可通过 `jcmd <pid> VM.flags`、`jcmd <pid> VM.info` 或启动日志确认。

## 对象大小估算

- 普通对象大小约等于对象头、实例字段和对齐填充之和。
- 64 位 HotSpot 开启压缩指针时，普通对象头常见为 12 字节，数组对象头常见为 16 字节，再按 8 字节对齐。
- 关闭压缩指针后，对象头和引用字段会变大，极小对象和引用密集对象内存成本明显上升。
- 精确布局建议使用 JOL Java Object Layout 工具验证，不同 JDK、字段顺序和参数都会影响结果。

## 对象访问方式

### 句柄访问

- reference 指向句柄池中的句柄。
- 句柄保存对象实例数据指针和类型数据指针。
- 对象移动时只需修改句柄中的实例数据指针，reference 不变。

### 直接指针访问

- reference 直接指向对象地址。
- 访问速度更快，少一次间接定位。
- HotSpot 主要采用直接指针访问。

## 对象头与锁

- 无锁状态下 Mark Word 可保存 identity hash code、GC 年龄等。
- 轻量级锁通过 CAS 把 Mark Word 替换为指向栈上锁记录的指针。
- 重量级锁会把对象头指向 Monitor。
- 一旦计算过 identity hash code，可能影响某些锁优化状态。

## 逃逸分析影响

- 方法内未逃逸的对象可能被标量替换，减少实际堆分配。
- 不逃出线程的对象同步可能被消除。
- 逃逸分析结果依赖 JIT 和运行时 Profile，不能作为语义保证。

## 常见问题

- 对象过大：检查数组、集合容量、缓存策略和序列化字段。
- 小对象过多：关注分配速率、Young GC 频率和对象复用是否合理。
- 对象头开销明显：大量极小对象可考虑压缩结构、基本类型数组或专用集合。
- 伪共享：多个线程频繁写入相邻字段时可能发生缓存行竞争，可用填充或 `@Contended` 处理。

## 伪共享

- CPU 缓存以缓存行为单位同步，常见缓存行大小为 64 字节。
- 多个线程修改同一缓存行中的不同变量，也会导致缓存行在核心之间频繁失效。
- `@Contended` 可用于填充隔离热点字段，但需要 `-XX:-RestrictContended` 才能对非 JDK 内部类生效。

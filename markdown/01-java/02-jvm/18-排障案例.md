# 排障案例

面试中的 JVM 排障题重点不是背命令，而是展示定位路径、证据链和取舍。

## CPU 飙高

### 现象

- 服务 CPU 接近 100%，接口延迟升高。
- GC 日志可能正常，也可能 Young GC 明显变多。

### 定位

1. `top -Hp <pid>` 找到高 CPU 线程。
2. 将线程 ID 转成十六进制，对应 `jstack` 中的 `nid`。
3. `jstack -l <pid>` 连续采集多次，确认热点栈是否稳定。
4. 如果栈在业务方法，检查死循环、复杂正则、大 JSON、加解密、排序或序列化。
5. 如果栈在 GC 线程，回到 GC 日志和分配热点分析。
6. 如果栈在锁或 Unsafe，自旋竞争可能是主因，结合 JFR 锁事件确认。

### 结论表达

- 说明哪个线程、哪个方法、持续多久占用 CPU。
- 说明是业务计算、GC、锁自旋还是 JIT 编译线程。
- 给出修复：优化算法、限流、减少分配、调整线程池或修复循环条件。

## Full GC 频繁

### 现象

- GC 日志出现频繁 Full GC。
- Full GC 后 Old 区下降很少，或者下降后很快涨回去。

### 定位

1. 看触发原因：allocation failure、metadata threshold、system gc、promotion failed、to-space exhausted。
2. 用 `jstat -gcutil <pid> 1s` 看 Old、Metaspace、YGC、FGC 趋势。
3. 如果 Old 持续高，导出 heap dump 分析 Retained Heap。
4. 如果 Metaspace 高，查 ClassLoader 和动态类生成。
5. 如果 Humongous 多，查大数组、大字符串、大 buffer 和批量查询。
6. 如果显式 `System.gc()`，用 JFR、代码搜索或 Arthas trace 找调用来源。

### 修复方向

- 泄漏：修复引用链，清理缓存、队列、ThreadLocal、监听器。
- 容量不足：调整堆大小或扩容实例，但要有对象存活证据。
- G1：检查 Mixed GC、IHOP、Humongous、G1ReservePercent。
- 元空间：修复 ClassLoader 泄漏或限制动态类生成。

## 内存持续上涨但堆不高

### 现象

- 监控显示容器 RSS 持续上涨。
- JVM heap used 不高，GC 正常，最后可能 OOMKill。

### 定位

1. 确认是否容器 OOMKill，而不是 Java OOM。
2. 查直接内存、线程数、元空间、代码缓存和 NMT。
3. Netty 服务重点看 direct buffer allocator 指标和引用计数泄漏。
4. 线程数上涨时查线程 dump，确认线程池是否无限创建。
5. 开启 NMT 后用 baseline 和 summary.diff 看 native 分类增长。

### 修复方向

- 限制 `MaxDirectMemorySize` 并修复 buffer release。
- 控制线程池上限，降低 `-Xss` 需评估递归和栈深度。
- 增加容器 limit 或降低 `-Xmx` 给 native 内存留空间。

## 线程池耗尽

### 现象

- 请求堆积，响应超时。
- 线程 dump 中大量线程处于 `WAITING`、`TIMED_WAITING` 或阻塞 IO。

### 定位

1. 看线程池 active、queue size、completed task、reject count。
2. `jstack` 查线程都阻塞在哪里，是数据库、RPC、Redis、文件 IO 还是锁。
3. JFR 查看 Socket Read/Write、Thread Park、Java Monitor Blocked。
4. 检查上游超时是否大于下游超时，是否缺少熔断和限流。

### 修复方向

- 设置合理超时、熔断、隔离和队列长度。
- 分离 CPU 密集和 IO 密集线程池。
- 不用单纯增大线程数掩盖下游慢，否则会增加内存和调度成本。

## 类加载器泄漏

### 现象

- Metaspace 持续上涨。
- 应用热部署或插件卸载后类数量不下降。

### 定位

1. `jcmd <pid> VM.classloader_stats` 查看 ClassLoader 数量。
2. heap dump 中查业务 ClassLoader 到 GC Roots 的路径。
3. 重点检查 ThreadLocal、线程上下文 ClassLoader、静态集合、MBean、日志框架、定时任务和回调注册。
4. 检查动态代理、CGLIB、ByteBuddy、脚本引擎是否无限生成类。

### 修复方向

- 卸载时注销回调、MBean、线程、定时任务和 SPI 注册。
- 清理 ThreadLocal，恢复线程上下文 ClassLoader。
- 缓存动态生成类，避免按请求生成新类。

## 接口延迟尖刺但 GC 不长

### 现象

- P99 偶发尖刺。
- GC pause 不长，CPU 也未必很高。

### 定位

1. 看 safepoint 日志，确认是否存在进入安全点耗时过长。
2. JFR 查看 Thread Park、Monitor Blocked、Socket Read、File IO、Class Loading、Compilation。
3. 检查容器 CPU throttling，低 CPU quota 会放大所有停顿。
4. 检查 JIT 编译和去优化事件，发布初期或流量形态变化时更明显。

### 修复方向

- 降低锁竞争和阻塞 IO。
- 调整 CPU limit 或 `ActiveProcessorCount`。
- 预热热点路径，减少启动初期编译抖动。
- 对关键链路设置超时、隔离和降级。

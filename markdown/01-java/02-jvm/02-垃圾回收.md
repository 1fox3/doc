# 垃圾回收

垃圾回收负责自动识别不可达对象并回收内存。调优目标不是“减少 GC 次数”，而是在吞吐、延迟、内存成本和稳定性之间取得平衡。

## 对象存活判定

### 引用计数

- 对象被引用时计数加一，引用失效时计数减一。
- 实现简单，但无法处理循环引用，HotSpot 主流 GC 不使用它作为主要判定方式。

### 可达性分析

- 从 GC Roots 出发沿引用链搜索，可达对象判定为存活，不可达对象可被回收。
- 常见 GC Roots 包括虚拟机栈中的引用、本地方法栈 JNI 引用、类静态字段、运行时常量池引用、同步锁持有对象、JVM 内部常驻对象。
- 方法区中类的回收要求实例已全部回收、Class 对象不可达、加载该类的 ClassLoader 可回收。

### GC Roots 细化

- 活跃线程栈帧中的局部变量和操作数栈引用。
- JNI 本地引用和全局引用。
- 已加载类的静态字段引用。
- 运行时常量池中的引用，例如字符串常量池中的引用。
- 被 synchronized 持有的对象监视器。
- JVM 内部引用，例如系统类加载器、基本类型 Class 对象、异常对象和 JMX/JVMTI 相关引用。

## 引用类型

- 强引用：普通对象引用，只要强引用存在就不会被回收。
- 软引用：内存不足前可能被回收，适合缓存但不适合可靠资源管理。
- 弱引用：下一次 GC 发生时会被回收，常用于规范化映射和辅助缓存。
- 虚引用：无法通过引用取得对象，配合 ReferenceQueue 跟踪对象回收事件。

## 对象死亡过程

- 可达性分析不可达后，对象不会立即必然死亡，JVM 还会判断是否需要执行 `finalize()`。
- 如果对象覆盖了 `finalize()` 且尚未执行过，可能被放入 F-Queue，由低优先级 Finalizer 线程执行。
- `finalize()` 中如果重新建立引用，对象可以自救一次，但该机制不可靠、执行时机不确定，已不推荐使用。
- 现代资源释放应使用 `try-with-resources`、显式 close、Cleaner 或框架生命周期管理。

## 回收算法

- 标记-清除：标记存活对象后清理未标记对象，速度较快但会产生碎片。
- 标记-复制：把存活对象复制到另一块空间，适合对象存活率低的新生代。
- 标记-整理：标记后移动存活对象并整理空间，适合老年代但移动成本高。
- 分代收集：基于多数对象朝生夕死、少数对象长期存活的假设，将堆划分为新生代和老年代或 Region。

### 算法取舍

- 标记-清除不移动对象，适合避免移动成本，但碎片会增加大对象分配失败风险。
- 标记-复制没有碎片，分配快，但需要额外空间，存活对象越多复制成本越高。
- 标记-整理没有碎片，但移动对象和更新引用成本高，常用于老年代或 Full GC。
- 现代收集器通常不是单一算法，而是按代、Region 和阶段组合使用。

## 并发标记关键机制

- 三色标记把对象分为白、灰、黑三类，用于描述并发可达性分析状态。
- 漏标风险来自应用线程并发修改引用关系，例如黑对象新增对白对象引用且灰对象删除对白对象引用。
- 增量更新关注新增引用，CMS 使用该思路。
- 原始快照 SATB 关注被删除引用，G1、Shenandoah 等使用该思路。
- 写屏障在引用写入时维护记忆集、卡表或并发标记需要的日志。

## 跨代引用

- 分代 GC 不能每次 Minor GC 都扫描整个老年代，否则新生代回收成本会失控。
- HotSpot 通常用卡表 Card Table 记录老年代哪些区域可能引用新生代。
- 写屏障在对象引用字段写入时把对应卡页标脏，Minor GC 只扫描脏卡对应区域。
- G1 使用 Remembered Set 记录其他 Region 指向当前 Region 的引用，降低 Region 级回收时的扫描范围。

## 安全点与安全区域

- 安全点是线程可以安全暂停并枚举根节点的位置，通常在方法调用、循环跳转、异常跳转等指令附近。
- 安全区域用于处理线程处于阻塞或 sleep 状态时无法主动到达安全点的场景。
- STW 时间受安全点到达时间、根扫描、对象移动、引用处理和系统负载影响。

### OopMap

- JVM 需要知道栈和寄存器中哪些位置是对象引用，才能准确枚举 GC Roots。
- HotSpot 在 JIT 编译时为安全点生成 OopMap，记录引用位置。
- 不是每条指令都能停下来做 GC，安全点选择要平衡停顿及时性和 OopMap 存储成本。

## 经典收集器

| 收集器 | 主要区域 | 特点 | 适用场景 |
| --- | --- | --- | --- |
| Serial | 新生代 | 单线程、复制算法、STW | 小堆、客户端或简单工具进程 |
| ParNew | 新生代 | Serial 的多线程版本 | 旧版本配合 CMS |
| Parallel Scavenge | 新生代 | 关注吞吐量，可自适应调节 | 后台批处理、吞吐优先服务 |
| Serial Old | 老年代 | 单线程、标记整理 | 小堆或 CMS 失败兜底 |
| Parallel Old | 老年代 | 多线程、标记整理 | 吞吐优先 |
| CMS | 老年代 | 并发标记清除、低停顿、会产生碎片 | JDK 8 旧系统，JDK 14 已移除 |
| G1 | 整堆 | Region、可预测停顿、复制整理 | JDK 9+ 默认，通用服务端场景 |
| Shenandoah | 整堆 | 并发移动对象、低停顿 | 大堆低延迟 |
| ZGC | 整堆 | 着色指针、读屏障、低停顿 | 大堆低延迟，JDK 17/21 常用 |

## CMS 流程

- 初始标记：STW，标记 GC Roots 直接可达对象。
- 并发标记：应用线程和 GC 线程并发执行。
- 重新标记：STW，修正并发期间变动的引用关系。
- 并发清除：清理不可达对象，不移动对象。
- 主要问题：CPU 敏感、浮动垃圾、空间碎片、并发失败后退化为 Full GC。

## G1 流程

- G1 将堆切分为多个 Region，以 Region 为单位进行回收。
- Young GC 回收 Eden 和 Survivor Region。
- Mixed GC 在并发标记后选择收益高的老年代 Region 与年轻代一起回收。
- Humongous 对象占用连续 Region，分配和回收不当会造成空间压力。
- `-XX:MaxGCPauseMillis` 是停顿目标，不是硬性保证。

### G1 关键概念

- CSet Collection Set 是一次 GC 选择回收的 Region 集合。
- RSet Remembered Set 记录其他 Region 对当前 Region 的引用，用于避免全堆扫描。
- Young GC 只回收年轻代 Region，Mixed GC 会同时回收年轻代和部分收益高的老年代 Region。
- G1 通过回收收益模型选择 Region，目标是在停顿预算内尽量回收更多垃圾。

## CMS 与 G1 对比

| 维度 | CMS | G1 |
| --- | --- | --- |
| 回收范围 | 主要回收老年代 | 整堆 Region 化管理 |
| 整理能力 | 标记清除，容易碎片 | 复制整理，减少碎片 |
| 停顿目标 | 低停顿但不可预测 | 可设置停顿目标，仍非硬保证 |
| 并发失败 | concurrent mode failure 后退化 Full GC | evacuation failure、to-space exhausted 等 |
| 适用状态 | JDK 14 已移除 | JDK 9+ 默认通用收集器 |

## ZGC 要点

- ZGC 以低停顿为目标，把大部分标记、转移和重定位工作放到并发阶段执行。
- 着色指针和读屏障用于在对象移动时维持引用访问正确性。
- 停顿时间通常不随堆大小线性增长，但需要足够 CPU 和堆余量让并发回收追上分配速度。
- 需要关注 Allocation Stall、并发周期耗时、Live Set、分配速率和 CPU throttling。
- JDK 21 引入分代 ZGC，能更好处理大量短生命周期对象。

## Shenandoah 要点

- Shenandoah 通过并发标记、并发疏散和读写屏障降低停顿。
- 与 ZGC 类似，停顿目标是尽量与堆大小解耦，但会消耗额外 CPU。
- 是否可用取决于 JDK 发行版，部分 Oracle JDK 版本不包含 Shenandoah。

## 常见问题

- 频繁 Young GC：对象分配速率过高、Eden 太小、TLAB 浪费、短生命周期对象过多。
- 频繁 Full GC：老年代晋升失败、元空间膨胀、大对象或 Humongous 对象、内存泄漏、显式 `System.gc()`。
- GC Overhead Limit Exceeded：大量时间用于 GC 但回收很少，通常是堆过小或对象泄漏。
- 晋升失败：Survivor 或老年代无法容纳 Minor GC 后的存活对象。
- G1 evacuation failure：转移存活对象时目标空间不足，常见于老年代压力大、Humongous 对象多或堆余量不足。
- ZGC Allocation Stall：应用分配速度超过并发回收速度，通常需要增大堆、降低分配速率或增加 CPU 余量。

## 排查顺序

- 开启统一 GC 日志，JDK 9+ 使用 `-Xlog:gc*,safepoint:file=gc.log:time,uptime,level,tags:filecount=10,filesize=100m`。
- 用 `jstat -gcutil <pid> 1s` 观察 Eden、Survivor、Old、Metaspace 和 GC 次数变化。
- 用堆转储分析大对象、重复对象和引用链。
- 用 JFR 分析对象分配热点、GC Pause、Safepoint、线程阻塞和锁竞争。
- 将 GC 日志和请求延迟、发布、定时任务、容器 CPU throttling 放在同一时间线，避免只看单次 GC。

# 垃圾回收

垃圾回收负责自动识别不可达对象并回收内存。调优目标不是“减少 GC 次数”，而是在吞吐、延迟、内存成本和稳定性之间取得平衡。

## 对象存活判定

### 引用计数

- 对象被引用时计数加一，引用失效时计数减一。
- 实现简单，但无法处理循环引用，HotSpot 主流 GC 不使用它作为主要判定方式。

### 可达性分析

- 从 GC Roots 出发沿引用链搜索，可达对象判定为存活，不可达对象可被回收。
- 常见 GC Roots 包括虚拟机栈中的引用、本地方法栈 JNI 引用、类静态字段、运行时常量池引用、同步锁持有对象、JVM 内部常驻对象。
- 方法区中类的回收要求实例已全部回收、Class 对象不可达、加载该类的 ClassLoader 可回收。

## 引用类型

- 强引用：普通对象引用，只要强引用存在就不会被回收。
- 软引用：内存不足前可能被回收，适合缓存但不适合可靠资源管理。
- 弱引用：下一次 GC 发生时会被回收，常用于规范化映射和辅助缓存。
- 虚引用：无法通过引用取得对象，配合 ReferenceQueue 跟踪对象回收事件。

## 回收算法

- 标记-清除：标记存活对象后清理未标记对象，速度较快但会产生碎片。
- 标记-复制：把存活对象复制到另一块空间，适合对象存活率低的新生代。
- 标记-整理：标记后移动存活对象并整理空间，适合老年代但移动成本高。
- 分代收集：基于多数对象朝生夕死、少数对象长期存活的假设，将堆划分为新生代和老年代或 Region。

## 并发标记关键机制

- 三色标记把对象分为白、灰、黑三类，用于描述并发可达性分析状态。
- 漏标风险来自应用线程并发修改引用关系，例如黑对象新增对白对象引用且灰对象删除对白对象引用。
- 增量更新关注新增引用，CMS 使用该思路。
- 原始快照 SATB 关注被删除引用，G1、Shenandoah 等使用该思路。
- 写屏障在引用写入时维护记忆集、卡表或并发标记需要的日志。

## 安全点与安全区域

- 安全点是线程可以安全暂停并枚举根节点的位置，通常在方法调用、循环跳转、异常跳转等指令附近。
- 安全区域用于处理线程处于阻塞或 sleep 状态时无法主动到达安全点的场景。
- STW 时间受安全点到达时间、根扫描、对象移动、引用处理和系统负载影响。

## 经典收集器

| 收集器 | 主要区域 | 特点 | 适用场景 |
| --- | --- | --- | --- |
| Serial | 新生代 | 单线程、复制算法、STW | 小堆、客户端或简单工具进程 |
| ParNew | 新生代 | Serial 的多线程版本 | 旧版本配合 CMS |
| Parallel Scavenge | 新生代 | 关注吞吐量，可自适应调节 | 后台批处理、吞吐优先服务 |
| Serial Old | 老年代 | 单线程、标记整理 | 小堆或 CMS 失败兜底 |
| Parallel Old | 老年代 | 多线程、标记整理 | 吞吐优先 |
| CMS | 老年代 | 并发标记清除、低停顿、会产生碎片 | JDK 8 旧系统，JDK 14 已移除 |
| G1 | 整堆 | Region、可预测停顿、复制整理 | JDK 9+ 默认，通用服务端场景 |
| Shenandoah | 整堆 | 并发移动对象、低停顿 | 大堆低延迟 |
| ZGC | 整堆 | 着色指针、读屏障、低停顿 | 大堆低延迟，JDK 17/21 常用 |

## CMS 流程

- 初始标记：STW，标记 GC Roots 直接可达对象。
- 并发标记：应用线程和 GC 线程并发执行。
- 重新标记：STW，修正并发期间变动的引用关系。
- 并发清除：清理不可达对象，不移动对象。
- 主要问题：CPU 敏感、浮动垃圾、空间碎片、并发失败后退化为 Full GC。

## G1 流程

- G1 将堆切分为多个 Region，以 Region 为单位进行回收。
- Young GC 回收 Eden 和 Survivor Region。
- Mixed GC 在并发标记后选择收益高的老年代 Region 与年轻代一起回收。
- Humongous 对象占用连续 Region，分配和回收不当会造成空间压力。
- `-XX:MaxGCPauseMillis` 是停顿目标，不是硬性保证。

## 常见问题

- 频繁 Young GC：对象分配速率过高、Eden 太小、TLAB 浪费、短生命周期对象过多。
- 频繁 Full GC：老年代晋升失败、元空间膨胀、大对象或 Humongous 对象、内存泄漏、显式 `System.gc()`。
- GC Overhead Limit Exceeded：大量时间用于 GC 但回收很少，通常是堆过小或对象泄漏。
- 晋升失败：Survivor 或老年代无法容纳 Minor GC 后的存活对象。

## 排查顺序

- 开启统一 GC 日志，JDK 9+ 使用 `-Xlog:gc*,safepoint:file=gc.log:time,uptime,level,tags:filecount=10,filesize=100m`。
- 用 `jstat -gcutil <pid> 1s` 观察 Eden、Survivor、Old、Metaspace 和 GC 次数变化。
- 用堆转储分析大对象、重复对象和引用链。
- 用 JFR 分析对象分配热点、GC Pause、Safepoint、线程阻塞和锁竞争。

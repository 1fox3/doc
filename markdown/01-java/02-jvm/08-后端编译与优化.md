# 后端编译与优化

后端编译指 JVM 在运行期把热点字节码编译为本地机器码，并基于运行时 Profile 做优化。HotSpot 的优化依赖方法调用次数、循环回边次数、类型分布和分支概率等信息。

## 热点探测

- 方法调用计数器统计方法被调用次数。
- 回边计数器统计循环体执行次数，用于触发栈上替换 OSR。
- 采样方式实现成本低但精度较低，计数器方式精度高但维护成本更高。

## 编译级别

- HotSpot 分层编译中常见层级包括解释执行、C1 无 Profile、C1 有基础 Profile、C1 有完整 Profile、C2 高优化代码。
- C1 关注编译速度和较低优化成本，C2 关注峰值性能。
- 不是所有方法都会进入最高优化级别，是否编译取决于热点程度、代码缓存、编译队列和 JVM 策略。

## 分层编译

- 解释器快速启动并收集 Profile。
- C1 编译器编译速度快，优化较轻，适合启动期和中等热点代码。
- C2 编译器优化更激进，编译成本更高，适合长期热点代码。
- 分层编译让应用在启动速度和峰值性能之间取得平衡。

## 去优化

- JIT 会基于类型分布、分支概率和内联假设生成投机优化代码。
- 当运行期假设失效，例如出现新的接收者类型或类被重新定义，JVM 会去优化并回退到解释执行。
- 去优化本身是正常机制，但频繁发生可能导致延迟抖动和编译开销上升。

## 编译对象

- 被多次调用的方法会触发方法级编译。
- 被多次执行的循环体会触发 OSR，把正在解释执行的栈帧切换到编译后代码。
- 过早或过度编译会消耗 CPU，因此阈值和代码缓存大小会影响性能。

## 优化技术

### 方法内联

- 将被调用方法体嵌入调用方，减少调用开销并暴露更多优化机会。
- 虚方法可结合类型 Profile 做去虚化和投机内联。
- 如果运行期类型假设失效，JVM 会触发去优化回退到解释执行或重新编译。

#### 内联收益

- 减少方法调用开销。
- 暴露跨方法常量传播、逃逸分析、锁消除、边界检查消除机会。
- 过度内联会导致机器码膨胀，占用 Code Cache，并可能降低指令缓存命中。

### 逃逸分析

- 判断对象是否逃出方法或线程作用域。
- 未逃逸对象可能进行标量替换，把对象字段拆成局部变量。
- 不逃出线程的对象同步可能被消除。
- 栈上分配更多是优化结果表达，HotSpot 常见收益来自标量替换而不是显式把对象放到 Java 栈上。

#### 逃逸级别

- 不逃逸：对象只在方法内部使用，可能标量替换或消除分配。
- 方法逃逸：对象被返回或传给外部方法，但不一定被其他线程访问。
- 线程逃逸：对象可被其他线程访问，同步和内存可见性不能随意消除。

## 常见 JIT 陷阱

- 微基准没有预热，测到的是解释执行或 C1 编译阶段。
- 结果未被消费，被 JIT 当作死代码消除。
- 输入是常量，计算被常量折叠。
- 分支数据分布和真实流量不同，导致线上去优化或预测失败。
- 使用 `System.nanoTime()` 手写基准容易被编译器、OS 调度和 GC 干扰，应使用 JMH。

### 公共子表达式消除

- 对相同输入产生相同结果的重复表达式只计算一次。

### 边界检查消除

- 对数组访问范围可证明安全时，消除重复边界检查。
- 常见于规范循环和数组遍历。

### 空值检查消除

- 如果对象引用已经被证明非空，可消除后续空指针检查。

### 锁优化

- 同步消除、锁粗化、自适应自旋等可能在 JIT 阶段发生。

## 代码缓存

- 编译后的机器码存放在 Code Cache。
- Code Cache 满会影响 JIT 编译，导致性能下降。
- 可通过 `jcmd <pid> Compiler.codecache` 查看代码缓存使用情况。
- 代码缓存压力常见于大量动态生成方法、代理类、脚本、复杂泛型桥接和长期运行的大型应用。

## 观察工具

- `jcmd <pid> Compiler.codecache`：查看代码缓存。
- `jcmd <pid> Compiler.queue`：查看编译队列。
- JFR：查看 Compilation、Code Cache、Method Profiling 事件。
- `-XX:+PrintCompilation`：打印编译日志，适合测试环境。
- JITWatch：分析 JIT 日志、内联决策和优化结果。
- `-XX:+PrintInlining`：观察内联决策，通常配合诊断参数在测试环境使用。

## 风险

- 微基准必须使用 JMH，避免被死代码消除、常量折叠、预热不足和逃逸分析误导。
- JIT 优化依赖运行时 Profile，同一段代码在不同流量分布下可能表现不同。
- 过度关注单个参数不如先用 JFR 和编译日志确认瓶颈。

# 线程安全与锁优化

线程安全问题来自多个线程并发访问共享可变状态。JVM 和 JDK 提供锁、volatile、CAS、线程本地存储、不可变对象和 JIT 锁优化等机制。

## 线程安全分类

- 不可变：对象状态创建后不再改变，天然线程安全，例如 `String`、部分值对象。
- 绝对线程安全：调用者无需额外同步也能在任意并发场景保持正确，实际很少。
- 相对线程安全：单个操作线程安全，组合操作仍需额外同步，例如很多同步容器。
- 线程兼容：对象本身不线程安全，但调用方可通过同步保证安全。
- 线程对立：即使外部同步也难以安全并发使用。

## 互斥同步

### synchronized

- 基于对象监视器 Monitor。
- 进入同步块对应 `monitorenter`，退出对应 `monitorexit`。
- 支持可重入，同一线程可重复获得同一把锁。
- 释放锁会把共享变量刷新到主内存，获得锁会使工作内存中相关共享变量失效，从而建立 happens-before。

### ReentrantLock

- 基于 AQS，实现可重入独占锁。
- 支持公平锁、非公平锁、可中断获取、限时获取和多个 Condition 队列。
- 需要在 `finally` 中显式 `unlock()`。

## 非阻塞同步

- CAS 比较内存位置当前值与期望值，相等则更新。
- Java 中常见入口包括 `Atomic*`、`VarHandle`、`Unsafe` 和 AQS 内部实现。
- CAS 常见问题包括 ABA、长时间自旋消耗 CPU、只能天然处理单变量原子更新。
- ABA 可通过版本号或 `AtomicStampedReference` 缓解。

## 无同步方案

- 可重入代码：不依赖共享可变状态或静态可变状态。
- 线程本地存储：`ThreadLocal` 为每个线程保存独立副本。
- Actor、消息队列、不可变数据结构可通过避免共享可变状态降低同步需求。

## 对象头与锁状态

- HotSpot 对象头包含 Mark Word 和类型指针，数组对象还包含数组长度。
- Mark Word 保存哈希码、GC 年龄、锁标志位、偏向线程 ID 等状态。
- JDK 15 起偏向锁默认禁用，JDK 18 起相关实现逐步废弃，现代 JDK 主要关注轻量级锁和重量级锁。

## 锁优化

### 自旋锁与自适应自旋

- 线程不立即阻塞，而是短时间循环等待锁释放。
- 适合锁持有时间很短的场景。
- 自适应自旋会根据历史竞争情况调整自旋时间。

### 锁消除

- JIT 通过逃逸分析发现锁对象不会被其他线程访问时，消除同步。
- 常见于局部对象上的无意义同步。

### 锁粗化

- 把连续多次加锁解锁合并为一次较大范围的锁，减少同步开销。
- 适合连续操作同一对象且中间没有必要释放锁的场景。

### 轻量级锁

- 通过 CAS 尝试把对象头 Mark Word 替换为指向栈中锁记录的指针。
- 适合无竞争或低竞争场景。
- 竞争加剧时可能膨胀为重量级锁。

### 重量级锁

- 依赖操作系统互斥量，竞争线程可能阻塞和唤醒。
- 上下文切换成本高，但高竞争下避免大量 CPU 自旋。

## 死锁条件

- 互斥：资源同一时间只能被一个线程持有。
- 持有并等待：线程持有资源同时等待其他资源。
- 不可抢占：资源只能由持有者主动释放。
- 循环等待：多个线程形成资源等待环。

## 排查

- 用 `jstack -l` 查看死锁、锁 owner、等待对象地址和线程状态。
- 用 JFR 查看 Java Monitor Blocked、Thread Park 和锁等待时间。
- 用 Arthas `thread -b` 查看阻塞当前线程的线程。
- 高竞争锁应结合业务临界区大小、锁粒度、线程池大小和对象分配热点分析。

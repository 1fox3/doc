# 内存排查实战

内存问题不等同于 Java 堆泄漏。生产排查要先区分堆、元空间、直接内存、线程栈、代码缓存、JNI/native 内存和容器限制。

## 现象分类

| 现象 | 优先怀疑 | 关键证据 |
| --- | --- | --- |
| `OutOfMemoryError: Java heap space` | 堆不足或堆泄漏 | GC 后 Old 降不下来，hprof 中存在大 retained 对象 |
| `GC overhead limit exceeded` | 堆过小或存活对象过多 | GC 时间占比高但回收很少 |
| `OutOfMemoryError: Metaspace` | 类加载过多或 ClassLoader 泄漏 | loaded class 持续上涨，ClassLoader 不可回收 |
| `OutOfMemoryError: Direct buffer memory` | 直接内存不足或释放不及时 | NIO/Netty direct memory 指标上涨 |
| `unable to create native thread` | 线程数过多或进程 native 内存不足 | 线程数、`-Xss`、系统 ulimit、容器 limit |
| 容器 OOMKilled | RSS 超过 cgroup limit | 没有 Java OOM，容器事件显示 killed |
| `OutOfMemoryError: CodeCache is full` | 代码缓存不足 | JIT 编译受限，Code Cache 使用接近上限 |
| `OutOfMemoryError: Requested array size exceeds VM limit` | 数组长度超过 JVM 限制 | 单个数组过大，常见于错误容量计算 |

## 快速判断路径

1. 看进程是否还活着，确认是 Java OOM、容器 OOMKill 还是系统 kill。
2. 用 `jcmd <pid> VM.flags` 确认堆、GC、容器和诊断参数。
3. 用 `jstat -gcutil <pid> 1s` 判断堆和元空间趋势。
4. 用 `jcmd <pid> GC.class_histogram` 快速查看对象数量，必要时再 heap dump。
5. 用 `jcmd <pid> VM.native_memory summary` 查看 native 内存，前提是已开启 NMT。
6. 对照 GC 日志、JFR、业务流量、发布和定时任务时间线。

## 堆泄漏排查

- 优先使用 `jmap -dump:live,format=b,file=heap.hprof <pid>` 或 `jcmd <pid> GC.heap_dump heap.hprof`。
- `live` 会触发 Full GC，能减少垃圾对象干扰，但生产执行前要评估 STW 风险。
- MAT 中先看 Dominator Tree、Leak Suspects、Histogram，再看 Path To GC Roots。
- 重点关注缓存、静态集合、线程本地变量、队列、监听器、未关闭资源和业务上下文对象。
- 如果只看 shallow size，容易低估集合、图结构和缓存入口对象的真实影响。

### 堆泄漏常见根因

- 无界缓存或缓存 key 设计错误。
- 队列积压，消费者处理变慢或停止。
- 静态集合持有请求级对象。
- ThreadLocal 未 remove。
- 监听器、回调、Future、定时任务未注销。
- ORM Session、RPC 上下文、日志 MDC、Trace 上下文没有清理。

## 元空间排查

- 用 `jcmd <pid> VM.classloader_stats` 查看 ClassLoader 数量和类占用。
- 用 `jcmd <pid> GC.class_stats` 查看类元数据，部分 JDK 需要诊断参数支持。
- 动态代理、CGLIB、ByteBuddy、Groovy、Janino、JSP、热部署和插件系统是常见来源。
- ClassLoader 泄漏通常来自线程上下文 ClassLoader、静态字段、ThreadLocal、注册中心、MBean 或未注销回调。

## 直接内存排查

- 直接内存常见来源包括 `ByteBuffer.allocateDirect`、Netty pooled direct buffer、JNI、Unsafe 和 mmap。
- 堆中只保存 `DirectByteBuffer` 对象和 Cleaner 引用，RSS 可能远大于 Java 堆。
- Netty 应同时看 allocator 指标、泄漏检测日志、池化配置和引用计数释放路径。
- `-XX:MaxDirectMemorySize` 未显式设置时，不同 JDK 版本和启动参数下默认值需要实际确认。

## 线程内存排查

- 每个 Java 线程都需要 Java 栈、native thread 结构和调度资源。
- 线程数过多会增加内存占用、上下文切换和调度延迟。
- `unable to create native thread` 要同时检查线程泄漏、线程池上限、`-Xss`、ulimit、pid limit、容器内存和宿主机线程限制。
- 线程 dump 中大量同名前缀线程通常能反推出创建来源。

## Native Memory Tracking

启动时开启：

```bash
-XX:NativeMemoryTracking=summary
```

查看概览：

```bash
jcmd <pid> VM.native_memory summary
```

建立基线并比较：

```bash
jcmd <pid> VM.native_memory baseline
jcmd <pid> VM.native_memory summary.diff
```

NMT 会带来额外开销，生产环境通常用 `summary`，只有需要更细粒度时才考虑 `detail`。

## ThreadLocal 泄漏

- 线程池线程生命周期长，`ThreadLocalMap` 中的 value 可能长期挂在线程上。
- key 是弱引用不代表 value 会自动释放，key 被回收后 value 仍可能残留到下一次清理。
- 使用完应在 `finally` 中调用 `remove()`。
- Web 容器和插件系统中，ThreadLocal value 持有业务 ClassLoader 会导致类卸载失败。

## 内存抖动

- 内存抖动指大量短生命周期对象快速创建和回收，表现为 Young GC 频繁、CPU 被 GC 消耗、延迟抖动。
- 常见来源包括大批量 JSON 序列化、日志字符串拼接、临时集合、正则、反射、BigDecimal 和重复 buffer 创建。
- 优先从 JFR Allocation、async-profiler alloc、对象采样和业务批量大小定位分配热点。
- 优化方向是减少临时对象、复用 buffer、控制批量大小、避免无意义装箱和字符串拼接。

## 排查输出建议

- 保留 JVM 参数、GC 日志、JFR、线程栈、堆转储、NMT 输出和容器事件。
- 每份证据都记录采集时间，和流量、发布、定时任务、告警放在同一时间线。
- 先证明是哪类内存上涨，再定位具体对象或 native 子系统。

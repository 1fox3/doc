# 红黑树

> 关联：[集合总览](README.md) | [HashMap](02-HashMap.md)

红黑树是 Java `HashMap`（JDK 8+）、`TreeMap`、`TreeSet` 的底层数据结构，面试高频考点。

## 什么是红黑树

红黑树是一种**自平衡二叉搜索树**，每个节点额外存储一个颜色位（红或黑）。通过在插入/删除后对颜色和结构进行调整，保证树的高度始终在 O(log n) 范围内。

## 五大性质

1. **颜色性**：每个节点是红色或黑色。
2. **根性**：根节点是黑色。
3. **叶性**：所有叶子节点（NIL / 哨兵节点）是黑色。
4. **红色性**：红色节点的两个子节点都是黑色（不能有连续的红节点，父子不能同时为红）。
5. **黑高性**：从任意节点到其任意叶子节点的所有简单路径上，黑色节点数相同（黑高相等）。

> **NIL 节点**：红黑树中所有"空指针"都视为黑色哨兵叶子节点，简化边界情况处理。Java 实现中通常用 `null` 表示。

## 从性质推导平衡

由性质 4 和 5 可以得出：

- **最短路径**：全黑路径（每一步都是黑色节点），长度等于黑高 h。
- **最长路径**：红黑交替（每个黑节点后跟一个红节点），长度为 2h。
- 因此 **最长路径 ≤ 2 × 最短路径**，树不会严重失衡。

对于有 n 个节点的红黑树，**树高 ≤ 2log₂(n+1)**，即 O(log n)。

## 与 AVL 树对比

| 对比项 | 红黑树 | AVL 树 |
| --- | --- | --- |
| 平衡条件 | 黑高相等（宽松） | 左右子树高度差 ≤ 1（严格） |
| 树高上界 | 2log₂(n+1) | 1.44log₂(n+1) |
| 查找性能 | O(log n)，常数较大 | O(log n)，常数较小 |
| 插入旋转 | 最多 2 次 | 最多 2 次 |
| 删除旋转 | 最多 3 次 | 最多 O(log n) 次 |
| 适用场景 | 频繁插入/删除 | 频繁查找 |
| Java 中应用 | `HashMap`、`TreeMap`、`TreeSet` | 通常不直接使用 |

AVL 树的严格平衡使查找路径更短，但删除后需要向上回溯所有祖先做旋转调整；红黑树删除最多 3 次旋转即可完成，对高频写入更友好。

## 节点结构

```java
class RBNode<K, V> {
    K key;
    V value;
    RBNode<K, V> left;
    RBNode<K, V> right;
    RBNode<K, V> parent;
    boolean red; // true = 红色，false = 黑色
}
```

## 旋转操作

旋转是调整树结构的基本操作，**不改变中序遍历顺序**（因此不破坏二叉搜索树性质），只改变父子关系。

### 左旋（Left Rotate）

将节点 x 的右子节点 y 提升为子树根，x 成为 y 的左子节点。

```
    x                    y
   / \                  / \
  α   y      →         x   γ
     / \               / \
    β   γ             α   β
```

```java
void rotateLeft(RBNode<K,V> x) {
    RBNode<K,V> y = x.right;

    // y 的左子树挂到 x 的右边
    x.right = y.left;
    if (y.left != null) y.left.parent = x;

    // y 接管 x 的父节点位置
    y.parent = x.parent;
    if (x.parent == null)
        root = y;
    else if (x == x.parent.left)
        x.parent.left = y;
    else
        x.parent.right = y;

    // x 成为 y 的左子
    y.left = x;
    x.parent = y;
}
```

### 右旋（Right Rotate）

与左旋完全对称，将节点 y 的左子节点 x 提升为子树根。

```
      y                  x
     / \                / \
    x   γ     →        α   y
   / \                     / \
  α   β                   β   γ
```

## 插入

**插入策略**：新节点始终着色为**红色**。

原因：插入红色节点不改变任何路径的黑高（不违反性质 5），只可能违反性质 4（红色节点的父节点也是红色）。而插入黑色节点必定破坏黑高，修复代价更大。

### 插入修复

设插入节点为 **N**（红），父节点为 **P**，祖父节点为 **G**，叔父节点为 **U**（P 的兄弟）。

只有当 **P 是红色**时才需要修复（P 是黑色时，插入 N 不违反任何性质）。

---

#### 情形 1：叔父节点 U 是红色

```
        G(B)                 G(R)  ← G 变红，继续向上修复
       / \                  / \
     P(R) U(R)    →       P(B) U(B)
     /                    /
    N(R)                 N(R)
```

**操作**：P 和 U 变黑，G 变红。然后以 G 为新的"当前节点"向上继续检查（G 的父节点可能也是红的）。若 G 是根节点，最终将根节点置黑。

---

#### 情形 2：U 是黑色，N 是 P 的内侧子节点（折线型）

以 N 是 P 的右子、P 是 G 的左子为例（另一侧对称）：

```
      G(B)                G(B)
     / \                 / \
   P(R)  U(B)   →     N(R)  U(B)
     \                /
     N(R)           P(R)
```

**操作**：以 P 为轴**左旋**（将 N 旋转到 P 的位置），然后将 P 作为当前节点，进入情形 3 继续处理。

情形 2 本身不违反性质 5，旋转后也不违反，只是为了转化为情形 3。

---

#### 情形 3：U 是黑色，N 是 P 的外侧子节点（直线型）

N 是 P 的左子，P 是 G 的左子（另一侧对称）：

```
        G(B)                P(B)
       / \                 / \
     P(R)  U(B)   →     N(R)  G(R)
     /                           \
    N(R)                          U(B)
```

**操作**：以 G 为轴**右旋**，然后交换 P 和 G 的颜色（P 变黑，G 变红）。修复完成，无需继续向上。

---

### 插入修复总结

| 情形 | 判断条件 | 操作 | 是否需要继续 |
| --- | --- | --- | --- |
| 1 | 叔父 U 是红色 | P/U 变黑，G 变红 | 是（以 G 向上） |
| 2 | U 黑，N 是内侧子节点 | 以 P 为轴旋转 | 是（转为情形 3） |
| 3 | U 黑，N 是外侧子节点 | 以 G 为轴旋转，换色 | 否 |

插入最多旋转 2 次（情形 2 旋转 1 次，情形 3 再旋转 1 次），修复路径长度 O(log n)（情形 1 的颜色修改向上传播）。

## 删除

删除比插入更复杂。

### 删除的基本步骤

**步骤 1**：按 BST 规则找到待删节点 D。

- 若 D 有两个非空子节点：找 D 的**中序后继**（右子树最小节点）S，将 S 的 key/value 复制到 D，然后删除 S（S 最多只有一个右子节点）。
- 若 D 有一个非空子节点：用该子节点替换 D。
- 若 D 是叶子节点：直接删除。

**步骤 2**：根据被删节点的颜色决定是否需要修复。

- 删除的是**红色节点**：不影响任何路径的黑高，无需修复。
- 删除的是**黑色节点**：减少了该路径的黑高，需要修复。

当用子节点替换黑色节点时，若子节点是红色，直接将子节点变黑即可。若子节点是黑色（或是 NIL），该节点变为"双重黑"（double black），需要进入修复流程。

### 双重黑修复

设双重黑节点为 **N**，其父节点为 **P**，兄弟节点为 **S**，S 的左子为 **SL**，S 的右子为 **SR**。

以 N 是 P 的左子为例（另一侧对称）。

---

#### 情形 1：S 是红色

因为 S 是红色，P 和 S 的两个子节点都必须是黑色。

```
      P(B)                    S(B)
     / \                     / \
   N(DB) S(R)     →       P(R)  SR(B)
         / \               / \
        SL  SR           N(B)  SL
```

**操作**：以 P 为轴**左旋**，P 变红，S 变黑。N 的双重黑还没解决，继续以 N 为当前节点进入其他情形（此时 N 的兄弟变成了原 SL，是黑色，进入情形 2/3/4）。

---

#### 情形 2：S 是黑色，SL 和 SR 都是黑色

```
      P(?)                    P(DB 或 B)
     / \                     / \
   N(DB) S(B)     →        N(B)  S(R)
         / \                      / \
       SL(B) SR(B)              SL(B) SR(B)
```

**操作**：S 变红，双重黑**上移至 P**（P 吸收 N 的一层黑）。
- 若 P 原来是红色：P 变黑，修复完成。
- 若 P 原来是黑色：P 变为双重黑，继续以 P 向上处理。
- 若 P 是根节点：P 的双重黑直接丢弃（根节点多一层黑不影响性质）。

---

#### 情形 3：S 是黑色，SL 是红色，SR 是黑色（N 在左边）

```
      P(?)                     P(?)
     / \                      / \
   N(DB) S(B)     →         N(DB) SL(B)
         / \                           \
       SL(R) SR(B)                      S(R)
                                           \
                                            SR(B)
```

**操作**：以 S 为轴**右旋**，S 变红，SL 变黑。转化为情形 4。

---

#### 情形 4：S 是黑色，SR 是红色（N 在左边）

```
      P(?)                     S(?)
     / \                      / \
   N(DB) S(B)     →         P(B)  SR(B)
         / \                / \
        SL  SR(R)         N(B)  SL
```

**操作**：以 P 为轴**左旋**，S 继承 P 的颜色，P 变黑，SR 变黑。N 的双重黑解除，修复完成。

---

### 删除修复总结

| 情形 | 判断条件 | 操作 | 是否继续 |
| --- | --- | --- | --- |
| 1 | S 是红色 | 左旋 P，P 变红，S 变黑 | 是（转入 2/3/4） |
| 2 | S 黑，SL/SR 都黑 | S 变红，双重黑上移至 P | 看 P 颜色 |
| 3 | S 黑，SL 红，SR 黑 | 右旋 S，SL/S 换色 | 是（转为情形 4） |
| 4 | S 黑，SR 红 | 左旋 P，S 继承 P 色，P/SR 变黑 | 否 |

删除最多旋转 3 次（情形 1 → 3 → 4，各一次旋转）。

## 复杂度汇总

| 操作 | 时间复杂度 | 旋转次数上界 | 说明 |
| --- | --- | --- | --- |
| 查找 | O(log n) | 0 | 纯二叉搜索，无需修复 |
| 插入 | O(log n) | ≤ 2 | 颜色修改 O(log n)，旋转 O(1) |
| 删除 | O(log n) | ≤ 3 | 颜色修改 O(log n)，旋转 O(1) |
| 构建（n 个节点）| O(n log n) | — | 逐个插入 |

## Java 中的红黑树实现

### HashMap 的 TreeNode

JDK 8 的 `HashMap` 当单桶链表长度达到 8 且 table 容量 ≥ 64 时，将链表转为红黑树：

```java
// HashMap.TreeNode 继承自 LinkedHashMap.Entry（后者继承 HashMap.Node）
static final class TreeNode<K,V> extends LinkedHashMap.Entry<K,V> {
    TreeNode<K,V> parent;
    TreeNode<K,V> left;
    TreeNode<K,V> right;
    TreeNode<K,V> prev;  // 双向链表前驱，反树化时用于还原链表顺序
    boolean red;
}
```

`TreeNode` 同时维护了链表结构（`next/prev`）和树结构（`parent/left/right`），树化和反树化只需要调整指针，不需要重新分配节点。

**树化阈值 8 的依据**：在理想随机哈希下，单桶节点数服从泊松分布（λ≈0.5），到达 8 个节点的概率约为 0.000000099（不足一千万分之一）。出现这种情况基本说明 `hashCode` 实现极差。

**反树化阈值 6**（而非 8）：避免在阈值附近频繁树化/反树化（thrashing）。

### TreeMap / TreeSet

`TreeMap` 直接以红黑树为核心存储所有条目：

```java
static final class Entry<K,V> implements Map.Entry<K,V> {
    K key;
    V value;
    Entry<K,V> left;
    Entry<K,V> right;
    Entry<K,V> parent;
    boolean color = BLACK; // 默认黑色
}
```

`TreeSet` 内部持有一个 `TreeMap`，元素作为 key 存入，value 是固定占位对象。

`TreeMap` 提供的有序能力直接来自红黑树的中序遍历：

```java
TreeMap<Integer, String> map = new TreeMap<>();
map.put(5, "five");
map.put(1, "one");
map.put(3, "three");

map.firstKey();          // 1
map.lastKey();           // 5
map.floorKey(4);         // 3（≤4 的最大 key）
map.ceilingKey(4);       // 5（≥4 的最小 key）
map.subMap(1, true, 4, false); // [1, 3]（闭开区间）
```

### ConcurrentHashMap 的 TreeBin

`ConcurrentHashMap` 的树化使用 `TreeBin` 包装：

```java
static final class TreeBin<K,V> extends Node<K,V> {
    TreeNode<K,V> root;
    volatile TreeNode<K,V> first; // 链表头，用于在加锁受阻时退化为链表遍历
    volatile Thread waiter;       // 等待写锁的线程
    volatile int lockState;       // 读/写锁状态
}
```

`TreeBin` 额外加了读写锁机制：写操作（put/remove）加写锁，`find`（读）检测到写锁存在时改为遍历链表，避免读写争用。

## 常见面试问题

### 为什么新插入节点要设为红色？

- 插入红色节点只可能违反性质 4（连续红节点），不影响黑高。
- 插入黑色节点必然违反性质 5（破坏黑高），需要在所有受影响路径上做调整，代价更大。
- 选择插入红色是"从代价更小的违规开始修复"的策略。

### 红黑树和哈希表如何搭配使用？

`HashMap` 结合了哈希表和红黑树：
- 正常情况：O(1) 哈希定位，O(1) 单链节点查找。
- 极端冲突（链表长度 ≥ 8）：转为红黑树，查找从 O(n) 降至 O(log n)，防止被有意构造的 hash 攻击导致服务降级。

### 黑高是什么？黑高相等为什么能保证平衡？

黑高（Black-Height）指从某节点到其任意叶子路径上的黑色节点数（不含该节点本身）。性质 5 要求所有路径黑高相等。

- 对于黑高为 h 的子树，最少有 2^h - 1 个内部节点（全黑路径），最多有 4^h - 1 个（红黑交替路径，每层节点数翻倍）。
- 反过来，n 个节点的红黑树黑高最大为 log₂(n+1)，树高最大为 2log₂(n+1)。
- 这保证了所有操作在 O(log n) 内完成。

# 序列化

序列化：对象 → 可存储/传输格式；反序列化：数据 → 对象。Java 原生序列化依赖 `Serializable`，但性能、安全性和跨语言能力较弱。

## 常见序列化方案

| 方案 | 特点 | 适用场景 |
|------|------|----------|
| JSON | 可读性好，体积较大 | HTTP API |
| Protobuf | 体积小、跨语言、性能好 | RPC、消息队列 |
| Kryo / Hessian | Java 生态性能好 | Java RPC、缓存 |
| Java 原生 | 简单，但性能差、安全风险高 | 不推荐生产使用 |

## transient

- 只能修饰实例变量，不能修饰类或方法。
- 被修饰字段不参与序列化，反序列化后恢复为类型默认值（引用类型为 `null`，基本类型为 `0`/`false`）。
- 典型用途：密码、密钥、可重算的缓存字段等不应持久化的数据。

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String username;
    private transient String password; // 不参与序列化，反序列化后为 null
}
```

## static 变量与序列化

- `static` 变量属于类，不属于实例，序列化只保存实例状态，因此 **static 字段无论是否加 `transient` 都不会被序列化**。
- 反序列化后 `static` 字段值由当前 JVM 类状态决定，与序列化时的值无关。

## serialVersionUID

- 序列化时写入字节流，反序列化时校验本地类的 UID 是否一致，不一致抛 `InvalidClassException`。
- 不显式声明时 JVM 根据类结构自动计算，类结构变动则 UID 变，破坏兼容性。
- 显式声明后新增字段只要 UID 不变，旧数据仍可反序列化，新字段取默认值（向前兼容）。
- 破坏性变更（如字段类型修改）应主动升版本号，触发快速失败而非静默错误。

```java
// v1
public class Product implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    private double price;
}

// v2：新增字段，UID 不变 → 旧数据可正常反序列化，category 取 null
public class Product implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    private double price;
    private String category; // 反序列化旧数据时取默认值 null
}

// v3：破坏性变更，主动升版本号防止静默错误
public class Product implements Serializable {
    private static final long serialVersionUID = 2L; // 升版本，旧数据反序列化直接报错
    private String name;
    private BigDecimal price; // 类型变更，与 v1/v2 不兼容
}
```

## 注意点

- 反序列化不可信数据可能造成安全漏洞（RCE）。
- 类结构变更需设计 schema 演进策略，避免兼容性问题。

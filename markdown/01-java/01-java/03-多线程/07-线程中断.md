# 线程中断

## 概述

Java 的线程中断是一种协作式机制，不是强制终止线程的手段。调用 `interrupt()` 只是给目标线程设置一个中断标志位（boolean），线程本身负责检查该标志并决定是否响应。这种设计避免了强制终止（如已废弃的 `Thread.stop()`）带来的资源不一致问题。

阻塞方法（`sleep`、`wait`、`join`）会在检测到中断标志时抛出 `InterruptedException`，并在抛出前清除中断标志。这意味着捕获异常后如果不重新设置标志，中断信息就会丢失。

## 核心概念

### interrupt() / isInterrupted() / interrupted()

这三个方法是中断机制的核心 API，行为差异明显：

| 方法 | 调用方式 | 作用 | 是否清除标志 |
|------|----------|------|--------------|
| `thread.interrupt()` | 实例方法 | 给目标线程设置中断标志 | 不涉及 |
| `thread.isInterrupted()` | 实例方法 | 检查目标线程的中断标志 | **不清除** |
| `Thread.interrupted()` | 静态方法 | 检查**当前线程**的中断标志 | **清除** |

`Thread.interrupted()` 是静态方法，始终作用于调用它的线程（当前线程），不管通过哪个引用调用。

### join/sleep/wait 的中断响应

这三个阻塞方法对中断的响应方式相同：抛出 `InterruptedException` 并清除中断标志。

**关键点：**
- 线程在阻塞状态时，若其他线程调用 `interrupt()`，阻塞立即解除，抛出 `InterruptedException`
- 抛出异常时中断标志被**自动清除**，捕获后需要调用 `Thread.currentThread().interrupt()` 恢复标志
- 如果线程尚未阻塞但已设置中断标志，下次进入阻塞方法时会立即抛出异常

`join()` 内部利用 `wait()` 实现，因此中断行为一致。

### yield vs sleep 的区别

两者都能让当前线程暂停执行，但机制不同：

| 对比维度 | yield | sleep |
|----------|-------|-------|
| 暂停时长 | 由 CPU 时间片划分决定，不可控 | 可指定具体毫秒数 |
| 目标状态 | 进入就绪状态，可能立即被重新调度 | 进入 TIMED_WAITING 状态，时间到才变为就绪 |
| 中断响应 | **不响应中断**，无法通过 interrupt() 唤醒 | 响应中断，抛出 `InterruptedException` |
| 锁释放 | 不释放已持有的锁 | 不释放已持有的锁 |
| 保证 | 不保证其他线程能获得 CPU | 保证暂停时长（精度受 OS 调度影响） |

`yield()` 主要用于测试和调优，生产代码中很少使用，因为行为高度依赖 JVM 和操作系统实现。

### sleep vs wait 的区别

| 对比维度 | Thread.sleep() | Object.wait() |
|----------|----------------|---------------|
| 所属类 | `Thread`（静态方法） | `Object`（实例方法） |
| 使用约束 | 任何地方均可调用 | 必须在 `synchronized` 块内调用 |
| 锁释放 | **不释放**持有的锁 | **释放**当前对象的监视器锁 |
| 唤醒方式 | 时间到期自动唤醒 | 需要 `notify()` / `notifyAll()` 或超时 |
| 使用场景 | 定时暂停、轮询间隔控制 | 多线程协作通信 |
| 中断响应 | 抛出 `InterruptedException` | 抛出 `InterruptedException` |

## 示例代码

```java
// 示例1：interrupt() 与 isInterrupted() / interrupted() 的区别
public class InterruptDemo {

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            // 使用 isInterrupted()：不清除标志，适合多次检查
            while (!Thread.currentThread().isInterrupted()) {
                System.out.println("working...");
                // 模拟轻量工作，不进入阻塞
            }
            // 此时中断标志仍为 true
            System.out.println("isInterrupted: " + Thread.currentThread().isInterrupted()); // true

            // Thread.interrupted() 会清除标志
            System.out.println("interrupted(): " + Thread.interrupted()); // true
            System.out.println("interrupted(): " + Thread.interrupted()); // false（已清除）
        });

        worker.start();
        Thread.sleep(10);
        worker.interrupt();
        worker.join();
    }
}
```

```java
// 示例2：阻塞方法对中断的响应，以及正确处理方式
public class BlockingInterruptDemo {

    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            try {
                System.out.println("about to sleep...");
                Thread.sleep(10_000); // 阻塞中收到中断
            } catch (InterruptedException e) {
                // sleep 抛出 InterruptedException 时已清除中断标志
                System.out.println("isInterrupted after catch: "
                    + Thread.currentThread().isInterrupted()); // false

                // 最佳实践：重新设置中断标志，让上层调用者感知
                Thread.currentThread().interrupt();

                System.out.println("isInterrupted after re-set: "
                    + Thread.currentThread().isInterrupted()); // true
            }
        });

        t.start();
        Thread.sleep(100);
        t.interrupt(); // 中断正在 sleep 的线程
        t.join();
    }
}
```

```java
// 示例3：生产场景中正确使用中断退出循环任务
public class TaskWorker implements Runnable {

    @Override
    public void run() {
        while (!Thread.currentThread().isInterrupted()) {
            try {
                doWork();
                Thread.sleep(1000); // 心跳间隔，可被中断
            } catch (InterruptedException e) {
                // 捕获后必须恢复标志，否则 while 条件无法感知中断
                Thread.currentThread().interrupt();
                // 此处也可以直接 break，取决于是否有后续清理逻辑
            }
        }
        cleanup(); // 正常退出前清理资源
        System.out.println("worker exited cleanly");
    }

    private void doWork() {
        System.out.println("processing...");
    }

    private void cleanup() {
        System.out.println("cleaning up...");
    }

    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(new TaskWorker());
        t.start();
        Thread.sleep(3500);
        t.interrupt(); // 停止任务
        t.join();
    }
}
```

```java
// 示例4：sleep 不释放锁 vs wait 释放锁的对比
public class LockReleaseDemo {

    private static final Object lock = new Object();

    public static void main(String[] args) throws InterruptedException {
        // 线程A：持有锁后 sleep —— 不释放锁，线程B无法进入
        Thread a = new Thread(() -> {
            synchronized (lock) {
                System.out.println("A acquired lock, sleeping...");
                try {
                    Thread.sleep(3000); // 不释放 lock
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
                System.out.println("A done sleeping");
            }
        }, "A");

        // 线程B：等待锁
        Thread b = new Thread(() -> {
            System.out.println("B waiting for lock...");
            synchronized (lock) {
                System.out.println("B acquired lock");
            }
        }, "B");

        a.start();
        Thread.sleep(100);
        b.start(); // B 会被阻塞直到 A 释放锁（sleep 结束后）

        // 若将 sleep 换成 lock.wait()，A 会释放锁，B 立即可进入
    }
}
```

## 常见问题

**1. 捕获 InterruptedException 后吞掉异常（最常见的坑）**

```java
// 错误写法：中断信号丢失，外层循环永远感知不到中断
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    // 什么都不做，或只打日志
}

// 正确写法：恢复中断标志
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
    // 或者直接向上抛出
}
```

**2. 混淆 Thread.interrupted()（静态）和 thread.isInterrupted()（实例）**

`Thread.interrupted()` 无论通过哪个引用调用，始终检查并清除**当前线程**的标志。在子线程中调用 `someOtherThread.interrupted()` 实际清除的是当前线程自己的标志，而非 `someOtherThread` 的标志。

**3. wait() 的虚假唤醒（spurious wakeup）**

`wait()` 可能在没有 `notify()` 的情况下自行返回（JVM 规范允许），因此必须在循环中检查条件：

```java
// 错误
synchronized (lock) {
    lock.wait(); // 可能虚假唤醒
    doWork();
}

// 正确
synchronized (lock) {
    while (!conditionMet()) {
        lock.wait();
    }
    doWork();
}
```

**4. yield() 不可用于线程同步或协作**

`yield()` 只是"建议"调度器切换线程，不保证其他线程实际运行。用 `yield()` 来控制线程执行顺序的代码在不同 JVM 或 OS 上行为不可预测，不能替代 `wait/notify` 或 `CountDownLatch` 等同步工具。

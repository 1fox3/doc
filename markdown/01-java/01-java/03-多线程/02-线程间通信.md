# 线程间通信

## 概述

线程间通信是多线程编程的核心问题：多个线程如何协调执行顺序、共享数据、传递信号。Java 提供了从低级到高级的一整套机制，底层是 `Object.wait/notify` 和 `LockSupport.park/unpark`，中层是 `ReentrantLock + Condition`，高层是 `CountDownLatch`、`CyclicBarrier`、`Semaphore`、`BlockingQueue`、`Exchanger` 等并发工具类。

选择哪种机制取决于场景：等待某个条件成立用 Condition 或 BlockingQueue，等待多个任务完成用 CountDownLatch，多个线程到达同一屏障点用 CyclicBarrier，控制并发访问数量用 Semaphore，两线程互换数据用 Exchanger。

## 核心概念

### synchronized + wait / notify / notifyAll

`wait()` 必须在 synchronized 块内调用，调用后线程释放监视器锁并进入等待队列（WAITING 状态）。`notify()` 随机唤醒等待队列中的一个线程，`notifyAll()` 唤醒全部线程，被唤醒的线程重新竞争锁后继续执行。

关键点：
- `wait()` 必须在循环中检查条件（`while (!condition)`），防止虚假唤醒（spurious wakeup）
- `notify()` 在等待队列有多个线程时存在信号丢失风险，大多数情况优先使用 `notifyAll()`
- `wait(long timeout)` 超时后自动恢复，但无法区分是超时还是被唤醒
- 调用 `notify/notifyAll` 的线程必须持有同一个监视器锁

### ReentrantLock + Condition

`Condition` 是 `Lock` 接口的配套等待/通知机制，比 `synchronized+wait` 更灵活：一个 `ReentrantLock` 可以创建多个 `Condition` 对象，实现精确的分组唤醒。

关键点：
- `condition.await()` 对应 `wait()`，`condition.signal()` 对应 `notify()`
- 支持 `awaitUninterruptibly()`（不响应中断）、`awaitNanos()`（纳秒精度超时）
- `ArrayBlockingQueue` 内部就用两个 Condition（`notEmpty` 和 `notFull`）分别控制生产者和消费者，避免 `notifyAll` 的无效唤醒

### CountDownLatch

用于等待一组操作完成。内部维护一个计数器，`countDown()` 将计数减 1，`await()` 阻塞直到计数归零。计数一旦归零无法重置（不可复用）。

典型场景：主线程等待多个子任务并行完成后汇总结果；服务启动时等待多个依赖组件就绪。

### CyclicBarrier

让一组线程在某个屏障点互相等待，全部到达后一起继续执行。与 CountDownLatch 的区别：
- `CyclicBarrier` 可以重复使用（`reset()` 或自动重置），适合分阶段并行计算
- 可以传入 `barrierAction`，在所有线程到达屏障后由最后一个到达的线程执行该 Runnable
- 如果某个线程在等待时被中断或超时，屏障进入 broken 状态，其他等待线程抛出 `BrokenBarrierException`

### Semaphore

信号量，控制同时访问某资源的线程数量。`acquire()` 获取许可（计数减 1，为 0 时阻塞），`release()` 释放许可（计数加 1）。

典型场景：数据库连接池限流、限制并发下载数、令牌桶限速的简单实现。

注意：`Semaphore` 不绑定线程，一个线程 acquire 后可以由另一个线程 release，因此可以用于跨线程的信号传递。

### BlockingQueue

生产者-消费者模式的最佳实践。主要实现：

| 实现类 | 特点 |
|---|---|
| `ArrayBlockingQueue` | 有界，基于数组，公平/非公平可选 |
| `LinkedBlockingQueue` | 默认无界（Integer.MAX_VALUE），链表实现，吞吐量较高 |
| `PriorityBlockingQueue` | 无界，按优先级出队 |
| `SynchronousQueue` | 容量为 0，每次 put 必须等待 take，适合直接传递任务 |
| `DelayQueue` | 元素到期才能出队，适合延迟任务调度 |

阻塞方法：`put/take`；超时方法：`offer(e, timeout, unit) / poll(timeout, unit)`；非阻塞方法：`offer/poll`。

### LockSupport

底层线程阻塞/唤醒工具，`park()` 挂起当前线程，`unpark(thread)` 唤醒指定线程。

与 `wait/notify` 的区别：
- `unpark` 可以先于 `park` 调用（提前发放许可），之后的 `park` 立即返回，不会阻塞
- 不需要在同步块内调用
- `AbstractQueuedSynchronizer`（AQS）内部就是用 `LockSupport` 实现线程的阻塞与唤醒

### Exchanger

两个线程之间的数据交换点。两个线程分别调用 `exchange(V)` 后，各自得到对方传入的值，然后同时继续执行。只支持两个线程配对，第三个线程调用会阻塞直到有新的配对方出现。

典型场景：双缓冲（一个线程填充数据，另一个线程消费数据，周期性交换缓冲区）。

## 示例代码

### synchronized + wait / notifyAll：生产者-消费者

```java
public class WaitNotifyExample {
    private final Queue<Integer> queue = new LinkedList<>();
    private final int capacity = 5;

    public synchronized void produce(int item) throws InterruptedException {
        while (queue.size() == capacity) {
            wait(); // 队列满，释放锁并等待
        }
        queue.offer(item);
        System.out.println("Produced: " + item + ", size=" + queue.size());
        notifyAll(); // 唤醒所有等待线程（含消费者）
    }

    public synchronized int consume() throws InterruptedException {
        while (queue.isEmpty()) {
            wait(); // 队列空，释放锁并等待
        }
        int item = queue.poll();
        System.out.println("Consumed: " + item + ", size=" + queue.size());
        notifyAll(); // 唤醒所有等待线程（含生产者）
        return item;
    }
}
```

### ReentrantLock + Condition：精确唤醒

```java
public class BoundedBuffer<T> {
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();
    private final Object[] items;
    private int head, tail, count;

    public BoundedBuffer(int capacity) {
        items = new Object[capacity];
    }

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) {
                notFull.await(); // 只有生产者在此等待
            }
            items[tail] = item;
            tail = (tail + 1) % items.length;
            count++;
            notEmpty.signal(); // 精确唤醒一个消费者
        } finally {
            lock.unlock();
        }
    }

    @SuppressWarnings("unchecked")
    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) {
                notEmpty.await(); // 只有消费者在此等待
            }
            T item = (T) items[head];
            head = (head + 1) % items.length;
            count--;
            notFull.signal(); // 精确唤醒一个生产者
            return item;
        } finally {
            lock.unlock();
        }
    }
}
```

### CountDownLatch：并行任务汇总

```java
public class CountDownLatchDemo {
    public static void main(String[] args) throws InterruptedException {
        int taskCount = 5;
        CountDownLatch latch = new CountDownLatch(taskCount);
        ExecutorService executor = Executors.newFixedThreadPool(taskCount);

        for (int i = 0; i < taskCount; i++) {
            final int id = i;
            executor.submit(() -> {
                try {
                    Thread.sleep(100 + id * 50L);
                    System.out.println("Task " + id + " done");
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                } finally {
                    latch.countDown(); // 无论成功失败都要 countDown
                }
            });
        }

        latch.await(5, TimeUnit.SECONDS); // 带超时，防止永久阻塞
        System.out.println("All tasks completed");
        executor.shutdown();
    }
}
```

### CyclicBarrier：分阶段并行计算

```java
public class CyclicBarrierDemo {
    public static void main(String[] args) {
        int threadCount = 3;
        CyclicBarrier barrier = new CyclicBarrier(threadCount, () ->
            System.out.println("--- Phase complete, all threads reached barrier ---")
        );

        for (int i = 0; i < threadCount; i++) {
            final int id = i;
            new Thread(() -> {
                try {
                    for (int phase = 1; phase <= 3; phase++) {
                        Thread.sleep(100 + id * 30L);
                        System.out.println("Thread " + id + " finished phase " + phase);
                        barrier.await(); // 等待所有线程完成当前阶段
                    }
                } catch (InterruptedException | BrokenBarrierException e) {
                    Thread.currentThread().interrupt();
                }
            }).start();
        }
    }
}
```

### Semaphore：限制并发数

```java
public class SemaphoreDemo {
    private static final Semaphore semaphore = new Semaphore(3); // 最多3个并发

    public static void accessResource(int threadId) throws InterruptedException {
        semaphore.acquire();
        try {
            System.out.println("Thread " + threadId + " accessing resource, permits left: "
                + semaphore.availablePermits());
            Thread.sleep(200);
        } finally {
            semaphore.release();
        }
    }

    public static void main(String[] args) {
        for (int i = 0; i < 8; i++) {
            final int id = i;
            new Thread(() -> {
                try { accessResource(id); }
                catch (InterruptedException e) { Thread.currentThread().interrupt(); }
            }).start();
        }
    }
}
```

### LockSupport：精确线程控制

```java
public class LockSupportDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            System.out.println("Worker: about to park");
            LockSupport.park(); // 挂起，等待 unpark
            System.out.println("Worker: unparked, continuing");
        });

        worker.start();
        Thread.sleep(500);

        System.out.println("Main: calling unpark");
        LockSupport.unpark(worker); // 唤醒指定线程
    }
}
```

### Exchanger：双缓冲数据交换

```java
public class ExchangerDemo {
    public static void main(String[] args) {
        Exchanger<List<Integer>> exchanger = new Exchanger<>();

        // 生产者：填充缓冲区后交换
        new Thread(() -> {
            List<Integer> buffer = new ArrayList<>();
            try {
                for (int round = 0; round < 3; round++) {
                    buffer.clear();
                    for (int i = 0; i < 5; i++) buffer.add(round * 5 + i);
                    System.out.println("Producer filled: " + buffer);
                    buffer = exchanger.exchange(buffer); // 交换，得到空缓冲区
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }).start();

        // 消费者：消费满缓冲区后交换
        new Thread(() -> {
            List<Integer> buffer = new ArrayList<>();
            try {
                for (int round = 0; round < 3; round++) {
                    buffer = exchanger.exchange(buffer); // 交换，得到满缓冲区
                    System.out.println("Consumer processing: " + buffer);
                    Thread.sleep(100);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }).start();
    }
}
```

## 常见问题

**1. wait() 必须在循环中检查条件，不能用 if**

`wait()` 返回后不代表条件一定成立，可能是被 `notifyAll()` 唤醒但条件仍不满足（其他线程抢先消费），也可能是虚假唤醒（JVM 规范允许）。正确写法是 `while (!condition) { wait(); }`，绝不能用 `if`。

**2. CountDownLatch.countDown() 必须放在 finally 块**

如果任务抛出异常导致 `countDown()` 未执行，`await()` 的调用方将永久阻塞。正确做法是在 finally 中调用 `countDown()`，并对 `await()` 设置超时：`latch.await(timeout, unit)`。

**3. CyclicBarrier 的 broken 状态会传染**

一个线程在 `barrier.await()` 时中断或超时，屏障进入 broken 状态，其他所有等待线程都会抛出 `BrokenBarrierException`。如果需要容错，必须捕获该异常并决定是 `reset()` 还是终止整个阶段。

**4. BlockingQueue 选型影响吞吐量和内存**

`LinkedBlockingQueue` 默认容量是 `Integer.MAX_VALUE`，在任务生产速度远快于消费速度时会无限堆积，导致 OOM。`ThreadPoolExecutor` 使用 `LinkedBlockingQueue` 时必须显式指定容量。`SynchronousQueue` 没有缓冲，适合线程池中任务直接交接，但要求生产消费速率基本匹配，否则触发拒绝策略。

# 进程间通信（IPC）- Java 后端视角

## 概述

进程间通信（Inter-Process Communication，IPC）是操作系统提供的机制，允许不同进程之间交换数据和协调执行。与线程间通信不同，进程之间的地址空间相互隔离，数据不能直接共享，必须借助操作系统内核或网络协议栈作为中介。

在传统 Unix/Linux 系统中，IPC 机制包括管道、消息队列、共享内存、信号量和 Socket。在 Java 后端分布式系统中，更常见的是通过 HTTP/RPC、消息中间件（MQ）、共享数据库或缓存来实现跨进程协作。理解这两个层次的差异，是区分 OS 级 IPC 和应用级 IPC 的关键。

## 核心概念

### OS 级 IPC 机制对比

| 机制 | 数据方向 | 内核参与 | 跨机器 | 典型场景 |
|------|----------|----------|--------|----------|
| 管道（Pipe） | 单向 | 是 | 否 | 父子进程间数据流 |
| 命名管道（FIFO） | 单向 | 是 | 否 | 同机无亲缘进程通信 |
| 消息队列（Message Queue） | 有结构的消息 | 是 | 否 | 异步、优先级消息传递 |
| 共享内存（Shared Memory） | 双向直接读写 | 映射时参与 | 否 | 高性能大数据量传输 |
| 信号量（Semaphore） | 无数据，仅同步 | 是 | 否 | 互斥访问共享资源 |
| Socket | 双向字节流/数据报 | 是 | 是 | 跨机器通信，最通用 |

#### 管道（Pipe / Named Pipe）

匿名管道只能在有亲缘关系的进程间使用（父子进程），数据单向流动，内部是内核维护的环形缓冲区，读端空时阻塞，写端满时阻塞。命名管道（FIFO）突破了亲缘限制，通过文件系统路径标识。

#### 消息队列（System V / POSIX Message Queue）

内核维护的消息链表，每条消息带有类型标签，支持按类型选择性接收，支持优先级。容量有限（`/proc/sys/fs/mqueue/msg_max`），进程崩溃后消息仍存在于内核（持久性）。主要问题：消息大小限制、不支持跨机器。

#### 共享内存（Shared Memory）

将同一块物理内存页映射到多个进程的虚拟地址空间，是最快的 IPC 方式（数据不经过内核拷贝）。但共享内存本身不提供同步机制，必须配合信号量或互斥锁使用，否则会有竞态条件。

#### 信号量（Semaphore）

不传递数据，仅用于进程间同步。System V 信号量是计数信号量，操作原子（P/V 操作）。与 Java 的 `java.util.concurrent.Semaphore` 概念相同，但作用于进程级别而非线程级别。

#### Socket

最通用的 IPC 机制，同时支持本机（Unix Domain Socket）和网络（TCP/UDP）通信。Unix Domain Socket 走内核路径，不经过网络协议栈，性能接近管道，且支持双向通信和传递文件描述符。

### Java 应用级 IPC 方式

#### Socket / TCP 直连

最基础的方式，适合低延迟点对点通信。需要自行处理连接管理、粘包/拆包、序列化。

#### HTTP / RESTful

基于 TCP，使用文本协议（HTTP/1.1）或二进制帧（HTTP/2）。无状态，天然适合服务间调用。Java 常用客户端：`HttpClient`（JDK 11+）、OkHttp、RestTemplate/WebClient（Spring）。

#### RPC（远程过程调用）

在 HTTP 或私有协议之上封装了序列化、服务发现、负载均衡。常见框架：
- **gRPC**：基于 HTTP/2 + Protobuf，强类型，支持流式调用
- **Dubbo**：阿里开源，支持多协议（Dubbo/Triple/REST），内置服务治理
- **Feign/OpenFeign**：声明式 HTTP 客户端，与 Spring Cloud 集成

RPC vs HTTP 的核心差异：RPC 框架通常包含服务注册/发现、负载均衡、熔断降级等服务治理能力，而裸 HTTP 调用需要自行处理。

#### 消息队列（MQ）

异步解耦，生产者和消费者不需要同时在线。适合高吞吐、允许短暂延迟的场景。

| 框架 | 协议 | 顺序性 | 事务 | 典型场景 |
|------|------|--------|------|----------|
| Kafka | 自定义 | 分区内有序 | 支持 | 日志收集、事件流 |
| RabbitMQ | AMQP | 队列内有序 | 支持 | 任务队列、延迟队列 |
| RocketMQ | 自定义 | 支持全局顺序 | 支持事务消息 | 电商订单、金融 |
| ActiveMQ | JMS/AMQP | 支持 | 支持 | 传统企业应用 |

## 示例代码

### ProcessBuilder 管道操作

```java
// 启动子进程并读取其标准输出（管道通信）
public class PipeExample {
    public static void main(String[] args) throws Exception {
        ProcessBuilder pb = new ProcessBuilder("ls", "-la", "/tmp");
        pb.redirectErrorStream(true); // 合并 stderr 到 stdout

        Process process = pb.start();

        // 父进程通过管道读取子进程输出
        try (BufferedReader reader = new BufferedReader(
                new InputStreamReader(process.getInputStream()))) {
            String line;
            while ((line = reader.readLine()) != null) {
                System.out.println(line);
            }
        }

        int exitCode = process.waitFor();
        System.out.println("Exit code: " + exitCode);
    }
}
```

### 内存映射文件（类共享内存）

```java
// 进程 A：写入共享数据
public class SharedMemoryWriter {
    private static final String FILE_PATH = "/tmp/shared_data.bin";
    private static final int SIZE = 1024;

    public static void main(String[] args) throws Exception {
        try (RandomAccessFile raf = new RandomAccessFile(FILE_PATH, "rw");
             FileChannel channel = raf.getChannel()) {

            MappedByteBuffer buffer = channel.map(
                FileChannel.MapMode.READ_WRITE, 0, SIZE);

            // 写入数据（其他映射了同一文件的进程可以读取）
            String message = "Hello from Process A";
            buffer.put(message.getBytes(StandardCharsets.UTF_8));
            buffer.force(); // 确保写入底层存储
            System.out.println("Written: " + message);
        }
    }
}

// 进程 B：读取共享数据
public class SharedMemoryReader {
    private static final String FILE_PATH = "/tmp/shared_data.bin";
    private static final int SIZE = 1024;

    public static void main(String[] args) throws Exception {
        try (RandomAccessFile raf = new RandomAccessFile(FILE_PATH, "rw");
             FileChannel channel = raf.getChannel()) {

            MappedByteBuffer buffer = channel.map(
                FileChannel.MapMode.READ_ONLY, 0, SIZE);

            byte[] bytes = new byte[20];
            buffer.get(bytes);
            System.out.println("Read: " + new String(bytes, StandardCharsets.UTF_8));
        }
    }
}
```

### TCP Socket 直连

```java
// 服务端
public class IpcServer {
    public static void main(String[] args) throws Exception {
        ExecutorService executor = Executors.newCachedThreadPool();

        try (ServerSocket serverSocket = new ServerSocket(8888)) {
            System.out.println("Server listening on port 8888");

            while (true) {
                Socket clientSocket = serverSocket.accept();
                executor.submit(() -> handleClient(clientSocket));
            }
        }
    }

    private static void handleClient(Socket socket) {
        try (socket;
             BufferedReader in = new BufferedReader(
                 new InputStreamReader(socket.getInputStream()));
             PrintWriter out = new PrintWriter(socket.getOutputStream(), true)) {

            String request = in.readLine();
            System.out.println("Received: " + request);
            out.println("Echo: " + request);

        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

// 客户端
public class IpcClient {
    public static void main(String[] args) throws Exception {
        try (Socket socket = new Socket("localhost", 8888);
             PrintWriter out = new PrintWriter(socket.getOutputStream(), true);
             BufferedReader in = new BufferedReader(
                 new InputStreamReader(socket.getInputStream()))) {

            out.println("Hello from client process");
            String response = in.readLine();
            System.out.println("Response: " + response);
        }
    }
}
```

### gRPC 跨进程调用（基于 Protobuf）

```proto
// message.proto
syntax = "proto3";
option java_package = "com.example.grpc";

service MessageService {
  rpc Send (MessageRequest) returns (MessageResponse);
}

message MessageRequest {
  string content = 1;
}

message MessageResponse {
  string result = 1;
}
```

```java
// gRPC 服务实现
public class MessageServiceImpl extends MessageServiceGrpc.MessageServiceImplBase {

    @Override
    public void send(MessageRequest request,
                     StreamObserver<MessageResponse> responseObserver) {
        String content = request.getContent();
        System.out.println("gRPC received: " + content);

        MessageResponse response = MessageResponse.newBuilder()
            .setResult("Processed: " + content)
            .build();

        responseObserver.onNext(response);
        responseObserver.onCompleted();
    }
}

// gRPC 客户端调用
public class GrpcClient {
    public static void main(String[] args) throws Exception {
        ManagedChannel channel = ManagedChannelBuilder
            .forAddress("localhost", 9090)
            .usePlaintext()
            .build();

        try {
            MessageServiceGrpc.MessageServiceBlockingStub stub =
                MessageServiceGrpc.newBlockingStub(channel);

            MessageResponse response = stub.send(
                MessageRequest.newBuilder()
                    .setContent("Hello gRPC")
                    .build());

            System.out.println("Response: " + response.getResult());
        } finally {
            channel.shutdown().awaitTermination(5, TimeUnit.SECONDS);
        }
    }
}
```

### Kafka 消息队列（异步 IPC）

```java
// 生产者（发送进程）
public class KafkaIpcProducer {
    public static void main(String[] args) throws Exception {
        Properties props = new Properties();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG,
            StringSerializer.class.getName());
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG,
            StringSerializer.class.getName());
        // 保证消息不丢失
        props.put(ProducerConfig.ACKS_CONFIG, "all");
        props.put(ProducerConfig.RETRIES_CONFIG, 3);

        try (KafkaProducer<String, String> producer = new KafkaProducer<>(props)) {
            ProducerRecord<String, String> record =
                new ProducerRecord<>("ipc-topic", "order-123", "{\"amount\": 100}");

            producer.send(record, (metadata, exception) -> {
                if (exception != null) {
                    System.err.println("Send failed: " + exception.getMessage());
                } else {
                    System.out.printf("Sent to partition %d, offset %d%n",
                        metadata.partition(), metadata.offset());
                }
            });
        }
    }
}

// 消费者（接收进程）
public class KafkaIpcConsumer {
    public static void main(String[] args) {
        Properties props = new Properties();
        props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ConsumerConfig.GROUP_ID_CONFIG, "ipc-consumer-group");
        props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG,
            StringDeserializer.class.getName());
        props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG,
            StringDeserializer.class.getName());
        // 手动提交 offset，避免消息丢失
        props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, "false");

        try (KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props)) {
            consumer.subscribe(Collections.singletonList("ipc-topic"));

            while (true) {
                ConsumerRecords<String, String> records =
                    consumer.poll(Duration.ofMillis(100));

                for (ConsumerRecord<String, String> record : records) {
                    System.out.printf("Received: key=%s, value=%s%n",
                        record.key(), record.value());
                    // 处理完成后手动提交
                    consumer.commitSync();
                }
            }
        }
    }
}
```

## 常见问题

### 1. 共享内存并发写导致数据损坏

`MappedByteBuffer` 本身不是线程安全的，多个进程同时写同一内存区域会导致数据撕裂。必须配合文件锁（`FileChannel.lock()`）或外部信号量进行互斥。生产中建议使用专门的共享内存库（如 Chronicle Map）而不是裸 `MappedByteBuffer`。

### 2. TCP Socket 粘包/拆包问题

TCP 是字节流协议，不保留消息边界。发送端连续发送两条消息，接收端可能一次 `read` 读到 1.5 条。解决方案：使用固定长度头部标识消息长度（Length-Prefix Framing），或使用换行符/特殊分隔符。使用 Netty 可以用 `LengthFieldBasedFrameDecoder` 自动处理。

### 3. MQ 消费幂等性

消息中间件的"至少一次"投递语义会导致重复消费。消费者必须实现幂等处理：可以用消息 ID 作为数据库唯一索引，或使用 Redis `SET NX` 做去重标记。切忌假设消息只会消费一次。

### 4. gRPC 超时与取消传播

gRPC 支持 Deadline 机制，客户端设置的超时会通过 `Context` 传播到服务端和下游调用链。若忽略 Deadline，上游已超时取消的请求，下游服务仍在消耗资源。应在服务端检查 `Context.current().isCancelled()`，并将 Deadline 传递给所有下游 stub 调用。

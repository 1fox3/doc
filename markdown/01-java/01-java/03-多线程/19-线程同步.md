# 线程同步工具：CountDownLatch / CyclicBarrier / Semaphore / Exchanger / LockSupport

## 概述

Java `java.util.concurrent` 包提供了一组高级同步工具，用于协调多线程之间的执行顺序、资源访问和数据交换。`CountDownLatch`、`Semaphore` 基于 AQS 共享模式实现；`CyclicBarrier` 基于 `ReentrantLock` + `Condition` 实现；`LockSupport` 则是 AQS 等同步器底层使用的阻塞/唤醒原语。

本文涵盖五个核心同步器：CountDownLatch（一次性计数门）、CyclicBarrier（可重置屏障）、Semaphore（信号量限流）、Exchanger（线程间数据交换）、LockSupport（底层阻塞/唤醒原语）。

## 核心概念

### CountDownLatch — 一次性计数器

CountDownLatch 初始化时设定一个计数值 N，允许一个或多个线程调用 `await()` 阻塞等待，直到其他线程调用 `countDown()` 将计数减为 0 后，所有等待线程才被释放。

关键特点：
- **一次性**：计数器归零后不可重置，不能复用（区别于 CyclicBarrier）。
- **await 可超时**：`await(long timeout, TimeUnit unit)` 超时后返回 false，不抛异常。
- **countDown 不阻塞**：调用方立即返回，仅减少计数。
- 底层使用 AQS 的共享模式，计数归零时唤醒所有等待线程。

典型场景：主线程等待多个子任务全部完成后汇总结果；服务启动时等待多个依赖初始化完毕。

底层映射：`state = N`，`countDown()` 调用 `releaseShared(1)` 递减计数，归零时唤醒所有 `await()` 线程；`await()` 调用 `acquireSharedInterruptibly(1)`，计数不为 0 时进入 AQS 队列等待。

### CyclicBarrier — 可重置屏障 + barrierAction

CyclicBarrier 设定参与方数量 parties，每个线程调用 `await()` 表示已到达屏障，当所有 parties 个线程均到达后，屏障打开，所有线程继续执行。之后屏障自动重置，可复用（Cyclic 的含义）。

关键特点：
- **parties**：构造时指定屏障拦截的线程数量。
- **barrierAction**：可选的 Runnable，在屏障打开前、由最后一个到达的线程执行（不是新线程）。常用于合并各线程的中间结果。
- **可重置**：屏障打开后自动进入下一轮，也可调用 `reset()` 强制重置（会导致当前等待线程抛 `BrokenBarrierException`）。
- **BrokenBarrierException**：任何一个等待线程被中断、超时或调用 reset，屏障进入 broken 状态，其他等待线程抛此异常。
- 与 CountDownLatch 的核心区别：CyclicBarrier 是等待"所有人都到"，CountDownLatch 是等待"计数归零"（调用者不必是等待者）。

典型场景：多轮迭代计算，每轮结束同步一次；并行测试中让所有线程同时开始压测。

与 CountDownLatch 的核心区别：

| 特性 | CountDownLatch | CyclicBarrier |
|---|---|---|
| 可复用 | 否，归零后不可重置 | 是，每轮通过后自动重置 |
| 等待方向 | 等待者等待计数归零 | 参与者互相等待到齐 |
| 触发动作 | 无内置回调 | 支持 `barrierAction` |
| 底层实现 | AQS 共享模式 | ReentrantLock + Condition |

### Semaphore — 信号量限流

Semaphore 维护一定数量的许可（permits），线程通过 `acquire()` 获取许可（数量不足时阻塞），通过 `release()` 归还许可。

关键特点：
- **permits**：构造时设定许可总数，控制同时进入临界区的线程数量。
- **fair**：构造参数，公平模式（true）按 FIFO 排队，避免饥饿；非公平模式（false，默认）吞吐更高。
- **tryAcquire**：`tryAcquire()` 不阻塞立即返回；`tryAcquire(timeout, unit)` 带超时。
- **acquire(n) / release(n)**：支持一次获取/释放多个许可。
- permits 可以超出初始值（release 次数多于 acquire），需注意避免逻辑错误。

典型场景：数据库连接池限制并发数；限流器控制 API 并发请求数；停车场模型。

`Semaphore` 更适合限制并发数量，不适合替代普通锁保护共享状态。多个线程同时持有许可时仍可能并发修改同一对象，临界区内的共享数据仍需额外同步。

### Exchanger — 两线程数据交换

Exchanger 是专为**恰好两个线程**之间同步交换数据设计的同步点。两个线程各自调用 `exchange(V)` 提交自己的数据，双方都到达后互换数据并继续执行。

关键特点：
- 严格限定两个线程：第一个调用 `exchange()` 的线程阻塞等待，第二个到达时完成交换，双方均返回对方的数据。
- 支持超时：`exchange(V, timeout, unit)`，超时抛 `TimeoutException`。
- 不适用于多于两个线程的场景（第三个线程调用会与下一个配对）。
- 常见于流水线模式：一个线程生产数据，另一个线程消费数据，通过 Exchanger 交换缓冲区。

### LockSupport — 底层阻塞/唤醒原语

LockSupport 是构建同步器的基础工具类，提供 `park` 和 `unpark` 两个核心方法，不需要持有锁。

关键特点：
- **park()**：阻塞当前线程，直到获得许可或被中断。不抛 InterruptedException，但中断后 `Thread.interrupted()` 标志位为 true。
- **unpark(Thread t)**：给指定线程发放一个许可。可以在 park 之前调用（许可预存），该线程之后调用 park 时不会阻塞。
- **许可是二值的**：一个线程最多持有一个许可，多次 unpark 不会累积（只有 0/1 两个状态）。
- **park 不需要锁**：相比 `wait/notify`，park/unpark 不依赖 synchronized 块，更灵活。
- **park 变体**：`parkNanos(long nanos)`、`parkUntil(long deadline)`、`park(Object blocker)` 可传入 blocker 对象用于调试（`jstack` 可见）。
- AQS 内部使用 LockSupport 实现线程的阻塞与唤醒。

## 示例代码

```java
import java.util.concurrent.*;
import java.util.concurrent.locks.LockSupport;

// ── 1. CountDownLatch：主线程等待3个子任务完成 ──────────────────────────
public class CountDownLatchDemo {
    public static void main(String[] args) throws InterruptedException {
        int taskCount = 3;
        CountDownLatch latch = new CountDownLatch(taskCount);

        for (int i = 1; i <= taskCount; i++) {
            final int taskId = i;
            new Thread(() -> {
                try {
                    System.out.println("Task " + taskId + " running...");
                    Thread.sleep(500 * taskId);
                    System.out.println("Task " + taskId + " done.");
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                } finally {
                    latch.countDown(); // 必须在 finally 中，防止异常导致永久阻塞
                }
            }).start();
        }

        System.out.println("Main waiting...");
        boolean completed = latch.await(5, TimeUnit.SECONDS);
        System.out.println("All tasks done: " + completed);
    }
}
```

```java
// ── 2. CyclicBarrier：3个线程两轮计算，每轮结束合并结果 ────────────────
public class CyclicBarrierDemo {
    static int[] results = new int[3];

    public static void main(String[] args) throws InterruptedException {
        // barrierAction 由最后一个到达的线程执行
        CyclicBarrier barrier = new CyclicBarrier(3, () -> {
            int sum = 0;
            for (int r : results) sum += r;
            System.out.println("[BarrierAction] Round result sum = " + sum);
        });

        for (int round = 0; round < 2; round++) {
            final int r = round;
            for (int i = 0; i < 3; i++) {
                final int idx = i;
                new Thread(() -> {
                    results[idx] = (r + 1) * (idx + 1); // 模拟计算
                    System.out.println("Thread-" + idx + " round " + r + " computed: " + results[idx]);
                    try {
                        barrier.await(); // 等待所有线程到达屏障
                    } catch (InterruptedException | BrokenBarrierException e) {
                        Thread.currentThread().interrupt();
                    }
                }).start();
            }
            Thread.sleep(200); // 确保第一轮线程完成，第二轮才启动（演示用）
        }
    }
}
```

```java
// ── 3. Semaphore：最多3个线程并发访问资源 ──────────────────────────────
public class SemaphoreDemo {
    public static void main(String[] args) {
        Semaphore semaphore = new Semaphore(3, true); // 3个许可，公平模式

        for (int i = 1; i <= 8; i++) {
            final int id = i;
            new Thread(() -> {
                try {
                    semaphore.acquire();
                    System.out.println("Thread-" + id + " acquired permit. Active: "
                            + (3 - semaphore.availablePermits()));
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                } finally {
                    semaphore.release(); // 必须在 finally 中释放
                    System.out.println("Thread-" + id + " released permit.");
                }
            }).start();
        }
    }
}
```

```java
// ── 4. Exchanger：生产者和消费者交换缓冲区 ─────────────────────────────
public class ExchangerDemo {
    public static void main(String[] args) {
        Exchanger<String> exchanger = new Exchanger<>();

        new Thread(() -> {
            try {
                String data = "data-from-producer";
                System.out.println("Producer sending: " + data);
                String received = exchanger.exchange(data);
                System.out.println("Producer received: " + received);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }, "Producer").start();

        new Thread(() -> {
            try {
                String ack = "ack-from-consumer";
                System.out.println("Consumer sending: " + ack);
                String received = exchanger.exchange(ack);
                System.out.println("Consumer received: " + received);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }, "Consumer").start();
    }
}
```

```java
// ── 5. LockSupport：手动 park/unpark，演示许可预存机制 ─────────────────
public class LockSupportDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            System.out.println("Worker: about to park");
            LockSupport.park(Thread.currentThread()); // blocker 参数用于 jstack 调试
            System.out.println("Worker: unparked, interrupted=" + Thread.interrupted());
        });

        worker.start();
        Thread.sleep(500);

        // unpark 可以在 park 之前或之后调用
        System.out.println("Main: calling unpark");
        LockSupport.unpark(worker);

        // 演示许可预存：unpark 在 park 之前调用，park 立即返回
        Thread instant = new Thread(() -> {
            System.out.println("Instant: will park after pre-unpark");
            LockSupport.park(); // 许可已预存，立即返回
            System.out.println("Instant: park returned immediately");
        });

        LockSupport.unpark(instant); // 先 unpark
        instant.start();             // 再启动线程
    }
}
```

## 常见问题

**1. CountDownLatch 中 countDown() 忘写在 finally 块**

若子线程抛出异常而 countDown() 未执行，主线程将永久阻塞在 await()。正确做法是始终将 countDown() 置于 finally 块中，或使用带超时的 await(timeout, unit) 作为保底。

**2. CyclicBarrier 屏障破损（BrokenBarrierException）未处理**

任一线程在 await() 期间被中断或超时，屏障进入 broken 状态，其余等待线程均抛 BrokenBarrierException。需捕获此异常并决定是 reset 屏障重试，还是整体终止任务。注意 reset() 本身也会对当前等待线程抛出此异常。

**3. Semaphore release() 没有与 acquire() 配对**

release() 不检查调用者是否曾经 acquire，多余的 release 会使 availablePermits 超过初始值，导致限流失效。应通过 try/finally 保证配对，且不要在未 acquire 的路径上调用 release。

**4. LockSupport.park 被虚假唤醒后未检查条件**

park 可能因线程中断或虚假唤醒提前返回（中断时不抛异常，仅设置中断标志）。必须在循环中配合条件判断使用，与 AQS 内部的实现方式一致：
```java
while (!conditionMet()) {
    LockSupport.park();
    if (Thread.interrupted()) { /* handle */ }
}
```

**5. Semaphore release() 可以超过初始许可数**

`release()` 不检查调用者是否真的成功 `acquire()`，多次误调用会使 `availablePermits()` 超过初始值，导致限流失效。应只在成功获取许可后进入 `try/finally`，并在 `finally` 中释放。

# 源码关键点

## 关键类

| 类 | 作用 |
| --- | --- |
| `AbstractAutowireCapableBeanFactory` | 负责创建 Bean、属性填充、初始化等完整流程 |
| `RootBeanDefinition` | 保存合并后的 Bean 定义，包括构造参数、工厂方法、方法覆盖等信息 |
| `InstantiationStrategy` | 实例化策略接口 |
| `SimpleInstantiationStrategy` | 默认反射实例化策略 |
| `CglibSubclassingInstantiationStrategy` | 需要方法注入时使用 CGLIB 子类实例化 |
| `MethodOverrides` | 保存 `lookup-method`、`replaced-method` 等方法覆盖配置 |

## 调用链概览

典型调用链可以简化为：

```text
getBean
  -> doGetBean
  -> createBean
  -> doCreateBean
  -> createBeanInstance
  -> instantiateBean / autowireConstructor / instantiateUsingFactoryMethod
  -> InstantiationStrategy.instantiate
```

`createBeanInstance` 会根据 BeanDefinition 信息决定走哪条实例化路径：

| 路径 | 触发条件 |
| --- | --- |
| `instantiateUsingFactoryMethod` | 配置了工厂方法 |
| `autowireConstructor` | 存在构造参数、候选构造方法或构造注入需求 |
| `instantiateBean` | 普通无参构造实例化 |

## SimpleInstantiationStrategy 判断逻辑

核心语义可以概括为：

```java
public Object instantiate(RootBeanDefinition bd, String beanName, BeanFactory owner) {
    if (!bd.hasMethodOverrides()) {
        Constructor<?> constructor = bd.getResolvedConstructorOrFactoryMethod();
        return BeanUtils.instantiateClass(constructor);
    }
    return instantiateWithMethodInjection(bd, beanName, owner);
}
```

无方法覆盖时，直接通过构造方法反射创建对象；有方法覆盖时，交给方法注入逻辑。

## CglibSubclassingInstantiationStrategy 做了什么

当存在 `methodOverrides` 时，Spring 需要创建目标类的子类，并在子类中覆盖指定方法。它通常会：

1. 使用 CGLIB `Enhancer` 设置父类为目标 Bean 类。
2. 设置回调过滤器，根据方法是否匹配 `lookup-method` 或 `replaced-method` 选择不同拦截器。
3. 生成子类并调用构造方法创建实例。
4. 当被覆盖方法被调用时，由拦截器从 BeanFactory 获取目标 Bean 或调用 `MethodReplacer`。

伪代码：

```java
class GeneratedReportService extends ReportService {
    private BeanFactory beanFactory;

    @Override
    protected ReportTask createTask() {
        return beanFactory.getBean("reportTask", ReportTask.class);
    }
}
```

## 与构造方法缓存

Spring 会缓存已解析的构造方法或工厂方法，避免每次创建 Bean 都重新做复杂匹配。对于单例 Bean，实例化通常只发生一次；对于原型 Bean，实例化会发生多次，构造方法解析结果缓存更有意义。

## 与循环依赖的关系

实例化是解决单例循环依赖的前置步骤。Spring 解决部分 setter 循环依赖时，会在对象实例化后、属性填充前提前暴露一个 ObjectFactory，用于后续获取早期引用。

需要注意：

- 构造器循环依赖无法通过提前暴露对象解决，因为对象还没有成功实例化。
- 方法注入生成的 CGLIB 子类和 AOP 代理不是同一个概念，但都可能影响早期引用暴露和最终 Bean 类型判断。

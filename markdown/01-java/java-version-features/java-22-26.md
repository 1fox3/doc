# Java 22-26 新特性

截至 2026-06，OpenJDK 当前 GA 版本是 JDK 26。JDK 27、JDK 28 仍属于 Early Access，不应按正式生产版本记录。Java 25 是继 Java 21 之后的新 LTS。

Java 22 到 Java 26 的重点是外部函数与内存访问、Stream Gatherers、Class-File API、Scoped Values、结构化并发、AOT 能力、HTTP/3、GC 改进和平台清理。

## Java 22：Foreign Function & Memory API

Java 22 标准化 Foreign Function & Memory API，用于替代 JNI 中一部分访问本地函数和堆外内存的场景。

```java
import java.lang.foreign.Arena;
import java.lang.foreign.MemorySegment;
import java.lang.foreign.ValueLayout;

try (Arena arena = Arena.ofConfined()) {
    MemorySegment segment = arena.allocate(ValueLayout.JAVA_INT);
    segment.set(ValueLayout.JAVA_INT, 0, 100);
    int value = segment.get(ValueLayout.JAVA_INT, 0);
    System.out.println(value);
}
```

价值：

- 用更安全、受 JVM 管理的方式访问堆外内存。
- 降低部分 JNI 使用复杂度。
- 适合数据库、序列化、向量计算、系统库调用等底层场景。

## Java 22：Unnamed Variables and Patterns

Java 22 标准化未命名变量和未命名模式，用 `_` 表示这个变量不关心、不会使用。

```java
try {
    riskyCall();
} catch (Exception _) {
    System.out.println("failed");
}
```

适合场景：

- catch 参数不需要使用。
- lambda 参数不需要使用。
- 模式匹配中只关心结构，不关心某个绑定值。

## Java 22：Launch Multi-File Source-Code Programs

Java 22 增强源码文件直接运行能力，允许直接启动由多个源码文件组成的小程序。

```bash
java Main.java
```

如果 `Main.java` 依赖同目录或源码路径下的其他 `.java` 文件，启动器可以按需编译相关源码。这个能力适合脚本、小工具、教学示例和快速验证，不适合替代正式项目构建系统。

示例目录：

```text
demo/
├── Main.java
└── MessageService.java
```

`Main.java`：

```java
public class Main {
    public static void main(String[] args) {
        var service = new MessageService();
        System.out.println(service.message());
    }
}
```

`MessageService.java`：

```java
public class MessageService {
    String message() {
        return "hello java 22";
    }
}
```

直接运行入口源码文件：

```bash
java Main.java
```

## Java 23：Markdown Documentation Comments

Java 23 支持在 Javadoc 中使用 Markdown 文档注释，降低编写 API 文档的成本。

```java
/// Creates a user.
///
/// - `name` must not be blank.
/// - `age` must be greater than zero.
///
/// @param name user name
/// @param age user age
public void createUser(String name, int age) {
}
```

价值：

- 文档列表、代码片段、段落更易读。
- 减少传统 HTML 风格 Javadoc 的书写成本。
- 对公共 API、SDK、基础库文档更友好。

## Java 23：ZGC Generational Mode by Default

Java 23 将 ZGC 的分代模式设为默认。分代 ZGC 会把堆划分为年轻代和老年代，更符合大多数 Java 应用对象“朝生夕死”的分配规律。

影响：

- ZGC 更适合常规服务端工作负载。
- 降低短生命周期对象带来的回收成本。
- 使用 ZGC 时仍需结合延迟、吞吐、内存占用压测评估。

启动示例：

```bash
java -XX:+UseZGC -Xms2g -Xmx2g -jar app.jar
```

观察 GC 日志：

```bash
java -XX:+UseZGC -Xlog:gc* -jar app.jar
```

Java 23 之后使用 `-XX:+UseZGC` 时默认就是分代 ZGC。升级前后不要只看平均响应时间，还要比较 P99/P999 延迟、CPU、内存占用和 GC 暂停。

## Java 24：Stream Gatherers

Java 24 标准化 Stream Gatherers。它为 Stream 提供中间操作扩展机制，可以表达滑动窗口、分组、限流、扫描等原来不容易用标准 Stream 写清楚的操作。

```java
import java.util.List;
import java.util.stream.Gatherers;

List<List<Integer>> windows = List.of(1, 2, 3, 4, 5)
        .stream()
        .gather(Gatherers.windowFixed(2))
        .toList();

System.out.println(windows); // [[1, 2], [3, 4], [5]]
```

价值：

- 扩展 Stream 中间操作。
- 减少复杂流式处理中的临时集合和样板代码。
- 适合数据处理、批处理、窗口计算等场景。

## Java 24：Class-File API

Java 24 标准化 Class-File API，用于解析、生成和转换 Java class 文件。

它主要面向框架、编译器、Agent、字节码工具作者。普通业务开发很少直接使用，但它对 Lombok、Byte Buddy、ASM 类工具、静态分析工具、构建插件等生态有长期影响。

解析 class 文件示例：

```java
import java.lang.classfile.ClassFile;
import java.lang.classfile.ClassModel;
import java.nio.file.Files;
import java.nio.file.Path;

public class ClassFileApiDemo {
    public static void main(String[] args) throws Exception {
        byte[] bytes = Files.readAllBytes(Path.of("target/classes/com/example/User.class"));
        ClassModel model = ClassFile.of().parse(bytes);

        System.out.println(model.thisClass().asInternalName());
        model.methods().forEach(method -> System.out.println(method.methodName().stringValue()));
    }
}
```

这个例子读取一个已经编译好的 `.class` 文件，打印类名和方法名。真实框架可以进一步读取字段、注解、方法字节码，或生成、转换新的 class 文件。

## Java 24：Synchronize Virtual Threads without Pinning

Java 24 改进虚拟线程与 `synchronized` 的协作，减少虚拟线程因为进入 `synchronized` 代码块而被固定到平台线程的情况。

意义：

- 降低虚拟线程使用传统同步代码时的性能风险。
- 老代码迁移到虚拟线程时更平滑。
- 仍然不能把虚拟线程理解为无限资源，连接池、限流、超时仍然必须治理。

示例：

```java
import java.util.concurrent.Executors;
import java.util.stream.IntStream;

public class VirtualThreadSynchronizedDemo {
    private static final Object LOCK = new Object();

    public static void main(String[] args) {
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            IntStream.range(0, 1000).forEach(i -> executor.submit(() -> {
                synchronized (LOCK) {
                    doShortWork(i);
                }
            }));
        }
    }

    private static void doShortWork(int value) {
        if (value == 0) {
            System.out.println(Thread.currentThread());
        }
    }
}
```

Java 24 改进的是虚拟线程进入 `synchronized` 时的底层调度表现，不代表可以随意扩大锁粒度。长时间持锁仍然会降低并发能力。

## Java 24：安全与平台清理

Java 24 永久禁用 Security Manager，并继续推进 `sun.misc.Unsafe` 内存访问方法的限制。

影响：

- 依赖 Security Manager 沙箱能力的老系统需要迁移。
- 使用 Unsafe 的框架、Agent、中间件需要关注兼容性。
- JDK 正在持续收紧内部 API 和危险能力的直接访问。

老代码示例：

```java
public class SecurityManagerDemo {
    public static void main(String[] args) {
        System.setSecurityManager(new SecurityManager());
    }
}
```

这类代码在新版本 JDK 中已经不再适合作为安全边界。升级老系统时，应删除对 Security Manager 的依赖，改用进程隔离、容器隔离、权限控制、代码审计等方式治理安全风险。

Unsafe 内存访问也应尽量迁移到标准 API，例如用 VarHandle、Foreign Function & Memory API 或框架提供的受支持能力替代。

## Java 25：Scoped Values

Java 25 标准化 Scoped Values。它用于在线程调用链中传递只读上下文，被视为 ThreadLocal 的现代替代方向之一，尤其适合虚拟线程。

```java
import java.lang.ScopedValue;

class RequestContextDemo {
    private static final ScopedValue<String> TRACE_ID = ScopedValue.newInstance();

    static void handle() {
        System.out.println(TRACE_ID.get());
    }

    public static void main(String[] args) {
        ScopedValue.where(TRACE_ID, "trace-001").run(RequestContextDemo::handle);
    }
}
```

相比 `ThreadLocal`：

- 作用域明确，离开作用域后不可访问。
- 默认不可变，适合传递请求上下文。
- 更适合大量虚拟线程，降低上下文泄漏风险。

## Java 25：Module Import Declarations

Java 25 标准化模块导入声明，可以一次导入某个模块导出的包，简化教学、脚本和小程序中的 import。

```java
import module java.base;

void main() {
    var names = List.of("Tom", "Jerry");
    System.out.println(names);
}
```

它不替代 JPMS 的 `module-info.java`，更像是源码层面的导入简化能力。

## Java 25：Compact Source Files and Instance Main Methods

Java 25 标准化紧凑源码文件和实例 main 方法，让简单程序可以省略类声明和传统 `public static void main(String[] args)`。

```java
void main() {
    println("hello java 25");
}
```

价值：

- 降低 Java 入门门槛。
- 适合教学、脚本、小工具。
- 正式工程代码仍建议保持清晰的类、包和模块结构。

## Java 25：Flexible Constructor Bodies

Java 25 标准化灵活构造方法体，允许在调用 `super(...)` 或 `this(...)` 之前执行不依赖当前实例的语句。

```java
class User extends Person {
    User(String name) {
        String normalized = name.trim();
        super(normalized);
    }
}
```

它主要解决构造参数校验、转换、规范化时必须绕到静态方法里的问题。注意，在 `super(...)` 之前仍然不能访问当前实例字段或调用实例方法。

## Java 25：Compact Object Headers

Java 25 引入紧凑对象头，减少对象头占用空间。对于对象数量巨大的应用，可能降低堆占用并改善缓存命中。

适合关注：

- 大量小对象场景。
- 内存敏感服务。
- 高密度缓存、索引、图结构等数据结构。

可以用大量小对象场景做压测对比：

```java
import java.util.ArrayList;

record Point(int x, int y) {}

public class CompactObjectHeadersDemo {
    public static void main(String[] args) throws Exception {
        var points = new ArrayList<Point>();
        for (int i = 0; i < 10_000_000; i++) {
            points.add(new Point(i, i));
        }

        System.out.println(points.size());
        Thread.sleep(60_000);
    }
}
```

运行时观察堆占用：

```bash
java -Xms1g -Xmx1g -Xlog:gc* CompactObjectHeadersDemo
```

紧凑对象头的收益取决于对象数量、对象大小、GC、压缩指针、JVM 参数和具体发行版实现，应通过压测和内存分析确认。

## Java 25：Generational Shenandoah

Java 25 标准化分代 Shenandoah。它把 Shenandoah 与分代理念结合，改善常见应用中年轻对象的回收效率。

选择 GC 时不要只看新特性名称，应结合应用目标判断：

- G1：默认选择，均衡稳定。
- ZGC：关注低延迟和大堆。
- Shenandoah：关注低暂停，也需要结合发行版支持情况。

启动 Shenandoah 示例：

```bash
java -XX:+UseShenandoahGC -Xms4g -Xmx4g -Xlog:gc* -jar app.jar
```

GC 选择建议用同一套压测流量对比：

```bash
java -XX:+UseG1GC -Xlog:gc* -jar app.jar
java -XX:+UseZGC -Xlog:gc* -jar app.jar
java -XX:+UseShenandoahGC -Xlog:gc* -jar app.jar
```

重点比较吞吐、平均延迟、P99/P999、GC 暂停、CPU 和内存，而不是只看某个 GC 名称。

## Java 25 LTS 意义

Java 25 是继 Java 21 后的新 LTS。相比 Java 21，它不是只增加某一个爆点特性，而是把多个预览能力推进到标准化，并继续完善虚拟线程、上下文传递、AOT、GC、文档和源码启动体验。

对后端的实际意义：

- 虚拟线程生态进一步成熟。
- Scoped Values 成为上下文传递的重要新工具。
- GC 和对象布局继续优化。
- JDK 对内部 API、Unsafe、Security Manager 等老机制继续收紧。

## Java 26：HTTP/3 for HTTP Client API

Java 26 为标准 HTTP Client API 增加 HTTP/3 支持。

```java
import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;

HttpClient client = HttpClient.newBuilder()
        .version(HttpClient.Version.HTTP_3)
        .build();

HttpRequest request = HttpRequest.newBuilder()
        .uri(URI.create("https://example.com"))
        .build();

HttpResponse<String> response = client.send(request, HttpResponse.BodyHandlers.ofString());
System.out.println(response.version());
```

HTTP/3 基于 QUIC，重点是连接迁移、减少握手开销和改善弱网体验。业务是否能受益，取决于服务端、网关、网络环境和客户端配置支持情况。

也可以只对单个请求指定 HTTP/3：

```java
HttpRequest request = HttpRequest.newBuilder(URI.create("https://example.com"))
        .version(HttpClient.Version.HTTP_3)
        .GET()
        .build();
```

HTTP/3 不是 Java 26 的默认协议版本。如果目标服务端不支持 HTTP/3，客户端可能回退到 HTTP/2 或 HTTP/1.1，具体结果可以通过 `response.version()` 观察。

## Java 26：Prepare to Make Final Mean Final

Java 26 开始为后续强化 `final` 语义做准备。长期方向是减少通过深反射、Unsafe 等方式修改 final 字段的能力，让 `final` 更可信。

影响：

- 依赖反射修改 `final` 字段的框架或测试工具需要关注兼容性。
- 不建议业务代码依赖修改 `final` 字段这种行为。
- 更强的 `final` 语义有利于 JVM 优化和代码可维护性。

不推荐的老写法示例：

```java
import java.lang.reflect.Field;

class User {
    private final String name = "Tom";
}

public class FinalReflectionDemo {
    public static void main(String[] args) throws Exception {
        User user = new User();

        Field field = User.class.getDeclaredField("name");
        field.setAccessible(true);
        field.set(user, "Jerry");
    }
}
```

这种代码破坏了 `final` 的语义，可能影响 JVM 优化，也会在后续 JDK 中面临更严格限制。业务代码应通过构造方法、工厂方法或 builder 在对象创建时确定 final 字段值。

## Java 26：Remove the Applet API

Java 26 移除 Applet API。Applet 早已退出主流浏览器和服务端开发场景，这属于平台历史包袱清理。

如果老项目仍然引用 `java.applet` 相关 API，需要迁移或删除相关代码。

老代码示例：

```java
import java.applet.Applet;

public class OldApplet extends Applet {
    @Override
    public void init() {
        System.out.println("init applet");
    }
}
```

在 Java 26 中，`java.applet` 已被移除，这类代码无法继续按原方式编译。迁移方向通常是删除 Applet 入口，改为普通桌面应用、Web 前端或服务端接口。

## Java 26：AOT 能力增强

Java 26 继续增强 Ahead-of-Time 相关能力，例如对象缓存和运行时镜像启动优化方向。

关注点：

- 改善启动速度。
- 降低冷启动成本。
- 对 CLI、Serverless、短生命周期服务更有价值。

AOT 缓存的典型流程是先训练，再创建缓存，最后使用缓存启动。示意命令如下：

```bash
java -XX:AOTMode=record -XX:AOTConfiguration=app.aotconf -jar app.jar
java -XX:AOTMode=create -XX:AOTConfiguration=app.aotconf -XX:AOTCache=app.aot -jar app.jar
java -XX:AOTCache=app.aot -jar app.jar
```

Java 26 的重点是让 AOT 对象缓存可以和任意 GC 更好地协作，例如 ZGC。需要显式生成可流式加载对象的 AOT 缓存时，可以关注类似参数：

```bash
java -XX:+AOTStreamableObjects -XX:AOTMode=create -XX:AOTConfiguration=app.aotconf -XX:AOTCache=app.aot -jar app.jar
```

AOT 适合启动敏感场景，但训练流量必须覆盖应用启动时真实会加载的类和路径，否则收益会下降。

## Java 26：G1 GC 改进

Java 26 继续优化 G1，通过减少同步开销改善吞吐。G1 仍是大多数服务端应用的默认和稳妥选择。

实践中应通过压测观察：

- 吞吐量。
- P99/P999 延迟。
- GC 暂停时间。
- CPU 和内存占用。

启动并观察 G1 日志：

```bash
java -XX:+UseG1GC -Xms4g -Xmx4g -Xlog:gc*,safepoint -jar app.jar
```

如果要设定暂停时间目标，可以使用：

```bash
java -XX:+UseG1GC -XX:MaxGCPauseMillis=100 -jar app.jar
```

`MaxGCPauseMillis` 是目标值，不是硬性保证。G1 会在吞吐、暂停和堆使用之间做权衡，最终效果仍以压测和线上监控为准。

## 版本关系与选型

| 版本 | 状态 | 重点 |
| --- | --- | --- |
| Java 22 | 非 LTS，已 GA | FFM API 标准化、未命名变量、多文件源码运行 |
| Java 23 | 非 LTS，已 GA | Markdown Javadoc、分代 ZGC 默认、模块导入预览 |
| Java 24 | 非 LTS，已 GA | Stream Gatherers、Class-File API、虚拟线程 pinning 改进 |
| Java 25 | LTS，已 GA | Scoped Values、紧凑源码、模块导入、构造方法增强、GC 改进 |
| Java 26 | 非 LTS，已 GA | HTTP/3、AOT 增强、G1 改进、Applet API 移除 |

## 面试回答模板

Java 22 到 26 的重点不是某一个单独语法，而是 Java 在平台现代化上的持续推进：FFM API 降低 JNI 替代成本，Stream Gatherers 扩展流处理能力，Scoped Values 改善虚拟线程上下文传递，Java 25 成为新的 LTS，Java 26 则继续补强 HTTP/3、AOT、G1 和平台清理。生产选型上，长期支持优先看 Java 21 或 Java 25，非 LTS 版本更适合学习、验证和跟进新能力。

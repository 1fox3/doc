# 线程池

> 专题：Java 语言基础
> 来源：面试内容汇总/Java.xmind

## 补充与实践

- 补充：JDK 21 的虚拟线程适合大量阻塞 IO 场景，但不适合 CPU 密集型任务，也不能消除共享状态同步问题。
- 实践：线程池参数应结合任务类型、队列长度、拒绝策略、监控指标和下游容量一起设计。
- 排障：线程问题优先看 `jstack`、线程池队列、活跃线程数、锁等待和上下文切换。

## 详细说明

### 核心参数

- `corePoolSize`：核心线程数。
- `maximumPoolSize`：最大线程数。
- `workQueue`：任务队列。
- `keepAliveTime`：非核心线程空闲存活时间。
- `RejectedExecutionHandler`：拒绝策略。

### 执行流程

- 线程数小于 core 时创建核心线程。
- 核心线程满后任务进入队列。
- 队列满后创建非核心线程，直到 maximum。
- 线程和队列都满时触发拒绝策略。

### 实践建议

- CPU 密集型任务线程数接近 CPU 核数。
- IO 密集型任务可适当增大线程数，但要受下游容量约束。
- 不要使用无界队列承接不可控流量，否则可能导致内存膨胀和延迟失控。

## 线程池

### ExecutorService是接口

### ThreadPoolExecutor是实现类

#### 核心参数

- corePoolSize
  - 核心线程数大小

- maximumPoolSize
  - 最大线程数大小

- keepAliveTime
  - 空余线程存活时间

- unit
  - 时间单位

- workQueue
  - 工作队列

- threadFactory
  - 线程工厂

- handler
  - 拒绝策略

#### 工作流程

- 线程数小于corePoolSize，创建新线程执行任务

- 线程数大于corePoolSize，任务暂时存储到workQueue

- workQueue满时，线程数小于maximumPoolSize，创建新线程执行

- 线程数达到maximumPoolSize时，执行拒绝策略

### 拒绝策略

#### AbortPolicy

- 直接抛异常

#### CallerRunsPolicy

- 在调用者线程中直接执行

#### DiscardPolicy

- 不处理新任务

#### DiscardOldestPolicy

- 抛弃最老的任务

### 内置线程池

#### FixedThreadPool

- 工作队列为LinkedBlockingQueue，最大为Integer最大值

#### SingleThreadExecutor

- 工作队列为LinkedBlockingQueue，最大为Integer最大值

#### CachedThreadPool

- 工作队列为SynchronousQueue直接提交队列

#### ScheduledThreadPool

- 对SingleThreadScheduledExecutor 的包装

- 使用了无界的延迟阻塞队列DelayedWorkQueue，最大长度为Integer最大值

#### WorkStealingPool

- 使用ForkJoinPool

### 执行线程

#### submit

- 有返回值，返回Future对象

- 通过Future抛出异常

- 支持Callable和Runnable

#### execute

- 没有返回值

- 无法获取具体异常，但出现异常会打印出来

- 仅支持Runnable

### 关闭

#### shutdown

- 不接受新任务，之前提交的任务处理完再关闭

#### shutdownNow

- 不接受新任务，视图停止正在执行的任务，放回所有未处理的线程列表

### CompletionService接口

#### ExecutorCompletionService实现类

- 底层结合ExecutorService和阻塞队列完成

#### 通过take方法可以取到任务的结果

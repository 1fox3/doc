# ThreadPoolExecutor 线程池：创建、提交、关闭与监控

## 概述

`ThreadPoolExecutor` 是 Java 并发框架中线程池的核心实现类，位于 `java.util.concurrent` 包。`ExecutorService` 是接口，`ThreadPoolExecutor` 是其底层实现。通过复用线程来降低频繁创建/销毁线程的开销，并提供任务队列、拒绝策略等流量控制机制。

`Executors` 工厂方法提供的内置线程池（FixedThreadPool、CachedThreadPool 等）本质上都是对 `ThreadPoolExecutor` 的封装，生产环境建议直接构造 `ThreadPoolExecutor` 并显式指定每个参数，避免使用默认无界队列或无上限线程数带来的内存风险。

## 核心概念

### 7 个核心参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `corePoolSize` | int | 核心线程数。线程池始终保持的线程数量，即使线程空闲也不会被回收（除非设置了 `allowCoreThreadTimeOut`） |
| `maximumPoolSize` | int | 最大线程数。当队列满时，允许创建的线程上限 |
| `keepAliveTime` | long | 非核心线程的最大空闲时间。超过此时间的空闲线程会被终止 |
| `unit` | TimeUnit | `keepAliveTime` 的时间单位 |
| `workQueue` | BlockingQueue | 任务等待队列。核心线程全忙时，新任务进入此队列 |
| `threadFactory` | ThreadFactory | 线程创建工厂，可用于设置线程名称、守护线程标志、优先级等 |
| `handler` | RejectedExecutionHandler | 拒绝策略。当线程数达到最大且队列已满时触发 |

### 工作流程

```
提交任务
  │
  ├─ 当前线程数 < corePoolSize ──→ 创建新核心线程执行任务
  │
  ├─ 当前线程数 >= corePoolSize
  │     │
  │     ├─ workQueue 未满 ──→ 任务入队等待
  │     │
  │     └─ workQueue 已满
  │           │
  │           ├─ 当前线程数 < maximumPoolSize ──→ 创建非核心线程执行任务
  │           │
  │           └─ 当前线程数 >= maximumPoolSize ──→ 触发拒绝策略
```

关键细节：
- 核心线程和非核心线程在代码层面没有区分，`ThreadPoolExecutor` 通过计数来判断是否超过 `corePoolSize`
- 队列选型直接影响线程扩展行为：`LinkedBlockingQueue`（有界/无界）、`SynchronousQueue`（直接交付，不缓存）、`ArrayBlockingQueue`（固定容量有界队列）、`DelayedWorkQueue`（定时任务）

### 拒绝策略

四种内置策略均实现 `RejectedExecutionHandler` 接口：

**AbortPolicy（默认）**  
直接抛出 `RejectedExecutionException`。调用方需要 catch 该异常并决定如何处理（重试、降级、告警等）。

**CallerRunsPolicy**  
由提交任务的线程（调用方线程）直接执行该任务。效果是让调用方线程阻塞，从而起到背压（back-pressure）作用，减缓任务提交速度。适合不能丢任务但也不需要严格隔离的场景。

**DiscardPolicy**  
静默丢弃新提交的任务，不抛异常也不通知。危险：任务丢失无任何日志，排查困难，除非业务允许丢失否则不建议使用。

**DiscardOldestPolicy**  
丢弃队列头部最老的任务，然后尝试重新提交当前任务。队列头部的任务往往等待最久，此策略优先保留新任务。

**自定义拒绝策略**  
实现 `RejectedExecutionHandler` 接口，可以在拒绝时打 metrics、写日志、持久化任务到数据库等。

### 内置线程池

| 工厂方法 | corePoolSize | maximumPoolSize | workQueue | 风险 |
|----------|-------------|-----------------|-----------|------|
| `FixedThreadPool(n)` | n | n | `LinkedBlockingQueue`（无界，容量 Integer.MAX_VALUE） | 任务积压导致 OOM |
| `SingleThreadExecutor` | 1 | 1 | `LinkedBlockingQueue`（无界） | 任务积压导致 OOM |
| `CachedThreadPool` | 0 | Integer.MAX_VALUE | `SynchronousQueue` | 线程数无上限，CPU/内存耗尽 |
| `ScheduledThreadPool(n)` | n | Integer.MAX_VALUE | `DelayedWorkQueue`（无界） | 延迟任务积压导致 OOM |
| `WorkStealingPool` | — | — | 使用 `ForkJoinPool`，work-stealing 算法 | 不保证任务执行顺序 |

**为什么生产环境不推荐直接用内置线程池：**  
`FixedThreadPool` 和 `SingleThreadExecutor` 使用无界 `LinkedBlockingQueue`，在任务生产速度远大于消费速度时，队列会无限增长，最终 OOM。`CachedThreadPool` 则会无限创建线程。

### 线程命名与 ThreadFactory

线程名是排查问题的基础。默认线程名如 `pool-1-thread-1`，无法定位业务含义。生产中应通过 `ThreadFactory` 指定业务前缀、是否守护线程和未捕获异常处理器。

```java
import java.util.concurrent.ThreadFactory;
import java.util.concurrent.atomic.AtomicInteger;

public class NamedThreadFactory implements ThreadFactory {
    private final String prefix;
    private final AtomicInteger counter = new AtomicInteger(1);

    public NamedThreadFactory(String prefix) {
        this.prefix = prefix;
    }

    @Override
    public Thread newThread(Runnable r) {
        Thread t = new Thread(r, prefix + "-" + counter.getAndIncrement());
        t.setDaemon(false);
        t.setUncaughtExceptionHandler((thread, e) ->
                System.err.println("Thread " + thread.getName() + " threw: " + e.getMessage()));
        return t;
    }
}
```

如果项目已引入 Guava，可使用 `ThreadFactoryBuilder` 简化命名：

```java
ThreadFactory factory = new ThreadFactoryBuilder()
        .setNameFormat("order-worker-%d")
        .setDaemon(false)
        .build();
```

### 线程池监控

`ThreadPoolExecutor` 暴露了运行时指标，应接入监控系统：

| 方法 | 含义 |
|---|---|
| `getPoolSize()` | 当前线程总数 |
| `getActiveCount()` | 正在执行任务的线程数 |
| `getQueue().size()` | 队列中等待的任务数 |
| `getTaskCount()` | 已提交任务总数 |
| `getCompletedTaskCount()` | 已完成任务总数 |

需要任务级耗时统计时，可以继承 `ThreadPoolExecutor` 并重写 `beforeExecute` / `afterExecute`。注意 `submit()` 的异常会被封装在 `Future` 中，`afterExecute` 的 `Throwable` 参数通常为 `null`，需要主动从 `Future.get()` 中提取异常。

### execute vs submit

| 对比点 | `execute(Runnable)` | `submit(Callable/Runnable)` |
|--------|--------------------|-----------------------------|
| 返回值 | void | `Future<T>` |
| 异常处理 | 异常由线程的 `UncaughtExceptionHandler` 处理，会打印堆栈 | 异常被封装在 `Future` 中，调用 `future.get()` 时才抛出 `ExecutionException` |
| 支持类型 | 仅 `Runnable` | `Runnable` 和 `Callable` |
| 阻塞等待结果 | 不支持 | 支持，通过 `future.get()` 或 `future.get(timeout, unit)` |

注意：使用 `submit` 时如果不调用 `future.get()`，任务内部的异常会被吞掉，导致任务静默失败。

### shutdown vs shutdownNow

| 方法 | 行为 |
|------|------|
| `shutdown()` | 停止接受新任务；等待已提交（队列中 + 执行中）的任务全部完成后关闭 |
| `shutdownNow()` | 停止接受新任务；对执行中的线程发送 `interrupt()`；返回队列中未开始的任务列表（`List<Runnable>`） |
| `awaitTermination(timeout, unit)` | 阻塞等待线程池完全终止，常与 `shutdown()` 配合使用 |

`shutdownNow()` 不能保证线程立即停止，只是发出中断信号，线程是否响应取决于任务代码是否检查中断状态（`Thread.currentThread().isInterrupted()` 或响应 `InterruptedException`）。

### isShutdown vs isTerminated

| 方法 | 含义 |
|---|---|
| `isShutdown()` | 已调用 `shutdown()` 或 `shutdownNow()`，线程池不再接收新任务 |
| `isTerminated()` | 所有任务结束、工作线程退出，线程池彻底终止 |

`isShutdown() == true` 不代表任务已执行完毕。判断线程池是否真正结束，应使用 `awaitTermination(timeout, unit)`，不要轮询 `isTerminated()`。

## 示例代码

```java
import java.util.concurrent.*;

public class ThreadPoolDemo {

    public static void main(String[] args) throws InterruptedException, ExecutionException {
        // 显式构造 ThreadPoolExecutor，避免使用 Executors 工厂方法的默认无界队列
        ThreadPoolExecutor executor = new ThreadPoolExecutor(
            2,                                      // corePoolSize：核心线程数
            4,                                      // maximumPoolSize：最大线程数
            60L,                                    // keepAliveTime：非核心线程空闲存活时间
            TimeUnit.SECONDS,                       // unit
            new ArrayBlockingQueue<>(10),           // workQueue：有界队列，容量为 10
            new ThreadFactory() {
                private int count = 0;
                @Override
                public Thread newThread(Runnable r) {
                    Thread t = new Thread(r, "pool-thread-" + (++count));
                    t.setDaemon(false);
                    return t;
                }
            },
            new ThreadPoolExecutor.CallerRunsPolicy() // 拒绝策略：调用方线程执行，起背压作用
        );

        // execute：无返回值，适合 fire-and-forget 场景
        executor.execute(() -> System.out.println("execute task on " + Thread.currentThread().getName()));

        // submit(Callable)：有返回值，异常封装在 Future 中
        Future<String> future = executor.submit(() -> {
            Thread.sleep(100);
            return "result from " + Thread.currentThread().getName();
        });

        // 必须调用 future.get() 才能感知任务内部的异常
        try {
            String result = future.get(5, TimeUnit.SECONDS);
            System.out.println("Got: " + result);
        } catch (TimeoutException e) {
            future.cancel(true);
            System.err.println("Task timed out");
        } catch (ExecutionException e) {
            System.err.println("Task threw exception: " + e.getCause());
        }

        // 优雅关闭：先 shutdown 停止接受新任务，再 awaitTermination 等待已有任务完成
        executor.shutdown();
        if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
            // 超时后强制停止
            executor.shutdownNow();
        }
    }
}
```

import java.util.concurrent.*;

// 自定义拒绝策略：记录日志 + 告警，不丢任务
public class LoggingRejectedHandler implements RejectedExecutionHandler {

    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        System.err.printf("[WARN] Task rejected. poolSize=%d, queueSize=%d, task=%s%n",
            executor.getPoolSize(),
            executor.getQueue().size(),
            r.toString()
        );
        // 可在此处写 metrics、持久化任务、或放入备用队列
        // 不抛异常则任务被静默丢弃，按需选择是否抛出
        throw new RejectedExecutionException("Task " + r + " rejected from " + executor);
    }
}

```java
import java.util.concurrent.*;

// ScheduledThreadPool 示例：固定频率执行 vs 固定延迟执行
public class ScheduledPoolDemo {

    public static void main(String[] args) {
        ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);

        // scheduleAtFixedRate：以固定周期执行（任务执行时间不计入间隔）
        // 若任务耗时超过 period，下次任务会在任务结束后立即执行
        scheduler.scheduleAtFixedRate(
            () -> System.out.println("Fixed rate at " + System.currentTimeMillis()),
            0L,     // initialDelay
            1L,     // period
            TimeUnit.SECONDS
        );

        // scheduleWithFixedDelay：任务结束后等待固定延迟再执行下次
        scheduler.scheduleWithFixedDelay(
            () -> System.out.println("Fixed delay at " + System.currentTimeMillis()),
            0L,     // initialDelay
            1L,     // delay
            TimeUnit.SECONDS
        );
    }
}
```

```java
import java.util.List;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.TimeUnit;

// 标准优雅停机模板：先优雅等待，超时再强制中断
public class GracefulShutdown {
    public static void shutdownGracefully(ExecutorService executor,
                                          long gracefulTimeout,
                                          long forceTimeout) {
        executor.shutdown();
        try {
            if (!executor.awaitTermination(gracefulTimeout, TimeUnit.SECONDS)) {
                List<Runnable> dropped = executor.shutdownNow();
                System.err.println("Dropped pending tasks: " + dropped.size());
                if (!executor.awaitTermination(forceTimeout, TimeUnit.SECONDS)) {
                    System.err.println("Executor did not terminate after shutdownNow");
                }
            }
        } catch (InterruptedException e) {
            executor.shutdownNow();
            Thread.currentThread().interrupt();
        }
    }
}
```

## 常见问题

**1. submit 吞掉异常导致任务静默失败**  
使用 `submit` 时，任务内部抛出的异常被封装在 `Future` 中。如果调用方从不调用 `future.get()`，异常永远不会被感知。解决方案：始终保留 `Future` 引用并在适当时机调用 `get()`，或在任务内部 catch 所有异常并记录日志。

**2. 使用 Executors.newFixedThreadPool / newCachedThreadPool 导致 OOM**  
`newFixedThreadPool` 的队列容量为 `Integer.MAX_VALUE`，任务积压时队列无限增长；`newCachedThreadPool` 的线程数上限为 `Integer.MAX_VALUE`，高并发时线程数暴增。生产环境应直接构造 `ThreadPoolExecutor` 并指定有界队列和合理的最大线程数。

**3. shutdownNow 不能立即停止线程**  
`shutdownNow()` 只是对执行中的线程调用 `interrupt()`，并不能强制终止。如果任务代码中没有检查中断状态（如死循环中不 sleep 也不检查 `isInterrupted()`），线程不会停止。任务代码应在关键循环点检查 `Thread.currentThread().isInterrupted()` 并提前退出。

**4. 线程数设置不当导致性能下降**  
- CPU 密集型任务：线程数 ≈ CPU 核数（`Runtime.getRuntime().availableProcessors()`），过多线程导致上下文切换开销
- IO 密集型任务：线程数可适当大于 CPU 核数，参考公式 `线程数 = CPU核数 × (1 + IO等待时间/CPU计算时间)`，但上限受下游服务（DB 连接池、HTTP 连接池）容量约束
- 无界队列 + 固定线程数在流量突增时无法通过扩线程来消化积压，所有新任务只能排队等待，尾延迟会急剧上升

**5. `CallerRunsPolicy` 可能阻塞调用方线程**  
`CallerRunsPolicy` 可以形成背压，但如果调用方是 Web 请求线程，会直接拉长接口响应时间甚至导致超时。关键链路可使用有界队列 + `AbortPolicy`，在上层捕获 `RejectedExecutionException` 后快速失败或降级。

**6. 忘记 `awaitTermination` 导致关闭不彻底**  
`shutdown()` 是异步的，调用后立即返回。必须配合 `awaitTermination()` 等待真正终止，超时后再调用 `shutdownNow()`，并处理其返回的未执行任务列表。

# 线程异常处理

## 概述

Java 线程中抛出的未捕获异常不会传播到调用线程，线程会静默终止。这一特性导致生产环境中任务失败极难被察觉。正确处理线程异常有四种主要手段：线程内部 `try-catch`、`Thread.UncaughtExceptionHandler`、`Future.get()` 捕获异步任务异常、以及重写 `ThreadPoolExecutor.afterExecute` 做统一兜底。

四种方式各有适用场景，通常在线程池环境中需要组合使用：`afterExecute` 负责兜底记录，`Future.get()` 负责业务层获取结果和异常，`UncaughtExceptionHandler` 负责处理非 `Future` 提交的原始 `Runnable` 任务异常。

## 核心概念

### try-catch

最直接的方式。在 `Runnable.run()` 或 `Callable.call()` 内部包裹 `try-catch`，可精确控制异常处理逻辑。

注意：只能捕获 `Exception`，`Error`（如 `OutOfMemoryError`）需单独捕获 `Throwable`。线程池任务如果不捕获异常，该线程会被终止并由池新建一个线程替换，异常本身被静默丢弃。

### Thread.UncaughtExceptionHandler

`Thread` 提供两级未捕获异常处理机制：

- **实例级**：`thread.setUncaughtExceptionHandler(handler)`，只对该线程生效。
- **全局默认级**：`Thread.setDefaultUncaughtExceptionHandler(handler)`，对所有未设置实例级 handler 的线程生效。

查找顺序：线程实例 handler → 线程组 handler → 全局默认 handler。

`ThreadFactory` 中统一设置 handler 是线程池场景下最常见的做法。

**重要限制**：`UncaughtExceptionHandler` 只对 `execute()` 提交的任务生效。通过 `submit()` 提交的任务，异常会被 `FutureTask` 内部捕获并封装，`UncaughtExceptionHandler` 不会被触发。

### Future.get()

`ExecutorService.submit()` 返回 `Future`，任务内部抛出的所有异常都会被包装为 `ExecutionException`，在调用 `future.get()` 时重新抛出。

```java
Future<?> future = executor.submit(() -> {
    throw new RuntimeException("task failed");
});
try {
    future.get(); // 这里才会抛出 ExecutionException
} catch (ExecutionException e) {
    Throwable cause = e.getCause(); // 获取原始异常
}
```

`future.get(timeout, unit)` 超时会抛出 `TimeoutException`，任务被取消会抛出 `CancellationException`。如果不调用 `get()`，异常永远不会被感知。

### ThreadPoolExecutor.afterExecute

`ThreadPoolExecutor` 提供钩子方法 `afterExecute(Runnable r, Throwable t)`，在每个任务执行完毕后被调用。

**关键陷阱**：通过 `submit()` 提交的任务，`afterExecute` 的第二个参数 `t` 始终为 `null`，即使任务抛出了异常。这是因为 `submit()` 将 `Callable` 包装成 `FutureTask`，`FutureTask` 内部 `catch` 了所有异常并保存在自身状态中。

要在 `afterExecute` 中获取 `submit()` 任务的异常，需要手动从 `FutureTask` 中提取：

```java
@Override
protected void afterExecute(Runnable r, Throwable t) {
    super.afterExecute(r, t);
    if (t == null && r instanceof Future<?>) {
        try {
            ((Future<?>) r).get();
        } catch (ExecutionException e) {
            t = e.getCause();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    if (t != null) {
        log.error("Task failed with exception", t);
    }
}
```

## 示例代码

```java
import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

public class ThreadExceptionDemo {

    // 1. try-catch：任务内部捕获，最基础
    static void demoTryCatch(ExecutorService executor) {
        executor.execute(() -> {
            try {
                riskyOperation();
            } catch (Throwable t) {
                System.err.println("[try-catch] caught: " + t.getMessage());
            }
        });
    }

    // 2. UncaughtExceptionHandler：捕获 execute() 提交的未捕获异常
    static ExecutorService buildExecutorWithHandler() {
        AtomicInteger count = new AtomicInteger();
        ThreadFactory factory = r -> {
            Thread t = new Thread(r, "worker-" + count.incrementAndGet());
            t.setUncaughtExceptionHandler((thread, ex) ->
                System.err.println("[UncaughtHandler] thread=" + thread.getName() + " ex=" + ex.getMessage())
            );
            return t;
        };
        return Executors.newFixedThreadPool(2, factory);
    }

    // 3. Future.get()：捕获 submit() 提交的任务异常
    static void demoFutureGet(ExecutorService executor) throws InterruptedException {
        Future<Integer> future = executor.submit(() -> {
            throw new IllegalStateException("compute failed");
        });
        try {
            future.get(1, TimeUnit.SECONDS);
        } catch (ExecutionException e) {
            System.err.println("[Future.get] original cause: " + e.getCause().getMessage());
        } catch (TimeoutException e) {
            future.cancel(true);
            System.err.println("[Future.get] timed out");
        }
    }

    // 4. afterExecute：线程池钩子统一兜底
    static ThreadPoolExecutor buildExecutorWithAfterExecute() {
        return new ThreadPoolExecutor(2, 4, 60, TimeUnit.SECONDS, new LinkedBlockingQueue<>()) {
            @Override
            protected void afterExecute(Runnable r, Throwable t) {
                super.afterExecute(r, t);
                // execute() 的异常直接通过 t 传入
                // submit() 的异常封装在 FutureTask 内，需手动提取
                if (t == null && r instanceof Future<?>) {
                    try {
                        ((Future<?>) r).get(0, TimeUnit.NANOSECONDS);
                    } catch (ExecutionException e) {
                        t = e.getCause();
                    } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                    } catch (TimeoutException ignored) {
                        // 任务还在运行，不处理
                    }
                }
                if (t != null) {
                    System.err.println("[afterExecute] task failed: " + t.getMessage());
                }
            }
        };
    }

    // 5. 全局默认 handler：兜底所有未设置 handler 的线程
    static void setGlobalHandler() {
        Thread.setDefaultUncaughtExceptionHandler((thread, ex) ->
            System.err.println("[GlobalHandler] thread=" + thread.getName() + " ex=" + ex)
        );
    }

    static void riskyOperation() {
        throw new RuntimeException("something went wrong");
    }

    public static void main(String[] args) throws Exception {
        setGlobalHandler();

        ExecutorService exec1 = buildExecutorWithHandler();
        demoTryCatch(exec1);
        exec1.execute(() -> { throw new RuntimeException("uncaught in execute"); }); // 触发 UncaughtHandler
        demoFutureGet(exec1);
        exec1.shutdown();

        ThreadPoolExecutor exec2 = buildExecutorWithAfterExecute();
        exec2.execute(() -> { throw new RuntimeException("execute task error"); });
        exec2.submit(() -> { throw new RuntimeException("submit task error"); });
        exec2.shutdown();
        exec2.awaitTermination(5, TimeUnit.SECONDS);
    }
}
```

## 常见问题

**1. submit() 任务异常被静默吞掉**

通过 `submit()` 提交任务后不调用 `future.get()`，任务异常永远不会被感知，日志里也看不到任何报错。线程池线程不会终止（与 `execute()` 不同），任务失败毫无痕迹。解决方式：要么调用 `get()`，要么在 `afterExecute` 中主动提取。

**2. UncaughtExceptionHandler 对 submit() 无效**

这是最容易踩的坑。通过 `ThreadFactory` 设置了 handler，但通过 `submit()` 提交的任务异常依然不会触发它。`FutureTask` 在 `run()` 内部已经 `catch (Throwable)` 并保存，不会再向外抛出。

**3. afterExecute 中的 t 参数对 submit() 始终为 null**

`afterExecute(r, t)` 的 `t` 只对 `execute()` 提交的任务携带实际异常。对 `submit()` 任务必须按上述代码模式手动调用 `((Future<?>) r).get()` 才能取到原因。

**4. Error 类异常不被 Exception 捕获**

`catch (Exception e)` 无法捕获 `OutOfMemoryError`、`StackOverflowError` 等 `Error` 子类。线程任务中应使用 `catch (Throwable t)` 做最外层兜底，并在捕获 `Error` 后考虑上报告警或重启服务，而不是简单记录后继续运行。

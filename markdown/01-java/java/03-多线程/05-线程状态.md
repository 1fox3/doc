# 线程状态

## 概述

Java 线程状态由 `Thread.State` 枚举定义，共六种：`NEW`、`RUNNABLE`、`BLOCKED`、`WAITING`、`TIMED_WAITING`、`TERMINATED`。这些状态由 JVM 层面维护，与操作系统的线程状态不是一一对应关系——例如 Java 的 `RUNNABLE` 同时涵盖了 OS 层的 ready 和 running 两种状态。

通过 `thread.getState()` 可在运行时查询线程状态，`jstack` 输出的线程快照也会显示每条线程当前所处的状态，是排查死锁、线程阻塞问题的主要手段。

## 核心概念

### NEW

线程对象已创建（`new Thread(...)`），但尚未调用 `start()`。此时 JVM 还没有为该线程分配任何系统资源，也未在 OS 层面创建对应的内核线程。一旦调用 `start()`，状态立即转变为 `RUNNABLE`。同一个 `Thread` 对象不能二次 `start()`，重复调用会抛出 `IllegalThreadStateException`。

### RUNNABLE（含 READY 和 RUNNING）

`RUNNABLE` 是一个宽泛的状态，对应 OS 层面的两种情况：
- **READY**：线程已具备运行条件，在就绪队列中等待 CPU 时间片调度。
- **RUNNING**：线程正在某个 CPU 核心上执行字节码。

Java 把两者合并为一个状态，原因是线程调度由 OS 完成，JVM 无法精确区分。值得注意的是，线程在执行 `native` 方法或进行非阻塞 I/O 时，JVM 层面同样显示为 `RUNNABLE`，即使该线程实际上正在等待 I/O 完成。

状态进入：`start()` 调用后；从 `BLOCKED`/`WAITING`/`TIMED_WAITING` 被唤醒并重新获得锁后。

### BLOCKED

线程在等待进入一个 `synchronized` 块/方法，而该锁当前被其他线程持有。只有 `synchronized` 关键字会产生此状态；`ReentrantLock.lock()` 阻塞时线程状态是 `WAITING`，不是 `BLOCKED`。

状态进入：试图进入已被占用的 `synchronized` 区域。  
状态退出：成功获取 monitor 锁，回到 `RUNNABLE`。

`jstack` 中 `BLOCKED` 线程后面通常跟着 `waiting to lock <0x...>` 和持锁线程信息，是排查死锁的直接线索。

### WAITING

线程进入无限期等待，必须由其他线程显式唤醒才能继续执行。

触发方法（调用后进入 WAITING）：
- `Object.wait()`：释放当前持有的 monitor 锁，等待 `notify()`/`notifyAll()`。
- `Thread.join()`：当前线程等待目标线程结束。
- `LockSupport.park()`：等待 `LockSupport.unpark(targetThread)` 唤醒。

调用 `Object.wait()` 必须在 `synchronized` 块内，否则抛 `IllegalMonitorStateException`。从 `WAITING` 被唤醒后，线程并不直接进入 `RUNNABLE`，而是先竞争锁（可能短暂进入 `BLOCKED`），拿到锁后才进入 `RUNNABLE`。

### TIMED_WAITING

有时限的等待，超时后自动转回 `RUNNABLE`（或 `BLOCKED`，如果还需要抢锁）。

触发方法：
- `Thread.sleep(millis)`：不释放锁，纯粹让出 CPU。
- `Object.wait(millis)`：释放 monitor 锁，超时或被唤醒后重新竞争锁。
- `Thread.join(millis)`：等待目标线程，超时后继续。
- `LockSupport.parkNanos(nanos)` / `parkUntil(deadline)`。
- `Condition.await(time, unit)`。

`Thread.sleep()` 和 `Object.wait()` 的关键区别：`sleep` 不释放锁，`wait` 释放锁。这是面试高频考点，也是实际编程中容易误用的地方。

### TERMINATED

线程执行完毕（`run()` 方法正常返回，或抛出未捕获异常导致退出）。处于此状态的线程不能再次启动，但线程对象本身仍存在于堆中，可以读取其属性。

## 状态转换总图

```
NEW ──start()──> RUNNABLE <──────────────────────────────────────────┐
                    │                                                 │
       ┌────────────┼──────────────────────────────┐                 │
       │            │                              │                 │
  synchronized   Object.wait()              Thread.sleep(ms)        │
  锁被占用        Thread.join()              Object.wait(ms)         │
       │         LockSupport.park()          LockSupport.parkNanos() │
       ▼            │                              │                 │
   BLOCKED       WAITING                    TIMED_WAITING            │
       │            │                              │                 │
  获取到锁      notify/notifyAll             超时 / 被唤醒            │
       └────────────┴──────────────────────────────┘                 │
                    └──────────────────────────────────────────────> ┘
                                                           run()结束
                                                              │
                                                         TERMINATED
```

## 示例代码

```java
import java.util.concurrent.locks.LockSupport;

public class ThreadStateDemo {

    private static final Object LOCK = new Object();

    public static void main(String[] args) throws InterruptedException {

        // 1. NEW 状态
        Thread t = new Thread(() -> {
            try {
                Thread.sleep(2000); // TIMED_WAITING
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }, "demo-thread");
        System.out.println("After new, before start: " + t.getState()); // NEW

        t.start();
        Thread.sleep(100); // 等 t 进入 sleep
        System.out.println("While sleeping: " + t.getState());          // TIMED_WAITING

        t.join();
        System.out.println("After run() returns: " + t.getState());     // TERMINATED

        // 2. BLOCKED vs WAITING 对比
        Object monitor = new Object();

        Thread holder = new Thread(() -> {
            synchronized (monitor) {
                try {
                    Thread.sleep(3000); // 持锁 sleep，t2 会 BLOCKED
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            }
        }, "lock-holder");

        Thread waiter = new Thread(() -> {
            synchronized (monitor) {
                // 拿到锁后立即结束
            }
        }, "lock-waiter");

        holder.start();
        Thread.sleep(100); // 确保 holder 先拿到锁

        waiter.start();
        Thread.sleep(100); // 等 waiter 尝试进入 synchronized
        System.out.println("waiter state (waiting for synchronized lock): " + waiter.getState()); // BLOCKED

        // 3. WAITING 状态（Object.wait）
        Thread waitThread = new Thread(() -> {
            synchronized (monitor) {
                try {
                    // 注意：此处 holder 仍持锁，这里仅演示写法
                    // 实际会先阻塞在 synchronized 外
                } catch (Exception e) {
                    Thread.currentThread().interrupt();
                }
            }
        });

        // LockSupport.park 演示 WAITING
        Thread parked = new Thread(() -> LockSupport.park(), "parked-thread");
        parked.start();
        Thread.sleep(100);
        System.out.println("parked thread state: " + parked.getState()); // WAITING
        LockSupport.unpark(parked);

        holder.join();
        waiter.join();
        parked.join();
    }
}
```

```java
// wait/notify 的正确用法模板（必须在 synchronized 块内）
public class WaitNotifyExample {

    private final Object lock = new Object();
    private boolean ready = false;

    public void producer() throws InterruptedException {
        synchronized (lock) {
            ready = true;
            lock.notifyAll(); // 唤醒所有在 lock 上 WAITING 的线程
        }
    }

    public void consumer() throws InterruptedException {
        synchronized (lock) {
            // 必须用 while，不能用 if，防止虚假唤醒（spurious wakeup）
            while (!ready) {
                lock.wait(); // 释放 lock，进入 WAITING
            }
            // 被唤醒后重新持有 lock，继续执行
            System.out.println("Condition met, proceeding.");
        }
    }
}
```

## 常见问题

### 1. sleep 不释放锁，wait 释放锁

`Thread.sleep()` 只让当前线程暂停执行，持有的 `synchronized` 锁不会释放。如果在持锁期间调用 `sleep()`，其他线程仍无法进入同一 `synchronized` 块，容易造成不必要的锁竞争。需要释放锁并等待条件时，应使用 `Object.wait()` 或 `Condition.await()`。

### 2. wait() 必须在循环中检查条件

被 `notify()` 唤醒后，条件不一定真正满足（可能是虚假唤醒，或其他线程已先消费了条件）。正确写法是 `while (!condition) { lock.wait(); }`，而不是 `if (!condition) { lock.wait(); }`。

### 3. BLOCKED 和 WAITING 在 jstack 中的区分

`BLOCKED` 线程在等待 `synchronized` 锁，输出含 `waiting to lock`；`WAITING` 线程通常因 `park()`/`wait()` 挂起，输出含 `waiting on` 或 `parking to wait for`。使用 `ReentrantLock` 时，阻塞的线程显示为 `WAITING`（`LockSupport.park` 内部实现），而非 `BLOCKED`，这与 `synchronized` 不同，排查时需注意。

### 4. interrupt() 对不同状态的效果不同

- 线程处于 `WAITING` 或 `TIMED_WAITING`（因 `sleep`/`wait`/`join` 进入）时，`interrupt()` 会抛出 `InterruptedException` 并清除中断标志。
- 线程处于 `BLOCKED`（等待 `synchronized` 锁）时，`interrupt()` 仅设置中断标志，线程继续阻塞，不会立即响应。
- 线程处于 `RUNNABLE` 时，`interrupt()` 只设置标志位，需要代码主动检查 `Thread.currentThread().isInterrupted()`。

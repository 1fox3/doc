# 线程异步

## 概述

Java 异步编程的核心目标是将耗时操作从主线程中剥离，避免阻塞，提高吞吐量。从早期的 `Future` 到 JDK 8 引入的 `CompletableFuture`，再到 Spring 生态的 `@Async`，异步编程模型持续演进。

生产中异步任务不只是"扔线程池就完事"，还必须处理线程池隔离、异常捕获、超时控制、上下文传递（如 MDC、TraceId）等问题，任何一个环节缺失都可能导致任务静默失败或雪崩。

## 核心概念

### Future 的局限性

`Future<V>` 是 JDK 5 引入的异步结果占位符，通过 `ExecutorService.submit()` 获得。主要问题：

- `get()` 是阻塞调用，调用线程会挂起直到任务完成，无法真正实现异步链式处理。
- 不支持回调，无法在任务完成时自动触发下一步逻辑。
- 多个 Future 之间无法编排（无法表达"A 和 B 都完成后执行 C"）。
- 异常只能通过 `get()` 抛出的 `ExecutionException` 感知，无法在任务内部传播回调链。
- `cancel()` 只是尝试中断，实际能否停止取决于任务本身是否响应中断。

`FutureTask<V>` 是 Future 的核心实现类，同时实现 `Runnable` 和 `Future`。它内部通过 `volatile int state` 管理状态，典型流转为 `NEW -> COMPLETING -> NORMAL`、`NEW -> COMPLETING -> EXCEPTIONAL`、`NEW -> CANCELLED` 或 `NEW -> INTERRUPTING -> INTERRUPTED`。任务完成后会通过 `LockSupport.unpark()` 唤醒等待 `get()` 的线程。

### CompletableFuture 异步编排

`CompletableFuture<T>` 实现了 `CompletionStage<T>` 接口，支持非阻塞的链式回调和多任务编排。

**常用操作分类：**

| 操作 | 方法 | 说明 |
|------|------|------|
| 创建 | `supplyAsync(supplier, executor)` | 有返回值的异步任务，必须指定线程池 |
| 创建 | `runAsync(runnable, executor)` | 无返回值的异步任务 |
| 转换 | `thenApply(fn)` | 对结果做映射，同步执行（在完成线程上） |
| 转换 | `thenApplyAsync(fn, executor)` | 对结果做映射，在指定线程池异步执行 |
| 消费 | `thenAccept(consumer)` | 消费结果，无返回值 |
| 组合 | `thenCompose(fn)` | 串联两个 CompletableFuture（flatMap 语义） |
| 合并 | `thenCombine(other, fn)` | 两个任务都完成后合并结果 |
| 并行等待 | `allOf(cf1, cf2, ...)` | 等待所有任务完成，返回 `CompletableFuture<Void>` |
| 竞速 | `anyOf(cf1, cf2, ...)` | 任意一个完成即返回 |
| 异常处理 | `exceptionally(fn)` | 捕获异常并返回默认值 |
| 异常处理 | `handle(fn)` | 无论成功失败都执行，可同时处理结果和异常 |
| 超时（JDK 9+） | `orTimeout(duration, unit)` | 超时后抛出 `TimeoutException` |
| 超时（JDK 9+） | `completeOnTimeout(value, duration, unit)` | 超时后返回默认值 |

**重要：不要使用默认的 `ForkJoinPool.commonPool()`**，生产中务必传入自定义线程池，否则所有业务共用同一个池，一个慢任务会拖垮全局。

`get()` 与 `join()` 的区别：`get()` 抛受检异常 `InterruptedException`、`ExecutionException`，适合边界处阻塞等待并设置超时；`join()` 抛非受检异常 `CompletionException`，适合在 `thenApply` 等回调内部取已经完成的子任务结果。

### Spring @Async

`@Async` 将方法调用委托给 Spring 管理的线程池异步执行，方法立即返回 `Future` 或 `CompletableFuture`，实际逻辑在后台线程执行。

**使用前提：**
1. 启动类或配置类上标注 `@EnableAsync`。
2. `@Async` 方法必须在 Spring Bean 上，且通过 Spring 代理调用（同类内部调用失效，因为绕过了代理）。
3. 方法必须是 `public`。

**自定义线程池：**

通过 `@Async("myExecutor")` 指定 Bean 名称，或实现 `AsyncConfigurer` 配置默认线程池。不配置时使用 `SimpleAsyncTaskExecutor`，该执行器每次都新建线程，没有复用，生产环境禁止使用。

**异常处理：**

`@Async` 方法如果返回 `void`，异常会被 Spring 的 `AsyncUncaughtExceptionHandler` 捕获，需要实现该接口并注册。如果返回 `Future`/`CompletableFuture`，异常会封装在返回值中，调用方通过 `get()` 或 `exceptionally()` 处理。

### 线程池隔离

不同业务类型的异步任务应使用不同的线程池，防止慢任务耗尽线程导致快任务也被阻塞：

- **IO 密集型任务**（数据库、RPC、文件读写）：线程数可设为 CPU 核数的 2~4 倍，因为大部分时间在等待 IO。
- **CPU 密集型任务**（计算、加密、压缩）：线程数设为 CPU 核数 + 1，过多反而增加上下文切换开销。
- **关键业务与非关键业务**：核心下单链路与发送营销短信应完全隔离，互不影响。

线程池参数设计要考虑：核心线程数、最大线程数、队列容量（无界队列会导致内存溢出）、拒绝策略（`CallerRunsPolicy` 适合需要背压的场景）。

### 异常处理

异步场景的异常极易被吞掉，常见的几种处理方式：

1. `CompletableFuture.exceptionally(e -> defaultValue)` — 捕获并返回默认值。
2. `CompletableFuture.handle((result, e) -> ...)` — 统一处理正常和异常分支。
3. `try-catch` 包裹 `future.get()` — 同步等待时捕获 `ExecutionException`。
4. Spring `AsyncUncaughtExceptionHandler` — 处理返回 void 的 `@Async` 方法抛出的异常。

**坑：** `allOf()` 组合多个任务时，若某个子任务异常，`allOf()` 本身也会以异常完成，但其他子任务异常不会自动汇聚，需要逐个检查。

### 超时控制

- JDK 9+ 推荐使用 `orTimeout()` 或 `completeOnTimeout()`，底层由 `ScheduledThreadPoolExecutor` 实现，无需额外阻塞线程。
- JDK 8 只能用 `future.get(timeout, TimeUnit.SECONDS)` 阻塞等待，超时抛 `TimeoutException`，此时仍需手动 `cancel()` 任务（但 cancel 不一定能停止正在执行的任务）。
- RPC 调用的超时通常由框架层（Feign、Dubbo）控制，不要依赖 CompletableFuture 超时来代替框架超时配置。

### allOf / anyOf 结果处理

`CompletableFuture.allOf()` 只返回 `CompletableFuture<Void>`，不携带子任务结果。正确做法是在 `allOf` 完成后，从原始子任务中逐个 `join()` 取值。若某个子任务失败，`join()` 会抛 `CompletionException`，需要在每个子任务上单独 `exceptionally`，或在聚合处统一处理。

## 示例代码

### Future 的阻塞问题演示

```java
ExecutorService executor = Executors.newFixedThreadPool(4);

Future<String> future = executor.submit(() -> {
    Thread.sleep(2000);
    return "result";
});

// 以下调用会阻塞当前线程 2 秒，失去异步意义
String result = future.get(); // 阻塞！
System.out.println(result);

executor.shutdown();
```

### CompletableFuture 链式编排

```java
// 必须显式传入线程池，不要依赖 commonPool
ExecutorService ioExecutor = new ThreadPoolExecutor(
    8, 16, 60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(200),
    new ThreadFactoryBuilder().setNameFormat("io-async-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()
);

// 串行编排：查询用户 -> 查询订单 -> 计算总价
CompletableFuture<Double> totalPrice = CompletableFuture
    .supplyAsync(() -> fetchUser(userId), ioExecutor)          // 异步查用户
    .thenApplyAsync(user -> fetchOrders(user.getId()), ioExecutor) // 异步查订单
    .thenApply(orders -> orders.stream()
        .mapToDouble(Order::getAmount).sum())                  // 同步计算总价
    .exceptionally(e -> {
        log.error("获取订单总价失败", e);
        return 0.0;  // 异常时返回默认值
    });

// 并行编排：同时查询库存和价格，都完成后组合
CompletableFuture<Integer> stockFuture =
    CompletableFuture.supplyAsync(() -> fetchStock(skuId), ioExecutor);
CompletableFuture<BigDecimal> priceFuture =
    CompletableFuture.supplyAsync(() -> fetchPrice(skuId), ioExecutor);

CompletableFuture<ProductVO> productVO = stockFuture.thenCombine(
    priceFuture,
    (stock, price) -> new ProductVO(skuId, stock, price)
);

// 批量并行，等待所有完成
List<CompletableFuture<Result>> futures = skuIds.stream()
    .map(id -> CompletableFuture.supplyAsync(() -> process(id), ioExecutor))
    .collect(Collectors.toList());

CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
    .thenRun(() -> System.out.println("所有任务完成"))
    .get(10, TimeUnit.SECONDS); // 整体超时控制

// 收集所有结果（注意异常处理）
List<Result> results = futures.stream()
    .map(f -> f.join()) // join() 不抛受检异常，但异常时抛 CompletionException
    .collect(Collectors.toList());
```

### 超时控制（JDK 9+）

```java
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> callRemoteService(), ioExecutor)
    .orTimeout(3, TimeUnit.SECONDS)              // 超时抛 TimeoutException
    .exceptionally(e -> {
        if (e instanceof TimeoutException) {
            return "timeout-fallback";           // 超时降级
        }
        throw new CompletionException(e);        // 其他异常继续抛出
    });

// 或者超时返回默认值（不抛异常）
CompletableFuture<String> withDefault = CompletableFuture
    .supplyAsync(() -> callRemoteService(), ioExecutor)
    .completeOnTimeout("default-value", 3, TimeUnit.SECONDS);
```

### Spring @Async 配置与使用

```java
// 配置类
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Bean("ioTaskExecutor")
    public Executor ioTaskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(8);
        executor.setMaxPoolSize(20);
        executor.setQueueCapacity(200);
        executor.setThreadNamePrefix("io-async-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }

    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        // 处理返回 void 的 @Async 方法的未捕获异常
        return (ex, method, params) ->
            log.error("@Async 方法异常: {}", method.getName(), ex);
    }
}

// 使用方
@Service
public class OrderService {

    // 指定线程池，返回 CompletableFuture 以便调用方处理异常和超时
    @Async("ioTaskExecutor")
    public CompletableFuture<Void> sendNotification(Long orderId) {
        try {
            notificationClient.send(orderId);
            return CompletableFuture.completedFuture(null);
        } catch (Exception e) {
            return CompletableFuture.failedFuture(e);
        }
    }
}
```

### 上下文传递（MDC TraceId）

```java
// 自定义线程池 Decorator，传递 MDC 上下文
public class MdcTaskDecorator implements TaskDecorator {
    @Override
    public Runnable decorate(Runnable runnable) {
        Map<String, String> contextMap = MDC.getCopyOfContextMap();
        return () -> {
            try {
                if (contextMap != null) {
                    MDC.setContextMap(contextMap);
                }
                runnable.run();
            } finally {
                MDC.clear();
            }
        };
    }
}

// 在 ThreadPoolTaskExecutor 中配置
executor.setTaskDecorator(new MdcTaskDecorator());
```

## 常见问题

**1. 使用默认 ForkJoinPool 导致线程饥饿**

`CompletableFuture.supplyAsync(fn)` 不传线程池时使用 `ForkJoinPool.commonPool()`，该池与并行流共享，线程数为 CPU 核数 - 1。IO 密集型任务会长期占用线程，导致其他异步操作排队。始终通过 `supplyAsync(fn, executor)` 传入业务专属线程池。

**2. 同类内部调用导致 @Async 失效**

Spring `@Async` 依赖 AOP 代理，同一个 Bean 内部方法调用（`this.asyncMethod()`）不经过代理，注解失效，任务在当前线程同步执行。解决方案：通过 `ApplicationContext.getBean()` 获取代理对象，或将异步方法抽取到另一个 Bean。

**3. CompletableFuture 异常被静默吞掉**

链式操作中如果没有 `exceptionally` 或 `handle`，异常会封装在 `CompletableFuture` 内部。如果没有人调用 `get()` 或 `join()`，异常永远不会被感知。对于不关心返回值的"即发即忘"任务，也要加 `exceptionally` 记录日志，否则排查问题时完全没有线索。

**4. allOf 后收集结果时 join 抛 CompletionException**

`allOf().get()` 只等待完成，不汇聚子任务的异常。之后逐个调用 `future.join()` 时，若某个子任务失败会抛出 `CompletionException`（包裹原始异常）。正确做法是在每个子 `CompletableFuture` 上单独加 `exceptionally` 处理，或在收集结果时统一 `try-catch CompletionException`。

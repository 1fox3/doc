# 安全地终止线程

## 概述

Java 没有提供任何机制来安全地强制终止线程。`Thread.stop()` 虽然存在，但已被废弃，因为它会在任意位置强制终止线程并释放所有锁，极易破坏共享对象的一致性。正确的做法是依赖**协作式终止**：请求线程停止，由线程自身在安全的检查点退出。

协作式终止有两种主要机制：一是使用 `volatile boolean` 标志位，线程主动检查标志并退出；二是利用 Java 内置的中断机制（`interrupt()`），这是更通用的方式，能与阻塞操作（如 `sleep`、`wait`、`BlockingQueue.take()`）配合使用。线程池则在这两种机制之上提供了 `shutdown()` 和 `shutdownNow()` 方法封装。

## 核心概念

### volatile boolean 停止标志

`volatile` 关键字保证对该变量的写操作对其他线程立即可见（通过禁止缓存和指令重排序）。线程在每次循环迭代时检查标志，看到 `true` 后自行退出。

优点：逻辑简单，适合无阻塞的计算型任务。
缺点：如果线程阻塞在 `sleep()`、`wait()` 或 I/O 操作上，标志的修改无法唤醒它，线程必须等待阻塞结束才能检查到标志变化。

```java
public class VolatileFlagTask implements Runnable {
    // 必须声明为 volatile，否则其他线程的修改可能对工作线程不可见
    private volatile boolean stopRequested = false;

    public void requestStop() {
        stopRequested = true;
    }

    @Override
    public void run() {
        while (!stopRequested) {
            // 执行工作...
            doWork();
        }
        // 到达这里时，线程已安全退出，不持有任何锁
        System.out.println("线程正常退出");
    }

    private void doWork() {
        // 模拟计算任务
    }
}
```

### interrupt() 协作终止

`Thread.interrupt()` 设置目标线程的中断状态位（一个内部布尔标志）。线程可以通过两种方式响应：

1. **阻塞方法自动响应**：`Thread.sleep()`、`Object.wait()`、`BlockingQueue.take()` 等方法在阻塞期间检测到中断后，会清除中断标志并抛出 `InterruptedException`。
2. **主动检查**：非阻塞循环中，线程调用 `Thread.currentThread().isInterrupted()` 检查中断标志。

关键规则：捕获 `InterruptedException` 后，要么重新抛出，要么重置中断标志（`Thread.currentThread().interrupt()`），否则中断信号会被吞掉，上层代码无法感知。

```java
public class InterruptTask implements Runnable {
    @Override
    public void run() {
        try {
            while (!Thread.currentThread().isInterrupted()) {
                doWork();
                // sleep 会在中断时立即抛出 InterruptedException
                // 并清除中断标志，所以循环条件中的 isInterrupted() 会返回 false
                // 因此需要在 catch 中恢复中断标志
                Thread.sleep(1000);
            }
        } catch (InterruptedException e) {
            // 不要 catch 后什么都不做！
            // 方式一：退出（适合顶层线程的 run 方法）
            System.out.println("收到中断信号，退出线程");
            // 方式二：重置中断标志（适合中间层方法，让上层决定如何处理）
            // Thread.currentThread().interrupt();
        }
    }

    private void doWork() throws InterruptedException {
        // 如果要将 InterruptedException 传播出去，方法签名必须声明
        System.out.println("working...");
    }
}
```

吞掉中断信号的错误示范：

```java
// 错误：中断信号被吞掉，线程无法被正常终止
public void badExample() {
    while (true) {
        try {
            Thread.sleep(1000);
        } catch (InterruptedException e) {
            // 什么都不做 —— 这会让 interrupt() 失效
        }
        doWork();
    }
}
```

### Thread.stop() 为何废弃

`Thread.stop()` 通过抛出 `ThreadDeath` 异常（`Error` 的子类）强制终止线程，并释放该线程持有的所有 `synchronized` 锁。问题在于：

- `ThreadDeath` 可以在任意字节码指令处抛出，包括正在修改共享对象的中途。
- 锁被强制释放后，其他线程立即可以访问该对象，此时对象处于**不一致的中间状态**（例如，转账操作扣款完成但还未入账就被 stop）。
- 由于 `ThreadDeath` 是 `Error` 而非 `Exception`，普通的 `catch (Exception e)` 不会捕获它，但 `finally` 块会执行，这反而可能掩盖问题。

```java
// Thread.stop() 导致数据不一致的场景演示
class BankAccount {
    private int balance = 1000;

    public synchronized void transfer(BankAccount target, int amount) {
        this.balance -= amount;          // 如果 stop() 在这行后发生
        // ---> 锁被强制释放，target.balance 还没增加
        // ---> 现在两个账户的总额已经不平衡了
        target.balance += amount;
    }
}
```

### 线程池 shutdown / shutdownNow

`ExecutorService` 提供了两个终止方法，语义不同：

| 方法 | 行为 | 适用场景 |
|------|------|----------|
| `shutdown()` | 不再接受新任务，等待已提交的任务（包括队列中的）全部执行完毕 | 优雅关闭，允许任务完成 |
| `shutdownNow()` | 不再接受新任务，尝试中断正在执行的任务，返回队列中未执行的任务列表 | 快速关闭，不等待任务完成 |

`shutdownNow()` 内部通过 `Thread.interrupt()` 中断工作线程，因此任务代码必须正确处理 `InterruptedException` 才能真正响应中断。仅调用 `shutdownNow()` 并不能保证线程立即停止。

标准的两阶段关闭模式（推荐）：

```java
public class ThreadPoolShutdownExample {

    public static void shutdownGracefully(ExecutorService executor) {
        // 第一阶段：优雅关闭，等待任务完成
        executor.shutdown();
        try {
            // 等待最多 60 秒让任务完成
            if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
                // 第二阶段：超时后强制中断
                executor.shutdownNow();
                // 再给一点时间让中断生效
                if (!executor.awaitTermination(10, TimeUnit.SECONDS)) {
                    System.err.println("线程池未能在规定时间内终止");
                }
            }
        } catch (InterruptedException e) {
            // 当前线程（调用 shutdownGracefully 的线程）被中断
            // 则再次发起强制关闭，并恢复中断标志
            executor.shutdownNow();
            Thread.currentThread().interrupt();
        }
    }

    public static void main(String[] args) {
        ExecutorService pool = Executors.newFixedThreadPool(4);

        // 提交能响应中断的任务
        pool.submit(() -> {
            try {
                while (!Thread.currentThread().isInterrupted()) {
                    System.out.println("任务执行中...");
                    Thread.sleep(500);
                }
            } catch (InterruptedException e) {
                System.out.println("任务被中断，正常退出");
                Thread.currentThread().interrupt();
            }
        });

        // 应用关闭时调用
        shutdownGracefully(pool);
    }
}
```

## 示例代码

结合 `volatile` 标志与 `interrupt()` 的完整示例，适用于既有阻塞操作又有普通循环的场景：

```java
public class CombinedStopExample {

    static class Worker implements Runnable {
        private volatile boolean stopRequested = false;
        private final BlockingQueue<String> queue;

        public Worker(BlockingQueue<String> queue) {
            this.queue = queue;
        }

        public void requestStop() {
            stopRequested = true;
            // 同时中断线程，以便从 queue.take() 的阻塞中唤醒
            Thread.currentThread().interrupt();
        }

        @Override
        public void run() {
            while (!stopRequested && !Thread.currentThread().isInterrupted()) {
                try {
                    // take() 在阻塞时可以响应 interrupt()
                    String item = queue.take();
                    process(item);
                } catch (InterruptedException e) {
                    // 恢复中断标志，让 while 条件检测到
                    Thread.currentThread().interrupt();
                }
            }
            System.out.println("Worker 正常退出");
        }

        private void process(String item) {
            System.out.println("处理: " + item);
        }
    }

    public static void main(String[] args) throws InterruptedException {
        BlockingQueue<String> queue = new LinkedBlockingQueue<>();
        Worker worker = new Worker(queue);
        Thread t = new Thread(worker);
        t.start();

        queue.put("task1");
        queue.put("task2");

        Thread.sleep(2000);

        // 外部请求停止：设置标志 + 中断
        worker.stopRequested = true;
        t.interrupt();
        t.join();
    }
}
```

## 常见问题

**1. 使用 `volatile boolean` 时线程迟迟不停止**

原因：线程阻塞在 `sleep()`、`wait()` 或 I/O 操作上，无法执行到标志检查的代码。
解决：对于有阻塞操作的线程，必须同时调用 `thread.interrupt()` 来唤醒阻塞，再结合标志检查。

**2. 调用 `shutdownNow()` 后任务没有真正停止**

原因：任务代码吞掉了 `InterruptedException`（`catch` 后什么都不做），或者任务是纯 CPU 计算循环，没有检查 `isInterrupted()`。
解决：任务代码必须正确传播或恢复中断标志；计算型任务需在循环中加入 `isInterrupted()` 检查点。

**3. 在框架代码（如 Spring 的 `@Scheduled`）中错误地吞掉中断**

原因：某些框架会捕获 `InterruptedException` 后继续调度，开发者自己的代码也没有恢复中断标志，导致整个调度链路的中断机制失效。
解决：在所有非顶层的 `catch (InterruptedException e)` 处，务必调用 `Thread.currentThread().interrupt()` 恢复标志。

**4. `awaitTermination()` 超时后未调用 `shutdownNow()`**

原因：开发者以为 `shutdown()` + `awaitTermination()` 就能保证线程一定停止，但如果任务长期不结束，线程池会一直存活，导致 JVM 无法正常退出。
解决：采用两阶段关闭模式，`awaitTermination()` 超时后必须跟进 `shutdownNow()`。

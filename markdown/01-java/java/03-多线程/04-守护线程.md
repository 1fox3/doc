# 守护线程（Daemon Thread）

## 概述

Java 线程分为两类：用户线程（User Thread）和守护线程（Daemon Thread）。用户线程是程序的主体逻辑线程；守护线程是在后台为用户线程提供服务的线程，其生命周期依附于用户线程。

当 JVM 中所有用户线程执行完毕后，无论守护线程处于何种状态，JVM 都会立即退出。守护线程不会阻止 JVM 的关闭。

JVM 自身启动的 GC 线程（垃圾回收线程）就是典型的守护线程。除此之外，JVM 内部的 Finalizer 线程、引用处理线程等也都是守护线程。

## 核心概念

### setDaemon(true)

通过 `Thread.setDaemon(true)` 将线程设置为守护线程。默认值为 `false`，即普通用户线程。

关键约束：**必须在 `thread.start()` 调用之前设置**，否则抛出 `IllegalThreadStateException`。

`Thread` 类内部通过 `boolean daemon` 成员变量记录该状态，子线程会继承父线程的守护状态——如果在守护线程中创建新线程，该新线程默认也是守护线程。

### 用户线程全部结束，JVM 退出

JVM 的退出条件是：所有非守护线程（用户线程）均已终止。JVM 不等待守护线程完成，直接强制终止所有仍在运行的守护线程并退出。

这意味着守护线程中执行到一半的逻辑可能在任意位置被中断，`finally` 块不保证执行，已分配的资源不保证释放。

### GC 线程是守护线程

JVM 启动时自动创建的垃圾回收线程（如 G1 的并发标记线程、CMS 的后台回收线程）全部是守护线程。这是合理的设计：GC 线程服务于用户线程，当没有用户线程时继续 GC 没有意义。

可以通过 `jstack <pid>` 查看线程列表，守护线程在输出中标记为 `daemon`：

```
"GC Thread#0" os_prio=31 cpu=0.23ms ... daemon
"G1 Main Marker" os_prio=31 cpu=0.05ms ... daemon
```

### 守护线程不适合做关键任务

由于 JVM 随时可能在守护线程执行期间退出，以下操作不应放在守护线程中：

- 数据库写入、事务提交
- 文件写入、流关闭
- 消息队列的消息确认（ACK）
- 任何需要"做完才能停"的业务逻辑

适合守护线程的场景：日志异步刷新（允许丢失最后几条）、心跳检测、内存缓存定时清理、监控指标采集等对完整性要求不高的后台任务。

## 示例代码

```java
public class DaemonThreadDemo {

    public static void main(String[] args) throws InterruptedException {
        Thread daemonThread = new Thread(() -> {
            int count = 0;
            while (true) {
                try {
                    Thread.sleep(500);
                    System.out.println("守护线程运行中... count=" + (++count));
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
            // 当 JVM 因用户线程结束而退出时，这行可能不会被执行
            System.out.println("守护线程正常结束");
        });

        // 必须在 start() 之前调用
        daemonThread.setDaemon(true);
        daemonThread.start();

        // 主线程（用户线程）只运行 2 秒
        Thread.sleep(2000);
        System.out.println("主线程结束，JVM 即将退出，守护线程被强制终止");
        // main 方法返回后，JVM 退出，守护线程无论执行到哪里都会被终止
    }
}
```

```java
public class DaemonInheritanceDemo {

    public static void main(String[] args) {
        Thread daemonParent = new Thread(() -> {
            // 在守护线程内创建的子线程，默认也是守护线程
            Thread child = new Thread(() -> {
                System.out.println("子线程 isDaemon: " + Thread.currentThread().isDaemon()); // true
            });
            child.start();
        });

        daemonParent.setDaemon(true);
        daemonParent.start();

        // 验证：在用户线程内创建的线程默认是用户线程
        Thread userChild = new Thread(() -> {
            System.out.println("用户子线程 isDaemon: " + Thread.currentThread().isDaemon()); // false
        });
        userChild.start();
    }
}
```

```java
public class DaemonSetAfterStartDemo {

    public static void main(String[] args) {
        Thread t = new Thread(() -> {});
        t.start();

        try {
            t.setDaemon(true); // 抛出 IllegalThreadStateException
        } catch (IllegalThreadStateException e) {
            System.out.println("错误：线程启动后不能设置 daemon 状态");
            // java.lang.IllegalThreadStateException
        }
    }
}
```

## 常见问题

**1. 在 start() 之后调用 setDaemon(true)**

这是最常见的错误，抛出 `IllegalThreadStateException`。正确做法是在构造线程后、启动前设置。使用线程池时无法直接控制线程创建，需通过自定义 `ThreadFactory` 来设置守护属性。

```java
ThreadFactory daemonFactory = r -> {
    Thread t = new Thread(r);
    t.setDaemon(true);
    return t;
};
ExecutorService executor = Executors.newFixedThreadPool(4, daemonFactory);
```

**2. 守护线程中的 finally 块不保证执行**

JVM 退出时强制终止守护线程，finally 块中的清理代码（如关闭文件句柄、提交事务）不会被执行。如果需要确保资源释放，应使用 `Runtime.addShutdownHook()` 注册关闭钩子（关闭钩子本身在独立线程中运行，与守护线程无关）。

**3. 守护线程继承父线程的 daemon 状态**

如果在守护线程中创建新线程而不显式设置，新线程默认也是守护线程。这在线程池场景中可能导致预期外的行为：用守护线程提交任务到线程池，池中线程继承 daemon=true，业务逻辑随 JVM 退出而中断。

**4. 误将关键持久化任务放入守护线程**

常见于日志异步写入或消息消费场景：将消费逻辑放入守护线程，主线程处理完即退出，导致最后一批消息未被 ACK 或未写入磁盘。关键任务必须使用用户线程，并在业务逻辑中明确处理退出信号（如 `CountDownLatch`、`shutdown hook`）。

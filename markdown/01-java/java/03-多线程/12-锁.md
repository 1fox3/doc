# Java锁分类与对比

## 概述

Java并发编程中，锁是协调多线程访问共享资源的核心机制。Java提供了两类锁：基于JVM内置实现的`synchronized`关键字，以及基于JDK `java.util.concurrent.locks`包的显式锁（`ReentrantLock`、`ReentrantReadWriteLock`、`StampedLock`）。

JDK 6之前`synchronized`性能较差，JDK 6引入锁升级机制（偏向锁 -> 轻量级锁 -> 重量级锁）大幅优化了竞争不激烈场景下的性能。显式锁则提供了更细粒度的控制，包括公平锁、条件变量、可中断等待、超时获取等能力。

选择依据：优先用`synchronized`（语义清晰、自动释放、JVM可优化）；需要公平锁、超时、中断、多条件队列时选`ReentrantLock`；读多写少场景选`ReentrantReadWriteLock`或`StampedLock`。

## 核心概念

### synchronized

**用法**

| 用法 | 锁对象 |
|------|--------|
| 修饰普通方法 | 当前实例（`this`） |
| 修饰静态方法 | 当前类的`Class`对象 |
| 同步块 `synchronized(this)` | 当前实例 |
| 同步块 `synchronized(obj)` | 指定对象实例 |
| 同步块 `synchronized(Foo.class)` | 类的`Class`对象 |

**可重入性**：同一线程可以重复获取同一把锁，内部通过`monitor`计数器实现，每次进入计数+1，退出计数-1，归零时释放。

**JDK 6锁优化**

- **锁消除（Lock Elimination）**：JIT编译器通过逃逸分析，对不可能存在共享竞争的对象上的锁进行消除。例如在方法内部对局部`StringBuffer`调用`append`，JVM会消除其内部的`synchronized`。
- **锁粗化（Lock Coarsening）**：连续对同一对象加锁解锁的操作会被合并为一个范围更大的锁，减少反复加锁解锁的开销。
- **自适应自旋（Adaptive Spinning）**：轻量级锁竞争失败后，线程不立即挂起而是自旋等待，自旋次数由JVM根据历史成功率动态调整。

**锁升级（单向，不可降级）**

```
无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁
```

| 状态 | 适用场景 | 实现方式 |
|------|----------|----------|
| 无锁 | 无竞争，无同步需求 | - |
| 偏向锁 | 只有一个线程进入同步块 | Mark Word记录线程ID，再次进入无需CAS |
| 轻量级锁 | 多线程交替执行，竞争不激烈 | CAS将对象Mark Word替换为锁记录指针 |
| 重量级锁 | 多线程真实竞争 | OS互斥量（Mutex），线程阻塞/唤醒涉及内核态切换 |

> 注意：JDK 15起偏向锁被标记为废弃（`-XX:-UseBiasedLocking`），JDK 18正式移除，因为维护复杂且现代多核场景收益有限。

---

### ReentrantLock

`ReentrantLock`是`java.util.concurrent.locks.Lock`的实现，基于AQS（AbstractQueuedSynchronizer）构建。

**核心特性**

- **可重入**：与`synchronized`相同，同一线程可重复加锁。
- **公平锁/非公平锁**：构造时传入`true`启用公平锁（按等待顺序获取），默认非公平锁（吞吐量更高）。
- **可中断等待**：`lockInterruptibly()`在等待锁时可响应中断，`synchronized`无法中断。
- **超时获取**：`tryLock(long timeout, TimeUnit unit)`，避免死锁等待。
- **Condition条件队列**：`newCondition()`创建多个等待集合，比`Object.wait/notify`更灵活。

**必须在`finally`中释放**：`unlock()`不会自动调用，异常时锁泄漏会导致死锁。

---

### ReentrantReadWriteLock

实现`ReadWriteLock`接口，内部维护一对锁：读锁（共享锁）和写锁（排他锁）。

**规则**

| 锁状态 | 读操作 | 写操作 |
|--------|--------|--------|
| 无锁 | 允许（多读并发） | 允许 |
| 有读锁 | 允许 | 阻塞 |
| 有写锁 | 阻塞 | 阻塞（非重入情况） |

**锁降级**：持有写锁的线程可以获取读锁，然后释放写锁，从而降级为读锁。反向（读锁升级为写锁）不支持，会死锁。

**适用场景**：读操作远多于写操作的场景（如缓存、配置中心）。写操作频繁时读写锁的开销可能超过`synchronized`。

---

### StampedLock

JDK 8引入，是`ReentrantReadWriteLock`的增强版，提供**乐观读**模式，显著减少读写冲突的开销。

**三种模式**

| 模式 | 方法 | 说明 |
|------|------|------|
| 写锁 | `writeLock()` | 排他锁，返回stamp |
| 悲观读锁 | `readLock()` | 共享锁，类似读写锁的读锁 |
| 乐观读 | `tryOptimisticRead()` | 不加锁，返回stamp，读完后用`validate(stamp)`验证是否有写入 |

**乐观读流程**：获取stamp -> 读数据到局部变量 -> `validate(stamp)` -> 若失败则升级为悲观读锁重新读。

**使用限制（重要）**

- **不可重入**：同一线程不能重复获取写锁，否则死锁。
- **不支持Condition**。
- **不能中断持有锁的线程**：中断会导致CPU 100%繁忙（busy-wait），必须避免对持有`StampedLock`的线程调用`interrupt()`。
- 不适合锁持有时间很长的场景。

---

### synchronized vs ReentrantLock 区别

| 维度 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 实现层 | JVM内置（monitorenter/monitorexit字节码） | JDK层（AQS） |
| 锁释放 | 自动（退出同步块/方法时） | 必须手动`unlock()`，需`finally`块 |
| 公平锁 | 不支持（非公平） | 支持（构造传`true`） |
| 可中断等待 | 不支持 | `lockInterruptibly()`支持 |
| 超时获取 | 不支持 | `tryLock(timeout)`支持 |
| 条件变量 | 单一等待集合（wait/notify） | 多个`Condition`，精确唤醒 |
| 锁粒度 | 方法或代码块 | 任意代码范围 |
| 性能 | JDK 6后差距缩小，低竞争下通常更优 | 高竞争、需要细粒度控制时更优 |
| 使用复杂度 | 简单 | 需要管理锁生命周期 |

---

### 锁优化手段

**减少锁持有时间**：只在真正需要保护共享数据的最小代码范围内加锁，避免在锁内执行耗时IO、RPC等操作。

**减少锁粒度**：将大对象拆分，每个子对象独立加锁。`ConcurrentHashMap`的分段锁（JDK 7分16段，JDK 8改为Node级别CAS）是经典案例。

**锁分离（Lock Splitting）**：读写分离，用`ReentrantReadWriteLock`替代单一互斥锁，读操作并发执行。

**锁粗化（Lock Coarsening）**：在循环内反复加锁解锁时，将锁范围扩展到循环外，减少加锁次数（JVM自动执行，也可手动优化）。

**锁消除（Lock Elimination）**：JIT逃逸分析确认对象不会逃逸时自动消除不必要的锁（JVM自动执行）。

**使用无锁数据结构**：`Atomic`类（`AtomicInteger`、`AtomicLong`、`AtomicReference`）基于CAS实现，无锁高并发；`LongAdder`比`AtomicLong`在高竞争下性能更好（分Cell分散竞争）。

## 示例代码

### synchronized基本用法与可重入

```java
public class SynchronizedDemo {

    private int count = 0;

    // 修饰实例方法，锁是 this
    public synchronized void increment() {
        count++;
    }

    // 修饰静态方法，锁是 SynchronizedDemo.class
    public static synchronized void staticMethod() {
        System.out.println("static synchronized");
    }

    // 同步块，锁是指定对象
    public void syncBlock() {
        synchronized (this) {
            count++;
        }
    }

    // 演示可重入：outer持有锁后可以直接调用inner（同一把锁）
    public synchronized void outer() {
        System.out.println("outer");
        inner(); // 不会死锁
    }

    public synchronized void inner() {
        System.out.println("inner");
    }
}
```

### ReentrantLock完整用法

```java
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantLock;
import java.util.concurrent.TimeUnit;

public class ReentrantLockDemo {

    // 公平锁：new ReentrantLock(true)
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    // 必须在finally中释放
    public void safeOperation() {
        lock.lock();
        try {
            // 临界区操作
        } finally {
            lock.unlock(); // 无论是否异常都释放
        }
    }

    // 可中断等待
    public void interruptibleLock() throws InterruptedException {
        lock.lockInterruptibly(); // 等待时可被interrupt()中断
        try {
            // 临界区
        } finally {
            lock.unlock();
        }
    }

    // 超时获取，避免死锁
    public boolean tryOperation() throws InterruptedException {
        if (lock.tryLock(500, TimeUnit.MILLISECONDS)) {
            try {
                // 临界区
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false; // 超时未获取到锁
    }

    // 多条件队列（生产者-消费者）
    private final Object[] items = new Object[10];
    private int putIndex, takeIndex, count2;

    public void put(Object item) throws InterruptedException {
        lock.lock();
        try {
            while (count2 == items.length) {
                notFull.await(); // 队列满，等待
            }
            items[putIndex] = item;
            putIndex = (putIndex + 1) % items.length;
            count2++;
            notEmpty.signal(); // 精确唤醒消费者
        } finally {
            lock.unlock();
        }
    }
}
```

### ReentrantReadWriteLock与锁降级

```java
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class CacheWithReadWriteLock {

    private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final ReentrantReadWriteLock.ReadLock readLock = rwLock.readLock();
    private final ReentrantReadWriteLock.WriteLock writeLock = rwLock.writeLock();

    private volatile Object cachedData;
    private volatile boolean cacheValid;

    public Object readData() {
        readLock.lock();
        try {
            if (cacheValid) {
                return cachedData; // 多线程并发读
            }
        } finally {
            readLock.unlock();
        }

        // 缓存失效，升级为写锁重新加载
        writeLock.lock();
        try {
            if (!cacheValid) { // double-check
                cachedData = loadFromDB();
                cacheValid = true;
            }
            // 锁降级：持有写锁时先获取读锁
            readLock.lock();
        } finally {
            writeLock.unlock(); // 释放写锁，降级为读锁
        }

        try {
            return cachedData;
        } finally {
            readLock.unlock();
        }
    }

    private Object loadFromDB() {
        return new Object(); // 模拟DB查询
    }
}
```

### StampedLock乐观读

```java
import java.util.concurrent.locks.StampedLock;

public class StampedLockDemo {

    private double x, y;
    private final StampedLock sl = new StampedLock();

    // 乐观读：不加锁，验证后使用数据
    public double distanceFromOrigin() {
        long stamp = sl.tryOptimisticRead(); // 获取乐观读stamp（不加锁）
        double currentX = x;
        double currentY = y;

        if (!sl.validate(stamp)) { // 检查读取期间是否有写操作
            // 乐观读失败，升级为悲观读锁
            stamp = sl.readLock();
            try {
                currentX = x;
                currentY = y;
            } finally {
                sl.unlockRead(stamp);
            }
        }
        return Math.sqrt(currentX * currentX + currentY * currentY);
    }

    public void move(double deltaX, double deltaY) {
        long stamp = sl.writeLock();
        try {
            x += deltaX;
            y += deltaY;
        } finally {
            sl.unlockWrite(stamp);
        }
    }
}
```

## 常见问题

**1. ReentrantLock忘记在finally中释放锁**

`lock.unlock()`如果不放在`finally`块，一旦临界区抛出异常，锁永远不会释放，导致其他线程永久阻塞（死锁）。正确模式必须是`lock.lock(); try { ... } finally { lock.unlock(); }`。

**2. synchronized锁对象变化导致失效**

`synchronized(obj)`依赖`obj`的引用，如果`obj`被重新赋值（如`obj = new Object()`），不同线程拿到的是不同锁对象，同步完全失效。应使用`final`修饰锁对象，或使用`synchronized`方法。

**3. StampedLock中断线程导致CPU 100%**

对持有`StampedLock`（无论读锁还是写锁）的线程调用`Thread.interrupt()`，会导致`StampedLock`内部自旋进入死循环，CPU飙升至100%。这是`StampedLock`的已知缺陷，使用时必须避免中断持有锁的线程。

**4. ReentrantReadWriteLock写锁饥饿**

在非公平模式下，持续的读请求可能导致写线程长期无法获取写锁（写锁饥饿）。解决方案：使用公平锁`new ReentrantReadWriteLock(true)`，或考虑`StampedLock`（其乐观读不阻塞写锁）。

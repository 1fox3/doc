#### 容错策略

- Failover Cluster（默认）
  - 失败重试,需配置重试次数
  - <dubbo:refreence retires="2"/>
  - <dubbo:method name=“sayHello” retires="2"/>
    - 针对具体方法

- Failfast Cluster
  - 快速失败
  - 仅调用一次

- Failsafe Cluster
  - 安全失败
  - 忽略调用时的异常

- Failback Cluster
  - 失败自动恢复
  - 调用失败后，在后台记录失败的请求，并按照一定的策略再进行重试

- Forking Cluster
  - 并行调用
  - 并行调用多个服务提供者，只要其中一个返回成功即可
  - forkes指定并行数

- Broadcast Cluster
  - 广播调用
  - 逐个调用所有的服务提供者，任意一台服务器调用异常则这次调用失败

- 自定义
  - 扩展Cluster接口

#### 负载均衡策略

- Random LoadBalance
  - 随机策略
  - 按照概率设置权重，可动态调节提供者的权重

- RoundRobin LoadBalance
  - 轮询策略

- LeastActive LoadBalance
  - 最小活跃调用数

- ConsistenHash LoadBalance
  - 一致性Hash策略
  - 保证相同参数的请求发到同一个提供者，如果某一台宕机了，将基于虚拟节点平摊给其他提供者

#### 线程模型

- 分类(根据消息类被I/O线程处理还是被业务线程池处理)
  - all（AllDispatcher）（默认）
    - 所有的消息都派发到业务线程池
  - direct（DirectDispatcher）
    - 所有消息都不派发到业务线程池
  - message (MessageOnlyDispatcher)
    - 只有请求响应消息派发到业务线程池
  - execution（ExecutionDispathcer）
    - 只把请求类消息派发到业务线程池
  - connection（ConnectionOrderedDispatcher）
    - 在I/O线程池上将连接事件，断开事件放入队列，有序逐个执行，其他消息配发到业务线程池
    - connect.queue.capacity指定线程池队列容量

- 线程模型确定时机
  - 服务提供方启动Nettyserver监听消费方链接时通过ChannelHanlers.wrap方法确定具体的线程模型

- 自定义线程模型
  - 实现Dispatcher接口
  - 实现自定义的ChannelHandler类
  - 在服务启动时试着线程模型

#### 线程池

- 分类
  - FixedThreadPool(默认)
    - 固定个数的线程池
    - 使用ThreadPoolExecutor创建
  - LimitedThreadPool
    - 线程池中的线程个数随着需求量动态增加，但不会超过配置的阈值，线程空闲也不会回收
    - 使用ThreadPoolExecutor创建
  - EagerThreadPool
    - 所有核心线程处于忙碌状态时，将创建新的线程来执行新任务，而不是将任务放入阻塞队列中
    - 与ThreadPoolExecutor不同
      - 核心线程数达到阈值后，会开启新线程处理任务，当线程数达到最大数时，才会把任务放入线程池队列
  - CachedThreadPool
    - 自适应线程池，线程空闲1分钟时会被回收，有新请求到来时会创建新请求
    - 使用ThreadPoolExecutor创建

- 线程池确定时机
  - 确定线程模型时通过ChannelHandler指定线程池模型

- 自定义线程池
  - 实现扩展接口ThreadPool
  - 增加接口配置（/META-INF/dubbo）
  - 服务提供者启动时指定线程池策略

#### 框架内核

- 分层架构概述
  - 分层说明
    - Service
    - Config
    - Proxy
    - Registry
    - Cluster
    - Montior
    - Protocol
    - Exchange
    - Transport
    - Serialize
  - 扩展点
    - 除了Service和Config两层外，其他层都可以通过扩展点来提供服务，而且仅当使用时才会被实例化
    - 使用扩展点
      - /META-INF/dubbo/接口的文本文件
      - myprotocol=com.alibaba.user.MyProtocol
      - <dubbo:protocol name="myprotocol"/>

- 远程调用
  - 基础过程
    - 服务提供者暴露服务
      - ServiceConfig类引用实现类
      - ProxyFactory.getInvoker()方法生成一个AbstractProxyInvoker实例
      - DubboProtocol类的expory方法时Invoker转为Exporter
    - 服务消费者消费服务
      - ReferenceConfig实例的init方法调用Protocol扩展接口实现类的refer方法生成Invoker实例
      - Invoker转换为客户端需要的接口
        - 发生在ProxyFactory接口的扩展实现类的getProxy方法中，主要使用代理对服务接口的调用转换为对Invoker的调用

- 适配器原理
  - 为每个SPI扩展接口动态生成一个对应的适配器类，并根据参数使用增强SPI以选择不同的SPI实现

- 动态编译原理
  - Compiler
    - JavassistCompiler(默认)
    - JdkCompiler

- 增强SPI
  - JDK标准SPI
    - 面向接口编程，服务规则提供者会在JRE的核心API里提供服务访问接口，而具体实现有其他开发商提供
  - Dubbo的增强SPI优点
    - 仅当需要时才加载类
    - 加载扩展点失败是，会通知异常
    - 增加了对扩展点的IoC和AOP支持

- 使用JavaAssit减少反射调用开销
  - Dubbo会给每个服务提供者的实现类生产一个Wrapper类，这个Wrapper类里面最终调用服务提供者的接口实现，Wrapper类的存在时为了减少反射的调用
  - 原理
    - 直接调用实现类的具体方法
  - 实现原理
    - Proxy层的扩展接口为ProxyFactory，其实现类主要有JavassistProxyFactory(默认使用)和JDKProxyFactory，其中JavassistProxyFactory为每个服务提供者实现类生成了Wrapper类

#### 基础使用

- 服务发布与调用
  - 发布
    - 构建ServiceConfig实例
    - 指定注册中心地址
    - 设置接口与实现类
    - 设置服务分组和版本
    - 设置线程池策略
    - 导出服务
    - 挂起线程，避免服务停止
  - 同步调用
    - 创建ReferenceConfig实例
    - 设置应用程序信息
    - 设置服务注册中心地址
    - 设置服务接口和超时时间
    - 设置自定义负载均衡策略与集群容错策略
    - 设置服务分组与版本
    - 引用服务
    - 设置隐式参数
    - 调用服务
  - 异步调用
    - 调用ReferenceConfig实例的setAsync(true)
    - 接口调用直接返回null
    - Future
      - RpcContext.getContext().getFuture()获取future对象
      - future.setCallback设置回调函数ResponseCallBack
        - done
          - 正常返回响应结果
        - caught
          - 调用出现错误
    - CompletableFuture
      - RpcContext.getContext().getCompletableFuture()获取Completablefuture对象
      - completableFuture.whenComplete

- 服务端异步执行
  - CompletableFuture.supplyAsync()

- 使用AsyncContext异步执行
  - RpcContext.startAsync()
  - asyncContext.signalContextSwitch()

- 服务消费端泛化调用
  - 泛化接口调用方式主要在服务消费端没有API接口及模型类元(比如入参和出参的POJO类)的情况下使用，其参数及返回值中没有对应的POJO类，所以所有的POJO参数均转换为Map表示。使用泛化调用时，服务消费块不在需要引入SDK二访包
  - 根据序列化方式不同，分为如下方式
    - true(generic=true)
      - 泛化参数固定为GenericService
        - ReferenceConfig<GenericService>
      - 设置泛化类型为true
        - referenceConfig.setGeneric(true)
      - 获取引用
        - 注意泛化值类型为GenericService
    - bean(generic=bean)
      - JavaBeanSerializeUtil.serialize把参数序列化
      - JavaBeanSerializeUtil.deserialize把结果反序列化
    - naticejava(generic=nativejava)
      - UnsafeByteArrayOutputStream
      - ExtensionLoader
      - 将参数序列化

- mock
  - 消费端本地mock
    - 首先实现服务接口的mock实现类，类名必须为接口包名.类名MOCK的形式
    - 开启mock
      - referenceConfig.setCheck(false)
      - referenceConfig.setMock(true)
    - 执行mock服务实现类mock()方法之前，会发起远程调用，当全程调用失败(服务不存在时)，才会降级执行mock功能
  - 服务降级
    - force
      - 直接返回客户端内设置的mock值
    - fail
      - 仅当调用服务提供方失败后返回mock值，否者返回正常结果

- 隐式参数传递
  - RpcContext.getContext().setAttachment(key, val)

- 本地服务暴露与引用
  - 在同一个JVM进程中，同时发布与调用同一个服务
  - referenceConfig.setCheck(false)

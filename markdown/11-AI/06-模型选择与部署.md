# 模型选择与部署

## 闭源模型与开源模型

### 闭源模型

代表：GPT、Claude、Gemini。

优点：

1. 能力强。
2. 接入简单。
3. 多模态和工具调用能力成熟。
4. 运维成本低。

缺点：

1. 数据合规和隐私风险需要评估。
2. 成本随调用量增长。
3. 受限于厂商 SLA 和模型版本变化。
4. 私有化控制弱。

### 开源模型

代表：Llama、Qwen、DeepSeek、Mistral。

优点：

1. 可私有化部署。
2. 数据不出内网。
3. 可做微调和定制。
4. 长期成本可能更可控。

缺点：

1. 部署和运维复杂。
2. 需要 GPU 资源。
3. 能力、稳定性和生态可能和顶级闭源模型有差距。
4. 需要处理扩缩容、监控和推理优化。

## 模型选择维度

1. 任务类型：问答、总结、代码、抽取、推理、多模态。
2. 效果：准确率、稳定性、幻觉率。
3. 延迟：首 Token 时间和总生成时间。
4. 成本：输入输出 Token 单价或 GPU 成本。
5. 上下文长度：是否需要处理长文档。
6. 工具调用：是否支持 function calling 或 JSON mode。
7. 合规：数据是否能出公网。
8. 可运维性：监控、限流、降级、版本管理。

## 常见模型类型

### 通用对话模型

适合问答、总结、改写、推理。

### Embedding 模型

用于把文本转换成向量，支撑语义检索。

选择时关注：

1. 中文效果。
2. 长文本支持。
3. 向量维度。
4. 检索评估集表现。

### Reranker 模型

用于重排检索结果，提高上下文质量。

### 代码模型

适合代码生成、代码解释、单测生成、代码审查。

### 多模态模型

处理图片、音频、视频和文本组合任务。

## 部署方式

### API 调用

适合快速上线和能力验证。

关注点：

1. API key 管理。
2. 网络稳定性。
3. 限流和重试。
4. 数据合规。
5. 成本监控。

### 私有化部署

适合数据敏感、调用量大、定制化强的企业场景。

需要考虑：

1. GPU 型号和显存。
2. 模型大小。
3. 并发和吞吐。
4. 推理框架。
5. 量化方案。
6. 监控和弹性伸缩。

## 推理框架

### vLLM

高吞吐 LLM 推理框架，常用于服务化部署。

特点：

1. PagedAttention 提高显存利用率。
2. 支持连续批处理。
3. 适合高并发推理服务。

### Ollama

适合本地开发和轻量部署。

特点：

1. 使用简单。
2. 适合本地验证。
3. 不一定适合大规模生产高并发。

### TensorRT-LLM

偏高性能推理优化，适合 NVIDIA GPU 场景。

## 量化

量化是把模型权重从高精度转换成低精度，降低显存和计算成本。

常见精度：

1. FP16/BF16。
2. INT8。
3. INT4。

收益：

1. 降低显存占用。
2. 提高部署可行性。
3. 降低推理成本。

风险：

1. 可能损失效果。
2. 对复杂推理和代码任务影响更明显。
3. 需要用业务评估集验证。

## 面试常问

### 如何选择模型？

我会先明确任务类型和约束，再用业务评估集对比效果、延迟和成本。简单分类抽取可以用小模型，复杂推理和生成用强模型。数据敏感场景优先私有化或合规模型，普通业务可以先用 API 快速验证。

### 什么时候需要私有化部署？

当数据不能出内网、调用量大导致 API 成本过高、需要模型深度定制，或者企业对可控性和稳定性要求很高时，可以考虑私有化部署。但要评估 GPU 成本、运维复杂度和模型效果。

### 量化会带来什么影响？

量化能降低显存和成本，但可能损失模型效果，尤其是复杂推理、代码和长上下文任务。上线前必须用真实业务评估集验证，而不是只看通用 benchmark。

# AI 应用工程架构

## 典型架构

```text
前端 -> API 网关 -> AI 应用服务 -> 模型网关 -> LLM Provider
                         |
                         +-> RAG 服务
                         +-> 工具服务
                         +-> 会话存储
                         +-> 日志与评估
```

核心目标：

1. 把模型能力封装成稳定服务。
2. 统一处理鉴权、限流、审计和成本。
3. 让业务系统不直接依赖某个模型厂商。
4. 为 Prompt、模型和检索策略迭代留下空间。

## 前后端集成

### 普通请求

适合短文本、非实时体验要求的任务，例如标题生成、分类、字段抽取。

### 流式输出

适合长答案和聊天场景，用户可以边生成边阅读。

常用技术：

1. SSE。
2. WebSocket。
3. HTTP chunked response。

工程注意：

1. 支持客户端中断。
2. 服务端要能取消模型请求或停止转发。
3. 前端要处理 Markdown 渲染、代码块、引用来源和错误提示。

## 会话管理

会话不是简单把所有历史消息塞给模型。

常见策略：

1. 最近 N 轮对话。
2. 历史摘要压缩。
3. 重要事实提取。
4. 按主题分段记忆。
5. RAG 方式检索历史。

需要避免：

1. 上下文无限增长。
2. 用户敏感信息长期保留。
3. 旧上下文污染新问题。
4. 不同用户上下文串扰。

## 模型网关

模型网关用于统一接入多个模型供应商。

能力：

1. Provider 路由。
2. API key 管理。
3. 超时和重试。
4. 失败降级。
5. Token 统计。
6. 成本统计。
7. Prompt 和响应日志。

好处：

1. 降低业务和模型厂商耦合。
2. 方便 A/B Test。
3. 方便按任务选择不同模型。
4. 统一合规和审计。

## 异步任务

长任务不要阻塞 HTTP 请求。

适合异步的任务：

1. 长文档总结。
2. 批量文档抽取。
3. 代码仓库分析。
4. 多步骤 Agent。

架构方式：

1. API 创建任务，返回 taskId。
2. 后台 Worker 消费队列。
3. 前端轮询或订阅任务进度。
4. 结果写入数据库或对象存储。

## 限流与重试

需要限制：

1. 用户级 QPS。
2. 租户级 Token 配额。
3. 模型 Provider 级并发。
4. 高成本任务调用次数。

重试策略：

1. 网络错误可以重试。
2. 429 需要退避。
3. 结构化输出解析失败可带错误信息重试。
4. 不要对所有错误无脑重试，避免成本放大。

## 成本控制

主要手段：

1. 模型分层：简单任务用小模型，复杂任务用强模型。
2. Prompt 压缩：减少无关上下文。
3. RAG 控制 topK：不要塞过多 chunk。
4. 缓存：相同问题、相同文档摘要可以缓存。
5. 批处理：批量抽取合并请求。
6. 限额：按用户、租户、功能设置预算。
7. 监控：统计 Token、延迟、失败率、单次成本。

## 监控与日志

需要记录：

1. 请求 ID。
2. 用户和租户。
3. 模型名称。
4. Prompt 版本。
5. 输入输出 Token。
6. 延迟和错误码。
7. 检索命中文档。
8. 工具调用轨迹。

敏感信息要脱敏，不能把用户隐私和密钥直接写入日志。

## 面试常问

### 如何设计一个企业 AI 应用的后端架构？

我会把 AI 能力封装在独立应用服务中，通过模型网关统一接入不同模型，通过 RAG 服务接入企业知识，通过工具服务访问内部系统。同时在网关和服务层处理鉴权、限流、审计、成本统计、Prompt 版本管理和失败降级。

### 为什么需要模型网关？

模型网关可以降低厂商绑定，统一鉴权、限流、重试、日志、成本统计和模型路由。业务代码只依赖内部统一接口，不直接依赖 OpenAI、Claude 或本地模型 API。

### 如何处理 AI 接口慢的问题？

可以使用流式输出改善体感延迟，对长任务异步化，减少上下文 Token，使用更小模型或缓存结果，并对模型调用设置超时、降级和并发控制。

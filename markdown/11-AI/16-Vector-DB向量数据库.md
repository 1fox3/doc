# Vector DB 向量数据库

## Vector DB 是什么

Vector DB 是专门用于存储、索引和检索向量数据的数据库。它通常把文本、图片、音频、代码等非结构化数据先通过 Embedding 模型转换成高维向量，再基于向量相似度找到语义相近的数据。

在 AI 应用里，Vector DB 最常见的用途是 RAG：把企业文档切分成 chunk，计算 embedding 后写入向量数据库，用户提问时再检索相关 chunk 交给大模型生成答案。

典型流程：

```text
原始数据 -> 清洗 -> 切分 -> Embedding -> Vector DB
用户 Query -> Embedding -> 向量检索 -> Metadata 过滤 -> Rerank -> LLM
```

## 为什么需要 Vector DB

传统数据库擅长精确查询，例如按 ID、状态、时间范围、关键词查询。Vector DB 擅长语义相似度检索，例如“报销流程怎么走”和“费用申请审批步骤”虽然词不同，但语义相近。

核心价值：

1. 支持语义搜索，不依赖完全相同的关键词。
2. 支持海量向量的近似最近邻检索。
3. 支持 metadata 过滤，例如租户、部门、文档类型、权限、时间。
4. 支持增量写入、更新、删除和重建索引。
5. 为 RAG、推荐、去重、聚类、异常检测提供底层检索能力。

## 核心概念

### Embedding

Embedding 是把文本、图片、代码等数据映射成向量的过程。相似内容的向量距离更近，不相似内容的向量距离更远。

注意点：

1. 不同 Embedding 模型的向量维度不同，例如 384、768、1024、1536、3072。
2. 入库和查询必须使用同一个或兼容的 Embedding 模型。
3. 更换 Embedding 模型通常需要重新计算全部向量并重建索引。
4. 多语言、代码、行业术语场景要选择匹配的 Embedding 模型。

### 向量相似度

常见相似度计算方式：

1. Cosine Similarity：看方向是否接近，文本语义检索常见。
2. Dot Product：点积，适合部分归一化或模型指定的场景。
3. Euclidean Distance：欧氏距离，看空间距离。

工程上不要只凭直觉选择距离函数，应该参考 Embedding 模型说明，并用真实 query 做离线评估。

### ANN 近似最近邻

高维向量如果暴力遍历计算相似度，数据量大时成本很高。Vector DB 通常使用 ANN（Approximate Nearest Neighbor，近似最近邻）算法，用少量精度损失换取更高检索速度。

常见索引：

1. HNSW：图索引，召回质量好，查询快，内存占用较高。
2. IVF：倒排文件索引，适合大规模数据，需要训练聚类中心。
3. PQ：向量压缩，降低存储和内存成本，但可能损失精度。
4. DiskANN：偏磁盘索引，适合更大规模和更低内存预算。

### Metadata

Metadata 是向量对应的结构化属性，例如：

1. `doc_id`：原始文档 ID。
2. `chunk_id`：切分片段 ID。
3. `tenant_id`：租户 ID。
4. `department`：部门。
5. `source_url`：来源链接。
6. `created_at` / `updated_at`：创建和更新时间。
7. `permission_tags`：权限标签。

RAG 系统里，metadata 很重要。很多线上问题不是“向量搜不到”，而是缺少租户、权限、时间、文档类型等过滤条件，导致召回结果不安全或不准确。

## Vector DB 和传统数据库的区别

| 维度 | 传统数据库 | Vector DB |
| --- | --- | --- |
| 查询方式 | 精确匹配、范围查询、聚合 | 相似度检索、近似最近邻 |
| 数据类型 | 结构化数据为主 | 非结构化数据的向量表示 |
| 典型索引 | B-Tree、Hash、倒排索引 | HNSW、IVF、PQ、DiskANN |
| 结果含义 | 满足条件的数据 | 与 query 最相似的数据 |
| 常见场景 | 订单、用户、交易、日志 | RAG、语义搜索、推荐、去重 |

两者不是替代关系。实际系统里通常是关系型数据库保存业务主数据，Object Storage 保存原文文件，Vector DB 保存 chunk 向量和检索索引。

## 常见产品

常见向量数据库和检索组件：

1. Milvus / Zilliz：开源向量数据库，适合大规模向量检索。
2. Pinecone：托管式向量数据库，运维成本低。
3. Weaviate：支持向量检索、GraphQL、混合检索。
4. Qdrant：Rust 实现，过滤能力和部署体验较好。
5. Chroma：轻量，常用于本地开发和原型验证。
6. FAISS：向量检索库，不是完整数据库，适合本地或自建检索服务。
7. Elasticsearch / OpenSearch：支持 BM25、向量检索和混合检索。
8. PostgreSQL + pgvector：适合中小规模、已有 PostgreSQL 技术栈的场景。

选型时不要只看“是否支持向量检索”，还要看数据规模、延迟、过滤能力、更新删除、混合检索、运维复杂度和成本。

## RAG 中怎么使用 Vector DB

### 入库阶段

```text
1. 解析文档：PDF、Word、HTML、Markdown、数据库记录。
2. 清洗内容：去噪、去页眉页脚、处理表格和代码块。
3. 文档切分：按标题、段落、固定长度或滑动窗口切 chunk。
4. 生成 Embedding：调用 embedding 模型得到向量。
5. 写入 Vector DB：保存 vector、text、metadata。
6. 建索引：选择 HNSW、IVF 等索引参数。
```

### 查询阶段

```text
1. 接收用户问题。
2. 可选 Query Rewrite，改写成更适合检索的表达。
3. 计算 query embedding。
4. 根据 tenant、权限、文档类型等 metadata 过滤。
5. 向量检索 top_k。
6. 可选 BM25 混合检索。
7. Reranker 重排。
8. 拼接上下文并调用 LLM。
```

### top_k 怎么设置

`top_k` 不是越大越好。

1. 太小：容易漏召回。
2. 太大：噪声增加，LLM 上下文成本上升。
3. 常见做法：Vector DB 先召回 top 20 到 top 100，再用 reranker 选 top 3 到 top 10。
4. 最终值应该基于评估集调参，而不是拍脑袋。

## 混合检索

纯向量检索适合语义相似问题，但对错误码、合同编号、API 名称、产品型号、人名等精确词不一定稳定。

企业 RAG 常用混合检索：

```text
向量检索 top_n + BM25 关键词检索 top_n -> 合并去重 -> Rerank -> top_k
```

优点：

1. 向量检索负责语义泛化。
2. BM25 负责关键词精确匹配。
3. Reranker 负责最终相关性排序。

## 工程设计要点

### 数据模型

一条向量记录通常包含：

```json
{
  "id": "chunk_001",
  "vector": [0.012, -0.031, 0.884],
  "text": "报销申请需要先提交发票和审批单...",
  "metadata": {
    "tenant_id": "t_001",
    "doc_id": "doc_123",
    "title": "员工报销制度",
    "section": "差旅报销",
    "source_url": "https://wiki.example.com/doc_123",
    "updated_at": "2026-06-16"
  }
}
```

线上系统不一定把全文都放在 Vector DB 中。如果原文较大，可以只存 chunk 摘要、位置和 metadata，完整内容放在对象存储或文档库。

### 权限控制

权限控制必须在检索阶段完成，不能只靠 Prompt 约束模型不要泄露。

正确做法：

1. 每个 chunk 带上租户、组织、角色、文档权限 metadata。
2. 查询时先做 metadata filter，再做向量召回或在召回过程中带过滤条件。
3. 返回结果前再次校验权限。
4. 日志中避免记录敏感原文和完整用户问题。

### 更新和删除

文档变化后要处理向量一致性：

1. 文档新增：解析、切分、embedding、写入。
2. 文档更新：删除旧 chunk 或按版本标记失效，再写入新 chunk。
3. 文档删除：删除向量记录，并确保缓存和索引不会继续返回旧内容。
4. 定期重建：当 embedding 模型、chunk 策略、索引参数变化时，需要批量重建。

### 可观测性

Vector DB 不是接上就完了，需要监控：

1. 写入成功率和失败原因。
2. 查询延迟 p50、p95、p99。
3. 召回结果数量和空召回比例。
4. top_k 相关性分数分布。
5. metadata filter 命中情况。
6. RAG 最终答案的引用命中率和人工反馈。

## 常见问题

### 召回结果不相关

可能原因：

1. Embedding 模型不适合业务领域。
2. chunk 太大或太小。
3. 文档清洗质量差。
4. top_k 设置不合理。
5. 缺少 reranker。
6. 相似度函数和模型不匹配。

### 召回不到精确术语

可能原因：

1. 只使用向量检索，没有 BM25。
2. 错误码、编号、缩写没有被 embedding 模型很好表达。
3. 文档切分时把关键上下文拆散了。

解决思路：引入混合检索、保留关键字段 metadata、对 query 做改写或扩展。

### 延迟太高

可能原因：

1. 索引参数不合适。
2. metadata filter 选择性差。
3. top_k 太大。
4. reranker 输入太多。
5. 向量维度高且数据规模大。

优化方向：调整索引、分区分片、降低候选集、缓存热门 query、异步批量入库。

### 数据隔离不安全

常见错误是把所有租户的数据放在一个集合里，但查询时忘记加 `tenant_id` filter。多租户系统必须把权限过滤做成检索服务的强制逻辑，而不是依赖业务调用方自觉传参。

## 选型建议

小规模原型：

1. Chroma。
2. FAISS。
3. PostgreSQL + pgvector。

中小规模生产：

1. PostgreSQL + pgvector，适合已有 PostgreSQL 体系且数据量不大。
2. Qdrant，适合希望部署简单、过滤能力较强的场景。
3. Elasticsearch / OpenSearch，适合本来就需要 BM25 和日志搜索的团队。

大规模生产：

1. Milvus / Zilliz。
2. Pinecone。
3. Weaviate。
4. 自建 FAISS 服务，但需要自己补齐存储、更新、权限、监控和扩缩容能力。

判断维度：

1. 数据规模：百万、千万还是十亿级向量。
2. 查询延迟：是否有严格 p95 / p99 要求。
3. 更新频率：离线批量还是实时增量。
4. 过滤复杂度：是否有多租户、权限、时间范围、标签过滤。
5. 混合检索：是否需要 BM25 + Vector。
6. 运维能力：团队是否能维护集群、索引、备份、扩容。
7. 成本：存储、内存、CPU/GPU、托管服务费用。

## 面试回答模板

如果被问“Vector DB 在 RAG 中起什么作用”，可以这样回答：

> Vector DB 负责存储文档 chunk 的 embedding，并在用户提问时做语义相似度检索。完整链路是文档解析、切分、embedding 入库，查询时对问题做 embedding，再结合 metadata 权限过滤召回 top_k，必要时做 BM25 混合检索和 rerank，最后把高相关 chunk 放进 Prompt 给 LLM。工程上我会重点关注 embedding 模型一致性、chunk 策略、索引参数、权限隔离、更新删除一致性、召回评估和查询延迟。

如果被问“为什么不用普通数据库”，可以这样回答：

> 普通数据库适合结构化条件查询和精确匹配，Vector DB 适合高维向量的语义相似度检索。RAG 里用户问题和文档表达经常不完全一致，只用关键词容易漏召回，所以需要 embedding 和向量检索。但 Vector DB 不替代业务数据库，通常是业务库保存主数据，文档库存原文，Vector DB 保存 chunk 向量和索引。

如果被问“Vector DB 怎么调优”，可以这样回答：

> 我会先构造评估集，看召回率、MRR、NDCG、空召回率和最终答案命中率。然后分别调 embedding 模型、chunk size、overlap、metadata filter、top_k、索引参数和 reranker。对于错误码、编号、专有名词较多的场景，会优先加 BM25 混合检索，而不是只调向量库参数。

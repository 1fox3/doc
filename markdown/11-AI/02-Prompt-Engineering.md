# Prompt Engineering

## Prompt 的作用

Prompt Engineering 是通过设计输入，让模型更稳定地完成任务。它不是简单写一句问题，而是把角色、任务、上下文、约束、输出格式和示例组织清楚。

一个好的 Prompt 通常包含：

1. Role：模型扮演什么角色。
2. Task：要完成什么任务。
3. Context：可依赖的信息。
4. Constraints：限制条件。
5. Output Format：输出格式。
6. Examples：必要时给出示例。

## 基础模板

```text
你是一个资深 Java 后端工程师。
请根据下面的上下文回答用户问题。

要求：
1. 只基于上下文回答。
2. 如果上下文没有答案，回答“资料中未提到”。
3. 输出必须包含结论、依据、风险。

上下文：
{{context}}

用户问题：
{{question}}
```

## 常见技巧

### Few-shot

给模型几个输入输出示例，让它模仿格式和思路。

适合：

1. 分类。
2. 信息抽取。
3. 固定格式改写。
4. 业务话术生成。

### Chain-of-Thought

让模型分步骤思考可以提升复杂推理效果，但生产环境不一定要把推理过程完整展示给用户。

更推荐的写法：

```text
请先在内部分析，再只输出最终结论和关键依据。
```

### ReAct

ReAct 是 Reasoning + Acting，模型一边思考，一边决定是否调用工具。

常见流程：

1. 理解用户目标。
2. 判断是否需要工具。
3. 调用搜索、数据库、代码执行等工具。
4. 观察工具返回结果。
5. 继续推理或给出答案。

### 结构化输出

对于程序后续处理，应该要求模型输出 JSON，并配合 schema 校验。

```text
请只输出 JSON，不要输出 Markdown。
字段：
{
  "summary": "一句话总结",
  "riskLevel": "low | medium | high",
  "items": ["问题列表"]
}
```

工程上不能只相信 Prompt，仍然要做 JSON parse、schema validate 和失败重试。

## System Prompt

System Prompt 用于定义模型的全局行为，例如身份、边界、安全规则和输出规范。

常见内容：

1. 模型角色。
2. 不允许做什么。
3. 优先级规则。
4. 遇到未知问题如何处理。
5. 工具调用规则。

注意：

1. System Prompt 不能当作安全边界。
2. 用户可能通过 Prompt Injection 诱导模型忽略规则。
3. 关键权限和数据过滤必须在服务端代码中实现。

## Prompt 优化思路

### 输出不稳定

解决方式：

1. 降低 temperature。
2. 增加输出格式约束。
3. 提供 few-shot 示例。
4. 拆分复杂任务。
5. 增加校验和重试。

### 答案太泛

解决方式：

1. 补充业务上下文。
2. 明确回答维度。
3. 要求给出依据和可执行建议。
4. 限制不要输出通用废话。

### 模型编造

解决方式：

1. 要求只基于给定上下文。
2. 要求引用来源。
3. 上下文不足时回答不知道。
4. 加入检索和事实校验。

## 面试常问

### Prompt Engineering 的价值是什么？

它能在不训练模型的情况下快速改善模型表现，尤其适合控制输出格式、任务边界和回答风格。但 Prompt 不是万能的，稳定性要求高的场景还需要代码校验、评估集、RAG、工具调用和监控。

### 如何让模型稳定输出 JSON？

我会从四层处理：Prompt 明确只输出 JSON，API 层使用支持 JSON mode 或 function calling 的能力，服务端做 schema 校验，失败时进行有限次数重试或降级处理。

### System Prompt 能防止越权吗？

不能。System Prompt 只能约束模型行为，不能代替后端权限控制。真正的权限过滤必须在检索、工具调用和数据访问层完成。

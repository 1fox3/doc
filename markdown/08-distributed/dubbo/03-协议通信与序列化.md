# 协议通信与序列化

## 协议选择

Dubbo 支持多协议，协议选择会影响性能、兼容性、治理能力和跨语言能力。

- Dubbo 协议：基于长连接和 NIO，适合内部高性能 Java RPC。
- Triple 协议：基于 HTTP/2，兼容 gRPC 语义，适合跨语言、云原生和流式通信。
- Rest 协议：适合 HTTP 生态和对外兼容，但性能和治理能力取决于实现。
- Injvm 协议：适合同 JVM 内本地调用。

## 网络传输

Dubbo 常用 Netty 作为通信框架。消费端和服务端通过长连接发送请求响应，减少频繁建连成本。

关键点：

- 长连接适合高频调用，但需要管理连接数、心跳和空闲检测。
- 请求响应通过 request id 匹配，支持并发复用连接。
- 大包会增加序列化、网络和内存压力，应限制请求体和响应体大小。
- 网络抖动、连接重置、服务端 Full GC 都可能表现为 RPC 超时。

## 编解码

编解码负责在对象、二进制协议和网络字节流之间转换。协议头通常包含魔数、版本、序列化标识、请求 id、消息长度和事件标记等信息。

排查编解码问题时重点看：协议是否匹配、序列化方式是否一致、类版本是否兼容、泛型字段是否丢失、请求包是否超过限制。

## 序列化

常见序列化方式包括 Hessian2、Fastjson2、Kryo、FST、Protobuf、Avro 等。选择时需要权衡性能、兼容性、安全性、跨语言和对象模型支持。

生产原则：

- 不随意切换序列化方式，尤其是已有多版本服务共存时。
- DTO 字段新增优于删除或改类型。
- 不把复杂领域对象直接作为 RPC 契约，优先使用稳定 DTO。
- 禁用存在安全风险的反序列化能力，限制可反序列化类型范围。

## Triple 与 Dubbo 协议差异

Dubbo 协议适合传统 Dubbo 内部调用，性能和治理成熟。Triple 基于 HTTP/2，支持 Unary、Server Streaming、Client Streaming、Bidirectional Streaming，更适合跨语言和云原生网关场景。

选择 Triple 时需要关注 HTTP/2 连接管理、代理兼容性、流控、Header 大小和可观测性集成。

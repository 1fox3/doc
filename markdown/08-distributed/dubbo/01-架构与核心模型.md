# 架构与核心模型

## 基本定位

Dubbo 是 RPC 框架，重点解决进程间服务调用、服务治理和扩展能力。它适合内部高频、低延迟、接口契约稳定的服务调用，不适合直接暴露给不可信公网客户端。

## 核心角色

- Provider：服务提供者，发布接口实现并注册服务元数据。
- Consumer：服务消费者，通过代理对象发起远程调用。
- Registry：注册中心，保存服务地址和元数据，例如 ZooKeeper、Nacos、Redis、Consul。
- Monitor：监控中心，收集调用次数、耗时、失败率等指标。
- Container：服务运行容器，负责启动、加载配置和优雅停机。

## 核心对象

- URL：Dubbo 的统一配置载体，协议、地址、参数、超时、重试、版本、分组等都会编码到 URL 中。
- Invoker：可执行调用实体，是 Dubbo 调用链的核心抽象。
- Protocol：负责服务暴露和引用，例如 DubboProtocol、TripleProtocol、InjvmProtocol。
- ProxyFactory：负责生成服务代理或把服务实现包装成 Invoker。
- Exporter：代表已暴露的服务，用于取消暴露。
- Cluster：将多个 Invoker 聚合为一个具备容错能力的 Invoker。
- Directory：维护服务目录和可用 Invoker 列表。
- Router：根据条件过滤 Invoker，例如标签路由、条件路由、脚本路由。
- LoadBalance：从候选 Invoker 中选择一个实例。
- Filter：调用链过滤器，用于上下文、监控、鉴权、限流和日志。

## 调用链路

一次 Dubbo 调用大致经过：消费端代理、ClusterInvoker、Directory、Router、LoadBalance、Filter、Client、Codec、网络传输、服务端 Codec、Filter、Invoker、业务实现。

排查问题时应沿这条链路定位：配置是否生效、路由是否过滤、负载均衡是否倾斜、线程池是否耗尽、序列化是否兼容、网络连接是否异常、服务端是否慢。

## URL 驱动

Dubbo 大量扩展点都通过 URL 参数驱动。常见参数包括 `timeout`、`retries`、`version`、`group`、`loadbalance`、`cluster`、`connections`、`serialization`、`threads`、`actives`、`mock`。

生产配置应明确作用范围：方法级优先于接口级，消费端配置通常优先于提供端默认配置。不要依赖隐式默认值承担关键治理策略。

## Dubbo 3.x 特性

- 应用级服务发现减少注册中心数据量，适合大规模实例场景。
- Triple 协议基于 HTTP/2，兼容 gRPC 生态并支持流式通信。
- 与 Kubernetes、Service Mesh、OpenTelemetry、Micrometer 等云原生体系集成更紧密。

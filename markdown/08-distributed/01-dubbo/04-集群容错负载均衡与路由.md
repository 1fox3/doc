# 集群容错负载均衡与路由

## Cluster

Cluster 将多个 Provider Invoker 聚合为一个逻辑 Invoker，并提供容错策略。它是 Consumer 面对多实例服务时的核心治理层。

## 容错策略

- Failover：失败自动切换并重试其他实例，适合幂等读操作，不适合非幂等写操作。
- Failfast：快速失败，只调用一次，适合新增、扣减、支付等非幂等操作。
- Failsafe：失败安全，异常被忽略，适合日志、审计等非核心操作。
- Failback：失败后后台定时重试，适合消息通知等最终一致场景。
- Forking：并行调用多个实例，任一成功即返回，适合低延迟但资源成本高的场景。
- Broadcast：广播调用所有实例，适合通知刷新缓存等场景，需要处理部分失败。

## 重试治理

重试会放大流量，必须和超时、幂等、熔断、限流配合。非幂等接口应使用 Failfast 或关闭重试。重试次数不应只看单次接口，还要看完整调用链预算。

## 负载均衡

- Random：按权重随机，默认常见策略，简单且分布较均匀。
- RoundRobin：按权重轮询，慢实例可能堆积请求。
- LeastActive：优先选择活跃调用数少的实例，适合性能差异明显的服务。
- ConsistentHash：一致性哈希，适合需要请求落到固定实例的场景，例如本地缓存命中。
- ShortestResponse：优先选择响应时间短的实例，需要依赖统计指标。

## Directory 与 Router

Directory 负责维护服务目录，通常从注册中心订阅 Provider 列表、配置规则和路由规则。Router 在负载均衡前过滤 Invoker，常用于灰度、标签路由、机房路由、黑白名单和参数路由。

路由规则应具备可观测和可回滚能力。错误路由可能导致 Consumer 看到服务存在但无可用 Provider。

## 服务降级与 Mock

Dubbo 支持 Consumer 侧 mock。常见策略：

- `force`：不发起远程调用，直接返回 mock 结果。
- `fail`：远程调用失败后返回 mock 结果。

mock 适合非核心读接口或兜底数据，不适合掩盖核心写失败。降级结果需要被日志和指标标记出来，否则会隐藏真实故障。

## 并发控制

并发控制可在 Consumer 或 Provider 侧配置，例如 `actives`、`executes`。Consumer 限制发起并发，Provider 限制接口执行并发。限流维度应结合接口成本、线程池容量和下游依赖容量设置。

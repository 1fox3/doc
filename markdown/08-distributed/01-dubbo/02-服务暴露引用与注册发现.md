# 服务暴露引用与注册发现

## 服务暴露流程

服务暴露是把本地接口实现转换为可被远程调用的 Invoker，并注册到注册中心。

典型流程：

- 解析服务配置，例如接口、实现类、协议、端口、版本、分组、超时、线程池。
- ProxyFactory 将服务实现包装为 Invoker。
- Protocol 将 Invoker 暴露为本地服务或远程服务。
- Netty 等通信框架启动端口监听。
- RegistryProtocol 将服务 URL 注册到注册中心。
- 订阅配置规则和动态治理规则。

## 本地暴露与远程暴露

- 本地暴露使用 InjvmProtocol，适合同 JVM 内部调用，避免网络开销。
- 远程暴露使用 Dubbo、Triple、Rest 等协议，面向跨进程调用。
- 同一服务可以同时本地暴露和远程暴露，消费端会根据引用方式和配置选择调用路径。

## 服务引用流程

服务引用是消费端根据接口生成代理，并把远程地址转换为 Invoker 列表。

典型流程：

- 解析 Reference 配置，例如接口、版本、分组、超时、重试和注册中心地址。
- 向注册中心订阅服务 Provider URL、路由规则和配置规则。
- Directory 维护可用 Invoker 列表。
- Cluster 将多个 Invoker 包装成一个具备容错能力的 Invoker。
- ProxyFactory 生成接口代理对象，业务代码像调用本地方法一样调用远程服务。

## 注册中心

Dubbo 支持 ZooKeeper、Nacos、Redis、Consul、Multicast 等注册中心。生产环境常用 ZooKeeper 或 Nacos。

注册中心职责：

- 保存 Provider 地址和元数据。
- 支持 Consumer 订阅变更。
- 支持路由规则、动态配置和禁用实例。
- 在 Provider 下线或会话失效时通知 Consumer 更新本地目录。

## Provider 失效下线

Provider 正常关闭时应先取消注册，再等待存量请求处理完成，最后关闭端口。异常宕机时依赖注册中心会话、心跳或健康检查感知下线。

Consumer 必须有本地缓存和容错策略，注册中心短暂不可用不应导致存量调用全部失败。

## 分组与版本

- `group` 用于区分同一接口的不同业务分组。
- `version` 用于接口兼容升级和灰度发布。
- 分组和版本必须在 Provider 与 Consumer 两侧匹配，否则会出现找不到服务或调用到错误实现。

## 配置优先级

Dubbo 配置可能来自注解、XML、API、配置中心、注册中心动态配置和系统属性。需要明确优先级和生效范围，避免 Provider 默认配置覆盖 Consumer 的关键治理配置。

常见治理参数包括超时、重试、负载均衡、集群容错、线程数、连接数、并发数和 mock 策略。

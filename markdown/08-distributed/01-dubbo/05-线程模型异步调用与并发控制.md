# 线程模型异步调用与并发控制

## 线程模型

Dubbo 调用涉及 IO 线程和业务线程。IO 线程负责网络读写和编解码，业务线程负责执行服务实现。不要在 IO 线程中执行阻塞业务逻辑，否则会影响同连接上的其他请求。

## Dispatcher

Dispatcher 决定请求事件如何派发到线程池。常见模型包括 all、direct、message、execution、connection 等。选择时要考虑请求处理、连接事件、心跳和响应回调是否会被阻塞。

## 线程池策略

- fixed：固定线程数，适合稳定吞吐场景，避免无限扩张。
- cached：按需创建线程，空闲回收，突发场景下可能导致线程过多。
- limited：限制最大线程数，避免资源耗尽。
- eager：队列满前优先扩线程，适合降低排队延迟。

线程池参数需要结合接口耗时、QPS、下游容量和机器资源设置。队列过大只会延迟失败，容易造成请求堆积和超时风暴。

## 异步调用

Dubbo 支持消费端异步、服务端异步和全链路异步。

- 消费端异步：调用后立即返回 Future 或 CompletableFuture，调用方后续获取结果或注册回调。
- 服务端异步：业务方法快速释放 Dubbo 线程，把耗时操作交给业务线程池处理，再通过异步结果返回。
- 全链路异步：从 Consumer 到 Provider 都避免阻塞等待，提高吞吐，但需要更严格的超时和异常处理。

## RpcContext 与隐式参数

RpcContext 可传递隐式参数，例如 TraceId、用户标识、租户、灰度标签和鉴权信息。隐式参数适合框架上下文，不适合承载大量业务数据。

注意事项：

- 线程切换时需要显式传递上下文。
- 不要透传敏感信息到不可信服务。
- Header 或 Attachment 命名需要统一，避免冲突。

## 泛化调用

泛化调用允许 Consumer 在没有接口 SDK 和模型类的情况下调用服务，参数和返回值通常用 Map 表达。它适合网关、测试平台、运维平台和低代码编排。

风险点：

- 缺少编译期类型检查。
- 序列化结构容易和 Provider DTO 不匹配。
- 权限必须严格控制，避免任意调用内部接口。

## 慢调用定位

定位 RPC 慢调用时按链路拆分：Consumer 线程池、路由和负载均衡、连接池、网络 RTT、序列化、Provider IO 线程、Provider 业务线程池、业务逻辑、数据库或缓存等下游依赖。

# 部署运维与故障排查

## 集群部署

ZooKeeper 通常部署 3 或 5 个节点。节点数越多，容错能力越强，但写入需要更多节点确认，延迟和运维成本也会上升。跨机房部署要谨慎，网络延迟会直接影响写性能和选举稳定性。

## 关键配置

- `tickTime`：基础时间单位，影响心跳和超时计算。
- `initLimit`：Follower 初始化连接 Leader 的最大 tick 数。
- `syncLimit`：Follower 与 Leader 同步和心跳的最大 tick 数。
- `dataDir`：快照目录。
- `dataLogDir`：事务日志目录，建议使用独立磁盘。
- `clientPort`：客户端连接端口，默认 2181。
- `server.x`：集群节点通信配置，通常包含选举端口和同步端口。

常见端口：

- `2181`：客户端端口。
- `2888`：Follower 与 Leader 数据同步端口。
- `3888`：Leader 选举端口。

## 四字母命令

四字母命令可用于运维诊断，生产环境应通过白名单限制。

- `ruok`：检查服务是否响应。
- `stat`：查看节点状态、连接数和模式。
- `mntr`：输出监控指标。
- `conf`：查看服务端配置。
- `cons`：查看客户端连接。
- `wchs`：查看 Watcher 汇总。
- `srvr`：查看服务端状态。

## 性能关注

- 事务日志磁盘延迟会影响写请求提交。
- 过多节点和 Watcher 会增加内存和事件分发压力。
- 大节点数据会增加网络传输和快照成本。
- JVM GC 暂停可能导致心跳异常和会话误判。
- 客户端会话超时设置过短会放大网络抖动影响。

## 故障排查

- 频繁选举：检查网络抖动、磁盘延迟、GC、节点负载和 tickTime 配置。
- 会话过期：检查客户端长时间 STW、网络断连、session timeout 和服务端负载。
- 写入变慢：检查事务日志磁盘、Follower ack 延迟、Leader 负载和请求大小。
- Watcher 风暴：检查是否大量客户端监听同一路径，是否重复注册未清理。
- 节点数过多：检查业务是否把 ZooKeeper 当数据库使用，必要时迁移到配置中心或数据库。

## 安全治理

- 开启 ACL，限制关键路径写权限。
- 四字母命令配置白名单。
- 管理端口和客户端端口限制来源网络。
- 定期备份快照和事务日志。
- 审计关键路径变更，避免误删根路径或核心元数据。

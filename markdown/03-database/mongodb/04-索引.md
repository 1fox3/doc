# MongoDB 索引

## 索引作用

索引用于减少扫描范围、支持排序、保证唯一性、实现过期删除和提升查询稳定性。索引不是越多越好，每个索引都会增加写入成本、占用内存和磁盘，并增加优化器选择成本。

## 常见类型

- 单字段索引：针对一个字段建立升序或降序索引。
- 复合索引：针对多个字段建立索引，字段顺序直接影响过滤和排序能力。
- 多键索引：数组字段索引，一个数组元素会生成多个索引项。
- 唯一索引：保证字段或字段组合唯一。
- TTL 索引：按时间自动删除过期文档。
- 哈希索引：对字段哈希值建立索引，常用于哈希分片键。
- 文本索引：支持基础文本搜索，复杂搜索通常使用专门搜索引擎。
- 地理位置索引：支持 `2d`、`2dsphere` 等地理空间查询。
- 部分索引：只索引满足过滤条件的文档，降低索引规模。
- 稀疏索引：只索引包含该字段的文档，使用时需注意查询结果语义。

## 复合索引顺序

复合索引遵循左前缀原则。设计顺序时通常考虑：

- 等值条件字段放前面。
- 高选择性字段优先，但要结合实际查询组合。
- 范围条件之后的字段通常难以继续用于精确过滤。
- 排序字段应和索引字段顺序、方向匹配。
- 覆盖查询需要把过滤字段、排序字段和返回字段纳入索引。

示例查询：

```javascript
db.orders.find({ userId: 1001, status: "PAID" }).sort({ createdAt: -1 })
```

可考虑索引：

```javascript
db.orders.createIndex({ userId: 1, status: 1, createdAt: -1 })
```

## 多键索引

数组字段建立索引后会成为多键索引。查询数组内嵌对象时应使用 `$elemMatch` 保证条件作用于同一个数组元素。

错误风险：

```javascript
db.items.find({ "attrs.k": "color", "attrs.v": "red" })
```

该查询可能分别匹配不同数组元素。更稳妥写法：

```javascript
db.items.find({ attrs: { $elemMatch: { k: "color", v: "red" } } })
```

## TTL 索引

TTL 索引用于自动删除过期文档：

```javascript
db.sessions.createIndex({ expireAt: 1 }, { expireAfterSeconds: 0 })
```

运行机制：

- 后台 TTL Monitor 周期性扫描 TTL 索引，通常约每 60 秒触发一次。
- 删除动作在主节点执行，并通过 oplog 复制到从节点。
- TTL 删除不是实时删除，业务不能依赖精确到秒的过期语义。

限制与注意点：

- TTL 索引只支持单字段。
- `_id` 字段不支持 TTL 索引。
- Capped Collection 不支持 TTL 删除，因为 capped 集合不能按任意条件删除文档。
- 同一字段不能建立多个不同过期策略的 TTL 索引。
- TTL 字段应使用日期类型或日期数组。

## 唯一索引

唯一索引用于数据库层防重。高并发创建业务对象时，唯一索引比应用层先查再插更可靠。

常见用法：

```javascript
db.users.createIndex({ tenantId: 1, email: 1 }, { unique: true })
```

如果字段可为空，需要明确 `null` 和字段缺失的唯一性语义。可使用部分唯一索引只约束有效数据：

```javascript
db.users.createIndex(
  { tenantId: 1, email: 1 },
  { unique: true, partialFilterExpression: { email: { $exists: true } } }
)
```

## 索引治理

- 根据真实查询建立索引，不为每个字段机械建索引。
- 定期用 `$indexStats`、慢查询和 profiler 识别低价值索引。
- 关注索引大小是否超过内存工作集，避免频繁磁盘读取。
- 新建大索引前评估构建时间、磁盘空间和线上负载。
- 分片集合的唯一索引必须满足分片键约束，否则无法保证全局唯一。

# 锁

## 锁粒度

- 全局锁：如 `FLUSH TABLES WITH READ LOCK`，会阻塞写入，常用于历史备份场景。
- 表级锁：包括显式表锁、元数据锁、意向锁和自增锁。
- 行级锁：InnoDB 基于索引实现，包含记录锁、间隙锁和临键锁。

## 元数据锁

- MDL 保护表结构和查询的一致性。
- 普通 DML 会持有 MDL 读锁，DDL 需要 MDL 写锁。
- 长事务未提交可能阻塞 DDL，DDL 等待又可能阻塞后续 DML，形成线上故障链路。

## 自增锁

- 自增锁用于分配 `AUTO_INCREMENT` 值。
- `innodb_autoinc_lock_mode` 控制自增锁策略，影响并发插入和复制安全性。
- 批量插入、`INSERT ... SELECT` 等语句可能持有更强的自增锁。

## 排查方式

```sql
SHOW PROCESSLIST;
SHOW ENGINE INNODB STATUS\G
SELECT * FROM performance_schema.metadata_locks\G
SELECT * FROM performance_schema.data_locks\G
```

## 锁等待处理流程

- 先确认等待 SQL 和阻塞 SQL，区分行锁、MDL 还是全局锁。
- 找到阻塞事务的开始时间、当前 SQL、来源主机和业务模块。
- 判断是否可以让业务自然完成；如果不能，再评估 kill 连接的影响。
- 事后从索引、事务范围、访问顺序和 DDL 流程上修复根因。

## 减少锁冲突

- 热点行更新可以拆分热点、使用队列串行化或改为乐观锁重试。
- 范围更新尽量按主键或明确索引分批推进。
- DDL 使用低峰窗口和在线变更工具，并设置锁等待超时。
- 避免在事务中执行慢查询、大分页或等待用户输入。

## 乐观锁示例

```sql
UPDATE account
SET balance = balance - 100,
    version = version + 1
WHERE id = 1
  AND version = 10
  AND balance >= 100;
```

受影响行数为 0 时，由应用决定重试或返回失败。

# 事务

## ACID

- 原子性：事务内操作全部提交或全部回滚，主要依赖 undo log。
- 一致性：事务前后数据满足约束和业务不变量，由数据库约束、事务机制和应用逻辑共同保证。
- 隔离性：并发事务之间的可见性由隔离级别、锁和 MVCC 控制。
- 持久性：事务提交后即使崩溃也能恢复，主要依赖 redo log 和刷盘策略。

## 提交与回滚

- `START TRANSACTION` 或 `BEGIN` 开启事务，`COMMIT` 提交，`ROLLBACK` 回滚。
- DDL、部分管理语句和隐式提交语句会结束当前事务。
- autocommit 开启时，每条独立 DML 默认都是一个事务。

## 工程风险

- 长事务会保留 ReadView，导致 undo 版本链无法及时清理，增加查询成本和磁盘占用。
- 大事务会产生大量 redo、undo 和锁持有时间，影响复制延迟和恢复时间。
- 事务中包含外部 RPC、用户交互或慢查询会扩大锁等待范围。

## 常用检查

```sql
SELECT * FROM information_schema.innodb_trx\G
SHOW ENGINE INNODB STATUS\G
SHOW VARIABLES LIKE 'transaction_isolation';
```

## 隐式提交语句

- 大多数 DDL 会隐式提交当前事务，例如 `ALTER TABLE`、`CREATE TABLE`、`DROP TABLE`。
- `LOCK TABLES`、`UNLOCK TABLES`、部分账户权限管理语句也可能触发隐式提交。
- 业务事务中混入 DDL 会破坏原子性预期，应用代码应避免这种模式。

## 事务边界设计

- 事务只包住必须保持一致的数据库操作，不要覆盖远程调用、文件操作、消息发送或用户交互。
- 先做不加锁的校验和数据准备，再开启事务执行关键写入。
- 多资源一致性场景中，数据库本地事务不能自动保证消息队列、缓存和外部系统一致，需要 outbox、事务消息或补偿机制。

## 常见异常

- `Lock wait timeout exceeded` 表示锁等待超过 `innodb_lock_wait_timeout`，不一定是死锁。
- `Deadlock found when trying to get lock` 表示 InnoDB 检测到死锁并回滚了一个事务。
- 应用捕获异常后必须明确回滚事务，连接池复用连接时尤其要防止脏事务状态泄漏。

## 事务模板

```sql
START TRANSACTION;

SELECT stock FROM inventory WHERE sku_id = 1001 FOR UPDATE;
UPDATE inventory SET stock = stock - 1 WHERE sku_id = 1001 AND stock > 0;
INSERT INTO order_item(order_id, sku_id, quantity) VALUES (1, 1001, 1);

COMMIT;
```

实际业务中应尽量用单条条件更新替代“先查再改”，减少锁持有时间和并发窗口。

## 长事务来源

- 应用连接池未关闭事务。
- 后台任务一次处理过多数据。
- 人工会话 `BEGIN` 后忘记提交。
- 备份或导出工具开启一致性快照后运行时间过长。

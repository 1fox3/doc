# MySQL 面试重点

## 架构主线

- Server 层负责连接、权限、解析、优化、执行器和 binlog。
- InnoDB 负责 Buffer Pool、索引、事务、锁、redo、undo、崩溃恢复。
- SQL 执行要从解析、优化、执行、存储引擎访问、返回结果串起来讲。

## 索引高频题

- InnoDB 聚簇索引叶子节点存整行数据，二级索引叶子节点存主键值，二级索引命中后可能回表。
- 联合索引遵循最左前缀原则，范围条件后面的列通常无法继续用于有序定位。
- 覆盖索引可以避免回表，但会增加写入和存储成本。
- 索引失效常见原因包括函数计算、隐式类型转换、前导模糊匹配、低选择性、统计信息不准、优化器认为全表更便宜。
- B+ 树适合范围查询和排序，Hash 索引不适合范围扫描。

## 事务与 MVCC

- ACID 中原子性主要依赖 undo，持久性主要依赖 redo，隔离性依赖锁和 MVCC，一致性由数据库约束和业务共同保证。
- ReadView 包含活跃事务集合、低水位和高水位，用于判断版本链中哪个版本可见。
- RC 通常每条语句生成 ReadView，RR 通常事务第一次快照读生成 ReadView。
- 当前读会加锁并读取最新已提交版本，快照读通过 MVCC 读取历史可见版本。

## 锁高频题

- InnoDB 行锁锁的是索引记录，不是直接锁物理行；没有合适索引可能扩大扫描和加锁范围。
- Record Lock 锁记录，Gap Lock 锁间隙，Next-Key Lock 是记录锁加间隙锁。
- RR 下 next-key lock 用于防止幻读；RC 下间隙锁使用更少，但外键和唯一性检查仍可能涉及。
- 死锁不可完全避免，要通过固定访问顺序、缩短事务、合理索引和重试降低概率。

## 日志高频题

- redo log 是物理日志，用于崩溃恢复，保证已提交事务持久化。
- undo log 是逻辑回滚日志，用于回滚和 MVCC 版本链。
- binlog 是 Server 层逻辑日志，用于复制和 PITR 恢复。
- 两阶段提交协调 redo 和 binlog，避免崩溃后主库恢复和从库复制不一致。

## 优化题回答框架

1. 用慢日志确认 SQL、耗时、扫描行数和返回行数。
2. 用 `EXPLAIN FORMAT=JSON` 看访问方法、索引、驱动表、rows、filtered、Extra。
3. 用 `optimizer_trace` 判断为什么选或不选某个索引。
4. 从索引、SQL 改写、分页、排序、Join 顺序、统计信息、表结构和业务访问模式优化。
5. 上线后对比延迟、扫描行数、Buffer Pool 命中率和锁等待。

# 存储引擎

## InnoDB

- 支持事务、行级锁、MVCC、外键和崩溃恢复。
- 使用聚簇索引组织表数据，二级索引叶子节点保存主键。
- 通过 Buffer Pool、redo log、undo log 实现高并发读写和恢复能力。

## MyISAM

- 不支持事务和行锁，主要使用表锁。
- 崩溃恢复能力弱，现代在线业务不建议使用。
- 历史上常用于读多写少场景，但多数已被 InnoDB 替代。

## MEMORY

- 数据存储在内存，重启后丢失。
- 适合临时数据或中间结果，不适合持久化业务数据。
- 内存限制和数据类型支持需要额外关注。

## 选择原则

- 默认使用 InnoDB，除非有明确且可验证的特殊需求。
- 引擎差异会影响事务边界、锁等待、备份恢复和复制一致性。
- 查看当前表引擎使用 `SHOW TABLE STATUS` 或 `information_schema.tables`。

## InnoDB 适用场景

- OLTP 业务表，要求事务、并发写入和崩溃恢复。
- 需要主从复制、备份恢复和在线 DDL 治理的生产系统。
- 需要通过行锁和 MVCC 支撑读写并发的场景。

## 不同引擎差异点

- 事务：InnoDB 支持，MyISAM 不支持。
- 锁粒度：InnoDB 主要行锁，MyISAM 主要表锁。
- 索引组织：InnoDB 聚簇索引，MyISAM 数据和索引分离。
- 崩溃恢复：InnoDB 依赖 redo，MyISAM 崩溃后更容易需要修复。

## 迁移建议

- 迁移前确认表是否依赖 MyISAM 的特殊全文索引、压缩或历史行为。
- 转换引擎会重建表，大表需要在线变更方案。
- 转换后重新验证执行计划、事务语义和备份恢复流程。

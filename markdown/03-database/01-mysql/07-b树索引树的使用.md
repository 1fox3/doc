# B+树索引树的使用

## 可使用索引的条件

- 等值条件：`col = const`、`col IN (...)` 可形成单点扫描区间。
- 范围条件：`>`、`>=`、`<`、`<=`、`BETWEEN`、前缀 `LIKE 'abc%'` 可形成范围扫描区间。
- 排序和分组：当 `ORDER BY` 或 `GROUP BY` 顺序与索引顺序一致时，可减少文件排序或临时表。
- 覆盖索引：查询列都在二级索引中时，可避免回表。

## 索引失效场景

- 对索引列使用函数、表达式或隐式类型转换，可能导致无法按有序键定位。
- `LIKE '%abc'` 不能利用普通 B+Tree 前缀有序性。
- 联合索引未使用最左前缀，或跳过中间列，无法形成连续扫描区间。
- 不同字符集或 collation 的列比较可能触发转换，影响索引使用。

## 索引下推

- Index Condition Pushdown 会在存储引擎层用索引中的列先过滤记录。
- ICP 不能减少扫描的索引范围，但可以减少回表次数。
- `EXPLAIN` 的 `Extra` 出现 `Using index condition` 表示使用了索引下推。

## 实践要点

- 建联合索引时按等值列、范围列、排序列、覆盖列综合排序，不只看单列选择性。
- 用 `EXPLAIN` 验证 `key`、`key_len`、`rows`、`filtered`、`Extra`，不要只看是否命中索引。
- 对低基数列可结合业务条件建复合索引，避免优化器认为全表扫描更便宜。

## 最左前缀细节

```sql
CREATE INDEX idx_a_b_c ON t(a, b, c);

-- 可以使用 a、b、c。
SELECT * FROM t WHERE a = 1 AND b = 2 AND c = 3;

-- 可以使用 a、b 范围；c 通常不能继续缩小扫描区间。
SELECT * FROM t WHERE a = 1 AND b > 10 AND c = 3;

-- 不能有效使用该联合索引定位，因为缺少最左列 a。
SELECT * FROM t WHERE b = 2 AND c = 3;
```

## 排序可用索引的条件

- `ORDER BY` 列顺序要与索引顺序一致，且前导列的过滤条件要能固定或参与同一顺序扫描。
- 混合升降序在 MySQL 8.0 可通过降序索引改善，但要确认实际索引定义。
- 如果排序列来自不同表，通常更难完全利用单个索引排序。
- `WHERE` 范围过宽时，即使能用索引排序，也可能因为扫描行数太多而不划算。

## SQL 改写方向

- 将函数计算移到常量侧，例如把 `DATE(created_at) = '2026-01-01'` 改为范围条件。
- 保持字段类型一致，避免字符串列和数字常量比较。
- 将大偏移分页改为基于游标或上次最大键值的翻页方式。

## 索引能力速查

- 覆盖索引：查询列都在索引中，`EXPLAIN Extra` 常见 `Using index`。
- 索引下推：在存储引擎层先用索引列过滤，减少回表，`Extra` 为 `Using index condition`。
- 排序消除：索引顺序满足 `ORDER BY`，减少 filesort。
- 范围扫描：利用 B+Tree 有序性扫描连续区间。

## 索引评审清单

- 这个索引服务哪些具体 SQL，而不是服务某个字段。
- SQL 的过滤、排序、分组、返回列是否与索引顺序匹配。
- 现有索引是否已经能覆盖该需求，避免重复索引和左前缀冗余索引。
- 新索引对写入、磁盘、Buffer Pool 和备份耗时的影响是否可接受。

## 删除索引前检查

- 通过慢日志和 `performance_schema` 确认索引近期是否被使用。
- 检查是否有外键、唯一约束或业务依赖。
- MySQL 8.0 可先设置 Invisible Index 观察计划影响，再决定是否删除。

```sql
ALTER TABLE t_user ALTER INDEX idx_old INVISIBLE;
ALTER TABLE t_user ALTER INDEX idx_old VISIBLE;
```

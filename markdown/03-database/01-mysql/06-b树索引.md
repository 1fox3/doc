# B+树索引

## 数据结构

- InnoDB 索引使用 B+Tree，所有真实记录都存储在叶子节点，非叶子节点保存键值和页号。
- 叶子节点按键值有序，并通过链表连接，适合等值查询、范围查询和排序扫描。
- B+Tree 高度通常很低，千万级数据常见高度为 3 到 4 层，查询主要成本来自页读取和回表。

## 聚簇索引和二级索引

- 聚簇索引叶子节点保存完整行数据，一张 InnoDB 表只有一个聚簇索引。
- 二级索引叶子节点保存二级索引列和主键值，查询非覆盖列时需要回表到聚簇索引。
- 主键过长会放大所有二级索引，因为二级索引叶子节点都要保存主键值。

## 索引能力

- 支持等值匹配、范围匹配、最左前缀、排序消除、覆盖索引和索引下推。
- 联合索引按定义顺序组织，`(a,b,c)` 可高效支持 `a`、`a,b`、`a,b,c` 的前缀匹配。
- 范围条件后的列通常不能继续用于缩小扫描范围，但可能用于索引下推过滤。

## 设计原则

- 优先为高选择性、常用于过滤或排序的列建立联合索引。
- 避免为低选择性列单独建索引，除非结合其他列形成有效联合索引。
- 控制索引数量，写入、更新、删除都需要维护索引并增加 redo、undo 和 Buffer Pool 压力。

## 联合索引示例

```sql
CREATE INDEX idx_status_created_user ON orders(status, created_at, user_id);

-- 可利用 status 等值和 created_at 范围。
SELECT * FROM orders
WHERE status = 1 AND created_at >= '2026-01-01';

-- 可利用 status 过滤和 created_at 顺序。
SELECT * FROM orders
WHERE status = 1
ORDER BY created_at DESC
LIMIT 20;
```

## 回表成本

- 二级索引扫描得到的是二级索引列和主键值。
- 如果查询列不在二级索引中，需要根据主键再访问聚簇索引，这一步叫回表。
- 扫描少量记录时回表成本可接受；扫描大量记录时，大量随机回表可能比全表扫描更慢。
- 覆盖索引可以减少回表，但不应为了覆盖所有查询而无限扩展索引列。

## 索引维护成本

- 每新增一个索引，插入都要写入一棵新的 B+Tree。
- 更新索引列可能等价于删除旧索引项再插入新索引项。
- 删除记录需要维护所有相关索引并产生 undo/redo。
- 索引越多，优化器枚举候选计划的成本和误选概率也会上升。

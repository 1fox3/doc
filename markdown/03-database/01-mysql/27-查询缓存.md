# 查询缓存

## 历史特性

- Query Cache 缓存完整 SQL 文本和结果集，相同 SQL 命中后可直接返回结果。
- 任意相关表发生写入，表上所有查询缓存都会失效。
- MySQL 8.0 已移除 Query Cache，生产优化不应依赖它。

## 不适合高并发的原因

- SQL 文本必须完全一致，参数、空格、大小写差异都可能影响命中。
- 写多读多场景下缓存频繁失效，维护成本超过收益。
- 全局互斥会导致高并发下竞争严重。

## 历史配置

```sql
SHOW VARIABLES LIKE 'query_cache%';
SELECT SQL_CACHE * FROM t WHERE id = 1;
SELECT SQL_NO_CACHE * FROM t WHERE id = 1;
```

## 替代方案

- 应用层缓存，如 Redis、本地缓存或 CDN。
- 读写分离、物化汇总表、索引优化和 SQL 改写。
- MySQL 8.0 中重点关注执行计划、Buffer Pool、直方图和慢日志治理。

## 应用层缓存设计

- 缓存 key 应包含影响结果的全部参数，例如用户、租户、分页、排序和过滤条件。
- 写入后需要明确失效策略，可以是删除缓存、更新缓存或基于版本号隔离。
- 热点 key 要防止缓存击穿，可使用互斥重建、逻辑过期或提前刷新。
- 缓存穿透可通过空值缓存、布隆过滤器或参数校验缓解。

## 数据库侧替代优化

- 为热点查询建立覆盖索引，减少回表和页读取。
- 对复杂聚合使用汇总表或异步物化结果。
- 对历史报表查询使用只读副本，避免污染主库 Buffer Pool。
- 用慢日志和 `performance_schema` 找出真正高频高成本 SQL。

## 历史兼容注意

- MySQL 5.7 及更早版本即使开启 Query Cache，也不建议在高并发写入系统依赖它。
- 从 5.7 升级到 8.0 时，如果应用依赖 `SQL_CACHE` 提示，需要确认兼容性和效果变化。

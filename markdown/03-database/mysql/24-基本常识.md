# 基本常识

## MySQL 架构

- Server 层负责连接管理、权限校验、SQL 解析、优化器、执行器、binlog 和内置函数。
- 存储引擎层负责数据读写、索引组织、事务、锁和崩溃恢复。
- InnoDB 是默认事务型引擎，支持行锁、MVCC、外键、崩溃恢复和聚簇索引。

## SQL 执行链路

- 客户端建立连接并完成认证。
- Server 层解析 SQL，生成语法树并进行语义校验。
- 优化器基于规则和成本生成执行计划。
- 执行器调用存储引擎接口读取或修改数据。
- 写入场景涉及 redo log、undo log、binlog 和事务提交。

## 常见对象

- 数据库、表、索引、视图、存储过程、触发器、事件。
- 系统库包括 `mysql`、`information_schema`、`performance_schema`、`sys`。
- 常用诊断入口是慢日志、`EXPLAIN`、`SHOW ENGINE INNODB STATUS` 和 `performance_schema`。

## 实践原则

- 默认使用 InnoDB、`utf8mb4`、显式主键和清晰的事务边界。
- 生产变更前关注锁、备份、回滚方案和复制延迟。
- 性能问题优先用数据和执行计划定位，不靠猜测加索引。

## 常见日志

- 错误日志记录启动、关闭、崩溃、复制、InnoDB 恢复和严重错误。
- 慢查询日志记录超过阈值的 SQL，可用于慢 SQL 治理。
- binlog 记录逻辑变更，用于复制和时间点恢复。
- redo log 是 InnoDB 物理重做日志，用于崩溃恢复。
- undo log 用于回滚和 MVCC 历史版本。

## 常见系统库

- `information_schema` 提供元数据视图，如表、列、索引、事务。
- `performance_schema` 提供性能事件、锁、等待、语句和内存等观测数据。
- `sys` 是基于 `performance_schema` 的易读视图集合。
- `mysql` 保存账号、权限、数据字典相关系统信息。

## 基础排查路径

- SQL 慢：慢日志 -> `EXPLAIN` -> 数据分布 -> 索引和 SQL 改写 -> 压测验证。
- 锁等待：`SHOW PROCESSLIST` -> `innodb_trx` -> `data_locks` -> 业务事务边界。
- 磁盘满：binlog -> undo -> 临时文件 -> 表空间 -> 日志文件。
- 复制延迟：主库大事务 -> 从库 SQL 线程 -> 锁等待 -> 网络和 I/O。

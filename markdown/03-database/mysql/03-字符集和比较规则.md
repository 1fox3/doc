# 字符集和比较规则

## 基本概念

- 字符集决定字符如何编码为字节，比较规则决定字符串如何排序和比较。
- MySQL 8.0 推荐使用 `utf8mb4`，不要使用历史上的 `utf8`，它实际是 `utf8mb3`，无法完整存储 4 字节 Unicode 字符。
- 常见比较规则中，`_ci` 表示大小写不敏感，`_cs` 表示大小写敏感，`_bin` 表示按二进制字节比较。

## 作用层级

- 字符集和比较规则可定义在服务器、数据库、表、列、连接和表达式层级。
- 列级定义优先级最高，连接变量影响客户端发送 SQL 和结果集返回时的编码转换。
- 相关变量包括 `character_set_server`、`character_set_database`、`character_set_client`、`character_set_connection`、`character_set_results`。

## 技术影响

- 字符集不一致可能导致乱码、索引长度变化、隐式转换和索引失效。
- 不同 collation 的列做比较时可能触发字符集转换，影响执行计划和排序结果。
- `LIKE`、`ORDER BY`、唯一索引判重都会受比较规则影响，例如大小写不敏感规则下 `abc` 和 `ABC` 可能被认为相等。

## 常用检查

```sql
SHOW VARIABLES LIKE 'character_set%';
SHOW VARIABLES LIKE 'collation%';
SHOW FULL COLUMNS FROM t_user;
SHOW CREATE TABLE t_user;
ALTER TABLE t_user CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci;
```

## 实践要点

- 新库新表统一使用 `utf8mb4` 和明确的 collation，避免依赖实例默认值。
- 迁移字符集前评估索引长度、唯一约束冲突和业务大小写敏感要求。
- Java/JDBC 连接串应显式声明编码，避免客户端和服务端编码不一致。

## 隐式转换示例

```sql
-- 字符列和数字比较可能触发类型转换，导致索引利用变差或结果异常。
EXPLAIN SELECT * FROM user WHERE phone = 13800138000;

-- 推荐写法：保持类型一致。
EXPLAIN SELECT * FROM user WHERE phone = '13800138000';
```

## 排序和唯一性

- `utf8mb4_0900_ai_ci` 中 `ai` 表示 accent insensitive，`ci` 表示 case insensitive，大小写和重音差异可能不参与比较。
- 如果用户名、编码、token 等字段要求严格区分大小写，优先使用 `_bin` 或业务层额外规范化。
- 唯一索引在大小写不敏感 collation 下可能认为 `Foo` 和 `foo` 冲突，迁移前必须扫描潜在重复数据。

## 迁移步骤建议

- 先统计库、表、列当前字符集和比较规则，找出混用情况。
- 对唯一索引列做冲突预检查，避免转换过程中失败。
- 大表使用在线 DDL 或在线变更工具，避免长时间 MDL 阻塞业务。
- 转换后验证连接层 `character_set_client`、`character_set_connection`、`character_set_results` 是否符合预期。

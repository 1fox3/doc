# 锁

## 行锁

- InnoDB 行锁锁的是索引记录，不是抽象的数据行。
- 条件未命中索引时，可能扫描并锁住大量记录。
- 唯一索引等值命中已有记录时通常只加记录锁，范围查询可能加 next-key lock。

## 表锁

- MDL 保护表结构，所有 DML/查询都会持有读类型元数据锁。
- DDL 需要写类型元数据锁，容易被长事务阻塞。
- 意向锁用于协调表级锁和行级锁，不表示实际锁住整张表数据。

## S 锁和 X 锁

- S 锁共享读，多个事务可兼容持有。
- X 锁排他写，与其他 S/X 锁冲突。
- `SELECT ... FOR SHARE` 加 S 锁，`SELECT ... FOR UPDATE` 加 X 锁。

## 自增锁

- `AUTO_INCREMENT` 分配受自增锁模式影响。
- 高并发插入热点表要关注自增主键、批量插入和复制模式的组合影响。

## 排查

```sql
SELECT * FROM performance_schema.data_locks\G
SELECT * FROM performance_schema.data_lock_waits\G
SELECT * FROM performance_schema.metadata_locks\G
```

## 锁范围和索引

```sql
-- idx_user_created(user_id, created_at) 可以缩小锁范围。
SELECT * FROM orders
WHERE user_id = 10 AND created_at >= '2026-01-01'
FOR UPDATE;

-- 如果没有合适索引，InnoDB 需要扫描更多记录，锁冲突范围扩大。
```

## 常见死锁模式

- 两个事务按相反顺序更新两行记录。
- 批量更新使用不同排序顺序扫描同一批数据。
- 先插入唯一键再更新相关表，多个事务顺序不一致。
- 范围锁和插入意向锁在高并发插入时冲突。

## 处理建议

- 所有事务按相同顺序访问表和记录。
- 批量任务按主键升序分批处理。
- 对死锁错误做重试，但重试次数和退避时间要有限制。
- 为范围更新补充合适索引，减少 next-key lock 覆盖范围。

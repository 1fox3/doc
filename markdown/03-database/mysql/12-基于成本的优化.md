# 基于成本的优化

## 成本模型

- 优化器会估算不同执行计划的 I/O 成本、CPU 成本、扫描行数、过滤率和排序成本。
- 候选计划包括访问方法、索引选择、连接顺序、连接算法、排序方式和是否使用临时表。
- 成本最低的计划不一定真实最快，因为统计信息、数据分布和缓存状态可能不准确。

## 影响因素

- 表和索引统计信息：基数、页数、记录数、直方图。
- 条件选择性：谓词过滤比例越高，索引访问通常越有价值。
- 回表成本：二级索引扫描返回大量行时，随机回表可能比全表扫描更慢。
- 排序和分组：无法利用索引顺序时，可能产生 filesort 和临时表。

## 工具

```sql
EXPLAIN FORMAT=JSON SELECT * FROM t WHERE a = 1 AND b > 10;
ANALYZE TABLE t;
ANALYZE TABLE t UPDATE HISTOGRAM ON a WITH 100 BUCKETS;
```

## 实践要点

- 慢 SQL 优化先看真实执行计划，再结合数据分布验证优化器估算是否偏差。
- 统计信息陈旧时可执行 `ANALYZE TABLE`，但大表要评估线上影响。
- 不要滥用 `FORCE INDEX`，它会锁死优化器选择，数据分布变化后可能成为性能隐患。

## 成本估算例子

- 如果二级索引能过滤到 10 行，即使需要回表，通常也比全表扫描便宜。
- 如果二级索引过滤到 50% 数据，且查询需要大量非索引列，优化器可能选择全表扫描。
- 如果 `ORDER BY` 可以利用索引顺序并结合 `LIMIT` 提前结束，优化器可能选择看似过滤性较差但能避免排序的索引。

## 执行计划突然变化

- 数据量增长或数据分布变化导致统计信息变化。
- `ANALYZE TABLE` 后统计信息更新，优化器重新估算成本。
- 新增索引后优化器有了新的候选计划，但新计划不一定更好。
- 参数、版本或 `optimizer_switch` 变化影响优化策略。

## 排查清单

- 对比慢 SQL 在问题前后的 `EXPLAIN FORMAT=JSON`。
- 查看相关列基数和数据分布，确认是否存在极端倾斜。
- 检查 `rows` 估算与实际返回行数差异。
- 必要时用 `optimizer_trace` 查看被放弃计划的成本原因。

# 连接的原理

## 连接类型

- 内连接只返回两表都匹配的记录。
- 左连接保留左表记录，右表无匹配时补 `NULL`。
- 右连接可等价改写为左连接，实际开发中通常统一使用左连接提升可读性。
- 半连接用于 `IN`、`EXISTS` 等只关心是否存在匹配行的场景。

## 执行算法

- Nested Loop Join：以驱动表结果逐行访问被驱动表，适合驱动表小且被驱动表有高效索引。
- Block Nested Loop：被驱动表无合适索引时，用 join buffer 批量比较，仍可能非常慢。
- Batched Key Access：批量收集键值后访问被驱动表，减少随机 I/O，依赖优化器开关和成本判断。
- Hash Join：MySQL 8.0 支持，用哈希表处理等值连接，适合无高效索引的大量等值连接场景。

## 优化要点

- 小结果集表更适合作为驱动表，但优化器会基于成本和统计信息选择连接顺序。
- 被驱动表连接列应有索引，否则连接可能退化为大量全表扫描。
- 外连接的语义会限制连接顺序改写，`WHERE` 条件写法还可能把外连接变成内连接。

## 排查命令

```sql
EXPLAIN FORMAT=JSON
SELECT * FROM orders o JOIN users u ON o.user_id = u.id WHERE o.created_at >= '2026-01-01';
```

关注 `nested_loop`、`attached_condition`、`rows_examined_per_scan`、`using_join_buffer` 等字段。

## 连接索引设计

```sql
-- orders 作为驱动表过滤后，再按 user_id 访问 users。
CREATE INDEX idx_orders_created_user ON orders(created_at, user_id);

-- users.id 通常是主键，天然支持被驱动表查找。
SELECT o.id, u.name
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.created_at >= '2026-01-01';
```

## 外连接条件位置

```sql
-- 保留没有支付记录的订单。
SELECT o.id, p.status
FROM orders o
LEFT JOIN payments p ON p.order_id = o.id AND p.status = 'SUCCESS';

-- WHERE 中过滤右表列会过滤掉 NULL，语义接近内连接。
SELECT o.id, p.status
FROM orders o
LEFT JOIN payments p ON p.order_id = o.id
WHERE p.status = 'SUCCESS';
```

## 常见问题

- 多表连接慢不一定是连接表多，通常是驱动表过滤后行数太大或被驱动表缺索引。
- `join_buffer_size` 只能缓解无索引连接，不能替代正确索引。
- `STRAIGHT_JOIN` 可以固定连接顺序，但应谨慎使用，避免数据分布变化后计划固化。

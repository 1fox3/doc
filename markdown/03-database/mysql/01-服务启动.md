# 服务启动

## 服务端启动方式

- `mysqld` 是 MySQL Server 主程序，直接读取配置并启动服务进程，适合容器或受进程管理器托管的环境。
- `mysqld_safe` 是守护脚本，会启动 `mysqld`、记录错误日志并在异常退出后尝试拉起，传统 Linux 部署中常见。
- `mysql.server` 是 SysV 风格管理脚本，内部通常调用 `mysqld_safe`，提供 `start`、`stop`、`restart`。
- `mysqld_multi` 用于同一机器管理多个 MySQL 实例，需要为每个实例指定独立的端口、socket、数据目录和配置组。

## 启动流程

- 解析配置文件和命令行参数，加载系统变量与插件。
- 初始化数据目录、错误日志、redo log、undo 表空间、临时表空间和 InnoDB Buffer Pool。
- 监听 TCP/IP、Unix Socket、命名管道或共享内存连接方式。
- 接收客户端连接后执行认证、权限加载、线程分配和会话变量初始化。

## 客户端连接

- `mysql -h host -P port -u user -p` 通过 TCP/IP 连接远程或本机端口。
- `mysql -S /path/mysql.sock -u user -p` 通过 Unix Socket 连接本机实例。
- Windows 可通过 `--protocol=pipe` 使用命名管道，通过 `--protocol=memory` 使用共享内存。

## 常用排查

- 启动失败先看错误日志，重点关注配置项拼写、端口占用、数据目录权限、redo/undo 文件不一致。
- 多实例启动失败检查 `server_id`、`port`、`socket`、`pid-file`、`datadir` 是否冲突。
- 连接失败区分网络不可达、监听地址不匹配、账号 host 不匹配、认证插件不兼容和 TLS 配置错误。

```sql
SHOW VARIABLES LIKE 'port';
SHOW VARIABLES LIKE 'socket';
SHOW VARIABLES LIKE 'datadir';
SHOW ENGINES;
```

## 启动失败定位流程

- 先确认进程是否存在：如果进程已经退出，优先查看错误日志；如果进程存在但不可连接，再检查监听地址、端口和 socket。
- 查看错误日志中的第一条关键错误，后续错误经常是连锁反应，例如数据目录权限错误会继续引发 redo 初始化失败。
- 检查配置文件是否被实际加载，`mysqld --verbose --help` 会打印默认读取路径和变量默认值。
- 检查数据目录所有者和权限，MySQL 进程用户必须能读写 `datadir`、日志目录、临时目录和 socket 目录。
- 检查端口和 socket 冲突，多实例部署时尤其容易多个实例使用相同 `port`、`socket` 或 `pid-file`。

## 生产部署建议

- 使用 systemd 或容器编排系统管理 `mysqld` 生命周期，不建议依赖手工后台启动。
- 错误日志、慢日志、binlog 和数据目录最好分盘或至少分目录管理，便于容量监控和故障隔离。
- 多实例必须显式配置 `server_id`、`datadir`、`port`、`socket`、`log_error`、`pid-file`，避免隐式默认值互相覆盖。
- 启动参数变更要记录变更单，尤其是 redo 容量、Buffer Pool、字符集、时区、隔离级别和 binlog 相关参数。

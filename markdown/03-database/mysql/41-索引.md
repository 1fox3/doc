# 索引

## 核心类型

- 聚簇索引：InnoDB 按主键组织整行数据。
- 二级索引：叶子节点保存索引列和主键值，查询非覆盖列需要回表。
- 联合索引：多个列按定义顺序组成键，遵循最左前缀原则。
- 唯一索引：保证键值唯一，注意 collation 会影响字符串唯一性判断。

## 关键能力

- 覆盖索引：查询列都在索引中，`EXPLAIN Extra` 常见 `Using index`。
- 索引下推：在存储引擎层先用索引列过滤，减少回表，`Extra` 为 `Using index condition`。
- 排序消除：索引顺序满足 `ORDER BY`，减少 filesort。
- 范围扫描：利用 B+Tree 有序性扫描连续区间。

## 索引失效

- 对索引列使用函数或表达式。
- 隐式类型转换或字符集转换。
- 联合索引不满足最左前缀。
- 前置通配符 `LIKE '%abc'`。
- `OR` 条件两侧不是都可有效使用索引。

## 设计原则

- 围绕高频 SQL 设计联合索引，而不是看到字段就建索引。
- 等值条件、范围条件、排序和覆盖需求要一起考虑。
- 控制索引数量，索引会增加写入成本、磁盘空间和优化器选择复杂度。

```sql
SHOW INDEX FROM t_user;
EXPLAIN SELECT id, name FROM t_user WHERE status = 1 ORDER BY created_at DESC LIMIT 20;
```

## 联合索引列顺序

- 等值过滤列通常放前面，便于快速缩小扫描范围。
- 范围列放在等值列之后，范围列后的列通常难以继续缩小扫描区间。
- 排序列是否放入索引，要结合过滤条件和 `LIMIT` 判断。
- 覆盖列可放在索引后部，但会增加索引体积和写入成本。

## 索引评审清单

- 这个索引服务哪些具体 SQL，而不是服务某个字段。
- SQL 的过滤、排序、分组、返回列是否与索引顺序匹配。
- 现有索引是否已经能覆盖该需求，避免重复索引和左前缀冗余索引。
- 新索引对写入、磁盘、Buffer Pool 和备份耗时的影响是否可接受。

## 删除索引前检查

- 通过慢日志和 `performance_schema` 确认索引近期是否被使用。
- 检查是否有外键、唯一约束或业务依赖。
- MySQL 8.0 可先设置 Invisible Index 观察计划影响，再决定是否删除。

```sql
ALTER TABLE t_user ALTER INDEX idx_old INVISIBLE;
ALTER TABLE t_user ALTER INDEX idx_old VISIBLE;
```

# MongoDB 复制集

## 作用

复制集提供高可用和数据冗余。一个复制集由多个 `mongod` 节点组成，通常包含一个 Primary 和多个 Secondary。客户端写入 Primary，Secondary 通过复制 Primary 的 oplog 应用变更。

## 节点类型

- Primary：接收写入，生成 oplog。
- Secondary：复制并重放 oplog，可承担读请求。
- Arbiter：只参与投票，不保存数据。现代生产环境应谨慎使用，优先使用奇数个数据节点。
- Hidden：隐藏节点，不被普通读流量选择，常用于备份、报表或延迟副本。
- Delayed：延迟应用 oplog，用于误删恢复窗口。

## Oplog

`oplog.rs` 位于 `local` 数据库，是 capped collection。它按时间顺序记录幂等的写操作，Secondary 持续拉取并重放。

关键点：

- oplog 大小决定可容忍的复制中断窗口。
- 写入高峰、批量更新、索引构建都可能加速 oplog 滚动。
- Secondary 落后超过 oplog 窗口后，需要重新同步。
- TTL 删除、事务提交等也会通过 oplog 传播。

## 选举

当 Primary 不可达或主动降级时，复制集发起选举。获得多数票的可选节点成为新的 Primary。

影响选举的因素：

- 节点优先级 `priority`。
- 节点是否有投票权 `votes`。
- 节点复制进度。
- 网络分区和心跳超时。
- 是否满足多数派。

只有多数派可用时才能选出 Primary。少数派节点不能继续接受写入，这是为了避免脑裂。

## 回滚

如果旧 Primary 上存在未复制到多数节点的写入，在重新加入复制集时可能发生 rollback。关键业务写入应使用 `w: "majority"` 降低回滚风险。

## 复制延迟

复制延迟常见原因：

- Primary 写入压力过高。
- Secondary 磁盘、CPU 或网络性能不足。
- 大事务或批量写入产生大量 oplog。
- Secondary 上执行重查询、备份或索引构建。
- oplog 应用线程被锁等待阻塞。

排查指标：

- `rs.status()` 中各节点 optime 差异。
- `replication lag` 监控。
- opcounters、锁等待、磁盘 IOPS、WiredTiger cache dirty bytes。
- 慢查询和后台任务。

## 读写路由

默认读写都走 Primary。读从节点可以分担压力，但会带来延迟读问题。若业务使用 `secondary` 或 `secondaryPreferred`，必须明确允许读旧数据。

## 部署建议

- 生产至少 3 个数据节点，跨可用区部署。
- 避免只有 2 个数据节点加 1 个 Arbiter 承载关键数据，因为 Arbiter 不保存数据，故障恢复能力较弱。
- 为备份、分析查询使用 Hidden Secondary，避免影响在线读写。
- 定期验证备份恢复，而不仅是备份成功。

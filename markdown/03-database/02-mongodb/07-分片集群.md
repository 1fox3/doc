# MongoDB 分片集群

## 作用

分片用于水平扩展数据容量和读写吞吐。分片集群把集合数据按分片键划分到多个 Shard，每个 Shard 通常是一个复制集。

## 组件

- Shard：保存业务数据的分片，通常为复制集。
- Config Server：保存集群元数据，例如数据库、集合、分片、Chunk 和路由信息。
- `mongos`：查询路由层，应用通常连接 `mongos`，不直接连接 Shard。

## 分片键

分片键决定数据分布和查询路由，是分片设计中最关键的决策。

要求与限制：

- 分片键必须是集合上的索引或索引前缀。
- 分片键字段应出现在集合文档中。
- 分片键大小有限制，设计时应保持短小稳定。
- 分片后修改分片键能力受版本和配置约束，应避免依赖频繁修改。

优秀分片键特征：

- 高基数，能够把数据分散到多个 Chunk。
- 写入分布均衡，避免单个 Shard 热点。
- 常出现在查询条件中，减少广播查询。
- 业务上稳定，不频繁变更。

## 分片策略

范围分片按分片键范围划分 Chunk。它适合范围查询，但如果分片键单调递增，例如纯时间戳或自增 ID，容易形成写入热点。

哈希分片对分片键哈希后分布。它能改善写入均衡，但范围查询能力较弱，且相邻业务值会被打散。

复合分片键可以结合租户、时间、业务 ID 等字段，但字段顺序需要同时服务数据分布和查询路由。

## Chunk

Chunk 是分片迁移和均衡的基本单位。集合数据按分片键范围切分为多个 Chunk。当 Chunk 过大时会发生 split，当 Shard 间 Chunk 分布不均衡时 Balancer 会迁移 Chunk。

Chunk 大小需要以当前版本配置为准，并结合文档大小、写入速率和迁移成本调整。较大的 Chunk 会降低元数据数量，但也会提高迁移和均衡成本。

## Balancer

Balancer 负责在 Shard 之间迁移 Chunk，使数据分布更均衡。迁移会带来网络、磁盘和复制压力，高峰期可能影响业务延迟。

迁移大致流程：

- 源 Shard 和目标 Shard 确认迁移范围。
- 目标 Shard 克隆 Chunk 范围内的数据。
- 目标 Shard 持续追赶迁移期间的增量修改。
- 进入临界区，短暂停止该 Chunk 范围写入。
- Config Server 提交路由元数据变更。
- 源 Shard 清理已迁移数据。

## 查询路由

如果查询条件包含完整或可用的分片键，`mongos` 可以把请求定向到少数 Shard。若查询不包含分片键，可能变成 scatter-gather，对所有 Shard 广播查询并合并结果。

查询设计原则：

- 高频查询尽量带上分片键或分片键前缀。
- 聚合和排序要关注是否跨 Shard 合并。
- 分页查询如果广播到多个 Shard，成本会显著上升。
- 唯一约束需要结合分片键设计，避免无法保证全局唯一。

## 常见问题

- 分片键选择错误，导致单 Shard 热点。
- 查询不带分片键，集群扩容后延迟反而升高。
- Jumbo Chunk 无法自动迁移，需要调整分片键或拆分数据。
- Balancer 在业务高峰迁移，造成延迟抖动。
- Config Server 性能或可用性异常导致路由元数据操作受影响。

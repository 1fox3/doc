# MongoDB 数据建模

## 建模原则

MongoDB 建模优先从访问模式出发，而不是先抽象实体关系。设计集合和文档结构时，需要明确：

- 最核心的查询条件是什么。
- 单次请求需要返回哪些字段和子对象。
- 写入是否会频繁修改数组或嵌套结构。
- 数据是否需要跨文档强一致。
- 数据增长是否会导致单文档过大或热点写入。

## 嵌入与引用

嵌入适合一对一、少量一对多、生命周期一致、经常一起读取的数据。例如订单和订单明细、文章和少量标签。优点是一次读取即可拿到完整聚合根，缺点是文档可能膨胀、局部更新竞争更集中。

引用适合多对多、大量一对多、生命周期独立、数据被多个聚合根共享的场景。例如用户与订单、商品与库存流水。优点是避免重复和超大文档，缺点是需要额外查询或聚合 `$lookup`，并增加一致性处理成本。

## 冗余字段

MongoDB 常通过冗余字段换取查询效率。例如在订单中冗余用户昵称、商品快照、结算价格。冗余字段必须明确更新策略：

- 快照型冗余：下单时固定，之后不随源数据变化。
- 同步型冗余：源数据变化后异步或同步更新副本。
- 派生型冗余：通过定时任务或变更流重建，例如统计字段。

冗余字段不是无成本优化，必须考虑更新放大、数据修复和一致性窗口。

## Schema 管理

MongoDB 是 flexible schema，不是 no schema。生产环境仍应治理字段类型、必填字段、索引、默认值和版本迁移。

可选手段：

- 在应用层使用 DTO、JSON Schema、ODM 或 validation library 校验。
- 在集合上配置 `$jsonSchema` validation，约束字段类型和枚举。
- 为文档增加 `schemaVersion` 字段，支持灰度迁移和兼容读。
- 禁止同一字段混用字符串、数字、日期等类型，否则会影响查询、排序和索引判断。

## 数组建模

数组字段适合保存有界列表。若数组会无限增长，应拆成独立集合。

常见风险：

- 大数组更新会造成文档移动、写放大和锁竞争。
- 数组字段建立多键索引后，索引项数量随数组元素增长。
- 同一复合索引中包含多个数组字段会受到限制，查询计划也更复杂。
- 对数组中元素排序、分页、筛选需求复杂时，独立集合通常更清晰。

## 常见模式

- 一对少量：嵌入，例如用户的多个收货地址。
- 一对大量：引用，例如用户的订单列表。
- 树结构：可使用父引用、祖先数组、物化路径或嵌套集合，按查询路径选择。
- 时间序列：使用 time series collection 或按时间分桶建模。
- 事件日志：追加写为主，避免频繁更新历史文档。

## 反模式

- 把关系型数据库的表结构直接迁移为大量集合，再用 `$lookup` 拼回对象。
- 把所有业务数据放入一个巨型文档。
- 高频更新同一个全局计数文档，造成单点热点。
- 不控制字段类型，导致同一查询命中不同类型值。
- 没有清理策略地保存无限增长日志。

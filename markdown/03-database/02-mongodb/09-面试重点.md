# MongoDB 面试重点

## 建模

- MongoDB 建模围绕访问模式，不是简单把关系表搬成集合。
- 嵌入适合强聚合、一起读写、生命周期一致的数据；引用适合高基数、独立增长、多对多和需要独立查询的数据。
- 反范式能减少查询次数，但要承担冗余更新和一致性成本。

## 索引

- 复合索引字段顺序要结合等值、排序、范围过滤设计。
- 多键索引用于数组字段，但多个数组字段组合会带来索引项膨胀和限制。
- TTL 索引适合过期数据清理，不适合强实时删除。
- 覆盖查询要求查询字段和返回字段都在索引中，且不需要回表读取文档。

## 一致性

- WriteConcern 决定写入确认级别，ReadConcern 决定读取隔离语义，ReadPreference 决定读哪个节点。
- 复制集通过 oplog 同步，oplog 窗口不足会导致落后节点需要重新全量同步。
- 多文档事务会增加锁、oplog、内存和延迟成本，不应替代合理建模。

## 分片

- 分片键决定数据分布和查询路由，是最关键设计。
- 低基数分片键容易热点，高单调递增键容易写热点。
- 查询不带分片键会广播到多个分片，成本高且不可预测。
- Chunk 迁移和 Balancer 会影响集群负载，生产要关注窗口期和热点。

## 排障主线

- 慢查询看 profiler、explain、索引、扫描文档数和返回文档数。
- 写入慢看锁、Journal、磁盘、复制确认级别和热点分片。
- 副本集问题看选举、心跳、oplog、复制延迟和网络分区。

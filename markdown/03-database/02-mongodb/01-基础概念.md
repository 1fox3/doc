# MongoDB 基础概念

## 定位

MongoDB 是面向文档的 NoSQL 数据库，使用 BSON 保存文档。它适合字段结构变化较快、读写吞吐要求高、数据天然以聚合根组织、需要水平扩展的场景，但不适合把复杂多表强关系模型原样迁移过来。

## 核心对象

- 数据库：逻辑命名空间，包含多个集合。
- 集合：一组文档的容器，类似关系型数据库中的表，但集合本身不强制固定 schema。
- 文档：MongoDB 的基本数据单元，结构是 BSON，类似 JSON 对象。
- 字段：文档中的 key/value，value 可为标量、数组、嵌套文档、日期、ObjectId 等类型。
- `_id`：每个文档必须有的主键字段，默认由驱动或服务端生成 `ObjectId`，也可以由业务显式指定。

## BSON

BSON 是 Binary JSON，支持比 JSON 更丰富的数据类型，例如 `ObjectId`、`Date`、`Decimal128`、二进制数据等。MongoDB 单个 BSON 文档大小上限是 16MB，超过该限制的大文件应使用 GridFS 或对象存储。

常见注意点：

- 字段名区分大小写。
- 字段顺序在 BSON 中有意义，虽然业务逻辑不应依赖字段顺序。
- 数字类型要明确区分 `int`、`long`、`double`、`decimal`，避免金额或计数精度问题。
- 日期建议统一使用 UTC 存储，由展示层处理时区。

## 特殊数据库

- `admin`：保存全局角色、用户和管理命令相关数据。许多集群级命令需要在 `admin` 上执行。
- `local`：只保存在单个节点本地，不参与复制。复制集的 `oplog.rs` 位于 `local` 库。
- `config`：分片集群中保存元数据，例如分片、数据库、集合、Chunk、Balancer 状态等。
- `test`：shell 默认数据库，不应在生产环境依赖。

## 命名空间

MongoDB 使用 `database.collection` 表示命名空间。集合名应保持稳定、语义清晰，避免在名称中编码租户、日期等高基数维度；这些维度更适合作为字段并通过索引或分片键治理。

## GridFS

GridFS 用于存储超过 16MB 的大文件，会把文件拆分为多个 chunk，并用两个集合保存元数据和分块数据：

- `fs.files`：文件元数据，例如文件名、长度、上传时间、md5、metadata。
- `fs.chunks`：实际分块数据，默认每个 chunk 255KB。

GridFS 适合需要和 MongoDB 元数据强绑定、需要通过数据库权限和备份统一管理的文件场景。纯文件分发、图片视频访问、高 CDN 命中场景通常更适合对象存储。

## MapReduce 与聚合

MongoDB 早期支持 MapReduce，但现代版本更推荐 Aggregation Pipeline。聚合管道在优化、可读性和生态兼容性上更好，只有在遗留系统中才应优先考虑 MapReduce。

# 数仓分层、维度建模与指标体系

数据仓库用于把分散的业务数据转化为统一、稳定、可分析、可治理的数据资产。核心能力包括分层设计、维度建模、指标口径治理和质量保障。

## 数仓分层目标

分层的核心目标是解耦数据接入、清洗建模、公共汇总和应用服务，降低重复开发和口径混乱。

- ODS：原始数据层，尽量保留源系统原貌。
- DWD：明细数据层，做清洗、规范化、去重和业务过程建模。
- DWS：汇总数据层，面向主题域沉淀公共聚合。
- ADS：应用数据层，面向报表、接口、产品、运营和算法应用。

## ODS 原始数据层

ODS 保留源系统原始数据，通常按业务系统、表名、日期、小时分区。

设计重点：

- 少做业务加工，只做必要格式转换、脱敏和入湖入仓。
- 保留源字段、操作类型、抽取时间、事件时间和批次号。
- 支持问题追溯、补数、重跑和重新建模。
- 对日志、CDC 和文件导入保留原始落地区。

## DWD 明细数据层

DWD 做清洗、去重、标准化和业务过程建模。

常见加工：

- 统一时间、ID、枚举、币种、状态和编码。
- 过滤测试数据、脏数据和无效记录。
- 按业务唯一键去重。
- 构建订单、支付、退款、点击、曝光等业务过程事实表。
- 补充维度键、来源、渠道和业务标签。

DWD 应保持明细粒度，不应过早做过多聚合，否则复用性和可追溯性会下降。

## DWS 汇总数据层

DWS 围绕主题域沉淀公共汇总，例如用户日汇总、商品日汇总、交易日汇总、渠道日汇总。

设计重点：

- 服务多个 ADS 应用，避免重复计算。
- 指标口径由统一指标体系管理。
- 聚合粒度要稳定，避免为单个报表过度定制。
- 对核心汇总表建立质量校验和血缘关系。

## ADS 应用数据层

ADS 面向具体查询、报表、接口、算法和产品功能。

常见形态：

- ClickHouse/Doris 宽表。
- Redis/HBase 实时特征表。
- BI 报表聚合表。
- API 服务查询表。
- 推荐、风控、搜索使用的样本和特征表。

ADS 可以为查询性能做宽表、预聚合和冗余，但必须标注来源、口径、负责人和生命周期。

## 维度建模

维度建模用事实表记录业务过程，用维度表描述分析视角。

事实表类型：

- 事务事实表：一笔订单、一笔支付、一条点击。
- 周期快照事实表：每天账户余额、库存快照。
- 累积快照事实表：订单从创建到完成的生命周期。

维度表描述业务实体，例如用户、商品、门店、地区、渠道、设备。维度字段通常用于 `where`、`group by` 和报表筛选。

## 星型、雪花与宽表

- 星型模型：事实表直接关联多个维表，查询简单，冗余较多。
- 雪花模型：维表继续规范化，冗余少，但查询复杂。
- 宽表：把常用维度冗余到事实表或 ADS 表中，查询快但更新和口径维护成本高。

离线 DWD/DWS 更强调复用和口径稳定，ADS 更强调查询性能和应用适配。

## 缓慢变化维与拉链表

维度属性会变化，例如用户等级、商品类目、门店状态。

处理方式：

- Type 1：直接覆盖，不保留历史。
- Type 2：新增版本记录，保留历史。
- Type 3：增加字段保存前一个状态。

拉链表通过开始时间和结束时间保存历史版本。

常见字段：

- `business_id`
- `start_date`
- `end_date`
- `is_current`
- `version`

拉链表适合回答“某个历史时间点用户或商品是什么状态”。代价是查询和维护复杂度更高。

## 指标体系

指标体系用于统一业务分析语言，避免不同团队对同一指标算出不同结果。

指标分类：

- 原子指标：不可再拆的基础度量，例如支付金额、订单数。
- 派生指标：原子指标 + 统计周期 + 维度 + 修饰词，例如最近 7 天新用户支付金额。
- 复合指标：由多个指标计算得到，例如转化率、客单价。

指标组成：

- 业务过程：下单、支付、退款、发货。
- 度量：金额、次数、人数、时长。
- 维度：时间、地区、渠道、商品、用户。
- 统计周期：日、周、月、实时窗口。
- 过滤条件：成功订单、有效用户、剔除测试数据。

## 口径治理

- 明确定义指标负责人。
- 指标公式统一沉淀，不允许 ADS 私自定义核心指标。
- 指标与数据表、任务、字段血缘关联。
- 指标变更有审批、版本和影响分析。
- 核心指标做实时和离线一致性校验。

常见口径问题：

- “订单数”是否包含取消订单。
- “GMV”是否包含退款、运费和优惠。
- “活跃用户”按登录、访问还是有效行为计算。
- 实时窗口和离线自然日窗口不一致。
- 测试数据、机器人流量和内部账号未剔除。

## 实时数仓分层

实时数仓把业务事件、日志、CDC 数据在秒级或分钟级加工成可查询、可分析、可告警的数据资产。

常见分层：

- ODS：原始实时数据，通常保留 Kafka 原始 Topic 或入湖原始表。
- DWD：实时明细层，做清洗、去重、标准化和宽化。
- DWS：实时汇总层，按主题域做窗口聚合或累计指标。
- ADS：应用层，写入 ClickHouse、Doris、Redis 或 HBase 支撑查询和服务。

关键问题：

- 乱序与迟到：使用 Event Time、Watermark 和 Allowed Lateness。
- 去重：基于业务唯一键、状态 TTL 和幂等 Sink。
- 维表关联：广播维表、异步 IO、Temporal Join 和维表缓存。
- 一致性：Checkpoint、两阶段提交、幂等写入和批次号。
- 回溯重算：保留 Kafka 数据，支持 Savepoint、湖仓快照和任务版本管理。

## 建模实践

- 分层不要为形式而分层，链路过长会增加延迟和维护成本。
- DWD 保持明细和稳定粒度，DWS 沉淀公共汇总，ADS 服务具体应用。
- 核心事实表必须定义业务主键、事件时间、分区字段和数据来源。
- 维度表要明确更新方式、历史保留策略和主键策略。
- 指标口径、SQL、产出表、负责人和质量规则要统一管理。

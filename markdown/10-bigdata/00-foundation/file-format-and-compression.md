# 数据格式与压缩

数据格式直接影响存储成本、查询性能、Schema 演进、多引擎兼容性和下游治理能力。

## 行式与列式

行式存储把一行数据连续存放，适合写入、点查和整行读取。

列式存储把同一列连续存放，适合分析型查询，因为 OLAP 通常只读取少量列并做过滤、聚合和统计。

列式格式的优势：

- 列裁剪减少读取字节数。
- 同列数据类型和分布相近，压缩率更高。
- 文件级、Row Group 级统计信息可支持谓词下推。
- 向量化读取和批量解码更适合分析引擎。

## 常见数据格式

### Text/CSV

- 优点：简单、可读、通用。
- 缺点：无强类型、体积大、解析成本高，不适合长期分析存储。

### JSON

- 优点：半结构化，适合日志、埋点和调试。
- 缺点：字段冗余、解析成本高、Schema 约束弱。

### Avro

- 行式二进制格式，支持 Schema 演进。
- 常用于 Kafka 消息、CDC 数据交换和跨系统传输。

### ORC

- Hive 生态常用列式格式。
- 支持压缩、索引、统计信息、谓词下推和 ACID 表能力。

### Parquet

- 通用列式格式，Spark、Flink、Trino、Iceberg、Hudi 广泛使用。
- 生态兼容性好，适合湖仓和跨引擎访问。

### Protobuf

- 强 Schema、体积小、序列化快。
- 适合服务间消息、Kafka 事件和实时链路，但不适合作为分析表的最终存储格式。

## 压缩算法

- Snappy：压缩比一般，速度快，适合计算中间结果和高吞吐链路。
- Gzip：压缩比高，但读取 CPU 成本高，且普通 gzip 文件不可切分。
- LZO：可切分，速度较快，但需要额外索引和部署支持。
- ZSTD：压缩比和速度平衡较好，现代数据湖和 OLAP 系统常用。
- LZ4：速度快，适合低延迟读写和日志链路。

## 选择建议

- Hive 离线数仓：优先 ORC 或 Parquet。
- Spark/Trino/湖仓：优先 Parquet，结合 Iceberg/Hudi 表格式。
- Kafka 消息：Avro、Protobuf 或 JSON，生产环境建议配合 Schema Registry。
- ClickHouse/Doris：使用引擎内部列式存储，重点设计分区、排序和导入批次。
- 临时导入或人工排查：CSV/Text 可用，但不建议长期保存核心数据。

## 常见问题

- 使用 JSON 长期保存明细数据，导致查询扫描和解析成本过高。
- 使用不可切分压缩格式保存大文件，导致并行读取能力差。
- Kafka 中没有 Schema 管理，导致字段新增、类型变化和兼容性不可控。
- 小文件过多，即使使用列式格式也无法发挥扫描优势。

## 工程实践

- 核心明细表使用列式格式，并控制单文件大小在计算引擎友好的范围内。
- 对宽表和历史表启用合适压缩，降低长期存储成本。
- 对高频查询字段保留统计信息，确保谓词下推和文件裁剪生效。
- Schema 变更要通过 Catalog、Schema Registry 或表格式能力管理，避免上下游不兼容。

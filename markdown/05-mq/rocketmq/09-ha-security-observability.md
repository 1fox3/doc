# 高可用、安全与可观测性

## 主从复制

传统部署中，同一 BrokerName 下 BrokerId 为 0 的节点是 Master，非 0 是 Slave。Producer 通常写 Master，Consumer 可根据配置从 Slave 读，缓解 Master 压力。

复制方式：

- 异步复制：Master 写入后即可返回，性能好但主节点宕机可能丢少量数据。
- 同步复制：等待 Slave 确认后返回，可靠性更高但延迟更高。

## 刷盘与复制组合

- 异步刷盘 + 异步复制：吞吐最高，可靠性最低。
- 同步刷盘 + 同步复制：可靠性最高，延迟最高。
- 生产环境要按业务等级拆 Topic 或集群，不能所有消息使用同一可靠性策略。

## DLedger

DLedger 基于 Raft 提供自动主从切换能力，解决传统主从需要人工切换的问题。它提升可用性，但对部署、磁盘、网络和多数派节点健康有更高要求。

## ACL 与 TLS

ACL 用于控制账号、签名、Topic 读写权限和管理权限。公网、跨部门或多租户环境应启用 ACL，并结合 TLS 保护链路传输。

## 消息轨迹

消息轨迹记录发送、存储、投递和消费关键事件，用于排查消息是否发送成功、是否到达 Broker、是否被消费以及消费失败原因。开启轨迹会带来额外存储和性能开销，应按需开启。

## 监控指标

- Broker 存活、主从同步延迟、刷盘耗时。
- CommitLog 写入 TPS 和延迟。
- ConsumeQueue 分发延迟。
- Topic 生产和消费 TPS。
- Consumer lag、重试消息数、死信消息数。
- 磁盘水位、PageCache 命中、GC、网络和线程池状态。

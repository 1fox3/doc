# 顺序消息与事务消息

## 顺序消息

顺序消息的核心是同一业务键固定进入同一 MessageQueue，并由消费者按队列顺序处理。

发送端实践：

- 使用订单号、用户 ID、设备 ID 等 sharding key。
- 不要在扩容队列时忽略 key 到队列映射变化。
- 避免把热点 key 写入单队列导致吞吐瓶颈。

消费端实践：

- 单队列串行消费。
- 消费失败不要跳过后续消息，除非业务允许乱序。
- 对长耗时任务拆分异步处理，避免阻塞整条队列。

## 事务消息流程

1. Producer 发送半消息到 Broker。
2. Broker 持久化半消息，暂不投递给 Consumer。
3. Producer 执行本地事务。
4. Producer 根据本地事务结果提交 Commit 或 Rollback。
5. Broker 对 Commit 消息恢复投递，对 Rollback 消息丢弃。
6. 如果状态未知，Broker 向 Producer 发起事务回查。

## 本地事务状态

- Commit：本地事务成功，消息可以投递。
- Rollback：本地事务失败，消息丢弃。
- Unknown：状态未知，等待后续回查。

## 工程落地

- 本地事务表记录事务消息状态，回查时以本地事务表为准。
- 回查接口必须幂等，不能在回查中重新执行业务写入。
- 事务消息解决本地事务和 MQ 消息的一致性，不解决消费者处理的最终一致性。
- 消费者仍需幂等，因为事务消息提交后仍可能重复投递。

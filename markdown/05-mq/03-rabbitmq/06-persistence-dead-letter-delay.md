# 持久化、死信与延迟队列

## 持久化

可靠持久化需要三者同时成立：

- Exchange 设置 durable。
- Queue 设置 durable。
- Message 设置 deliveryMode=2。

只持久化消息而队列不持久，Broker 重启后队列不存在，消息也无法恢复。持久化提升可靠性，但不等于绝不丢失，还需要 confirm、集群复制和磁盘可靠性。

## 死信触发条件

消息会进入死信交换机的常见原因：

- 消费者 reject/nack 且 `requeue=false`。
- 消息 TTL 过期。
- 队列长度达到上限，旧消息被挤出或新消息无法进入。

## DLX 设计

队列通过 `x-dead-letter-exchange` 指定死信交换机，可通过 `x-dead-letter-routing-key` 指定死信路由键。死信队列应包含原始消息、失败原因、重试次数和业务 key，便于补偿。

## 延迟队列方案

- TTL + DLX：消息先进入延迟队列，TTL 到期后成为死信，再路由到真实消费队列。
- delayed message exchange 插件：通过插件支持延迟投递，使用更直接。

## TTL + DLX 局限

- 队列级 TTL 只能统一延迟。
- 消息级 TTL 可能受队头阻塞影响，前面长 TTL 消息会阻塞后面短 TTL 消息。
- 大量长延迟消息会占用队列资源。

## 实践建议

- 业务重试建议设置最大次数，超过后进入死信。
- 死信处理要有可视化、告警和人工重投能力。
- 大规模长时间定时任务优先考虑专业调度系统或延迟消息服务。

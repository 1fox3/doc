# 消费确认与 QoS

## Ack 模式

- 自动 ack：Broker 投递后立即认为成功，消费者宕机会丢消息。
- 手动 ack：业务处理成功后调用 `basicAck`，可靠性更高。

生产环境核心消息应使用手动 ack。

## nack 与 reject

- `basicAck`：确认成功，消息删除。
- `basicReject`：拒绝单条消息，可选择是否 requeue。
- `basicNack`：可批量拒绝消息，也可选择是否 requeue。

如果反复 requeue 毒丸消息，会造成消息不断被重新投递，拖垮消费者。应结合死信队列处理不可恢复错误。

## Prefetch/QoS

`basicQos(prefetchCount)` 控制单个消费者未确认消息数量。它影响：

- 消费公平性。
- 内存占用。
- 吞吐和并发。
- 慢消费者对队列的影响。

prefetch 过大可能导致单个消费者缓存大量消息，其他消费者空闲；过小则吞吐不足。

## 消费幂等

RabbitMQ 至少一次投递是常见实践。消费者必须能够处理重复消息：

- 数据库唯一约束。
- 消费记录表。
- 状态机校验。
- 幂等下游接口。

## 消费失败分类

- 临时失败：网络抖动、下游超时，可短期重试。
- 业务失败：参数非法、状态不允许，应进入死信或人工处理。
- 毒丸消息：每次消费都失败，应限制重试次数并隔离。

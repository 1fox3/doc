# 优先级、队列类型与集群

## 优先级队列

通过 `x-max-priority` 声明队列最大优先级，消息发布时设置 priority。优先级队列会增加 Broker 排序成本，不适合大量优先级层级和极高吞吐场景。

## Classic Queue

经典队列是传统队列类型，适合普通业务消息。历史上的 mirrored queue 已不推荐作为新方案。

## Quorum Queue

Quorum queue 基于 Raft 复制，面向高可用和数据安全，是新版本中推荐的高可靠队列方案。它牺牲部分吞吐换取更好的复制一致性和故障恢复能力。

适用场景：

- 核心业务消息。
- 需要节点故障后自动恢复。
- 不希望使用已过时 mirrored queue。

## Stream Queue

Stream queue 面向日志流和大规模顺序读取，提供类似流的语义。它不是传统队列的完全替代，适用场景更接近事件流和历史回放。

## Lazy Queue

Lazy queue 倾向于把消息落盘，减少内存占用，适合消费慢、堆积多的场景，但读取延迟可能更高。

## 集群与网络分区

RabbitMQ 集群共享元数据，队列数据位置取决于队列类型和复制策略。网络分区可能导致可用性和一致性冲突，应配置合理的 partition handling 策略，并优先使用 quorum queue 承载核心消息。

## 实践建议

- 新核心队列优先评估 quorum queue。
- 不要在单个 vhost 下创建过多高频短生命周期队列。
- 队列复制不是消息幂等的替代，消费者仍要处理重复投递。

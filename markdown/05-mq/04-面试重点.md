# MQ 面试重点

## 通用问题

- MQ 解决异步解耦、削峰填谷、事件驱动和最终一致性，不适合强同步强一致链路。
- 消息语义通常是至少一次，业务必须设计幂等；至多一次可能丢消息，严格 exactly once 通常有边界和成本。
- 可靠链路包括生产确认、Broker 持久化、副本复制、消费确认、失败重试和死信补偿。
- 顺序性通常只在分区、队列或同一业务 key 内成立，不能轻易承诺全局顺序。

## Kafka

- 高吞吐来自顺序写、page cache、批量、压缩、零拷贝和分区并行。
- ISR、acks、min.insync.replicas 共同影响可靠性和可用性。
- Rebalance 会导致短暂停止消费，频繁 Rebalance 通常来自消费者超时、实例波动或分区分配策略不合理。
- 幂等生产者解决单分区单会话内重复写，事务用于跨分区原子写和 consume-transform-produce 场景。

## RocketMQ

- CommitLog 存消息主体，ConsumeQueue 是按 Topic/Queue 的逻辑消费索引。
- 顺序消息依赖同一业务 key 发送到同一队列，并使用顺序消费。
- 事务消息通过 half message、事务回查和本地事务状态协调最终一致。
- 延迟、重试、死信是业务消息治理常见能力。

## RabbitMQ

- Exchange 负责路由，Queue 负责存储，Binding 定义路由关系。
- Publisher Confirm、mandatory return、持久化、consumer ack 和 quorum queue 共同影响可靠性。
- Prefetch 控制消费者未确认消息数量，影响吞吐、公平性和内存压力。
- RabbitMQ 适合复杂路由和业务队列，不适合超大吞吐日志流替代 Kafka。

## 项目表达

- 讲 MQ 项目时要说明选型理由、投递语义、幂等设计、重试死信、监控告警和容量评估。
- 不要只说“用了 MQ 解耦”，要讲失败场景：发送失败、Broker 挂、重复消费、消费慢、顺序要求和补偿机制。

# 消息类型

## 普通消息

普通消息是最基础类型，适合异步解耦、削峰和事件通知。它只保证同一队列内存储顺序，不保证全局顺序。

## 顺序消息

顺序消息分为：

- 全局顺序：Topic 只有一个队列，吞吐低，不适合高并发。
- 分区顺序：同一 sharding key 进入同一队列，常用于同一订单、同一用户、同一设备的事件顺序。

顺序消息的核心约束是发送端固定队列、消费端按队列串行处理。消费失败会阻塞该队列后续消息。

## 延迟消息

RocketMQ 4.x 使用固定延迟级别，消息先进入 `SCHEDULE_TOPIC_XXXX`，到期后再投递到真实 Topic。RocketMQ 5.x 提供更灵活的定时/延时消息能力。

适用场景：订单超时关闭、延迟通知、重试退避。不适合超大规模长时间定时任务替代专业调度系统。

## 事务消息

事务消息用于保证本地事务和消息发送最终一致。典型链路是发送半消息、执行本地事务、提交或回滚半消息，Broker 在状态未知时发起事务回查。

## 批量消息

批量消息减少网络往返，提高吞吐。限制是同一批消息 Topic 必须一致，消息体总大小受限制，且过大批次会增加延迟和失败重试成本。

## 过滤消息

- Tag 过滤：Broker 端基于 Tag 进行轻量过滤，最常用。
- SQL92 属性过滤：表达能力更强，但有额外开销和配置要求。
- 复杂业务过滤不建议全部压到 MQ，应在消费端或索引系统中处理。

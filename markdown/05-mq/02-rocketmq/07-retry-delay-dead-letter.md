# 重试、延迟与死信

## 生产重试

生产发送失败后，客户端可重试其他 Broker。生产重试解决的是发送链路瞬时失败，不代表业务只会收到一次消息。只要存在重试，就要假设可能重复投递。

## 消费重试

集群消费下，消费者返回失败、抛异常或超时未确认时，RocketMQ 会按重试策略重新投递。重试消息通常进入重试 Topic，例如 `%RETRY%ConsumerGroup`。

## 死信队列

消费失败超过最大重试次数后，消息进入死信队列，例如 `%DLQ%ConsumerGroup`。死信队列不是发送失败的常规归宿，主要用于消费端长期无法成功处理的消息。

死信处理建议：

- 记录业务 key、messageId、失败原因和堆栈。
- 提供人工修复、重新投递或忽略能力。
- 对毒丸消息设置隔离机制，避免反复拖垮消费端。

## 延迟消息机制

RocketMQ 4.x 固定延迟级别不是任意时间戳调度。延迟消息先写入定时调度内部 Topic，到期后再恢复原 Topic 投递。

适用场景：

- 订单超时未支付关闭。
- 失败后延迟重试。
- 定时通知。

不适合场景：

- 数千万级长周期精确调度。
- 复杂 cron 任务。
- 强实时定时触发。

## 幂等设计

- 以订单号、流水号、业务事件 ID 做唯一约束。
- 消费状态机只允许合法状态迁移。
- 对下游请求传递幂等 token。
- 处理失败时区分可重试、不可重试和需要人工介入的异常。

# MQ 排障案例

## 消息丢失

- 生产端：未等待确认、异步发送回调未处理、重试后仍失败。
- Broker：未持久化、刷盘策略弱、副本不足、leader 故障时未同步。
- 消费端：先提交 offset/ack 后处理业务失败。
- 修复：生产确认、持久化、副本确认、消费成功后 ack、失败重试和补偿表。

## 消息重复

- 原因：生产重试、Broker 重投、消费处理成功但 ack/commit 失败、Rebalance。
- 修复：业务幂等，常见方案是唯一业务键、去重表、状态机幂等、版本号或幂等写。

## 消息堆积

- 排查：生产速率、消费速率、单条耗时、分区/队列热点、下游依赖、消费者实例数。
- 修复：扩分区和消费者、优化消费逻辑、批量处理、隔离慢下游、限流生产、临时旁路消费。
- 注意：分区数不足时单纯增加消费者无效。

## 顺序错乱

- 原因：同一业务 key 没有路由到同一分区/队列，并发消费，失败重试插队。
- 修复：按业务 key 选择分区/队列，单队列有序消费，业务侧按版本或状态机校验。

## 延迟抖动

- 原因：Broker IO、页缓存压力、GC、消费者慢、批量参数不合理、网络抖动。
- 排查：端到端时间戳、Broker 指标、消费 lag、慢消费日志、磁盘和网络。

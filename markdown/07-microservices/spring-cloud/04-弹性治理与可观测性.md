# 弹性治理与可观测性

## 目标

弹性治理的目标是在局部失败时保护整体系统，避免慢调用、错误重试和资源耗尽引发级联故障。核心手段包括超时、重试、熔断、限流、降级、隔离和背压。

## 超时

每个远程调用都必须设置连接超时、读取超时和总耗时预算。超时应按链路预算倒推，而不是所有服务统一配置一个大值。

设计原则：

- 上游超时时间应大于下游单次调用超时加重试耗时。
- 核心交易链路优先短超时和明确失败，后台任务可适当放宽。
- 数据库、缓存、消息队列、HTTP 客户端都需要独立超时配置。

## 重试

重试只适合处理短暂网络抖动、连接复用失效、临时限流等瞬时故障。重试会放大流量，必须和幂等、退避、最大次数、熔断配合。

不应自动重试的场景：

- 非幂等写操作，例如支付、扣库存、创建订单。
- 下游明确返回业务失败。
- 下游已过载或熔断打开。
- 请求耗时已经接近上游超时预算。

## 熔断

熔断器根据错误率、慢调用比例或连续失败次数判断下游是否不可用。当熔断打开时，直接返回降级结果或失败，避免继续压垮下游。

常见状态：

- Closed：正常放行请求并统计指标。
- Open：短路请求，不再调用下游。
- Half-Open：放行少量探测请求，根据结果决定恢复或继续熔断。

Hystrix 已进入维护状态，新项目通常使用 Resilience4j 或 Sentinel。Resilience4j 提供熔断、限流、舱壁、重试、限时器等模块；Sentinel 更强调流量控制、热点参数限流和实时规则治理。

## 限流与降级

限流用于保护服务容量，常见算法包括固定窗口、滑动窗口、令牌桶和漏桶。限流维度可以是接口、用户、租户、IP、应用、Header、热点参数或调用来源。

降级是在下游不可用或系统过载时返回可接受的替代结果，例如返回缓存数据、关闭非核心功能、异步处理或提示稍后重试。降级不是吞掉错误，而是有意识地牺牲非核心能力。

## 隔离

- 线程池隔离：为不同下游或不同业务分配独立线程池。
- 信号量隔离：限制并发数，开销低但不能隔离阻塞线程。
- 连接池隔离：数据库、Redis、HTTP 客户端按依赖拆分连接池。
- 舱壁模式：把核心链路和非核心链路分配到不同资源池。

## 日志、指标、追踪

可观测性包括日志、指标和链路追踪，三者需要围绕同一个 TraceId 或请求上下文关联。

- 日志：使用结构化日志记录服务名、实例、TraceId、SpanId、错误码和关键业务字段，避免记录敏感信息。
- 指标：关注请求量、成功率、错误率、P95/P99 延迟、线程池队列、连接池使用率、熔断状态、限流次数和降级次数。
- 追踪：网关入口生成或透传 TraceId，HTTP、RPC、消息队列都要传递上下文。

## Sleuth、Micrometer Tracing 与 Spring Boot Admin

Spring Cloud Sleuth 在新版本中被 Micrometer Tracing 接替，常与 OpenTelemetry、Zipkin、Jaeger、Tempo 等系统集成。Spring Boot Admin 可集中查看 Actuator 信息，例如健康状态、环境属性、日志级别、线程、内存和指标。生产环境应开启认证授权，并避免暴露敏感环境变量和配置内容。

## 组件

### Spring Cloud Eureka

#### 自我保护机制

- 自我保护背景
  - Eureka注册中心的节点都是平等的，及时N-1个节点挂掉也不影响其他节点的正常运行
  - 默认情况下，Eureka Server在一定时间内（默认90s）没有收到某个微服务实例的心跳，就会移除该实例
  - 如果是因为网络问题，微服务与Eurkea Server无法正常通信，但微服务本身是正常运行的，此时不应该移除这个微服务，所以就引入了自我保护机制

- 进入自我保护机制判定
  - 如果15分钟内超过85%的客户端没有正常心跳 ，则Eureka就认为客户端与注册中心出现了网络故障，Eureka Server进入自我保护机制

- 自我保护机制的处理
  - Eureka Server不再从注册列表中移除因长时间没有收到心跳而应该过期的服务
  - Eureka Server仍然能够接受新服务的注册和查询请求，但不会同步到其他节点上，保证当前节点可用
  - 网络稳定后，当前Eureka Server新的注册信息会被同步到其它节点中

- 开关
  - eureka.server.enable-self-preservation=true
  - eureka.server.eviction-interval-timer-in-ms:3000

### Spring Cloud Feign

#### 介绍

- Feign是声明式、模板化的web service客户端，它让微服务之间的调用变得更简单，类似controller调用service

- Spring Cloud集成了Ribbon和Eureka，在使用Feign时提供负载均衡的http客户端，使用HTTP请求远程服务，就像调用本地方法一样

### Spring Cloud OpenFeign

#### 介绍 (6)

- 它是更高一层的抽象，使用它可以更简单的调用无误接口，只要通过一个接口，然后使用一个注解就能生成动态代理类简单完成复杂的调用过程，像调用本地接口一样完成远程接口调用

- 支持可插拔式的编码器和解码器，支持Spring MVC注解，并使用Spring Web中相同的HttpMessageConverts转换器

- Spring Cloud继承了Eureka、Spring Cloud CircuitBreaker以及Spring Cloud LoadBalancer，在使用OpenFeign时都可以提供负载均衡的HTTP声明式客户端

#### 以Feign的区别

- OpenFeign是Spring Cloud旗下子项目，在Feign的基础之上，又支持了JAX_RS和Spring MVC的注解，并通过动态代理的方式动态生成实现类，在实现类中实现负载均衡并调用其他服务

- Feign本身不支持Spring MVC注解的，它有自己的一套注解

#### 连接池

- JDK的HttpURLConnection原生连接（默认）
  - 每次请求都会创建一个新的连接，不使用连接池，不重用连接

- httpclient
  - 启用参数feign.httpclient.enabled
  - 依赖feign-httpclient

- HttpComponents Client 5
  - 启用参数feign.httpclient.hc5.enbaled
  - 依赖feign-hc5

- okhttp
  - 启用参数feign.okhttp.enabled
  - 依赖feign-okhttp

### Spring Cloud Gateway

#### 介绍 (12)

- 是Spring Cloud Finchley版推出的新组件，替代旧网关Zuul

- Spring Cloud微服务的一个子项目，属于Spring开源社区，依赖名：spring-cloud-starter-gateway

- 响应式的非阻塞API，同时支持websockets

#### 工作原理

- 客户端向Spring Cloud Gateway发出请求

- 如果Gateway Handler Mapping确定请求与路由匹配，则将其发送到Gateway Web Handler，它通过特定于请求的过滤器链运行请求

- 过滤器可以在发送代理请求之前和之后运行相关逻辑，在PRE前置过滤器执行后发出代理请求，代理请求执行后，再执行POST后置过滤器

#### 组件

- Route（路由）
  - 网关的基本构建单元，它由ID，目标URO、谓词集合（predicates）和过滤器集合（filters）组成

- Predicate（谓词/断言）
  - 这是一个Java8函数谓词，输入类型为Spring框架中的ServerWebExchange，可以匹配来自HTTP请求的任何内容，例如标头或参数等，如果谓词匹配为真，则转发路由
  - 相关实现类
    - 时间相关
      - BeforeRoutePredicateFactory
      - AfterRoutePredicateFactory
      - BetweenRoutePredicateFactory
    - 请求头
      - HeaderRoutePredicateFactory
    - 请求方法
      - MethodRoutePredicateFactory
    - 路径
      - PathRoutePredicateFactory

- Filter(过滤器)
  - 使用特定工厂构建的GatewayFilter实例，可以在路由转发请求之前和之后修改请求和响应参数
  - 生命周期
    - PRE（路由转发前执行）
    - POST（路由转发后执行）
  - 分类
    - 全局过滤器
      - 相关实现
        - ForwardPathFilter
          - forward://协议过滤器
        - AdaptCachedBodyGlobalFilter
          - 缓存HTTP Body过滤器
        - RouteToRequestUrlFilter
          - 路由转发过滤器
        - NettyRoutingFilter
          - Netty路由过滤器
        - SentinelGatewayFilter
          - Sentinel过滤器
    - 局部过滤器
      - 仅对部分路由生效
      - 相关实现
        - AddRequestHander
          - 为请求添加请求头
        - AddRequestParameter
          - 为请求添加请求参数
        - PrefixPath
          - 为请求添加前缀
        - StripPath
          - 转发时移除路径前缀
        - SetStatus
          - 设置最终返回的HTTP状态码

### Spring Cloud Stream

#### 介绍 (27)

- 构建消息驱动的微服务应用程序的框架，可以屏蔽MQ底层的选型的类型，以及操作差异，降低了切换成本，同意了消息编程模型

- 基于Spring Boot建立独立的生产级Spring应用程序，并使用Spring Integration提供与消息代理的连接

#### Binder与Binding

- Binder
  - 用于与外部消息中间件集成的组件，用于创建Binding
  - 具体实现
    - Kafka
      - KafkaMessageChannelBinder
    - RabbitMQ
      - RabbitMessageChannelBinder
    - RocketMQ
      - RocketMQMessageChannelBinder

- Binding
  - 划分
    - 输入Binding
    - 输出Binding
  - 充当消息中间件与应用程序提供者和消费者之间的桥梁

#### 与Spring Cloud Bus区别

- Bus是在Stream基础之上的再次进行抽象封装，使我们可以不用理解消息发送、监听等概念的基础上使用消息来完成业务逻辑的处理

- Stream中，异步调用能让各个服务充分解耦而且也更加灵活

- Bus借助消息驱动来实现将消息广播到各个服务中，然后服务对这些消息进行消费

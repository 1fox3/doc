# 安全与一致性

## 安全边界

微服务安全需要同时覆盖入口安全、服务间安全、数据安全和运维安全。网关只能解决统一入口问题，不能替代服务内部权限校验和数据访问控制。

## Spring Security

Spring Security 提供认证、授权、过滤器链、安全上下文、密码编码、CSRF 防护和方法级权限控制。Spring Boot Security 自动配置会根据 classpath 和配置启用默认安全能力，生产项目应显式定义安全过滤链。

关键点：

- 密码必须使用 BCrypt、Argon2、PBKDF2 等安全哈希算法，不保存明文或可逆密文。
- 使用 `SecurityFilterChain` 定义不同路径的认证授权规则。
- 对浏览器表单场景关注 CSRF，对纯 Token API 场景可按需关闭 CSRF。
- 方法级权限适合保护细粒度业务操作，但不能替代数据权限过滤。

## OAuth2 与 JWT

OAuth2 是授权框架，常见角色包括 Resource Owner、Client、Authorization Server、Resource Server。OIDC 在 OAuth2 基础上提供用户身份认证语义。

JWT 是自包含 Token，包含 Header、Payload 和 Signature。它适合无状态鉴权，但一旦签发，在过期前默认难以撤销。

JWT 治理要点：

- Access Token 有效期应较短，Refresh Token 单独管理。
- 签名算法固定白名单，避免 `none` 或算法混淆问题。
- Token 中只放必要声明，不放敏感信息。
- 密钥支持轮换，Header 中使用 `kid` 标识密钥版本。
- 高风险场景需要黑名单、版本号或会话状态实现撤销。

## 网关安全

网关适合实现统一认证解析、基础鉴权、限流、防刷、IP 黑白名单、请求大小限制、Header 清洗和审计日志。服务不能盲目信任外部传入的用户 Header，用户身份 Header 应由网关在认证后写入，并在进入网关前清除同名 Header。

## 分布式一致性

服务拆分后，原本单库事务会变成跨服务、跨数据库、跨消息系统的协作。需要根据业务选择强一致、最终一致或人工补偿。

常见问题：

- 下单成功但库存扣减失败。
- 支付成功但订单状态未更新。
- 消息发送成功但消费失败。
- 重试导致重复扣款或重复发货。
- 缓存、读模型和主库数据短暂不一致。

## 解决方案

- 本地事务加可靠消息：业务数据和消息记录在同一个本地事务中提交，再由后台任务或事务消息投递到消息队列。
- TCC：Try 预留资源，Confirm 确认提交，Cancel 释放资源，必须处理幂等、空回滚和悬挂。
- Saga：将长事务拆成多个本地事务，每个步骤有对应补偿动作，适合跨服务长流程。
- Seata AT：通过代理数据源记录 undo log，并由全局事务协调器管理提交或回滚，需要关注全局锁、热点行、长事务和 SQL 支持范围。

## 幂等与补偿

幂等是分布式系统的基础能力。可以通过幂等键、业务唯一约束、状态机、消息处理表或回调流水号去重实现。最终一致系统必须有对账和补偿任务，超过自动补偿能力的异常进入人工处理队列。

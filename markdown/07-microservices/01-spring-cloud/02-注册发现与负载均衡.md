# 注册发现与负载均衡

## 注册中心职责

注册中心保存服务实例地址、端口、元数据、健康状态和集群分组信息，使调用方无需硬编码下游地址。Spring Cloud 常见注册中心包括 Eureka、Consul、Nacos 和 ZooKeeper。

## 服务注册流程

- 服务启动后向注册中心提交服务名、IP、端口、协议、权重、集群、版本和元数据。
- 注册中心保存实例信息，并根据实例类型进行健康检查或接收心跳。
- 消费者订阅服务列表，本地缓存可用实例，并在列表变化时更新。
- 服务下线时主动注销；异常宕机时由心跳超时或健康检查剔除。

## 服务发现模式

- 客户端发现：消费者从注册中心拉取实例列表，在本地做负载均衡。Ribbon、Spring Cloud LoadBalancer 属于此类。
- 服务端发现：消费者请求网关或代理，由网关根据注册中心信息选择后端实例。
- DNS 发现：通过 DNS 返回服务地址，通用但元数据表达能力和实时性较弱。

## 负载均衡策略

- 轮询：适合同质实例，实现简单。
- 随机：实例数量较多时分布较均匀。
- 权重：适合机器规格不同、灰度发布或容量差异场景。
- 最少连接或最小响应时间：适合请求耗时差异大的服务，需要采集运行时指标。
- 一致性哈希：适合会话黏性或缓存命中场景，扩缩容时要关注流量倾斜。

## Eureka

Eureka 强调 AP 和客户端缓存。注册中心短暂不可用时，消费者仍可依赖本地缓存访问已有实例。它适合服务发现可用性优先的场景，但 Netflix 体系已进入维护模式，新项目需要评估替代方案。

关键点：

- 服务端通过心跳判断实例状态。
- 客户端缓存会带来实例上下线延迟。
- 自我保护机制可避免网络抖动导致大量实例误剔除，但也可能短期保留不可用实例。

## Consul

Consul 支持服务发现、健康检查、KV、ACL 和多数据中心，偏基础设施治理。它支持 HTTP、TCP、脚本等健康检查方式，适合多语言、多平台场景。

## Ribbon 与 Spring Cloud LoadBalancer

Ribbon 是 Netflix 客户端负载均衡组件，已进入维护模式。Spring Cloud LoadBalancer 是 Spring Cloud 官方负载均衡抽象，配合 DiscoveryClient 使用，是新项目更合适的选择。

迁移时需要关注：

- 默认负载均衡策略差异。
- 重试配置差异。
- Feign、RestTemplate、WebClient 的集成方式。
- 实例元数据和灰度路由逻辑是否需要重写。

## 元数据治理

实例元数据可用于版本路由、灰度、机房路由、权重、协议标记和能力描述。常见字段包括 `version`、`zone`、`region`、`weight`、`protocol`、`secure`。元数据应统一命名，避免不同服务用不同字段表达同一含义。

## 故障处理

- 消费端必须使用本地缓存，注册中心短暂不可用不应导致存量调用全部失败。
- 实例剔除不能过于激进，否则网络抖动会造成大面积实例抖动。
- 服务下线应先从注册中心摘除，再等待连接耗尽，最后停止进程。
- 多注册中心或多集群部署时，需要明确数据同步策略和故障切换策略。

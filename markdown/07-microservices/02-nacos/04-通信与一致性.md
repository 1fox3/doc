# 通信与一致性

## Nacos 2.x 通信

Nacos 2.x 引入 gRPC 长连接，提升配置推送、服务注册和服务订阅性能。默认端口和 `server.port` 有偏移关系。

常见端口：

- `8848`：HTTP 控制台和 OpenAPI 默认端口。
- `9848`：客户端 gRPC 请求端口，通常为 `server.port + 1000`。
- `9849`：服务端之间 gRPC 通信端口，通常为 `server.port + 1001`。
- `7848`：部分集群通信或旧版本协议相关端口，部署时需结合版本确认。

生产部署必须放通 HTTP 和 gRPC 端口，否则会出现控制台可访问但客户端注册、订阅或动态刷新异常的问题。

## 寻址机制

- 单机寻址：客户端直接配置单个 Nacos 地址，适合本地开发。
- 文件寻址：客户端从本地配置读取服务端列表。
- 地址服务器寻址：客户端从地址服务器获取 Nacos 集群地址，适合大规模环境统一管理。

生产环境应使用域名、负载均衡或地址服务器降低服务端节点变更对客户端配置的影响。

## 一致性协议

Nacos 根据数据类型选择不同一致性机制。

- Distro：面向临时实例，强调最终一致性和高可用。节点间会进行数据同步和校验，适合实例频繁上下线的场景。
- Raft/JRaft：面向需要强一致的数据，例如持久实例和部分配置元数据。通过 Leader 处理写入并复制日志，保证多数派一致。

理解 AP/CP 时要结合具体数据类型，不能简单说 Nacos 整体只属于 AP 或 CP。

## 集群同步

Nacos 集群需要保持节点列表一致，并确保节点间网络、端口、时钟和数据库连接稳定。临时实例数据主要在内存中同步，配置和权限等数据依赖数据库持久化。

## 一致性风险

- 网络分区可能导致不同节点短时间看到不同实例列表。
- 客户端本地缓存可能带来实例上下线延迟。
- 数据库异常会影响配置发布、权限和命名空间等持久化数据。
- 直接修改数据库可能绕过 Nacos 缓存和版本机制，造成服务端状态不一致。

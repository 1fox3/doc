# 配置中心与动态刷新

## 配置中心职责

Nacos Config 用于集中管理不同环境、应用、租户和集群的配置，并支持动态变更、权限控制、版本管理、灰度发布和回滚。

## 配置组织

推荐命名方式：

- `Namespace` 表示环境或租户，例如 prod、staging、tenant-a。
- `Group` 表示业务域或应用组，例如 ORDER、PAYMENT。
- `DataId` 表示具体配置文件，例如 `order-service.yaml`。

配置优先级必须明确，否则容易出现本地配置、环境变量、启动参数和配置中心互相覆盖的问题。

## 配置格式

常见格式包括 properties、yaml、json、xml 和纯文本。微服务项目中推荐使用 yaml 或 properties 管理应用配置，复杂规则可以使用 json 表达。

治理要求：

- 同一应用尽量使用统一格式。
- 配置项命名稳定，避免随代码版本频繁改名。
- 数值类配置标明单位，例如毫秒、秒、字节、百分比。
- 列表和 Map 类型配置需要明确空值语义。

## 动态刷新机制

动态刷新通常包含四步：

- Nacos 保存新版本并生成变更事件。
- 客户端监听配置变化并拉取最新内容。
- 客户端解析配置并更新本地环境上下文。
- 受影响的 Bean、规则或缓存重新加载。

不是所有配置都适合动态刷新。线程池核心参数、日志级别、限流规则适合动态调整；服务端口、数据源 URL、重要安全参数通常更适合重启发布。

## 灰度与回滚

配置灰度用于先让部分实例、用户或租户使用新配置。每次配置变更应有版本号、变更人、变更原因和差异记录。回滚应恢复到明确历史版本，而不是手工复制旧内容。

## 加密与敏感配置

敏感配置包括数据库密码、Redis 密码、MQ 密码、OAuth2 Client Secret、JWT 密钥、证书和第三方 API Key。

治理原则：

- 不把明文密钥提交到代码仓库。
- Nacos 开启权限控制和操作审计。
- 客户端日志禁止打印完整配置。
- 密钥定期轮换，支持双密钥过渡。
- 高安全要求场景接入 KMS、Vault 或云厂商密钥管理服务。

## 并发修改

多人同时修改同一配置时，可能出现覆盖发布。应通过版本号、MD5、历史记录、发布前差异对比和审批流程降低风险。

## 排查路径

- 检查 Namespace、Group、DataId 是否完全匹配。
- 检查客户端是否已成功连接 Nacos 并注册监听。
- 检查配置格式是否解析失败。
- 检查 Bean 是否支持刷新，例如是否使用 `@RefreshScope` 或规则监听器。
- 检查权限、网络、防火墙和 Nacos 2.x gRPC 端口是否可达。
- 检查本地缓存是否仍在使用旧版本配置。

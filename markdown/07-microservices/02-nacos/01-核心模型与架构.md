# 核心模型与架构

## 核心能力

Nacos 的核心能力包括服务发现和健康检查、动态配置服务、动态 DNS、服务及元数据管理。注册中心关注实例可用性和服务发现，配置中心关注配置变更传播、版本管理、灰度和回滚。

## 配置模型

- `Namespace`：命名空间，常用于环境或租户隔离，例如 dev、test、prod、tenant-a。
- `Group`：配置分组，常用于业务域或应用组隔离，例如 ORDER、PAYMENT。
- `DataId`：配置标识，通常对应一个配置文件，例如 `order-service.yaml`。
- 配置内容：具体配置文本，支持 properties、yaml、json、xml、text 等格式。

## 服务模型

- `Namespace`：隔离不同环境或租户的服务注册数据。
- `Group`：隔离不同业务组的同名服务。
- `Service`：服务名，代表一组可提供相同能力的实例。
- `Cluster`：集群或机房分组，用于就近访问和容灾。
- `Instance`：具体服务实例，包含 IP、端口、权重、健康状态、上线状态和元数据。

## 架构组成

- Client：应用侧 SDK，负责注册实例、发送心跳、订阅服务列表、监听配置变化和维护本地缓存。
- Server：Nacos 服务端，负责请求处理、权限校验、数据存储、集群同步和推送变更。
- Console：控制台，提供配置管理、服务列表、权限管理和操作入口。
- Storage：集群模式通常使用 MySQL 保存配置、用户、权限、命名空间等数据。

## 元数据

实例元数据可以承载版本、权重、机房、协议、灰度标记和自定义路由标签。元数据应有统一命名规范，避免消费者解析逻辑散落在不同服务中。

常见字段：

- `version`：服务版本，用于灰度和兼容性路由。
- `zone` 或 `region`：机房和地域，用于就近访问。
- `weight`：实例权重，用于流量分配。
- `protocol`：HTTP、gRPC、Dubbo 等调用协议。

# 部署安全与排障

## 数据存储

Nacos 单机模式可使用内嵌数据库，集群模式通常使用 MySQL 保存配置、用户、权限、命名空间等数据。服务实例运行时数据主要保存在内存并通过一致性协议同步。

数据库治理要求：

- 集群模式使用独立 MySQL，并做好备份和高可用。
- 配置表变更要通过 Nacos 控制台或 OpenAPI，避免直接改库造成缓存不一致。
- 关注配置历史表增长，定期归档或清理。

## 认证与权限

Nacos 支持用户、角色和权限控制。生产环境不能使用默认账号密码，控制台和 OpenAPI 都应纳入权限治理。

安全要点：

- 开启认证并修改默认密钥和默认用户。
- 按命名空间、Group 或操作类型分配权限。
- 控制台入口限制网络来源。
- OpenAPI 调用记录审计日志。
- 敏感配置配合加密插件或外部密钥管理。

## 部署检查

- 节点数量满足高可用要求，避免单点。
- 所有节点的端口、防火墙和安全组配置一致。
- MySQL 连接池、账号权限、备份策略和慢 SQL 监控已配置。
- JVM、磁盘、网络、线程池和日志采集纳入监控。
- 控制台入口只对可信网络开放。

## 客户端配置检查

- Nacos 地址使用域名或负载均衡入口，避免写死单节点 IP。
- 客户端版本与服务端版本兼容。
- Namespace、Group、DataId、服务名和集群名命名规范统一。
- 注册发现和配置中心的认证信息分级管理，不写入代码仓库。

## 常见故障

- 注册失败：检查服务名、Namespace、网络、认证、gRPC 端口、客户端版本兼容性。
- 发现不到实例：检查 Group、Namespace、实例健康状态、上线状态、本地缓存和订阅日志。
- 配置不刷新：检查 DataId、Group、Namespace、监听状态、配置格式、权限和 Bean 刷新机制。
- 集群异常：检查节点列表、MySQL、时钟、端口、Leader 状态和节点间网络。

## 变更规范

- 重要配置发布前进行差异对比和格式校验。
- 变更后观察错误率、延迟、业务指标和客户端日志。
- 灰度配置先覆盖少量实例或租户，再逐步放量。
- 回滚使用历史版本，不手工拼接旧配置。

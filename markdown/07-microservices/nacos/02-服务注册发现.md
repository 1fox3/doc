# 服务注册发现

## 注册流程

- 服务启动后向 Nacos 提交服务名、IP、端口、命名空间、Group、集群、权重和元数据。
- Nacos 保存实例信息，并根据实例类型接收心跳或进行服务端健康探测。
- 消费者订阅服务列表，本地缓存可用实例，并在列表变化时更新。
- 服务下线时主动注销；异常宕机时由心跳超时或健康检查剔除。

## 临时实例与持久实例

- 临时实例依赖客户端心跳，心跳超时后会被自动删除，适合普通应用实例。
- 持久实例由服务端主动探测健康状态，实例不会因心跳丢失直接删除，适合 DNS、外部服务或需要人工管理的实例。
- 临时实例更偏 AP，强调可用性；持久实例更偏 CP，强调注册数据一致性。

## 健康状态与上线状态

服务实例有健康状态和上线状态两个维度。健康状态反映实例是否可用，上线状态反映实例是否参与流量。实例健康但被手动下线时，不应再被消费者选中。

下线发布建议：

- 先将实例标记为下线或从注册中心摘除。
- 等待客户端服务列表刷新和存量请求处理完成。
- 再停止进程，避免流量打到正在关闭的实例。

## 服务订阅与本地缓存

Nacos 客户端订阅服务后，会维护本地实例缓存。注册中心短暂不可用时，消费者仍可基于本地缓存访问已有实例。缓存提升可用性，但会引入实例上下线传播延迟，需要配合合理的健康检查和下线等待时间。

## 集群与权重

Cluster 可用于机房、地域或逻辑分组。消费者可以优先访问同集群实例，跨集群作为兜底。权重可用于流量倾斜、灰度发布或机器规格差异场景。

## 排查要点

- 注册失败：检查服务名、Namespace、Group、认证、网络、gRPC 端口和客户端版本兼容性。
- 发现不到实例：检查 Namespace、Group、实例健康状态、上线状态、本地缓存和订阅日志。
- 流量未按预期分配：检查权重、集群策略、消费者负载均衡逻辑和实例元数据。

# 升级路线与检查项

## 升级策略

大版本升级应分阶段进行，避免同时修改框架版本、JDK、业务逻辑和基础设施。

推荐顺序：

1. 先升级到当前大版本的最新补丁版本。
2. 清理废弃 API 和过期配置。
3. 升级构建工具和插件。
4. 升级 Spring Boot 大版本。
5. 修复编译错误和配置错误。
6. 运行自动化测试和关键链路回归。
7. 做压测、灰度和监控验证。

## 依赖检查

- Spring Cloud 版本必须与 Spring Boot 版本匹配。
- ORM、连接池、数据库驱动、Redis、MQ、搜索、任务调度等依赖需要确认兼容矩阵。
- APM、日志采集、字节码增强、Mock 工具可能受 Java 版本和框架版本影响。
- 内部 Starter 需要同步迁移自动配置声明和 Jakarta API。

## 配置检查

- 检查废弃配置项和改名配置项。
- 检查 Profile 激活方式。
- 检查配置中心加载顺序。
- 检查 Actuator 暴露端点和安全策略。
- 检查日志格式、日志级别和链路追踪字段。

## 代码检查

- 替换 `javax.*` 为 `jakarta.*`。
- 修复 Spring Security 配置方式。
- 检查自调用导致的事务、异步、缓存代理失效。
- 检查循环依赖。
- 检查路径匹配策略对 Controller、拦截器和接口文档工具的影响。
- 检查反射、动态代理和资源加载对 Native Image 的影响。

## 验证清单

- 编译和单元测试通过。
- 集成测试覆盖数据库、缓存、消息、外部 HTTP 调用。
- 关键接口回归通过。
- Actuator 健康检查、指标和追踪正常。
- 启动时间、内存、线程数、连接池指标符合预期。
- SQL 执行计划和慢 SQL 无异常退化。
- 灰度期间错误率、延迟、GC、连接池和队列积压稳定。

## 回滚准备

升级发布前应准备可执行回滚方案，包括旧镜像、数据库兼容策略、配置回滚、消息兼容和缓存 key 兼容。数据库变更应优先采用向前兼容方式，避免新旧版本无法同时运行。

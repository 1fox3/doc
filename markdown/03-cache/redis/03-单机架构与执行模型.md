# 单机架构与执行模型

## 单机数据库结构

Redis 服务器通过 `redisServer` 保存全局状态，包括数据库数组、客户端列表、持久化状态、复制状态、发布订阅信息和慢查询日志。

每个逻辑库由 `redisDb` 表示，核心结构包括：

- `dict`：保存 key 到 value 的映射。
- `expires`：保存 key 到过期时间的映射。
- `watched_keys`：保存事务 `WATCH` 监控关系。

客户端状态由 `client` 保存，包括连接 fd、当前数据库、输入缓冲区、输出缓冲区、命令参数、认证状态、事务状态和阻塞状态。

## 命令执行流程

- 读取客户端请求并解析 RESP 协议。
- 根据命令表查找 `redisCommand`。
- 校验参数数量、认证、ACL、只读状态、内存限制和集群槽位。
- 执行命令函数，修改 keyspace 或返回结果。
- 更新统计信息、慢查询、AOF 缓冲区、复制传播和键空间通知。
- 将响应写入客户端输出缓冲区，等待 socket 可写时发送。

## 单线程命令执行

Redis 常说的单线程，主要指命令执行和数据结构修改在主线程完成。这样能避免复杂锁竞争，保证单条命令原子性。

单线程并不代表 Redis 只有一个线程。后台线程会处理关闭文件、异步刷盘、惰性释放等任务；Redis 6 之后还可以启用 IO 多线程处理网络读写。

## 事件模型

Redis 使用事件循环处理文件事件和时间事件。

- 文件事件：封装 socket 可读、可写事件，底层可使用 `select`、`epoll`、`kqueue` 等机制。
- 时间事件：执行 `serverCron` 等周期任务。
- `serverCron`：默认由 `hz` 控制频率，负责过期 key 清理、客户端清理、统计更新、持久化调度、复制心跳和集群维护。

## 事务

Redis 事务由 `MULTI`、`EXEC`、`DISCARD`、`WATCH` 组成。

- `MULTI` 后的命令进入队列，不立即执行。
- `EXEC` 按入队顺序执行命令。
- 入队阶段语法错误会导致事务整体拒绝执行。
- 执行阶段某条命令失败不会回滚其他命令。
- `WATCH` 是乐观锁，被监控 key 在 `EXEC` 前被修改时，事务取消。

Redis 事务适合轻量原子批处理，不适合需要回滚和复杂隔离级别的业务事务。

## Lua 与函数

Lua 脚本通过 `EVAL`、`EVALSHA` 执行，脚本执行期间具有原子性。适合封装多步读写、分布式锁释放、限流计数和库存扣减。

使用 Lua 要注意：

- 脚本不能执行过久，否则会阻塞主线程。
- Cluster 中脚本访问的 key 应位于同一 hash slot。
- 脚本要显式传入 key，避免隐藏访问影响迁移和审计。
- Redis 7 的 Functions 更适合长期注册和版本化管理服务端逻辑。

## 发布订阅

Pub/Sub 通过 `PUBLISH`、`SUBSCRIBE`、`PSUBSCRIBE` 实现消息广播。

它不持久化消息，也没有消费确认。消费者断开期间的消息会丢失，因此适合在线通知、配置广播和失效通知，不适合作为可靠消息队列。

## 阻塞点

- 大 key 读写、删除和序列化。
- O(N) 命令，例如 `KEYS`、超大 `SMEMBERS`、大范围 `ZRANGE`。
- `SAVE` 同步生成 RDB。
- AOF `appendfsync always` 带来的刷盘等待。
- AOF 重写期间的 fork 和 copy-on-write 内存压力。
- Cluster 迁移大 key 或热点槽。

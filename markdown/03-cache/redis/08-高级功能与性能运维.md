# 高级功能与性能运维

## Stream 消息流

Stream 适合轻量级消息队列和事件流水。

常用命令：

- `XADD`：追加消息。
- `XREAD`：按 ID 读取消息。
- `XGROUP CREATE`：创建消费者组。
- `XREADGROUP`：按消费者组消费。
- `XACK`：确认消息。
- `XPENDING`：查看待确认消息。
- `XAUTOCLAIM`：转移长时间未确认消息。
- `XTRIM`：裁剪消息流，控制内存。

使用 Stream 要设置裁剪策略，否则消息会持续堆积。消费者处理必须具备幂等性，因为故障恢复和消息转移可能导致重复消费。

## 模块系统

Redis Modules 允许通过外部模块扩展命令和数据类型，例如搜索、时序、JSON 等能力。

使用模块要关注：

- 模块版本与 Redis 版本兼容性。
- RDB/AOF 编码兼容性，避免升级后无法加载历史数据。
- 模块命令的复杂度和阻塞风险。
- 主从、Cluster、备份恢复是否完整支持模块数据。

## 异步删除

`UNLINK` 是 `DEL` 的异步释放版本，会先从 keyspace 解除引用，再由后台线程释放内存。

适用场景：

- 删除大 key。
- 清理批量临时 key。
- 降低主线程释放复杂对象的延迟。

`FLUSHDB ASYNC` 和 `FLUSHALL ASYNC` 可异步清空数据库，但仍是高风险操作，生产环境需要严格权限和审计。

## 后台线程与 IO 多线程

后台线程主要用于：

- `bio_close_file`：关闭 AOF/RDB 过程中的临时文件。
- `bio_aof_fsync`：异步执行 AOF fsync。
- `bio_lazy_free`：释放异步删除的大对象。

Redis 6 引入 IO 多线程，可通过以下配置启用：

- `io-threads 4`：设置 IO 线程数。
- `io-threads-do-reads yes`：开启多线程读请求解析。

IO 多线程只加速网络读写和协议处理，命令执行仍主要在主线程。CPU 瓶颈如果来自慢命令或大 key，多线程 IO 帮助有限。

## 慢查询与延迟

慢查询只记录命令执行耗时，不包含网络排队和客户端读取耗时。

常用配置和命令：

- `slowlog-log-slower-than`：慢查询阈值，单位微秒。
- `slowlog-max-len`：慢查询保留条数。
- `SLOWLOG GET`：查看慢查询。
- `LATENCY DOCTOR`：分析延迟事件。
- `INFO commandstats`：查看命令调用次数和总耗时。

排查慢请求要同时看 Redis、客户端、网络和系统资源。

## Big Key 排查

常用方法：

- `redis-cli --bigkeys`：扫描各类型较大的 key。
- `MEMORY USAGE key`：查看 key 近似内存。
- `STRLEN`、`HLEN`、`LLEN`、`SCARD`、`ZCARD`：查看元素数量。
- `SCAN` 分批扫描，避免 `KEYS` 阻塞。

治理方式包括拆分、压缩、限制写入、异步删除和迁移前预处理。

## Hot Key 排查

常用方法：

- `redis-cli --hotkeys`：依赖 LFU 统计，需启用 LFU 相关策略才更有意义。
- 客户端埋点统计 key 访问频率。
- 代理层采样。
- Redis 监控或 `MONITOR` 临时观察，`MONITOR` 开销较高，不适合长时间生产运行。

治理方式包括本地缓存、请求合并、读写分离、key 分片和限流。

## 常见阻塞原因

- 使用 `KEYS`、`SORT`、大范围集合运算等 O(N) 命令。
- 同步删除大 key 或清空数据库。
- `SAVE`、AOF fsync、AOF 重写和 fork 抖动。
- 客户端输出缓冲区过大，慢客户端导致内存堆积。
- 主从全量同步或复制缓冲区积压。
- Cluster 迁移大 key 或槽位热点。

## 运维检查清单

- `INFO memory`：内存使用、碎片率、淘汰数量。
- `INFO stats`：命中率、过期数、淘汰数、连接数。
- `INFO persistence`：RDB/AOF 状态和 fork 耗时。
- `INFO replication`：主从角色、延迟、offset、积压缓冲区。
- `CLIENT LIST`：客户端数量、输出缓冲区和阻塞状态。
- `SLOWLOG GET`：慢命令。
- `LATENCY LATEST`：最近延迟事件。
- `MEMORY STATS`：内存细分。

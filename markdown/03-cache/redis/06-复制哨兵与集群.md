# 复制哨兵与集群

## 主从复制

复制用于提升读能力、容灾能力和故障恢复速度。Redis 复制默认异步，主节点写入成功不代表从节点已经同步完成。

复制流程：

- 从节点配置主节点地址，建立连接。
- 发送 `PING`、认证、上报端口和能力。
- 执行同步，可能是全量同步或部分重同步。
- 进入命令传播阶段，主节点持续发送写命令。
- 周期性心跳检测连接状态和复制偏移量。

## 全量同步与部分重同步

Redis 2.8 后使用 `PSYNC`。

- 全量同步：主节点生成 RDB 发给从节点，同时用缓冲区记录期间写命令，RDB 加载后再发送增量命令。
- 部分重同步：从节点根据 runid 和复制 offset 请求缺失部分，主节点从复制积压缓冲区补发。
- 如果复制积压缓冲区太小或 runid 不匹配，会退化为全量同步。

相关配置：

- `repl-backlog-size`：复制积压缓冲区大小。
- `repl-timeout`：复制超时。
- `replica-read-only`：从节点只读。
- `min-replicas-to-write`、`min-replicas-max-lag`：限制主节点在从节点不足时继续写入。

## Sentinel

Sentinel 用于监控主从、自动故障转移和通知客户端新主节点。

核心概念：

- 主观下线：单个 Sentinel 认为实例超时不可达。
- 客观下线：足够数量 Sentinel 达成下线判断。
- 纪元：用于故障转移选举和配置版本推进。
- 领头 Sentinel：负责执行本轮故障转移。

故障转移流程：

- 监控主节点不可达并达成客观下线。
- Sentinel 之间选举 leader。
- 从候选从节点中选择新主节点。
- 对新主执行 `SLAVEOF no one`。
- 让其他从节点复制新主。
- 将旧主恢复后配置为新主的从节点。

Sentinel 只解决高可用，不解决数据分片。由于复制异步，故障切换可能丢失已在旧主确认但未复制的数据。

## Cluster

Redis Cluster 通过 16384 个 hash slot 做分片，每个 key 根据 `CRC16(key) % 16384` 映射到槽。带 `{}` 的 hash tag 可让多个 key 落到同一槽，便于多 key 操作。

Cluster 特点：

- 每个主节点负责一部分 slot。
- 每个主节点可配置从节点用于故障转移。
- 节点之间通过 gossip 协议传播状态。
- 客户端需要支持 `MOVED`、`ASK` 重定向。
- Cluster 模式只支持 0 号数据库。

## 槽迁移

迁移 slot 时，源节点进入 `MIGRATING`，目标节点进入 `IMPORTING`。

- 已迁移的 key 访问可能返回 `ASK`，客户端需要先向目标节点发送 `ASKING` 再执行命令。
- 稳定状态下访问错误节点返回 `MOVED`，客户端应更新槽缓存。
- 迁移大 key 会阻塞和拉长迁移窗口，应先拆分大 key。

## 高可用取舍

- 主从复制提升读能力，但读从节点可能读到旧数据。
- Sentinel 适合单分片高可用，不做水平扩容。
- Cluster 适合容量和吞吐扩展，但多 key、事务、Lua 和运维复杂度更高。
- Redis 不是 CP 系统，网络分区和故障切换下更偏可用性，业务要能容忍短暂不一致。

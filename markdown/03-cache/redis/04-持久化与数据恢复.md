# 持久化与数据恢复

## 持久化目标

Redis 持久化用于在进程退出、机器重启或故障恢复时重建内存数据。持久化不能等同于强一致存储，仍需结合写入策略、复制和备份设计 RPO 与 RTO。

## RDB

RDB 是某一时刻的数据快照。

- `SAVE`：主线程同步生成 RDB，会阻塞服务。
- `BGSAVE`：fork 子进程生成 RDB，主线程继续处理请求。
- 自动快照由 `save <seconds> <changes>` 配置触发。

优点：

- 文件紧凑，适合备份、全量复制和快速加载。
- 对运行时写入路径影响较小。

缺点：

- 两次快照之间的数据可能丢失。
- fork 大进程会带来延迟抖动和 copy-on-write 内存放大。

## AOF

AOF 通过追加写命令记录数据变化。

`appendfsync` 决定刷盘策略：

- `always`：每条写命令刷盘，数据安全性高，延迟最大。
- `everysec`：每秒刷盘，常用折中方案，最多丢失约 1 秒数据。
- `no`：交给操作系统决定，性能较好但故障丢失窗口不可控。

AOF 重写通过当前内存状态生成更短的新 AOF 文件，而不是逐条压缩旧日志。重写期间新写入会进入 AOF 重写缓冲区，最后追加到新文件并原子替换。

## 混合持久化

开启 `aof-use-rdb-preamble yes` 后，AOF 重写文件前半部分使用 RDB 格式，后半部分追加增量 AOF。这样兼顾加载速度和 AOF 的数据完整性，是常见生产选择。

## 过期 key 与持久化

- 生成 RDB 时会忽略已过期 key。
- 主节点加载 RDB 时会忽略已过期 key。
- 从节点加载 RDB 时可能先载入，后续由主从同步清理。
- AOF 中 key 过期删除时会追加 `DEL` 或等价删除命令。
- AOF 重写会忽略已过期 key。

## 恢复顺序

同时存在 RDB 和 AOF 且开启 AOF 时，Redis 通常优先加载 AOF，因为 AOF 数据更新。恢复前应确认：

- 文件是否完整，可使用 `redis-check-rdb`、`redis-check-aof`。
- 是否接受 AOF 修复截断带来的最后部分数据丢失。
- 目标实例配置、版本和模块是否兼容。
- 恢复后是否需要重新建立复制或重分片。

## 生产建议

- 缓存场景可只开 RDB 或不开持久化，但要能承受冷启动回源。
- 状态数据建议开启 AOF everysec 和混合持久化。
- 避免在内存接近上限时频繁 fork，预留 copy-on-write 空间。
- 对重要数据做异地备份，不能只依赖本机持久化文件。
- 监控 `rdb_last_bgsave_status`、`aof_last_bgrewrite_status`、`aof_delayed_fsync`、`latest_fork_usec`。

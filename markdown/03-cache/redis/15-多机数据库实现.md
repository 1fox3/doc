### 复制

#### 命令

- SLAVEOF

#### 旧版复制(2.8以前)

- 同步
  - 1、从服务器项主服务器发送SYNC命令
  - 2、主服务收SYNC命令后，执行BGSAVA命令，在后台生成一个RDB文件，并使用一个缓冲区记录下从现在开始执行的所有写命令
  - 3、主服务器BGSAVA执行完毕后，将RDB文件发送给从服务器，从服务器接受并载入RDB文件，且动态同步至主服务器开始执行BGSAVAE的状态
  - 4、主服务器将缓冲区中的所有写命令发送给从服务器，从服务器根据这些命令更新至主服务器当前所处的状态

- 命令传播
  - 写命令传播

#### 新版复制(2.8以后)

- PSYNC
  - 完整重同步
    - 与SYNC过程一致
  - 部分重同步
    - 复制迁移量
    - 复制积压缓冲区
      - 默认1MB
    - 服务器运行ID
  - 调用方式
    - SLAVEEOF ？ -1
      - 从未复制过任何主服务器
    - PSYNC <runid> <offset>
      - 复制过某个主服务器

### Sentinel(哨兵)

#### 启动哨兵

- redis-sentinel /path/to/your/sentinel.conf

- redis-server /path/to/your/sentinel.conf --sentinel

#### 启动逻辑

- 1、初始化服务器

- 2、将普通的redis服务器使用的代码替换成Sentinel专用代码

- 3、初始化Sentinel状态
  - sentinelState
    - current_epoch
      - 当前纪元用于故障转移
    - masters
      - 主服务器
        - 字典对象sentinelRedisInstance
          - flags
            - 标识值，记录了实例的类型和当前状态
          - name
            - 实例的名字
          - runid
            - 运行id
          - config_epoch
            - 配置纪元，用于实例故障转移
          - addr
            - 实例地址
          - down_after_period
            - 实例无响应多少毫秒之后才判断为主观下线
          - quorum
            - 判断这个实例为客观下线所需要的支持投票数量
          - parallel_syncs
            - 执行故障转移操作时，可以同时对新的主服务器进行同步的从服务器数量
          - failover_timeout
            - 刷新故障迁移状态的最大时限
          - slaves
            - 从服务器
    - tilt
      - 是否进入TILT模式
    - running_scripts
      - 目前正在执行的脚本数量
    - tilt_start_time
      - 进入TILT模式的时间
    - previous_time
      - 最后一次执行时间处理器的时间
    - scripts_queue
      - 一个FIFO队列，包含了所有需要执行的用户脚本

- 4、根据跟定的配置文件，初始化Sentinel的监视主服务器列表

- 5、创建连向主服务器的网络连接
  - 命令连接
  - 订阅连接

### 集群

#### 集群通过分片来进行数据共享，并提供复制和故障转移功能

- CLUSTER NODES

- 启动节点

- 集群数据结构
  - clusterNode
    - ctime

    - name

    - flags
      - 节点表示
    - configEpoch
      - 配置纪元
    - ip
      - 节点的IP地址
    - port
      - 节点端口号
    - link
      - 保存连接节点所需的有关信息
        - clusterLink
          - ctime
            - 创建时间
          - fd
            - TCP套接字描述符
          - sndbuf
            - 输出缓冲区
          - rcvbug
            - 输入缓冲区
          - node
            - 与这个连接相关的节点
    - state
      - 集群目前所处的状态
        - clusterState
          - myself
            - 指向当前节点的指针
          - currentEpoch
            - 集群当前的配置纪元
          - state
            - 集群当前的状态
          - size
            - 集群中至少处理这一个槽的节点数量
          - nodes
            - 集群节点名单
          - slots[16383]
            - 槽的指派信息，clusterNode属性
          - clusterNode importing_slots_form
            - 当前节点正在从其他节点导入的槽

            - 当前节点正在迁移至其他节点的槽

      - 节点复制处理槽的数量

    - int numslaves
      - 从节点数量
    - slaves
      - 从节点数组
    - fail_reports
      - 一个链表，记录了所有其他节点对该节点的下线报告

- 计算属于哪个槽
  - CRC16(key) & 16383

#### 节点数据库实现

- 集群节点仅能使用0号数据库

- 跳跃表 slots_to_keys
  - 记录数据库键与槽号键的关系

#### 重新分片

- redis-trib负责重新分片
  - 1、对目标节点发送CLUSTER SETSLOT <slot> IMPORTING <source_id>命令，让目标节点准备好从源节点导入槽<slot>的键值对
  - 2、对源节点发送CLUSTER SETSLOT <slot> MIGRATING <target_id>命令，让源节点准备好将属于槽slot的键值对迁移至目标节点
  - 3、从源节点获取特定数量属于slot的键值对
  - 4、将键值对迁移至目标节点
  - 5、重复3、4步直到所有键值对迁移完毕
  - 6、先任意一个节点发送槽指派命令，后续统一更新至整个集群

- ASK错误
  - 用于处理迁移过程中的数据查询
  - 数据迁移时，如果数据已迁移至目标服务器，则返回ASK错误，并指引客户端转向目标节点，并重新执行想要执行的命令

- ASKING
  - 打开客户端的REDIS_ASKING标识

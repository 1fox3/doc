# 缓存设计与一致性

## 缓存模式

最常见的是 Cache Aside：应用先读缓存，未命中再读数据库并写入缓存；更新时先更新数据库，再删除缓存。

选择缓存模式时要关注读写比例、数据时效要求、失败补偿、并发写和数据库承压能力。

## 缓存穿透

穿透是请求查询不存在的数据，缓存和数据库都无法命中，导致请求持续打到数据库。

治理方式：

- 参数校验，拦截非法 ID 或明显无效查询。
- 缓存空值，并设置较短 TTL。
- 使用布隆过滤器提前判断 key 是否可能存在。
- 对异常来源限流和熔断。

空值缓存要防止缓存大量无效 key。布隆过滤器存在误判，不能替代最终数据库校验。

## 缓存击穿

击穿是热点 key 过期瞬间，大量并发请求同时回源。

治理方式：

- 热点 key 不设置物理过期，改用逻辑过期和异步刷新。
- 使用互斥锁，只允许一个请求回源构建缓存。
- 请求合并，将同一 key 的并发回源合并为一次。
- 提前刷新，在 TTL 接近过期时异步续期。

互斥锁必须有超时时间和唯一 token，释放锁要校验持有者。

## 缓存雪崩

雪崩是大量 key 同时失效或 Redis 故障导致请求集中打到数据库。

治理方式：

- TTL 增加随机抖动，避免同一时间过期。
- 多级缓存，本地缓存兜底热点数据。
- Redis 高可用部署，避免单点故障。
- 限流、熔断、降级，保护数据库。
- 热点数据预热，避免重启后冷缓存冲击。

## 缓存一致性

常用更新策略：

- 先更新数据库，再删除缓存：最常见，配合重试或消息补偿。
- 延迟双删：写库前后删除两次缓存，降低并发读旧值回填的概率。
- 订阅数据库变更：通过 binlog、CDC 或消息队列异步失效缓存。
- 写穿缓存：写请求同时更新缓存和数据库，链路复杂且失败处理困难。

工程上没有绝对强一致的低成本方案。要明确允许的不一致窗口，并对删除缓存失败、消息丢失、并发回填设计补偿。

## 分布式锁

基础写法：`SET lock:key token NX PX 30000`。

释放锁时必须比较 token 后删除，通常用 Lua 保证原子性。

关键注意点：

- 锁过期时间要覆盖业务执行时间，或实现安全续期。
- token 必须唯一，不能直接 `DEL`。
- 锁只能保护 Redis 能感知的临界区，不能替代数据库唯一约束。
- 主从异步复制下，主节点故障可能导致锁安全性下降。
- 对强一致锁需求更高的场景，应评估 ZooKeeper、etcd 或数据库锁。

## 热 key 治理

- 本地缓存热点数据，降低 Redis 单点压力。
- key 分片，例如 `hot:sku:1:0` 到 `hot:sku:1:N`。
- 请求合并，减少同一热点的重复查询。
- 限流和降级，避免热点拖垮整体实例。
- 通过代理、客户端统计、`--hotkeys` 或监控采样定位热点。

## 缓存容量设计

- 区分核心缓存、可淘汰缓存和状态数据，尽量不要混在同一实例。
- 给缓存 key 设置 TTL，并根据业务访问模式选择淘汰策略。
- 预估 value 大小、key 数量、对象编码和内存碎片。
- 为 fork、复制缓冲区、客户端输出缓冲区和碎片预留内存。
- 设置数据库保护策略，缓存异常时要限流而不是无限回源。

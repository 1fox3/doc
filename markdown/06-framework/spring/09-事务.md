# 事务

> 专题：Spring
> 来源：Spring.xmind

## 补充与实践

- 补充：Spring AOP 基于代理，事务、缓存、异步等注解都依赖代理边界。
- 实践：事务方法避免自调用，异常回滚规则要明确，跨服务事务优先采用最终一致性方案。
- 误区：`@Transactional` 默认只对运行时异常回滚，且 private/final 方法通常不会被代理增强。

  - 4、在执行目标方法前获取事务并收集事务信息

        - 1、获取事务：doGetTransaction
        - 2、如果当前线程存在事务，则转向嵌套事务处理handeExistingTransaction
          - PROPAGATION_REQUIRES_NEW
            - 通过suspend讲原事务挂起，以便当前事务执行完毕后将原事务还原
          - PROPAGATION_NESTEND
            - Spring云溪嵌入事务，则首选设置保存点的方式作为异常处理的回滚
            - 对于其他方式，这处理方式与PROPAGATION_NEW相同
        - 3、事务超时设置验证
        - 4、事务porpagationBehavior属性的设置验证
        - 5、构建DefaultTransactionStatus
        - 6、完善transaction
          - doBegin
            - 1、尝试获取链接
            - 2、设置隔离级别和只读标识
            - 3、更改默认的提交设置
            - 4、设置标志为，标识当前连接已被事务激活
            - 5、设置过期时间
            - 6、将connectionHolder绑定到当前线程
  - 5、执行目标方法
  - 6、出现异常，尝试异常处理
    - 回滚有保存点的事务
    - 保存的事务信息中的事务为新事务，则直接回滚
    - 当前事务信息中表面是存在事务，但又不属于以上2种，则制作回滚标识，等提交的时候统一不提交
  - 7、提交事务前的事务信息清理
    - 1、设置状态是对事务信息做完成标识以避免重复调用
    - 2、当事务是新的同步状态，需要讲绑定到当前线程的事务信息清楚
    - 3、如果是新事物需要做些清除资源的工作
    - 4、如果事务执行前有事务挂起，那么当前事务执行结束后续将挂起的事务恢复
  - 8、提交事务
    - 1、当前事务中有保存点信息是，不提交事务
    - 2、当事务非新事务是也不会去提交事务

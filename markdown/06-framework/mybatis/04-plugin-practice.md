# 插件机制与生产实践

## 插件原理

MyBatis 插件基于 JDK 动态代理和拦截器链实现。可拦截的核心接口包括：

- `Executor`
- `StatementHandler`
- `ParameterHandler`
- `ResultSetHandler`

插件通过 `@Intercepts` 和 `@Signature` 声明拦截点，在 `intercept` 方法中增强目标调用。

## 常见插件场景

- 分页：拦截 SQL 并追加分页语法。
- 慢 SQL 记录：统计执行耗时并输出 SQL、参数和调用上下文。
- 审计字段：自动填充创建时间、更新时间、操作人。
- 多租户：追加租户条件。
- 数据权限：按用户角色追加过滤条件。
- SQL 防护：阻止全表 update/delete。

## 插件注意事项

- 插件顺序会影响最终 SQL 和参数，多个插件组合时需要明确顺序。
- 改写 SQL 必须同步处理参数映射，否则会出现参数错位。
- 分页 count SQL 对复杂查询可能不准确或性能很差。
- 多租户和数据权限应防止被 `${}` 或手写 SQL 绕过。
- 日志插件不能输出敏感字段。

## 流式查询

大结果集查询可使用游标或 ResultHandler 流式处理，避免一次性加载到内存。

注意事项：

- 需要保持连接和 SqlSession 打开直到消费完成。
- 不应在流式处理过程中长时间阻塞。
- MySQL 等数据库可能需要额外 fetchSize 或连接参数配合。
- 处理逻辑应避免把结果重新累积到大集合中。

## SQL 注入防护

- 默认使用 `#{}` 绑定参数。
- 动态列名、排序字段、表名必须白名单校验。
- 禁止把前端传入片段直接拼接到 SQL。
- 对批量条件限制最大集合大小，避免超长 SQL。
- 对管理后台的自由查询能力做权限、审计和限流。

## 性能治理

- 为核心查询建立合适索引并关注执行计划。
- 避免 N+1 查询，必要时使用 join、批量查询或显式缓存。
- 控制 resultMap 嵌套查询，避免隐式触发大量 SQL。
- 大批量写入使用 BatchExecutor 或数据库批量语法。
- 对慢 SQL 做采样、归因和持续优化。

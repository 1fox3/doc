# 分析器与相关性

## Analyzer 流程

Analyzer 将原始文本转换为 token 流，流程包括：

- character filter：字符级预处理，例如 HTML 清理、字符映射。
- tokenizer：将文本切分为 token，一个 analyzer 只能有一个 tokenizer。
- token filter：对 token 做小写、停用词、同义词、词干化、拼音等处理。

索引 analyzer 和搜索 analyzer 可以不同。中文搜索常见做法是索引用更细粒度分词，搜索用更粗粒度分词，降低召回噪声。

## 常见 Tokenizer

- `standard`：标准分词器，适合英文和通用场景。
- `letter`：按非字母切分。
- `lowercase`：按非字母切分并转小写。
- `whitespace`：按空白字符切分。
- `keyword`：整个输入作为一个 token，常用于精确字段或自定义 filter。
- 中文场景通常使用 IK、jieba、smartcn 或业务词典方案。

## 分析调试

使用 `_analyze` 验证 analyzer 输出，使用 `_termvectors` 查看文档实际入库词项。排查搜索无结果时，应先检查索引 analyzer、搜索 analyzer、字段类型和查询类型是否匹配。

## 精确率与召回率

- 精确率：返回结果中相关文档占比。
- 召回率：所有相关文档中被返回的占比。

搜索调优通常在精确率和召回率之间取舍。电商搜索可能先保证召回，再通过业务排序提升结果质量；站内帮助文档搜索可能更重视精确率。

## BM25

BM25 是 Elasticsearch 5+ 的默认相似度算法。它综合考虑词频、逆文档频率和字段长度归一化。

核心直觉：

- 查询词在文档中出现越多，得分通常越高，但增长会逐渐饱和。
- 查询词在整个索引中越稀有，区分度越高。
- 字段越短且命中词占比越高，相关性通常越强。

## Function Score

`function_score` 用于将文本相关性和业务因子组合，例如销量、库存、时间衰减、距离、会员权重。

常见函数：

- `weight`：固定权重。
- `field_value_factor`：使用字段值调整分数。
- `script_score`：脚本评分，灵活但成本高。
- `gauss`、`linear`、`exp`：衰减函数，常用于距离和时间。

使用建议：

- 先让召回 query 足够准确，再叠加业务分。
- 控制 `boost_mode` 和 `score_mode`，避免业务分完全淹没文本相关性。
- 脚本评分只应用在候选集足够小的场景。

## Rescore

`rescore` 对每个分片返回的 top N 文档进行二次打分，适合使用更昂贵的 phrase query、脚本或模型重排。

它不能改变召回范围，只能重排候选窗口内的文档。`window_size` 过大会明显增加查询成本。

## 同义词

同义词可以在索引阶段或搜索阶段处理。

- 索引阶段扩展：查询快，但更新同义词需要重建索引，且会放大索引。
- 搜索阶段扩展：更灵活，但查询成本更高，复杂同义词可能影响相关性。

同义词要区分等价词、上下位词、品牌词、错别字和业务词。不要把所有词都粗暴互相扩展，否则会严重降低精确率。

## 搜索体验功能

- term suggester：针对单词拼写纠错。
- phrase suggester：针对短语纠错，考虑词序和上下文。
- completion suggester：基于 FST 的前缀补全，延迟低但需要专门字段。
- context suggester：在补全时加入分类、地域等上下文过滤。
- highlighter：高亮命中片段，需关注字段大小和高亮器成本。

补全、纠错和同义词应结合搜索日志、点击数据和人工词库持续迭代。

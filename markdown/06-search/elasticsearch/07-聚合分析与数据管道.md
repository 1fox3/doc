# 聚合分析与数据管道

## 聚合类型

Elasticsearch 聚合用于实时分析和统计，常见类型包括指标聚合、桶聚合、管道聚合和矩阵聚合。

聚合通常依赖 doc values。高基数字段、大范围时间查询和深层嵌套聚合可能带来明显内存和 CPU 压力。

## 指标聚合

常见指标聚合：

- `min`、`max`、`sum`、`avg`：基础统计。
- `stats`、`extended_stats`：一次返回多项统计指标。
- `cardinality`：近似唯一值计数，基于 HyperLogLog++，可通过 precision threshold 调整精度和内存。
- `percentiles`：百分位统计，适合延迟分析。
- `top_hits`：返回桶内样本文档，成本较高，要控制 size。

## 桶聚合
常见桶聚合：

- `terms`：按字段值分桶。
- `date_histogram`：按时间分桶。
- `range`、`date_range`：范围分桶。
- `filters`：多个过滤条件分桶。
- `nested`、`reverse_nested`：nested 文档聚合。

`terms` 聚合要关注 `size`、`shard_size` 和高基数字段。需要遍历全部桶时优先考虑 `composite` 聚合分页。

## 管道聚合
管道聚合基于其他聚合结果继续计算。

- parent pipeline：在父桶内部计算，例如 derivative、moving_fn、bucket_script。
- sibling pipeline：基于兄弟聚合计算，例如 max_bucket、avg_bucket、sum_bucket。

管道聚合不会减少前置聚合成本，前置桶数量过大仍会消耗大量内存。

## 请求缓存

分片请求缓存主要缓存 `size=0` 的查询结果，适合仪表盘和重复分析查询。

注意点：

- 索引 refresh 会使相关缓存失效。
- 带 `now` 的时间范围查询缓存效果差。
- 高频写入索引上的聚合缓存收益有限。

## Ingest Pipeline

Ingest Pipeline 在写入前对文档做转换处理。结构包括 `description`、`processors` 和 `on_failure`。

常见 processor：

- `set`、`append`、`rename`、`remove`：字段处理。
- `convert`：类型转换。
- `date`：日期解析。
- `grok`：日志文本解析。
- `geoip`：IP 地理位置解析。
- `script`：自定义处理，成本较高。

复杂清洗不应全部放到 ingest 节点。高吞吐日志场景通常在 Beats、Logstash、Flink 或应用侧预处理，ES 只承担必要转换。

## Logstash、Beats 与 Kibana
Elastic Stack 常见组件：

- Beats：轻量采集器，例如 Filebeat、Metricbeat、Packetbeat、Winlogbeat。
- Logstash：数据采集、过滤和转发，适合复杂 grok、路由和多输出。
- Kibana：数据发现、可视化、Dashboard、Dev Tools、告警和管理界面。

选择建议：

- 简单日志采集优先 Beats 加 ingest pipeline。
- 复杂日志解析、多源汇聚、多输出路由使用 Logstash 或流处理系统。
- 业务数据同步通常由应用、CDC 或专门同步任务写入 ES。

## Transform 与运行时字段

Transform 可将源索引聚合转换为目标索引，适合宽表、汇总表和持续统计。

Runtime fields 可在查询时计算字段，适合低频探索和临时修正。高频查询应尽量索引时固化字段，避免每次查询重复计算。

## 数据管道可靠性

- 数据源要有 offset 或 checkpoint。
- 写入 ES 要支持重试和幂等。
- 对不可解析数据设置 dead letter queue 或错误索引。
- Mapping 变更要和管道版本同步。
- 监控采集延迟、写入失败、bulk rejected 和管道积压。

# 高性能实践与故障排查

## 高性能来源

Netty 的高性能主要来自非阻塞 I/O、Reactor 线程模型、直接内存、内存池、零拷贝、异步 Future、批量 flush、减少锁竞争和 native transport。

## EventLoop 不阻塞

EventLoop 被阻塞会导致同线程管理的所有 Channel 读写延迟。常见阻塞来源包括慢 SQL、同步 HTTP、文件 I/O、复杂计算、锁等待、Thread.sleep 和大对象序列化。

处理方式：

- 耗时业务切到独立业务线程池。
- 对业务线程池设置队列和拒绝策略。
- 使用指标监控 EventLoop 任务队列延迟。
- 对 Handler 执行耗时打点。

## 写缓冲与背压

Netty 通过 ChannelOutboundBuffer 和写水位控制出站缓冲。WRITE_BUFFER_WATER_MARK 包含低水位和高水位。当缓冲超过高水位时，Channel 变为不可写；低于低水位时恢复可写。

实践要点：

- 监听 channelWritabilityChanged，停止或恢复上游发送。
- 不要无视 isWritable 持续写入，否则可能导致内存膨胀。
- 大消息和慢客户端要有超时、限速或断开策略。

## 超时设置

常见超时包括连接超时、读超时、写超时、空闲超时、业务请求超时和心跳超时。超时需要和重试、连接关闭、上游等待时间配合，避免请求已经超时但连接仍堆积大量数据。

## JDK Selector 空轮询

部分 JDK 版本在 epoll 场景下可能出现 Selector 空轮询，表现为 select 方法未阻塞却没有就绪事件，导致 CPU 飙高。Netty 会检测连续空轮询并重建 Selector，将 Channel 迁移到新 Selector 上。

排查时关注：

- 单个 EventLoop 线程 CPU 是否异常高。
- 日志中是否出现 selector rebuild 相关信息。
- JDK 版本和 Netty 版本是否存在已知问题。

## Native Transport

Linux 下可使用 netty-transport-native-epoll，macOS/BSD 下可使用 kqueue。Native transport 通常能获得更好的性能和更多 TCP 参数能力，但需要处理平台依赖和部署兼容性。

## 常见故障排查

- 连接数异常：检查 accept 速率、连接泄漏、心跳、空闲关闭、客户端重连风暴和系统 fd 限制。
- CPU 飙高：检查 EventLoop 阻塞、空轮询、编解码死循环、日志过多和业务线程池忙等。
- 内存上涨：检查 DirectMemory、ByteBuf 泄漏、写缓冲堆积、大包和对象池配置。
- 请求延迟高：检查 EventLoop 队列、业务线程池、GC、网络 RTT、flush 策略和下游依赖。
- 半包粘包：检查协议长度字段、解码器顺序、readerIndex 回滚和最大帧长度。

## 编码规范

- Handler 标注是否可共享，使用 `@Sharable` 前确认没有可变实例状态。
- 异常路径必须关闭连接或释放资源。
- 出站写入结果必须监听失败并记录原因。
- 自定义协议必须保留版本字段，便于灰度升级。
- 对所有外部输入设置长度、格式和速率限制。

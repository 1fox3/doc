# 启动流程与核心组件

## Bootstrap

Netty 使用 Bootstrap 和 ServerBootstrap 启动客户端和服务端。

服务端启动核心步骤：

- 创建 BossGroup 和 WorkerGroup。
- 配置 ServerBootstrap 的 Channel 类型，例如 NioServerSocketChannel。
- 设置服务端 Channel 选项，例如 SO_BACKLOG。
- 设置子 Channel 选项，例如 TCP_NODELAY、SO_KEEPALIVE、WRITE_BUFFER_WATER_MARK。
- 配置 childHandler 初始化每个连接的 Pipeline。
- bind 端口并注册到 EventLoop。

客户端启动核心步骤：

- 创建 EventLoopGroup。
- 配置 Bootstrap 的 Channel 类型。
- 设置远端地址和连接参数。
- 配置 Handler 初始化 Pipeline。
- connect 并等待连接结果。

## Channel

Channel 是网络连接的抽象，代表 Socket、ServerSocket 或其他传输通道。它负责注册到 EventLoop、触发读写事件、维护 ChannelPipeline 和保存 ChannelConfig。

常见 Channel：

- NioServerSocketChannel：服务端监听连接。
- NioSocketChannel：TCP 客户端或服务端已接受连接。
- EpollServerSocketChannel、EpollSocketChannel：Linux native transport。

## EventLoop

EventLoop 是 Netty 的事件循环线程，一个 EventLoop 可以管理多个 Channel，一个 Channel 通常只绑定到一个 EventLoop。这样同一 Channel 的事件串行执行，减少锁竞争。

EventLoop 负责：

- Selector 事件轮询。
- 处理 I/O 读写事件。
- 执行普通任务和定时任务。
- 触发 Pipeline 事件传播。

不要在 EventLoop 中执行阻塞调用，例如慢 SQL、远程 HTTP、文件大读写、长时间计算和 Thread.sleep。

## Future 与 Promise

Netty 的 I/O 操作是异步的，write、connect、bind 等方法通常返回 ChannelFuture。调用方可以添加 Listener 处理成功或失败，避免阻塞等待。

Promise 是可写 Future，通常由框架或异步任务持有，用于在未来某个时间设置成功或失败结果。

使用原则：

- 优先使用 listener，不要在 EventLoop 中调用 sync 或 await 阻塞。
- 异步失败必须记录原因，否则连接失败、写失败容易被忽略。
- 回调中不要执行耗时业务。

## 设计模式

Netty 中常见设计模式包括 Reactor、责任链、工厂、建造者、观察者、适配器和享元。理解这些模式有助于掌握 Pipeline、EventLoopGroup、ByteBufAllocator 和 Future Listener 的设计。

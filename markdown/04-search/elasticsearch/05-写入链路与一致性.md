# 写入链路与一致性

## 写入路由

写入请求先到协调节点，协调节点根据 `_routing` 或 `_id` 计算目标主分片。请求转发到 primary shard，primary 写入成功后再复制到 replica shard。

写入响应成功通常表示主分片和要求数量的副本完成写入，但搜索可见性仍取决于 refresh。

## 文档 ID

使用业务自定义 `_id` 可以幂等更新，但写入前需要查找旧版本位置，随机 ID 写入吞吐通常更好。

选择建议：

- 日志和行为流水使用自动 ID。
- 商品、用户、订单等实体索引用业务 ID。
- CDC 同步场景使用源库主键，保证可重复写入。

## 更新与删除

Lucene segment 不可变，更新本质是读取旧文档、合并变更、标记旧文档删除、写入新文档。删除也是先打删除标记，空间在 merge 后释放。

高频更新会带来：

- segment 增多。
- merge 压力增大。
- deleted docs 比例升高。
- IO 和 CPU 抖动。

频繁变更字段应评估是否真的适合进 ES，或者拆成独立索引、降低刷新频率、批量更新。

## Refresh

`refresh` 让内存中的新数据生成可搜索 segment。默认 `index.refresh_interval` 通常为 1 秒。

调优建议：

- 批量导入时可临时设置 `refresh_interval=-1`，导入完成后恢复并手动 refresh。
- 写多读少场景增大 refresh interval，减少小 segment。
- 不要在每条写入都使用 `refresh=true`。

## Flush 与 Translog

translog 是事务日志，用于在故障后恢复尚未提交到 Lucene commit 的操作。flush 会执行 Lucene commit 并清理旧 translog。

关键配置：

- `index.translog.durability=request`：每次请求后 fsync translog，安全性更高。
- `index.translog.durability=async`：按 `sync_interval` 异步刷盘，吞吐更好但可能丢最近数据。
- `index.translog.sync_interval`：异步刷盘间隔。
- `index.translog.flush_threshold_size`：translog 达到阈值后触发 flush。

## 并发控制

旧版本使用 `_version` 做乐观并发控制。现代版本推荐使用 `_seq_no` 和 `_primary_term`。

典型方式：

- 读取文档时拿到 `_seq_no` 和 `_primary_term`。
- 更新时带 `if_seq_no` 和 `if_primary_term`。
- 如果期间文档被其他请求修改，更新会返回版本冲突。

这种机制只能解决单文档并发冲突，不能提供跨文档事务。

## Bulk

Bulk 是提升写入吞吐的核心手段。

实践建议：

- 控制单个 bulk 大小，常从 5 MB 到 15 MB 或 1000 到 5000 条开始压测。
- 客户端并发数要逐步增加，观察 rejected、CPU、IO、merge 和 heap。
- 对 429 或 rejected 做指数退避重试。
- 批量导入期间降低副本数或 refresh 频率，导入后恢复。

## Reindex

Mapping 变更、数据修复、索引拆分合并通常使用 `_reindex` 或应用层重建。

注意点：

- `_reindex` 会消耗源和目标集群资源。
- 大规模 reindex 要使用 slices 并限速。
- 重建完成后用 alias 原子切换。
- 对写入中的业务要设计双写、补偿或 CDC 追增量。

## 一致性边界

Elasticsearch 不是强一致数据库。

- GET 实时性强于 search。
- 副本复制、refresh、集群恢复都可能带来短暂不一致。
- 跨文档事务、唯一约束、复杂约束应放在主数据库或业务层。
- ES 更适合作为搜索和分析视图，而不是唯一事实源。
